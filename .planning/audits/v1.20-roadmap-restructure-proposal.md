# v1.20 Roadmap Restructure Proposal: Codex Runtime Substrate

**Date:** 2026-04-08
**Author:** Orchestrator (Claude Opus 4.6), synthesizing findings from 7 audit agents
**Status:** Proposal — awaiting review and user decision

## Problem Statement

The v1.20 roadmap (10 phases, 57 requirements) was designed with Codex CLI as a "graceful degradation" target — hooks degrade to advisory, enforcement defers to agent compliance, cross-runtime adaptation lives in Phase 60. Three independent audit streams have revealed this framing is inadequate:

1. **prix-guesser forensics** (14 issues found): Running upstream GSD on Codex produced zero product work across an entire session. Recursive orchestration loops, model ambiguity, hand-authored substitute plans, premature stall diagnosis, confabulated capability reports. The user had to invent `portable-gsd` infrastructure and write AGENTS.md guardrails from scratch.

2. **Claude Sonnet harness audit** (10 gaps): Sensor discovery silently returns zero on Codex (*.md glob vs *.toml agents), signal synthesizer/reflector/spike-runner blocked by missing sandbox write permissions, 12/14 workflows have no capability_check, `automation.cjs` hardcodes Codex as no-task_tool despite capability matrix update.

3. **GPT-5.4 xhigh cross-model audit** (6 additional gaps + 6 drift findings): `automation.cjs` caps Codex in CODE not just docs. Codex NOW HAS hooks (`codex_hooks: under development true`). `codex.toml` doesn't exist (real file is `config.toml`). Subagent API signatures already drifted. No parity monitoring infrastructure. `.codex/` pollutes brownfield detection. Phase 58-60 sequencing is wrong — substrate fixes are prerequisites for enforcement gates, not follow-on work.

## The Structural Problem

The current roadmap has a **circular dependency**:

```
Phase 58 (Structural Enforcement Gates)
  ├── Claims: "cannot be circumvented by agent discretion"
  ├── Mechanism: hooks (Claude Code) + "workflow-enforced steps" (Codex)
  ├── Problem: Codex substrate is broken — enforcement degrades to advisory
  └── Depends on: Phase 60 for cross-runtime fixes (but Phase 60 depends on 58)

Phase 60 (Cross-Runtime Adaptation)
  ├── Contains: the actual Codex substrate fixes
  ├── Problem: these are PREREQUISITES for Phase 58, not follow-on
  └── Depends on: Phase 58 (current roadmap says "structural gates provide enforcement context")
```

XRT-01 ("every hook-dependent v1.20 feature specifies its Codex CLI degradation path in phase CONTEXT.md before implementation begins") was supposed to bridge this gap, but:
- XRT-01 is itself in Phase 58, creating a self-referential gate
- XRT-01 is a documentation requirement with no structural enforcement
- The prix-guesser experience shows documentation requirements fail 100% on Codex agents

**Source:** `.planning/audits/codex-harness-audit-gpt54-2026-04-08.md` lines 196-212; `.planning/REQUIREMENTS.md` lines 102-107 (GATE-07, XRT-01)

### New Phase 57: Codex Runtime Substrate

**Goal:** The harness correctly detects, adapts to, and verifies Codex CLI capabilities — runtime detection is code-level accurate, agent/sensor discovery works across formats, and a parity monitoring mechanism prevents silent drift.

**Depends on:** Phase 56 (KB must exist for signal provenance model updates)

**Justification for placement before telemetry:**
- Telemetry baseline (Phase 58) should capture the REAL cross-runtime state, not a state where Codex is fundamentally broken. Measuring a broken baseline then fixing it creates a confounded before/after comparison. (Reference: `.planning/ROADMAP.md` line 68 — "baseline must be captured before any Phase 58 structural gates are deployed" — the same logic applies to Codex fixes.)
- Phase 59 (structural enforcement) cannot credibly claim cross-runtime enforcement without the substrate. GPT-5.4 audit: "Until those are fixed, Phase 58's Codex story is advisory in a deeper sense than the requirements currently admit."

### Impact on Existing Phases

**Phase 58 (now Measurement & Telemetry):**
- Unchanged in scope. Delayed by 1 phase.
- Benefits: telemetry baseline captures post-Codex-fix state, making before/after comparisons for Phase 59 enforcement more meaningful.

**Phase 59 (now Structural Enforcement Gates):**
- Unchanged in scope EXCEPT: XRT-01 now has a real substrate to reference. The Codex degradation paths documented in CONTEXT.md files will reference actual working infrastructure (runtime resolver, adapter blocks) rather than fictional advisory paths.
- Risk reduced: enforcement claims on Codex are backed by tested substrate.

**Phase 61 (now Sensor Pipeline, reduced scope):**
- Cross-runtime sensor discovery already done in Phase 57 Tier 1.
- Cross-runtime parity verification already done in Phase 57 Tier 3.
- Remaining scope: log sensor implementation, patch sensor implementation, progressive deepening pipeline. These are NEW capabilities, not substrate fixes.
- Estimated reduction: ~30% smaller (the XRT-02, SENS-05, SENS-06 requirements move to Phase 57).

**All other phases:** Renumbered +1, otherwise unchanged.

### Trade-offs

**Costs:**
- +1 phase to the milestone (11 phases, ~62 requirements)
- Telemetry baseline delayed by 1 phase
- Renumbering existing phase references in STATE.md, REQUIREMENTS.md
- New CODEX-* requirements need drafting

**Benefits:**
- Phase 59 (enforcement) can credibly claim cross-runtime support
- Phase 61 (sensors) scope reduces ~30%
- Telemetry baseline is more accurate (captures real cross-runtime state)
- Prevents repeating the prix-guesser disaster on any GSD Reflect deployment
- XRT-01 gets actual enforcement instead of being aspirational
- Establishes parity monitoring that catches future Codex drift

**What this does NOT address:**
- The `portable-gsd` infrastructure problem (repo-local Codex installs with `.codex/` gitignored) — deeper than one phase
- Codex model behavioral deficiencies (confabulation, premature stall) — structural guardrails help but can't fix the model
- Whether `codex_hooks` (under development) should be adopted now or waited on — should be a discuss-phase decision
- Upstream GSD's own quality regression (13 issues in one day) — separate concern from our Codex story
- Phase 64 (parallel execution) gating criteria — unchanged

## Decision Requested

1. Accept the restructure as proposed
2. Accept with modifications (specify)

**Recommendation:** Option 1. The evidence is overwhelming that the current ordering produces enforcement claims on a broken substrate. The cost is 1 additional phase and a renumbering pass. The alternative is shipping Phase 58 "structural enforcement" that is structurally unenforced on Codex.

# Requirements Review: v1.20 GSD Reflect

**Reviewer:** Claude Opus 4.6 (System Architect)
**Date:** 2026-04-08
**Documents reviewed:** REQUIREMENTS.md (46 requirements), MILESTONE-CONTEXT.md, SUMMARY.md (9 research docs synthesized), PITFALLS.md (6 critical / 7 moderate), FEATURES.md, ARCHITECTURE.md, STACK.md
**Verification:** Codebase inspection of discuss-phase adoption state, gsd-tools.cjs (676 lines, 16 lib modules), package.json engines constraint, signal files

---

## Executive Summary

The requirements document is unusually well-grounded for a pre-implementation artifact -- every requirement traces to audit evidence, signals, or research, and the research base is genuinely deep. However, the document has a structural problem: it conflates three distinct types of work (one-line configuration fixes, multi-file workflow rewrites, and novel epistemological framework operationalization) under the same "requirement" format, which obscures massive variance in implementation risk. The 46-requirement count also hides a scope that, if fully implemented, would constitute the largest milestone in the project's history across the most diverse set of capability domains (KB infrastructure, telemetry, spike epistemology, cross-runtime adaptation, parallel execution, workflow commands, sensor pipeline, structural gates) -- and there is no explicit capacity model, timeline, or phasing constraint that would force hard cuts if the team runs long. The milestone thesis ("infrastructure correction, not feature expansion") is contradicted by the actual contents, which include at least 6 genuinely novel features that have no precedent in the codebase.

## Category-by-Category Review

### KB Infrastructure (KB-01 through KB-08)

**Completeness: Strong.** The research coverage is thorough. The File+SQLite pattern is well-validated by three open-source precedents, and the PITFALLS document names the specific failure mode (C1: SQLite-as-truth) with a concrete prevention strategy.

**Specificity issues:**

- **KB-01** specifies lifecycle states and "transition validation" but does not define what valid transitions are. Can a signal go from `detected` directly to `remediated`, skipping `proposed` and `in_progress`? Can it go backward (`remediated` -> `in_progress` if a fix regresses)? The state machine is listed without its transition table. This will produce ambiguity during implementation.

- **KB-04** bundles rebuild, FTS5 search, lifecycle queries, AND relationship traversal into a single requirement. Each of these is independently testable and has different complexity. FTS5 search alone requires schema design decisions (what fields are indexed, what tokenizer, how to handle YAML frontmatter vs body content). This should be decomposed.

- **KB-06** lists 7 subcommands (query, stats, health, transition, link, search, rebuild) as one requirement. The "done" criterion for this requirement is a 7-feature CLI tool. This is not atomic.

- **KB-08** says "uses SQLite queries instead of grep-through-index" but does not specify what happens when `kb.db` does not exist (fresh clone, first run before `kb rebuild`). ARCHITECTURE.md mentions graceful fallback to grep, but the requirement itself does not.

**Dependencies:** KB-04 depends on KB-01 (schema must exist before index schema), and KB-07 depends on KB-04 (lifecycle transition commands must exist before collect-signals can invoke them). These structural dependencies are not stated in REQUIREMENTS.md but are documented in FEATURES.md's dependency graph. The requirements document should be self-contained on dependency ordering.

**Feasibility:** The `node:sqlite` live verification on Dionysus is strong evidence for feasibility. The risk is not "can we build it" but "how many edge cases emerge when indexing 198 existing signal files with inconsistent frontmatter." PITFALL N5 (source field ambiguity) names one; there may be others. The requirement for backward-compatible schema migration (KB-01/KB-02 add fields to existing signals) has no explicit requirement for a migration script or strategy, just the assumption that defaults will be sufficient.

**Missing requirement:** There is no requirement for a schema migration tool or validation pass over existing signals. The research notes that `source` field ambiguity must be resolved (PITFALL N5), but no requirement captures "resolve source field to detection_method + origin." This is an implementation prerequisite that fell between the research and requirements.

### Measurement & Telemetry (TEL-01 through TEL-04)

**Completeness: Adequate with one critical gap.**

**Specificity issues:**

- **TEL-01** lists 5 subcommands (summary, session, phase, baseline, enrich) as one requirement. Each subcommand is independently testable and has different complexity. `baseline` requires statistical computation (percentiles); `enrich` requires facets join logic; `phase` requires STATE.md timestamp correlation. This is 3-4 requirements hiding as one.

- **TEL-02** says "baseline captured before any v1.20 structural interventions ship" but does not define what "before" means operationally. Does baseline capture happen in Phase 1? Is it a gate on Phase 2? Who runs it? The SUMMARY's Phase 1 section says telemetry ships in Phase 1, and ARCHITECTURE anti-pattern 4 says baseline must precede interventions -- but the requirement itself does not state the sequencing constraint.

- **TEL-03** is a spike (validation experiment), not a requirement. It reads: "Token count validation spike confirms session-meta reliability." This is an investigative action, not a deliverable. The requirement should be: "Telemetry baseline annotates data source reliability rating based on validation results" or similar. Encoding a spike as a requirement blurs the line between "what we must build" and "what we must learn."

**Critical gap:** TEL-04 mentions facets data as "AI-generated session quality assessments" but has no requirement for labeling facets outputs as estimates, not ground truth. PITFALL M3 explicitly warns about this: "Correlations should be treated as hypothesis-generating, not hypothesis-confirming." The requirement says "joined with session-meta for quality signal enrichment" -- this phrasing treats facets as enrichment (adding value) when the research says the accuracy is unknown. There should be a requirement for facets reliability annotation.

**Feasibility concern:** The implicit assumption is that 268 session-meta files span enough time and project diversity to produce meaningful baselines. No requirement addresses minimum sample size for statistical validity of baselines. If the project has only 30 sessions against get-shit-done-reflect specifically (after project_path filtering), percentile distributions may not be meaningful.

### Structural Enforcement (GATE-01 through GATE-08)

**Completeness: Strong.** These are well-motivated by audit evidence with specific recurrence counts.

**Specificity issues:**

- **GATE-01** says "structural gate, not advisory text" but does not specify the enforcement mechanism. Does the workflow refuse to proceed? Does it loop asking for CI confirmation? Does it allow override with justification? "Structural" is an adjective, not a specification. Compare with GATE-02, which says "--merge is passed explicitly in `gh pr merge` invocation" -- that is testable. GATE-01 is not.

- **GATE-04** specifies ".continue-here files marked consumed on read and deleted/archived after session start" plus "staleness check if file predates last session." This is actually two requirements: (a) consumption lifecycle and (b) staleness detection. These have different implementation paths (a is file management; b requires timestamp comparison against session-meta).

- **GATE-06** says "SessionStop hook on Claude Code, workflow-enforced step on Codex." This requires TWO different implementations for the same requirement. The Codex path ("workflow-enforced step") is precisely the advisory text approach that the milestone thesis says consistently fails. The requirement acknowledges the asymmetry but does not resolve it.

- **GATE-08** claims "5 missing files" for discuss-phase adoption but codebase inspection shows `discuss-phase-assumptions.md` (279 lines) already exists, `discuss_mode` config routing exists in the command file, and `discuss_mode` appears in `config.json`. The signal (sig-2026-04-03) was accurate when filed, but partial remediation has occurred since. The requirement's scope may be stale -- the actual gap is narrower than stated (primarily `gsd-assumptions-analyzer.md` agent and `docs/workflow-discuss-mode.md` plus mode-aware gates in plan-phase.md/progress.md).

**Cross-requirement tension:** GATE-06 (automation postlude fires structurally) conflicts with the milestone's thesis that advisory text fails. On Codex, GATE-06 degrades to advisory text. If advisory text is known to fail, specifying advisory text as the Codex degradation path is specifying known failure. The requirement should either accept this limitation explicitly or propose a different mechanism for Codex (e.g., a required manual checklist step, or a post-session reminder file).

**Feasibility:** GATE-05 (sensor model echoed and logged before dispatch) is trivial to implement but hard to verify -- "logged" where? In the session JSONL? In a GSD artifact? In stdout? The verification path is unspecified.

### Sensor Pipeline (SENS-01 through SENS-06)

**Completeness: Adequate.**

**Specificity issues:**

- **SENS-01** describes a 4-stage progressive deepening architecture (fingerprinting, triage, expansion, signal construction) as a single requirement. The log sensor research doc describes this in detail, but the requirement itself does not state done criteria for each stage. Is it done when the sensor produces any signals? When it handles edge cases (subagent sessions, compacted sessions, very long sessions)? The SUMMARY flags TS-9 as the "largest single engineering effort in the milestone" but it is specified as one requirement.

- **SENS-02** says "Codex adapter handles response_item/event_msg container types" -- this is an implementation detail, not a requirement. The requirement should be "log sensor produces equivalent signal candidates from both Claude Code and Codex session logs." How it handles container types is design, not specification.

- **SENS-03** specifies "Codex session discovery uses state_5.sqlite" -- again, this is an implementation decision encoded as a requirement. What if Codex's storage format changes? The requirement should specify the capability (discover Codex sessions by project), not the implementation mechanism (query state_5.sqlite).

**Missing requirement:** There is no requirement for sensor error handling or graceful degradation when session log files are corrupt, truncated, or in an unexpected format. The SUMMARY flags Codex JSONL edge cases as a research need, but no requirement says "sensor reports parse failures as warnings rather than crashing."

**Feasibility:** SENS-04/SENS-05 (patch sensor) depend on the installer manifest having SHA256 hashes for all installed files. The existing `saveLocalPatches()` mechanism uses content comparison, but whether the manifest stores hashes in a queryable format needs verification.

### Spike Methodology (SPIKE-01 through SPIKE-11)

**Completeness: Ambitious to the point of concerning.**

This is 11 requirements spanning agent design, template revision, epistemological framework operationalization, and multi-spike programme infrastructure. It is the most intellectually ambitious category and also the one most likely to suffer from the gap between specification and operationalization.

**Specificity issues:**

- **SPIKE-01** specifies that the design reviewer "must use a different model from the designer" but does not specify how this is enforced at runtime. Cross-model dispatch is flagged as an open question in SUMMARY.md (Open Question 2). Encoding a requirement before resolving the implementation mechanism is a risk -- if cross-model dispatch turns out to be infeasible with the current agent dispatch infrastructure, the requirement itself becomes a blocker.

- **SPIKE-02** (three-level confidence) is well-specified conceptually but has no acceptance criteria. How do you determine whether an implementation of "measurement validity / interpretation validity / extrapolation validity" is correct? This is a template change that passes or fails on the quality of the guidance text, not on code behavior. The requirement needs a different kind of "done" criterion -- perhaps example assessments that demonstrate correct application.

- **SPIKE-04** (auxiliary hypothesis register) and **SPIKE-05** (metric limitation documentation) are template additions. Both risk PITFALL N4 (filled by rote). The requirements specify the templates but not the mechanism for preventing rote compliance. PITFALL N4 suggests the design reviewer should flag generic entries -- but this creates a circular dependency: the reviewer (SPIKE-01) must be built and effective before the templates (SPIKE-04/05) have their anti-rote safeguard.

- **SPIKE-10** (spike programme infrastructure) is the most complex requirement in the entire document. It includes programme directory structure, shared data assets, cross-spike tracking, progressive refinement, AND the 2-round limit exception. This is at least 4 requirements. FEATURES.md recommends deferring D-7 to v1.21; REQUIREMENTS.md includes it. The SUMMARY acknowledges this as Tension 1 but does not resolve it -- it just says "needs explicit roadmap decision." Including it as a requirement while simultaneously flagging it as a tension is indecisive.

- **SPIKE-11** is the broadest requirement in the entire document. It asks to "operationalize Lakatos, Mayo, and institutional critique" -- this is a research programme, not a requirement. The parenthetical elaborations (progressiveness ledger, severity assessment, three-tier enforce/encourage/warn model) are three separate capabilities. The Feyerabend warning ("without rigid procedures") is a design constraint that actively opposes the specificity that requirements need. This requirement is aspirational, not implementable as stated.

**Over-formalization risk: HIGH.** This is the category where PITFALL C5 (reviewer as checklist) and PITFALL C6 (over-formalization kills productive practices) are most likely to bite. The requirements specify a large amount of formal infrastructure (reviewer agent, confidence framework, auxiliary register, sample design section, protocol adherence checkpoints, findings reviewer, programme infrastructure, epistemological framework) for what is currently a 4-spike empirical base. The ratio of formalization to empirical base is high. Even with the three-tier model correctly applied, the sheer volume of new process in the spike methodology category creates weight that may suppress the trial-before-formalize practice that MEMORY.md records as a methodological practice.

**Missing:** There is no requirement for spike methodology rollback or simplification if formalization proves counterproductive. No metric for "did the methodology overhaul actually improve spike outcomes" beyond PITFALL C5's "compare before/after on premature closure rate." If SPIKE-01 through SPIKE-11 are implemented and spike quality does not improve, what happens? There is no off-ramp.

### Workflow Gaps (WF-01 through WF-03)

**Completeness: Thin.** Three workflow commands is light for a category.

**Specificity issues:**

- **WF-01** (cross-model review) says "committed audit spec before launch, background execution, structured response protocol" -- this is specifying implementation approach, not capability. The requirement then adds "opt-in and flexible, must not be over-formalized" -- which contradicts "committed audit spec before launch." Committing an audit spec before every invocation IS formalization. The requirement contains its own contradiction.

- **WF-02** says "formal path for mid-phase scope changes with ROADMAP.md and REQUIREMENTS.md update, commit, and re-discuss." This is clear and testable.

- **WF-03** (research command) says "lightweight pre-milestone research producing committed knowledge artifacts without code changes or spike overhead." How is this different from an agent reading files and writing a Markdown document? The "without spike overhead" framing suggests the value is in what it omits, not what it adds.

**Missing workflow:** The MILESTONE-CONTEXT.md gap note identifies that no `/gsdr:discuss-milestone` workflow exists. The milestone-level steering brief was created informally. There is no requirement to close this gap. If the current milestone identified the gap and did not capture a requirement for it, the gap will persist into v1.21.

### Cross-Runtime Parity (XRT-01 through XRT-03)

**Completeness: Adequate but underspecified.**

- **XRT-01** says "every hook-dependent v1.20 feature specifies its Codex CLI degradation path in phase CONTEXT.md before implementation begins." This is a process requirement (write a degradation spec), not a feature requirement (implement degradation). It guarantees documentation, not functionality.

- **XRT-02** says "validates patches against target runtime before cross-runtime application" but does not define what "validates" means. Syntax check? Content equivalence check? Runtime behavior test?

- **XRT-03** duplicates SENS-06 (post-install parity verification). Both say the same thing: post-install check that installed files match source. This should be consolidated.

**Missing:** There is no requirement for maintaining a cross-runtime capability matrix as a living artifact. The research produced a capability matrix (cross-runtime-parity-research.md) but no requirement says "this matrix is updated when features ship." Without this, the parity assessment is a snapshot that goes stale immediately.

### Parallel Execution (PAR-01 through PAR-03)

**Completeness: Minimal.**

- **PAR-01** and **PAR-02** are well-specified (per-worktree JSON files, composite view).

- **PAR-03** says "Phases with non-overlapping file scopes can execute in parallel on separate worktrees with coordinated merge" -- this is aspirational. "Coordinated merge" is unspecified. Who coordinates? What happens when file scopes overlap unexpectedly? This is the kind of requirement that sounds simple and hides enormous complexity in "coordinated."

**Feasibility:** The SUMMARY rates parallel execution as Phase 6 (separately gated) and FEATURES.md calls it Tier 3. Including it as 3 requirements in v1.20 while simultaneously planning to separately gate it sends mixed signals about scope commitment.

## Gaps Identified

### 1. No Schema Migration Requirement
PITFALL N5 (source field ambiguity) and the 198 existing signal files with the old schema need a migration path. No requirement addresses "migrate existing signals to new schema" or "resolve source field to detection_method + origin." This is a prerequisite for KB-04 (SQLite index build) but is not captured.

### 2. No Sensor Error Handling Requirement
SENS-01/02 specify what the log sensor should produce but not how it handles malformed, truncated, or unexpected input. For a sensor processing files from two different runtimes with different schemas, robustness is not optional.

### 3. No `/gsdr:discuss-milestone` Workflow Requirement
MILESTONE-CONTEXT.md explicitly identifies this gap. No requirement captures it. The gap persists.

### 4. No Node.js Version Bump Requirement
STACK.md says `engines.node` must change from `>=16.7.0` to `>=22.5.0`. This is a breaking change for any downstream user on Node 16-21. No requirement captures this, and no requirement addresses the graceful degradation path (version guard in kb.cjs) for users who cannot upgrade.

### 5. No Facets Reliability Annotation Requirement
PITFALL M3 warns that facets data is AI-generated and of unknown accuracy. TEL-04 uses facets for "quality signal enrichment" without requiring reliability labeling. The research recommends treating facets as hypothesis-generating; the requirement treats them as enrichment data.

### 6. No Backward Compatibility Testing Requirement
KB-01/KB-02 add fields to signal schema. PITFALL M2 warns that old signal reads may break. No requirement says "existing signal files parse correctly with new schema" or "kb rebuild succeeds on current signal corpus."

### 7. No Off-Ramp for Spike Methodology
11 spike methodology requirements constitute major formalization. No requirement addresses "evaluate whether formalization improved outcomes" or "simplify if compliance theater is detected." PITFALL C5 names the risk but no requirement mitigates it.

### 8. No Cross-Runtime Capability Matrix Maintenance
Research produced a capability matrix. No requirement keeps it current as features ship.

### 9. No `gsd-tools` Module Size / Complexity Budget
gsd-tools.cjs is currently 676 lines (router) with 16 modules. v1.20 adds 2 new modules (kb.cjs, telemetry.cjs) and extends the router. No requirement or constraint addresses when gsd-tools.cjs becomes too large or complex, or when the module should be refactored.

### 10. No Requirement for `/gsdr:research` Acceptance Criteria
WF-03 proposes a research command but does not specify what makes a research output valid or complete. "Committed knowledge artifacts" is the output format, but what quality standard applies?

## Recommendations

### Add / Modify

1. **Add a schema migration requirement (KB-09):** "Existing 198 signal files parse successfully with new schema. `source` field resolved to `detection_method` + `origin`. `kb rebuild` on current corpus succeeds without data loss. Migration script provided for one-time field resolution." This is a prerequisite for KB-04 and should block it.

2. **Decompose KB-04 into at least 3 requirements:** (a) SQLite index rebuild from signal files, (b) FTS5 full-text search, (c) lifecycle state queries and relationship traversal. Each is independently testable and has different complexity.

3. **Decompose KB-06 into at least 2 requirements:** (a) query/search/stats/health read operations, (b) transition/link write operations. Write operations have the C1 dual-write invariant; read operations do not.

4. **Decompose TEL-01 similarly:** baseline and enrich are notably more complex than summary/session/phase.

5. **Reclassify TEL-03 as a spike prerequisite, not a requirement.** It is an experiment that gates TEL-02 (baseline trust), not a deliverable.

6. **Add a facets reliability annotation requirement (TEL-05):** "All telemetry outputs that include facets-derived data annotate those fields as AI-generated estimates with unknown accuracy."

7. **Add a Node.js version constraint requirement:** "package.json engines.node updated to >=22.5.0 with version guard in kb.cjs that prints actionable error on older Node versions."

8. **Add a backward compatibility requirement (KB-10):** "kb rebuild and all query operations succeed against the current signal corpus without modification. New fields default gracefully when missing from old files."

9. **Modify GATE-01 to specify enforcement mechanism:** Replace "structural gate, not advisory text" with the specific behavior: "offer_next blocks phase advancement until PR is created and CI passes, or user provides explicit override with documented justification." Testable.

10. **Modify GATE-08 to reflect current state:** The discuss-phase adoption gap has partially closed since the signal was filed. Update the requirement to list specifically what remains missing (gsd-assumptions-analyzer agent, docs, mode-aware gates in plan-phase/progress) rather than "5 missing files."

11. **Add a sensor robustness requirement (SENS-07):** "Log sensor reports parse failures, format mismatches, and unexpected structures as warnings with diagnostic context rather than crashing or silently dropping data."

12. **Add a discuss-milestone workflow requirement (WF-04):** "A `/gsdr:discuss-milestone` workflow produces a MILESTONE-CONTEXT.md with structured steering brief, working assumptions, and open questions." The current milestone identified this gap.

### Modify Scope

13. **Resolve SPIKE-10 in or out.** The requirement is included but FEATURES.md recommends deferral and SUMMARY.md flags it as Tension 1. Either commit to it with decomposed requirements, or remove it to v1.21 deferred. Including it as a single unresolved requirement invites scope creep.

14. **Rewrite SPIKE-11 as a design constraint, not a requirement.** "Operationalize Lakatos, Mayo, and institutional critique" is not testable. The concrete deliverables (progressiveness ledger, severity assessment dimension, three-tier enforcement model) should be extracted as separate, specific requirements. SPIKE-11 as stated should become an epigraph or design principle in MILESTONE-CONTEXT.md, not a checkbox.

15. **Merge XRT-03 and SENS-06** -- they describe the same feature (post-install cross-runtime parity verification). Consolidate into one requirement with clear ownership.

16. **Add explicit dependency annotations to every requirement that has structural prerequisites.** Currently, dependencies are documented in FEATURES.md but not in REQUIREMENTS.md. The requirements document should be self-contained.

### Remove / Defer

17. **Consider deferring PAR-03** ("coordinated merge" for parallel phases). PAR-01 and PAR-02 provide the infrastructure. PAR-03's "coordinated merge" is underspecified and the SUMMARY rates it as separately gated. Deferring it reduces scope while keeping the enabling infrastructure.

18. **Consider deferring SPIKE-08** (protocol adherence checkpoints). This requires the spike design reviewer (SPIKE-01) and findings reviewer (SPIKE-09) to be operational before it can verify adherence. It is a second-order feature building on first-order features that do not yet exist. Ship SPIKE-01 and SPIKE-09 first; add adherence checking once there is operational experience.

## Tensions and Risks

### Tension 1: "Infrastructure Correction" vs. Actual Scope

The milestone thesis explicitly states "v1.20 is an infrastructure correction, not a feature expansion." But the requirements include:
- A novel SQLite query layer (KB-04 through KB-08)
- A new telemetry extraction system (TEL-01)
- Two entirely new agent types (SPIKE-01 design reviewer, SPIKE-09 findings reviewer)
- An epistemological framework operationalization (SPIKE-11)
- Spike programme infrastructure (SPIKE-10)
- Parallel execution infrastructure (PAR-01 through PAR-03)
- Three new workflow commands (WF-01 through WF-03)

This is not infrastructure correction. It is the most ambitious milestone in the project's history, spanning 8 capability domains with 46 requirements. Calling it "infrastructure correction" creates a framing mismatch that may lead to underestimating implementation effort and timelines. The framing should be honest: this is a major infrastructure and methodology milestone.

### Tension 2: Advisory Text Fails vs. Codex Has No Hooks

The milestone thesis is built on the finding that advisory text consistently fails. Multiple GATE requirements (GATE-06, and implicitly GATE-01/GATE-04) degrade to advisory text on Codex because Codex has no hook mechanism. This means the core thesis of the milestone -- structural enforcement over advisory text -- is structurally unachievable on one of the two target runtimes. XRT-01 addresses this by requiring degradation path documentation, but documentation of known failure is not the same as solving it.

### Tension 3: Over-Formalization Risk vs. Formalization Volume

PITFALLS C5 and C6 warn about over-formalization. The three-tier model (enforce/encourage/warn) is the proposed resolution. But SPIKE-01 through SPIKE-11 represent 11 new formalizations applied to a domain (spike methodology) that currently has 4 empirical examples. The ratio of formalization to empirical base is high. Even with the three-tier model correctly applied, the sheer volume of new process in the spike methodology category creates weight that may suppress the trial-before-formalize practice that MEMORY.md records as a methodological practice.

### Tension 4: KB-07 (Lifecycle Wiring) Depends on Workflow Changes AND CLI Tools

KB-07 says collect-signals reads `resolves_signals` from completed plan frontmatter and auto-transitions matching signals to `remediated`. This requires:
- `resolves_signals` in plan frontmatter (exists since Phase 34, never read)
- `gsd-tools kb transition` command (KB-06, new)
- `collect-signals.md` workflow modification
- Integration test proving the chain works (PITFALL C2)

These span the workflow layer, CLI layer, and knowledge store layer simultaneously. The requirement reads as one item but the implementation crosses three architectural boundaries. Failure at any junction produces the "silently does nothing" outcome that C2 warns about.

### Tension 5: WF-01 Contains Its Own Contradiction

WF-01 requires "committed audit spec before launch" (formalization) and "must not be over-formalized" (anti-formalization) in the same requirement. A committed audit spec before every invocation IS formalization. Either the audit spec is required (formalized) or the review is flexible (informal). The requirement should decide which constraint wins or define the specific boundary between them.

### Risk: No Capacity Model

46 requirements across 8 categories with no timeline, velocity estimate, or capacity constraint. The SUMMARY proposes 6 phases but does not estimate duration. If the team hits Phase 3 and realizes the pace is slower than expected, which requirements get cut? There is no priority-ordered cut line in the requirements -- only the FEATURES.md tier system, which is not referenced from REQUIREMENTS.md. The roadmapper will need to make these cuts, but the requirements document should anticipate them with explicit must-have vs. should-have annotations.

### Risk: gsd-tools.cjs Complexity Ceiling

The router is currently 676 lines with 16 modules. Adding kb.cjs and telemetry.cjs (both "high" complexity per FEATURES.md) will make this the largest single-file artifact in the project. No requirement addresses maintainability of the CLI tool itself. If v1.20 succeeds, gsd-tools.cjs may need refactoring before v1.21 can add more capabilities -- but there is no requirement for managing this growth.

## Summary Assessment

**What is strong:**
- Evidence tracing from requirements to audit findings and signals
- Pitfall documentation with concrete prevention strategies
- Anti-feature list with specific rationale for exclusion
- KB infrastructure design (File+SQLite pattern well-validated)
- Structural gate requirements (GATE-01 through GATE-03) directly address highest-recurrence failures

**What needs work:**
- Requirement atomicity -- several requirements bundle 3-7 independent deliverables
- SPIKE-10 and SPIKE-11 need resolution (in/out/decomposed), not ambiguity
- Missing requirements for migration, error handling, version constraints, backward compatibility
- Dependency annotations should be in REQUIREMENTS.md, not only in FEATURES.md
- Scope honesty -- the milestone thesis understates the actual ambition
- WF-01's internal contradiction needs resolution
- GATE-08's stale scope needs verification against current codebase state

**Overall risk level: MEDIUM-HIGH.** Not because individual requirements are poorly conceived -- they are generally well-motivated and well-researched -- but because the aggregate scope is very large, the capability domains are diverse (KB + telemetry + spike epistemology + cross-runtime + parallel execution is 5 distinct engineering disciplines), and the formalization-to-empirical-base ratio in spike methodology is high. The strongest mitigation is the SUMMARY's phased build order, but the requirements document itself does not encode phasing constraints or priority tiers.

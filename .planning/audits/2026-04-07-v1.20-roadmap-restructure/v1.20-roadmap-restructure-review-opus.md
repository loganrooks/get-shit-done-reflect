## 1. Where I Agree With the Sonnet Review

### 1.1 The Circular Dependency Framing Is Overstated

Sonnet is correct. The ROADMAP.md dependency chain is:

- Phase 58 depends on Phase 57 (telemetry baseline must be captured first) -- ROADMAP.md line 91
- Phase 60 depends on Phase 58 (structural gates provide enforcement context for sensors) -- ROADMAP.md line 115
- Phase 58 does NOT declare a dependency on Phase 60

The proposal frames this as "Phase 58 depends on Phase 60 for cross-runtime fixes, but Phase 60 depends on Phase 58" -- this is not what the ROADMAP declares. What actually exists is a credibility gap: Phase 58 makes claims about cross-runtime enforcement (success criterion 5: "every hook-dependent v1.20 feature specifies its Codex CLI degradation path") that are undermined by broken substrate code. That is a real problem, but it is not a circular dependency. The proposal's argument is stronger when stated honestly: "Phase 58 enforcement claims will be partially false on Codex without substrate repairs."

### 1.2 The "14 Issues" Count Is Unverifiable

I searched for any forensics document in `.planning/` and found none. The proposal cites `phases/01-authored-round-contract/.continue-here.md` as a prix-guesser artifact, but this file does not exist -- the actual phase directory contains `01-CONTEXT.md`, `01-DISCUSSION-LOG.md`, `01-RESEARCH.md`, `01-VALIDATION.md`, and four plan files. There is no `.continue-here.md` in it. The number "14" has no traceable source document. Sonnet is right to flag this. The underlying problems are real (the 83-line AGENTS.md and the portable-gsd infrastructure confirm the user had a catastrophic session), but the quantification is confabulated.

### 1.3 Tier 3 Contains v1.21 Pull-Forward Risk

I verified that REQUIREMENTS.md line 194 explicitly defers AUT-02 ("Full Issue #17 -- cross-runtime drift hardening, 9 acceptance criteria") to v1.21. The Tier 3 "behavioral guardrails appendix," "Codex skill adapter block," and "reasoning effort in model profiles" have significant overlap with AUT-02's acceptance criteria. This is a legitimate scope concern.

### 1.4 The GPT-5.4 Tool Self-Reports Are Model Self-Assertions

The drift audit's evidence for the spawn_agent parameter names (`agent_type`, `model`, `reasoning_effort`) comes from GPT-5.4 describing its own session tool surface. This is first-party evidence from the model about its own runtime, which is higher quality than hearsay, but it is not independently verified against an API contract or documentation. Adapter blocks built on this could encode stale assumptions. Sonnet is right to flag this risk.

## 2. Where I Disagree With the Sonnet Review

### 2.1 The "Circular Dependency Is False" Claim Is Right But Misses the Real Structure

Sonnet correctly identifies that the ROADMAP declares a one-directional dependency (Phase 60 depends on Phase 58, not vice versa). But Sonnet then dismisses this as a "sequencing inconvenience," which understates the problem.

The actual structural defect is subtler: **Phase 58 success criterion 5 is unrealizable without substrate work that currently lives in Phase 60.** Success criterion 5 reads: "Every hook-dependent v1.20 feature specifies its Codex CLI degradation path in phase CONTEXT.md before implementation begins, and capability-matrix.md is updated." But the capability-matrix.md is factually wrong about Codex right now (hooks listed as "N" when `codex_hooks` is `under development true`; config file listed as `codex.toml` when it is `config.toml`; agents listed as "via AGENTS.md" when they use `config.toml` + `agents/*.toml`). Updating capability-matrix.md to be accurate IS substrate work, and it is currently scoped to Phase 60.

This is not a sequencing inconvenience. It is a contradiction: Phase 58 requires updating a document that is factually wrong, and the work to make it factually accurate is scheduled two phases later. Calling this a "sequencing inconvenience" is too generous to the current roadmap. Calling it a "circular dependency" is too generous to the proposal. The right term is **prerequisite inversion**: work that is a prerequisite for Phase 58 credibility is currently scheduled as Phase 60 follow-on work.

### 2.2 Tier 3 Is Not Entirely "Scope Creep Into v1.21 Territory"

Sonnet treats Tier 3 as a monolith and recommends deferring most of it. But Tier 3 contains at least two items that are proportionate to v1.20:

1. **Post-install Codex parity smoke test** -- Sonnet keeps this, and I agree.
2. **Reasoning effort in model profiles** -- Sonnet defers this, and I think that is wrong. The signal-detection.md already encodes `opus-class (claude-opus-*)` and `sonnet-class (claude-sonnet-*)` as the profile-mismatch detection heuristic (lines 67-76, verified). This heuristic is actively wrong for Codex right now. If Phase 57 telemetry ships and Phase 58 enforcement ships, signal-detection will generate false mismatches or miss real mismatches on Codex sessions. This is not v1.21 work -- it is a prerequisite for accurate telemetry and honest enforcement on a runtime that is already deployed.

What IS v1.21 territory: the behavioral guardrails appendix and the full skill adapter block. The guardrails appendix is generalizing what the prix-guesser AGENTS.md does at the project level -- that generalization is AUT-02. The full adapter block depends on GPT-5.4 self-reported tool signatures that may drift. Both should be deferred.

### 2.3 "Fold Into 55.1" Underestimates the Scope

Sonnet's preferred recommendation (Option A) is to fold all Tier 1 fixes into Phase 55.1 and Tier 2 into Phase 55.1 or 55.2. Phase 55.1 is currently described as:

```
**Goal:** [Urgent work - to be planned]
**Requirements**: TBD
**Depends on:** Phase 55
**Plans:** 0 plans
```

Phase 55.1 is already scoped to three specific upstream bugs: #2005 (details-wrapped ROADMAP corruption), #1972 (incomplete atomicWriteFileSync), #1981 (worktree reset --soft data loss). These are upstream bug patches with known upstream PRs.

Adding 6 Tier 1 code fixes + 6 Tier 2 documentation fixes + parity smoke test + reasoning effort model fix to Phase 55.1 would:
- Nearly triple the scope of a phase designed for surgical upstream patches
- Conflate upstream bug patches (where we pull from upstream) with fork-specific Codex substrate work (where we fix our own code)
- Create a phase with two unrelated workstreams (upstream patching and Codex substrate repair) that have different review/merge characteristics

This is the kind of phase overloading that this project's own audit methodology would flag. Sonnet's alternative (55.2) is more proportionate, but Sonnet presents it as "acceptable" rather than recommended.

### 2.4 The Telemetry Delay Is Neither Pure Cost Nor Pure Benefit

Sonnet claims the telemetry delay is a cost, not a benefit, citing TEL-02: "baseline captured before any v1.20 structural interventions ship." The proposal claims it is a benefit because "telemetry baseline captures post-Codex-fix state."

Both framings are partially correct and partially misleading:

**Sonnet's framing is partially wrong** because TEL-02 says "before any v1.20 structural interventions." Codex substrate fixes (fixing `automation.cjs` to recognize `codex-cli`, fixing sensor discovery globs) are NOT structural interventions in the TEL-02 sense. TEL-02's concern is about structural enforcement gates (Phase 58) contaminating the baseline. Fixing a broken runtime resolver does not introduce a new structural intervention -- it corrects a pre-existing defect. Delaying telemetry by one phase to fix a broken runtime resolver does not contaminate the baseline; it makes the baseline more accurate by ensuring it measures the system as intended, not as accidentally broken.

**The proposal's framing is partially wrong** because it overstates the benefit. The baseline captures the state of the system whenever it is captured. If the Codex resolver is broken when the baseline is captured, the baseline documents that broken state, which is still useful information. There is no "confounded before/after comparison" -- the Codex fix would simply appear as a change between baseline and post-intervention measurement, which is what measurement infrastructure is designed to detect.

**My assessment:** The telemetry delay is a minor cost (one phase later) with a minor benefit (slightly cleaner baseline). Neither framing should drive the decision. The decision should rest on the prerequisite inversion in Phase 58, not the telemetry ordering.

### 2.5 The "14 Issues" Claim -- Can I Verify?

Sonnet claims this is "confabulated." I partially agree and partially disagree:

- The number "14" has no traceable source document -- **verified, Sonnet is right**
- The `.continue-here.md` artifact cited does not exist -- **verified, Sonnet is right**
- However, the prix-guesser AGENTS.md (83 lines, verified) and portable-gsd infrastructure (verified at `prix-guesser/tooling/portable-gsd/`) confirm a catastrophic Codex session occurred
- The prix-guesser `.planning/phases/01-authored-round-contract/` has 8 files including a full research doc, 4 plans, discussion log, and validation -- this is a complete phase that was planned and executed
- Whether "14 issues" is the right count is unverifiable, but the underlying experience (session produced operational lessons that required 83 lines of guardrails) is first-hand evidence

Sonnet is right that the number should not be cited. The qualitative evidence is strong enough without it.

## 4. Bias Analysis

### 4.1 The Proposal: Recency and Availability Bias

**Recency bias: Confirmed.** The proposal was written the same day as the three GPT-5.4 audits, Phase 55 completion, and Phase 56 completion. All evidence is from 2026-04-08. Sonnet correctly identifies this pattern.

**Availability bias: Confirmed but overstated by Sonnet.** The prix-guesser experience is vivid and recent. But Sonnet's counterpoint (that a successful Claude Code pipeline ran the same day) is a weak challenge: the successful pipeline ran on Claude Code, not Codex. The relevant question is not "do things work on Claude Code?" (they do) but "is the Codex substrate so broken that Phase 58 enforcement claims are false?" The proposal answers that question with code-level evidence. The bias is real but the conclusion is still sound.

**Scope anchoring: Confirmed.** The proposal develops Tier 1/2/3 in full detail while alternative framings get two-sentence dismissals. The anchoring is clear in how the "fold into 55.1" alternative is not even explored as a viable option despite being structurally simpler.

### 4.2 The Sonnet Review: Status Quo Bias and Minimization

**Status quo bias: Present.** Sonnet's preferred option (fold into 55.1, no renumbering, keep telemetry at 57) is explicitly motivated by avoiding disruption: "no new phase number is needed, no renumbering occurs." The cost of renumbering is real but small. Sonnet weighs it as if it were a major cost, when in practice it is a mechanical find-and-replace in 3-4 files. This is status quo bias operating as change aversion.

**Minimization of the substrate problem: Present.** Sonnet frames the Tier 1 fixes as "10-line fix," "targeted fix," "targeted fix," "documentation fixes," "new document." This undersells the scope. The `automation.cjs` fix is not a 10-line change -- it requires designing a runtime capability resolver that recognizes `codex-cli` as a first-class runtime, which means deciding what capabilities to assign, adding test coverage for the new runtime path, and ensuring the resolver is consistent with the capability matrix. The sensor discovery fix requires designing multi-format discovery that works across `.md` and `.toml` in both `.claude/` and `.codex/` paths. These are not trivial patches.

**Proportionality argument has a blind spot:** Sonnet argues that a full phase for a secondary runtime is disproportionate. This is valid in general, but it elides the fact that Phase 58's success criteria explicitly include Codex cross-runtime claims. If Phase 58 is going to make claims about Codex, the substrate for those claims must be sound. The question is not "how important is Codex in general?" but "can Phase 58's specific claims survive contact with the actual Codex runtime?"

### 4.3 Are Either Neglecting the Milestone's Core Goals?

The milestone goal (MILESTONE-CONTEXT.md, ROADMAP.md line 15) has four pillars:
1. Replace advisory with structural enforcement
2. Mature signal system to full lifecycle management
3. Overhaul spike methodology
4. Establish measurement infrastructure

**The proposal** is laser-focused on pillar 1 (structural enforcement credibility on Codex) at the expense of acknowledging pressure on pillars 3 and 4. Sonnet is right that adding a phase risks compressing Phases 62-64 (spike methodology, spike programmes). The proposal does not engage with this tension at all.

**The Sonnet review** is laser-focused on minimizing disruption to the existing plan, which implicitly prioritizes pillars 3 and 4 (keeping their phase numbers and timeline intact) over pillar 1 (ensuring enforcement claims are credible on Codex). Neither document asks the right question: "What is the minimum substrate work required for Phase 58 to be honest about its Codex claims?"

## 5. The Core Question

> Does Codex substrate work belong BEFORE telemetry + enforcement, or can it be handled incrementally?

### The Evidence

**Phase 58 success criterion 5** requires updating `capability-matrix.md` when features ship. The capability matrix is factually wrong about Codex right now (verified: hooks, config file name, agent registration, skill format). Updating it during Phase 58 means doing Tier 2 work during Phase 58, which means the "fixes live in Phase 60" framing is already broken -- some fixes MUST happen during Phase 58 or its success criterion is unrealizable.

**Phase 58 success criterion 4** requires "workflow-enforced step on Codex" for automation postlude. This requires the runtime capability resolver to correctly identify `codex-cli`. Currently it does not (verified: `automation.cjs:114-117`). This fix MUST happen before or during Phase 58.

**Phase 57 (telemetry) does NOT have Codex-specific success criteria.** None of TEL-01a through TEL-05 mention Codex. Telemetry extraction works on `session-meta` files that already exist. The telemetry phase is not blocked by Codex substrate issues.

**Phase 60 (sensors) IS blocked by substrate issues.** SENS-02 (cross-runtime log adapter), SENS-06 (post-install parity), XRT-02 (patch compatibility) all depend on correct runtime detection.

### My Position

The substrate work belongs **before enforcement (Phase 58)** but **not necessarily before telemetry (Phase 57)**. The telemetry phase does not depend on Codex substrate. The enforcement phase does. The optimal ordering is:

```
Phase 57: Measurement & Telemetry Baseline   (unchanged)
Phase 57.1: Codex Substrate Repair            (inserted before enforcement)
Phase 58: Structural Enforcement Gates        (unchanged, but now credible on Codex)
```

This preserves the telemetry ordering (baseline before interventions, as TEL-02 requires), fixes the prerequisite inversion for Phase 58, avoids renumbering all phases, and keeps the scope focused.

**Why not 55.1?** Phase 55.1 is scoped to upstream bug patches. Adding fork-specific Codex substrate work conflates two different workstreams. If the work exceeds 55.1's scope, the right move is a decimal phase near the work it enables, not a scope overload on a phase designed for something else.

**Why not a full new Phase 57?** The proposal's Phase 57 (renumbering everything) is disproportionate to the actual blocking dependency. Telemetry does not depend on Codex substrate. Only enforcement does. A decimal phase (55.2 or 57.1) scoped to verified blocking fixes avoids the renumbering overhead and keeps scope honest.

## 6. Verdict: ACCEPT WITH MODIFICATIONS (Third Option)

Neither the proposal's full restructure nor Sonnet's "fold into 55.1" is the right answer.

### Recommended Structure

**Phase 55.1: Upstream Bug Patches** -- Keep as-is. Scope to #2005, #1972, #1981.

**Phase 55.2: Codex Substrate Repair** (NEW decimal phase)
- Scope: Tier 1 verified code fixes + Tier 2 documentation drift fixes + parity smoke test + signal-detection model provenance fix
- Does NOT include: behavioral guardrails appendix, full skill adapter block, or runtime-agnostic spawn/wait abstraction (those are AUT-02/v1.21)
- DOES include: reasoning effort in model provenance for signal-detection.md (this is a prerequisite for honest signal detection on Codex, not v1.21 scope)
- Success criteria: 4 items (runtime resolver recognizes codex-cli, sensor discovery handles .toml, brownfield detection excludes .codex/, capability-matrix.md has zero factual inaccuracies vs live CLI)
- New requirements: CODEX-01 (runtime resolver), CODEX-02 (multi-format discovery), CODEX-05 (parity research doc). Keep to 3 new requirements, not 5.

**Phase 57: Measurement & Telemetry Baseline** -- Unchanged. Stays at Phase 57. Telemetry baseline captured before enforcement.

**Phase 58: Structural Enforcement Gates** -- Unchanged in scope. Now has accurate substrate to build on. XRT-01 references Phase 55.2 infrastructure rather than fictional advisory paths.

**Phase 60: Sensor Pipeline (reduced scope)** -- Cross-runtime sensor discovery already done in Phase 55.2. Remaining scope: log sensor, patch sensor, progressive deepening, cross-runtime log adapter. SENS-05 and SENS-06 satisfied by Phase 55.2 parity test.

**All phases 57+: No renumbering.** Only one decimal phase inserted.

### Why This Is Better Than Both Alternatives

**vs. the Proposal (full Phase 57 insertion):**
- No renumbering of 10 phases
- Telemetry stays at its designed position (baseline before interventions)
- Tier 3 scope creep eliminated
- Smaller blast radius for the milestone plan
- Still addresses the prerequisite inversion for Phase 58

**vs. Sonnet's Option A (fold into 55.1):**
- Does not overload 55.1 with unrelated work
- Gives the substrate work its own scope, plan, and verification -- appropriate for work that touches 6+ source files across code, docs, tests, and installer
- The scope is not "surgical" -- it requires design decisions about runtime resolution, multi-format discovery, and model provenance. It deserves its own planning phase, not cramming into an upstream-patch phase.

### What Must Change in the Proposal

| Change | Priority |
|--------|----------|
| Drop circular dependency framing; use "prerequisite inversion" | High |
| Insert as Phase 55.2, not new Phase 57; no renumbering | High |

### Risk Assessment

If accepted as Phase 55.2 with scoped Tier 1 + Tier 2 + parity test + model provenance:
- No renumbering disruption
- No telemetry ordering change
- No v1.21 pull-forward
- Phase 58 enforcement claims become credible on Codex
- Phase 60 scope reduced by verified substrate work already done
- 3 new requirements (CODEX-01, CODEX-02, CODEX-05) is proportionate

Primary risk: Phase 55.2 scope creep during execution. Mitigation: strict scope gate -- only items verified against code in this audit trail. No "discovered during execution" additions.

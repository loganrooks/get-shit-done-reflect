# Adversarial Review: v1.20 Roadmap Restructure Proposal

**Date:** 2026-04-08
**Reviewer:** Claude Sonnet 4.6 (adversarial reviewer — independent of proposal author)
**Proposal:** `.planning/audits/v1.20-roadmap-restructure-proposal.md`
**Verdict:** ACCEPT WITH MODIFICATIONS

---

## Executive Summary

The proposal identifies real defects in the Codex substrate and makes a structurally coherent argument for front-loading fixes. The core evidence is mostly solid. However, the proposal overstates the "circular dependency" (it is a sequencing inconvenience, not a logical cycle), contains unverified GPT-5.4 capability claims, conflates harness bugs with substrate design work, and proposes a phase scope (Tier 3 "adapter infrastructure") that exceeds what the verified evidence warrants. The proposal should be accepted with the Tier 3 adapter infrastructure scoped down to what can be directly verified, and the circular dependency framing dropped in favor of honest sequencing logic.

## 2. Bias Analysis

### 2.1 Recency Bias

The proposal was authored on 2026-04-08, the same day the trois GPT-5.4 audits were generated, Phase 55 completed, and Phase 56 completed. The Codex audit cluster is fresh material. The proposal responds almost entirely to this audit cluster.

The v1.20 milestone goal has four equal pillars: (1) structural enforcement, (2) signal lifecycle/KB maturity, (3) spike methodology overhaul, (4) measurement infrastructure. The proposal gives zero space to whether adding a Phase 57 Codex substrate phase might crowd out Phases 61-63 (spike methodology, workflow commands, spike programmes). It mentions "Phases 61-63 renumbered" but says nothing about whether they might be deprioritized, deferred, or cut under time pressure.

This is recency bias operating at the milestone level. The Codex gap just happened; the spike methodology overhaul is older, more abstract, and was already extensively researched. The proposal treats "Codex is broken" as more urgent than "spike methodology is epistemically weak" — but the latter was independently derived from 4 sources (REQUIREMENTS.md) and addresses a pattern the project documented as ongoing failure.

### 2.2 Availability Bias

The prix-guesser experience is vivid: zero product work across a session, manual infrastructure invented, behavioral guardrails written from scratch. The proposal treats this as the canonical Codex failure mode. But the signal log (`sig-2026-04-08-autonomous-discuss-plan-execute-pr-merge-pipeline.md`) documents a successful full autonomous pipeline on Claude Code the same day. The asymmetry in how these experiences are framed is worth noting.

The proposal implicitly weights the bad Codex experience heavily. The question it does not ask is: how often does a Codex session fail this catastrophically vs. run adequately? One session is not a statistical base. The proposal's recommendation hinges on this single experience as evidence that "the prix-guesser disaster" is something that will repeat — but there is no frequency estimate.

### 2.3 Anchoring Bias

The proposal is anchored on "what Phase 57 should fix" before it asks whether a new phase is the right structure. The alternative framings examined (fold into Phase 58, distribute across phases, decimal phase 55.2) are listed in Section 4 of the proposal but dismissed quickly with two-sentence arguments. The anchoring shows in how the proposal's Tier 1, 2, and 3 scope is fully developed while the alternatives remain skeletal.

The anchor also shows in the framing of the milestone goal. The proposal reads XRT-01 (CONTEXT.md degradation paths) as "advisory with no enforcement" and uses this to justify a new structural phase. But XRT-01 was designed as documentation, not code — the proposal is critiquing a documentation requirement for not being code-level enforcement. This is a fair critique, but it justifies the XRT-01 requirement changing, not necessarily a new phase.

### 2.4 Scope Creep

The Tier 3 "adapter infrastructure" scope explicitly includes:
- Codex skill adapter blocks in generated SKILL.md (tool translation table)
- Behavioral guardrails appendix for Codex agents
- Reasoning effort in model profiles
- Post-install Codex parity smoke test

This is new feature development, not substrate repair. The first three items in Tier 3 are in direct competition with what REQUIREMENTS.md already deferred: `AUT-02` ("Full Issue #17 — cross-runtime drift hardening, 9 acceptance criteria") is explicitly v1.21. The proposal is creating requirements that sound different from AUT-02 but overlap significantly with it. If Phase 57 ships behavioral adapter blocks and reasoning-effort model profiles, what remains for AUT-02?

The adapter block scope is appropriate, but there is a risk of informal v1.21 requirement pull-forward under the cover of "substrate repair."

## 3. What's Being Neglected

### 3.1 Telemetry Baseline Delay Compounds the Measurement Problem

The proposal delays Phase 57 (telemetry baseline) by 1 phase and frames this as a benefit: "telemetry baseline captures post-Codex-fix state." But REQUIREMENTS.md `TEL-02` says the baseline must be captured "before any v1.20 structural interventions ship." The proposal's justification (measuring a broken baseline is confounded) inverts the problem: the baseline is supposed to be captured *before* interventions precisely so that the pre-intervention state is documented, even if imperfect.

There is a well-understood principle in measurement: a contaminated baseline captured early is still better than no baseline, because it anchors the pre-intervention state. If telemetry is delayed until after Codex is fixed, the baseline will reflect "post-Codex-fix but pre-enforcement-gate" — which is itself already a post-intervention state relative to the original roadmap intent. The measurement story is weaker after this change, not stronger.

The proposal's own language (`ROADMAP.md:83`: "baseline must be captured before any Phase 58 structural gates are deployed") is applied selectively — the proposal uses this logic to argue Codex fixes must precede enforcement gates, but fails to apply the same logic consistently to the telemetry baseline.

### 3.2 Phases 61-63 Silent Deprioritization

The proposal adds 1 phase (11 total). With fixed-bandwidth development, adding a phase means something else gets compressed or cut. The proposal asserts "all other phases: renumbered +1, otherwise unchanged" — but this cannot be guaranteed unless Phase 57 scope is held to strict Tier 1 (bug fixes only). Tier 2 and Tier 3 represent significant development work.

Phases 62-64 (Workflow Commands, Spike Programme Infrastructure, Parallel Execution) are already last in the execution order and the first candidates for deferral under time pressure. The spike methodology overhaul (now Phase 62) and spike programme infrastructure (now Phase 64) are the most epistemologically ambitious work in v1.20. Adding a new Phase 57 increases pressure on these. The proposal does not acknowledge this tension.

### 3.3 Adapter Layer as v1.21 Territory

The Tier 3 "Codex skill adapter block in generated SKILL.md" and "behavioral guardrails appendix" are generator-level features, not substrate repairs. The substrate repairs (Tier 1) fix broken code in `automation.cjs`, `sensors.cjs`, `init.cjs`, and `install.js`. The adapter layer generates new content. These are different categories of work.

The `Out of Scope` section of REQUIREMENTS.md explicitly excludes "Full subagent parity in Codex CLI" and delegates AUT-02 ("Full Issue #17 — cross-runtime drift hardening") to v1.21. The behavioral adapter blocks and tool translation tables look like a v1.21 concern that is being pulled into v1.20 under the "substrate" label.

### 3.4 Spike Methodology Already Has Evidence of Neglect

The proposal mentions SPIKE-01 through SPIKE-12 (10 requirements) in Phase 62 (now). These were grounded in "4 independent sources" (REQUIREMENTS.md SPIKE-01) and "3 independent derivations" (SPIKE-02). The audit that grounded v1.20 flagged spike methodology as a high-priority structural gap — the same audit that grounded the GATE-* requirements.

The proposal does not ask whether delaying enforcement gates by 1 phase (to accommodate Codex substrate) further delays the spike methodology overhaul. If Phase 62 was already borderline for the milestone timeline, adding Phase 57 makes it more likely to slip.

## 4. Alternative Approaches Not Considered

### 4.1 Fold Tier 1 Bug Fixes Into Phase 55.1 (Upstream Bug Patches)

Phase 55.1 already exists as "INSERTED" urgent work with "Requirements: TBD" and "Plans: 0 plans." The Tier 1 bug fixes (`automation.cjs:114-147`, `sensors.cjs:13-25`, `init.cjs:205-228`, `install.js:21-34`) are exactly the kind of surgical fixes that Phase 55.1 was designed for. The phase already exists; it just needs its requirements defined.

The proposal does not explain why these fixes require a new Phase 57 rather than being added to the already-planned Phase 55.1. The only acknowledged difference is that Phase 55.1 is described as "upstream bug patches (#2005, #1972, #1981)" while Tier 1 are harness-level Codex compatibility bugs. This is a category distinction, not an execution boundary — both are pre-enforcement substrate repairs.

**This is the strongest alternative not considered.** If Phase 55.1 absorbs Tier 1, no new phase number is needed, no renumbering occurs, and telemetry baseline stays at Phase 57.

### 4.2 Distribute Codex Compatibility as XRT-01 Enforcement

XRT-01 already requires "every hook-dependent v1.20 feature specifies its Codex CLI degradation path in phase CONTEXT.md before implementation begins." The proposal correctly identifies this as documentation-only. But the fix is to strengthen XRT-01 to require *code-level* degradation checks, not to create a new phase. If XRT-01 required that each phase also ship a passing Codex capability test before moving forward, the substrate fixes would happen organically per-phase.

This approach keeps fixes local to the phase that introduces the feature they protect. The proposal rejects this because "documentation requirements fail 100% on Codex agents" — but this is a framing problem. XRT-01 could be changed from a documentation requirement to a code-level gate requirement without needing a new phase.

### 4.3 Decimal Phase 55.2 for Codex Substrate

If the Tier 1 fixes are too substantial for Phase 55.1 (which is scoped to upstream patches), a decimal Phase 55.2 could hold the Codex substrate work. This would avoid renumbering 10 phases, keep telemetry at Phase 57, and scope the Codex work to Tier 1+2 (verified bugs and documentation drift) without the Tier 3 adapter infrastructure pulling v1.21 work forward.

### 4.4 Adapter Layer as Backlog Item

The Tier 3 behavioral adapter blocks and reasoning effort model profiles could be added to the backlog under `AUT-02` territory with a "promote to v1.20 if Phase 55.2 scope stays below X hours" gate. This would preserve the option without committing to scope now.

## 5. The Circular Dependency Claim

The proposal's central structural argument is a circular dependency between Phase 58 and Phase 60:

> Phase 58 depends on Phase 60 (for cross-runtime fixes) but Phase 60 depends on Phase 58 (structural gates provide enforcement context for sensors).

**This is not a logical circular dependency. It is a sequencing inconvenience misframed as a circularity.**

The ROADMAP.md dependency chain is:
- Phase 60 (`Sensor Pipeline & Cross-Runtime Adaptation`) depends on Phase 58 (`structural gates provide enforcement context for sensors`)
- Phase 58 does NOT depend on Phase 60. ROADMAP.md:91 states "Depends on: Phase 57 (baseline must be captured first)"

The proposal's framing — "Phase 58 claims enforcement but Phase 60 contains the actual Codex substrate fixes" — describes a semantic gap (Phase 58 makes claims it cannot fully support without Phase 60 fixes), not a dependency cycle. A circular dependency would mean neither phase could execute without the other having completed first. The actual situation is: Phase 58 can execute, but its Codex enforcement story will be weaker than claimed until Phase 60 fixes land.

The proposal identifies a real ordering problem: the Codex fixes in Phase 60 (later) would have been more useful before Phase 58 (earlier). That is a valid sequencing concern. But it does not constitute a circular dependency. Inflating a sequencing concern into a "circular dependency" overstates the structural defect.

A honest version of the argument: "Phase 58 makes cross-runtime enforcement claims that will be partially false until Phase 60 ships. Pulling the Codex substrate fixes earlier makes Phase 58 claims more credible when delivered." That is a good argument. It does not need the circular dependency framing to be persuasive.

The circular dependency claim also surfaces a deeper issue: the proposal itself notes that Phase 60 depends on Phase 58 because "structural gates provide enforcement context for sensors." If Phase 60 depends on Phase 58, and Phase 58 needs Codex substrate before it can credibly claim cross-runtime enforcement, then inserting Phase 57 breaks the Phase 60 dependency on Phase 58 *and* makes Phase 60's enforcement context valid. This is actually the right argument — but it requires acknowledging that the dependency is one-directional, not circular.

## 6. Proportionality

### 6.1 Primary vs Secondary Runtime

GSD Reflect is explicitly described in PROJECT.md as a "runtime-agnostic enhancement" and lists "4 runtimes supported: Claude Code, OpenCode, Gemini CLI, OpenAI Codex CLI" with "GSDR namespace co-installation enables side-by-side with upstream GSD." However, the 628-test suite is validated via "Claude Code + mechanical installer tests" (PROJECT.md). Runtime-specific test suites per runtime are explicitly out of scope.

The milestone itself (MILESTONE-CONTEXT.md) focuses on Claude Code as the primary surface: hooks, structural enforcement, sensor pipeline — all are Claude Code features. Codex degrades gracefully (or is supposed to).

The project decision log includes: "OpenAI Codex CLI as 4th runtime — User's primary secondary runtime alongside Claude Code." Codex is explicitly secondary.

A full dedicated phase for a secondary runtime is disproportionate if the primary failures are:
1. `automation.cjs` caps Codex incorrectly (10-line fix)
2. Sensor glob misses `.toml` (targeted fix)
3. `init.cjs` doesn't exclude `.codex/` (targeted fix)
4. Documentation drift in capability-matrix.md (documentation fixes)
5. Missing `cross-runtime-parity-research.md` (new document)

These are Tier 1 and Tier 2 fixes. They belong in Phase 55.1 or a decimal phase, not a full new numbered phase with 5 new CODEX-* requirements, 3 Tier 3 infrastructure deliverables, and 6 success criteria.

### 6.2 Actual Codex Deployment Scale

The proposal does not quantify how many Codex deployments exist vs Claude Code deployments. Evidence from the repository suggests: this project's get-shit-done-reflect runs on Claude Code; prix-guesser runs Codex on upstream GSD (not GSDR). The global Codex installation exists at `~/.codex/` on this machine. There is no evidence of additional GSDR-on-Codex deployments. The proposal justifies a full phase ("prevents repeating the prix-guesser disaster on any GSD Reflect deployment") without evidence that there are multiple GSD Reflect deployments on Codex.

### 6.3 Is a Full Phase Justified?

For Tier 1+2 (bug fixes + documentation drift): No. These are surgical and should be handled in Phase 55.1 or a decimal phase without renumbering. The fixes are real and urgent; they do not require a new milestone-phase-level organizational unit.

For Tier 3 (adapter infrastructure): Only partially. The post-install parity smoke test is appropriate and proportional. The behavioral adapter block and reasoning effort work approach v1.21 scope. The parity research document is appropriate and lightweight.

## 7. Verdict: ACCEPT WITH MODIFICATIONS

### Accept

The following elements of the proposal are well-grounded and should be implemented:

**Tier 1 bug fixes are real and urgent:**
- `automation.cjs` Codex runtime capability resolver bug (verified first-hand)
- Sensor discovery glob incompatibility with `.toml` agents (well-evidenced)
- `init.cjs` brownfield detection must exclude `.codex/` (well-evidenced)
- Signal model provenance must support `{model, reasoning_effort}` (directionally correct)
- CODEX_AGENT_SANDBOX expansion for synthesizer/reflector/spike-runner (stated evidence)
- Fix stale task_tool claim in execute-phase.md (stated evidence)

**Tier 2 documentation drift is real and should be fixed:**
- `config.toml` not `codex.toml` (verified: docs vs tests already disagree within repo)
- Hooks: `under development true`, not `N` (verified: live CLI confirms)
- Subagent API: `agent_type`, `model`, `reasoning_effort` params (verified: session self-report)
- Skills format: `skills/<name>/SKILL.md` not `skills/*.md` (verified: live filesystem)
- Agent registration: `config.toml` + `agents/*.toml` (verified: installer vs docs divergence)
- `cross-runtime-parity-research.md` must be created (verified: file does not exist)

**Sequencing argument has merit:**
Phase 58 enforcement claims are stronger if the Codex substrate is repaired first. This is a valid sequencing argument, even though the "circular dependency" framing overstates it.

### Do Not Accept (require modification)

**The circular dependency framing should be dropped.** The ROADMAP.md dependency chain shows Phase 60 depends on Phase 58, but Phase 58 does not depend on Phase 60. The problem is a sequencing inconvenience and credibility gap, not a logical cycle. Misnaming this weakens the proposal's analytical integrity.

**Tier 3 adapter infrastructure should be scoped down:**
- Keep: Post-install Codex parity smoke test (proportional, verifiable)
- Keep: `cross-runtime-parity-research.md` as a living document (Tier 2, appropriate)
- Defer: Behavioral guardrails appendix in generated SKILL.md (this is what AGENTS.md already does at the project level; generalizing it is AUT-02/v1.21 territory)
- Defer: Codex skill adapter block with tool translation (adapter vocabulary is correct but building it into generator is v1.21 scope; the drift audit's session self-report of tool names is unverified as authoritative)
- Defer: Reasoning effort in model profiles (deserves its own requirement and spike, not incidental Phase 57 work)

**The new Phase 57 structure should be reconsidered:**
Option A (preferred): Fold Tier 1 into Phase 55.1, Tier 2 into Phase 55.1 or a 55.2, keep Tier 3 parity test in Phase 55.2. No phase renumbering. Telemetry baseline stays at Phase 57. The only new requirement is updating Phase 55.1 scope.

Option B (acceptable): Insert Phase 55.2 "Codex Substrate Repair" with Tier 1 + Tier 2 scope. No phase 57+ renumbering. Telemetry stays at 57. Tier 3 deferred.

Option C (as-proposed, with scope reduction): Accept the full Phase 57 insertion and renumbering but scope Phase 57 to Tier 1 + Tier 2 + parity smoke test only. Drop the behavioral adapter blocks and reasoning effort requirements from Phase 57. This keeps the proposal's structure while trimming the v1.21 pull-forward.

**The 14-issue forensics count should not be cited as evidence.** The number is unverifiable. The actual evidence (AGENTS.md guardrails, portable-gsd infrastructure, session signal) is strong enough without it.

**Telemetry delay should not be framed as a benefit.** The proposal's argument that "telemetry baseline captures post-Codex-fix state, making before/after comparisons more meaningful" inverts the baseline logic in REQUIREMENTS.md `TEL-02`. A pre-intervention baseline is the point. If telemetry is genuinely delayed by one phase, this should be acknowledged as a cost, not reframed as a benefit.

### Summary of Modifications Required

| Modification | Priority |
|---|---|
| Drop "circular dependency" framing; use "sequencing inconvenience" | High |
| Scope Phase 57 to Tier 1 + Tier 2 + parity smoke test only | High |
| Evaluate folding into Phase 55.1 before accepting Phase 57 insertion | High |
| Drop/defer behavioral adapter blocks and reasoning effort requirements | Medium |
| Acknowledge telemetry delay as a cost, not a benefit | Medium |
| Drop unverifiable "14 issues" citation | Low |
| Note that GPT-5.4 tool self-reports are model self-assertions, not verified API contracts | Low |

### Risk Assessment

If accepted as-proposed (without modifications), the primary risks are:
1. Tier 3 scope pulls v1.21 work into v1.20, increasing pressure on Phases 62-64 (spike methodology)
2. Adapter blocks built on GPT-5.4 session self-reporting may themselves encode stale assumptions
3. 10-phase renumbering creates documentation debt across STATE.md, REQUIREMENTS.md, and all planning artifacts
4. Milestone goal ("replace advisory with structural enforcement, mature signal system, overhaul spike methodology") is diluted by adding a Codex substrate phase that competes for execution bandwidth

If the Tier 1 + Tier 2 fixes are routed through Phase 55.1 (Option A), none of these risks materialize. The fixes are still urgent and real; they simply do not require a new phase number.

## Tensions

### T1: Advisory text fails on Codex (Tension 2)

**Reviewer says:** The core thesis (structural > advisory) is unachievable on Codex, which has no hooks. Specifying advisory text as the degradation path specifies known failure.

**My response: Acknowledge the tension, accept the limitation.**

This is a real tension with no clean resolution. Codex may add hooks in future versions — we can't know. For v1.20, the honest position is: structural enforcement on Claude Code (primary runtime), best-effort enforcement on Codex (secondary runtime). The degradation paths should be explicit about their limitations. We should not pretend advisory text will work on Codex just because we have no alternative.

**Proposed action:** Add a note to REQUIREMENTS.md preamble: "Structural enforcement requirements (GATE-*) achieve full enforcement on Claude Code via hooks. On Codex CLI (no hook mechanism), enforcement degrades to workflow-level checks that depend on agent compliance — the same advisory pattern the audit found unreliable. This is an accepted limitation for v1.20; Codex hook support is tracked as a dependency for full cross-runtime enforcement parity."

**Decision:** `________________`

### T2: WF-01 contradiction (Tension 5)

**Reviewer says:** "Committed audit spec before launch" AND "must not be over-formalized" in the same requirement.

**My response: Accept — resolve the contradiction.**

The audit spec is a lightweight artifact (what to review, what model, what to focus on) — not a formal methodology. The "must not be over-formalized" applies to the review *protocol* (don't enforce rigid rounds, don't prescribe model choice, don't make it mandatory). These are different aspects.

**Proposed revision:** WF-01: "`/gsdr:cross-model-review` command launches background cross-model review with: (a) committed audit spec documenting what is being reviewed and by whom (traceability), (b) structured response template (accept/accept-with-nuance/dispute-with-evidence) for the receiving agent. The command is opt-in, model choice is configurable not prescribed, and review round count is user-determined. Over-formalization risk: the review protocol must remain flexible — the command provides infrastructure, not a rigid procedure."

**Decision:** `________________`

---

### T3: Formalization volume vs empirical base (Tension 3)

**Reviewer says:** 11 spike requirements for a 4-spike empirical base. The ratio of formalization to evidence is high.

**My response: Accept with qualification.**

The reviewer's ratio concern is valid but incomplete. The 4 spikes produced 11 gaps and 5 failure patterns — the evidence density per spike is very high. And the user explicitly said spike programme infrastructure is blocking other projects, making this urgent not speculative. The mitigation is SPIKE-12 (the off-ramp requirement) — if formalization proves counterproductive, we simplify.

**Proposed action:** No change to scope, but add SPIKE-12 (the off-ramp) and note in REQUIREMENTS.md preamble that spike methodology requirements are monitored for compliance theater.

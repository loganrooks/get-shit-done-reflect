## Cross-Session Patterns

### Pattern 1: User Prompts That Should Have Been Agent Self-Initiating

Across all three major sessions (cb3ee1b7, 2c1aa264, bb8a9df5), there is a consistent pattern: the user prompts the agent to do something the agent should have done proactively, and the agent confirms "you're right, I should have done that." Examples:

- "have we checked the plan though?" → agent skipped plan checker
- "Shouldn't this be considered a signal?" → agent hadn't self-signaled
- "have these gone through the proper release workflow?" → agent hadn't used release workflow
- "did you look online?" → agent hadn't cross-validated with external sources
- "is there any alternative explanation?" → agent hadn't questioned test methodology

The user has a persistent meta-worry about this: "any time I have to prompt you to do something you should have, even if that's 'hey have you checked this, or considered this, or reflected on this' and you don't have a good justification why you explicitly didn't consider X, then that's a gap." This suggests a design opportunity: proactive checklists or verification triggers that fire at natural workflow boundaries.

### Pattern 2: Quality Gates Are Opt-Out Rather Than Mandatory

Two separate sessions show quality gates being bypassed: the plan checker in f1-modeling, and the release workflow in get-shit-done-reflect. Both bypasses were agent-initiated, both were caught by the user, and both did catch real issues when eventually run. The harness currently relies on agent self-discipline to run quality gates. There is no structural enforcement — nothing prevents an agent from saying "this is small, I'll skip it."

### Pattern 3: Signal Lifecycle Completion Is Universally Absent

Every session that touched signals (cb3ee1b7, eb9541ff, 2c1aa264) found the same problem: signals accumulate but don't close. The signal audit found 171/187 stuck in "active" including 7 already fixed. The integration checker mislabeled severities. The audit-milestone workflow found tech debt that was mislabeled as "low." There is no automated mechanism that checks "does this signal still describe the current state of the code?" This is a systemic gap, not a single instance.

### Pattern 4: Headless Delegation Is Unreliable in the Current Environment

Multiple attempts to delegate work to headless Claude sessions in cb3ee1b7 failed: git index.lock conflicts, silent exits, stdout capture issues. The agent had to develop workarounds (isolated worktrees, `-p` flag, separate working directory) through trial and error. This suggests headless Claude is not a reliable primitive in the current setup, but there is no documented guidance on how to use it correctly.

### Pattern 5: Philosophical Depth of User vs Pragmatic Response of Harness

The pre-milestone deliberation session (cb3ee1b7) shows the user operating at a sophisticated philosophical level (Levinas, Popper, Stiegler, complexity theory) while the harness and agent operate at a practical engineering level. The user repeatedly points out when the agent makes epistemically overconfident claims or when proposed solutions are not adequately grounded. This tension is productive — the user's philosophical orientation consistently improves the rigor of agent proposals — but it also reveals that the harness has no designed capacity to engage at this level. The deliberation system exists but has no workflow integration, no evaluation of predictions against outcomes, and no operational bridge from philosophical to engineering decisions.

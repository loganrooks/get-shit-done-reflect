  <name>Task 1: Add phase correlation command and phase time window parser</name>
  <files>get-shit-done/bin/lib/telemetry.cjs</files>
  <action>
Add the following to the Helpers section of telemetry.cjs:

**`parsePhaseTimeWindows(cwd)`** -- reads `.planning/STATE.md` from `cwd` (using `resolveWorktreeRoot(cwd)` to find the planning directory). Parses the Performance Metrics section to extract phase plan entries. Each entry in STATE.md looks like:
```
- NN-PP: Xmin, N tasks, N files
```
These entries do not contain timestamps directly. The time windows need to be derived from session data: find sessions whose `project_path` matches the cwd project and whose `start_time` falls during the phase's active period.

Alternative approach (more robust): Parse STATE.md `last_activity` entries and phase completion dates from ROADMAP.md Progress table. Each phase has a "Completed" date in the ROADMAP table. The time window for a phase is: (completion date of previous phase, completion date of this phase].

Implementation:
1. Read ROADMAP.md and parse the Progress table for `| NN. Name | X/Y | Status | Date |` rows
2. Extract phase number and completion date for each completed phase
3. For the requested phase number, define the time window as: after previous phase's completion date through this phase's completion date
4. If the phase is not yet completed, use "now" as the end bound
5. Return `{ phase, start: Date, end: Date }` or null if phase not found

**`cmdTelemetryPhase(cwd, phaseNum, raw, homeDir)`** -- uses `parsePhaseTimeWindows(cwd)` to get the time window for the requested phase, then loads session-meta filtered to the project path and the derived date range. Computes the same summary statistics as `cmdTelemetrySummary`. Also loads facets and reports facets coverage for the phase window.

Human-readable output: Header with phase number and date range, then same table format as summary. `--raw` returns JSON with phase metadata plus session data.

If phaseNum is not provided, call `error('Usage: gsd-tools telemetry phase <phase-num>')`.

**Key constraints:**
- Use `resolveWorktreeRoot(cwd)` to find .planning/ directory.
- The phase number match should handle both "57" and "57.1" formats.
- Date range is inclusive on both ends.
  </action>

Run: `node get-shit-done/bin/gsd-tools.cjs telemetry phase 56` -- should show human-readable phase telemetry.
  </verify>

<task type="auto">
  <name>Task 2: Add baseline command with token validation and all 8 metrics</name>
  <files>get-shit-done/bin/lib/telemetry.cjs</files>
  <action>
Add the following to telemetry.cjs:

**`validateTokenCounts(homeDir, sampleSize)`** -- performs the inline token validation task (5-session comparison per CONTEXT.md decision). This is NOT a formal spike; it is a bounded binary check.

Implementation:
1. Load session-meta from `{homeDir}/.claude/usage-data/session-meta/`
2. For each session, look for a matching JSONL file in `{homeDir}/.claude/projects/`. The JSONL directory structure uses dash-encoded project paths. List all subdirectories of `{homeDir}/.claude/projects/`, then check for `{sessionId}.jsonl` in each.
3. Select up to `sampleSize` (default 5) sessions that have both session-meta and JSONL files
4. For each selected session, aggregate JSONL token counts: read the `.jsonl` file line by line, parse each JSON line, sum `message.usage.input_tokens` and `message.usage.output_tokens` from entries where `type === 'assistant'`
5. Compare session-meta tokens vs JSONL-aggregated tokens. Compute ratios.
6. Return a validation result object:
```javascript
{
  status: 'completed',
  sessions_compared: N,
  results: [ { session_id, meta_input, meta_output, jsonl_input, jsonl_output, input_ratio, output_ratio } ],
  finding: '<derived from comparison>',
  recommendation: '<derived from comparison>'
}
```
7. The finding should describe whether session-meta tokens are reliable or not. Based on RESEARCH.md: input_tokens are post-cache residuals (unreliable), output_tokens are closer (within ~1.4x median).

**`cmdTelemetryBaseline(cwd, options, raw, homeDir)`** -- produces `.planning/baseline.json` with all 8 metric dimensions.

Implementation:
1. Load all session-meta with optional `--project` filter
2. Load all facets data
3. Run `validateTokenCounts()` to get token validation result
4. Compute all 8 metrics:

   a. **tokens_per_session**: `computePercentiles()` on input_tokens and output_tokens arrays. Add `_caveat` about token reliability per validation result.

   b. **token_to_commit_ratio**: For sessions with `num_commits > 0`, compute `output_tokens / num_commits` per session, then `computePercentiles()` on those values. Exclude sessions with 0 commits. Report excluded count.

   c. **tool_error_rate**: For each session, compute `tool_errors / (sum of all tool_counts values)`. If no tool usage, skip session. `computePercentiles()` on rates.

   d. **interruption_rate**: For each session, compute `user_interruptions / user_message_count`. Skip sessions with 0 user messages. `computePercentiles()` on rates.

   e. **session_outcome_distribution**: From facets-matched sessions only. Count each `outcome` value. Report as `{ value: count, ... }` with `n` and `_annotation`.

   f. **friction_frequency**: From facets-matched sessions only. Aggregate `friction_counts` across all sessions (sum values by friction type). Report with `n` and `_annotation`.

   g. **session_duration_distribution**: Compute duration in minutes from `start_time` to `end_time` (or last activity timestamp). `computePercentiles()` with `unit: "minutes"`.

   h. **agent_usage_rate**: Count sessions where `agent_usage` (or equivalent field) > 0, divide by total. Report `{ rate, sessions_with_agent, total_sessions }`.

5. Build the baseline JSON structure per RESEARCH.md schema:
```javascript
{
  generated: new Date().toISOString(),
  project_filter: options.project || null,
  session_count: { total, matched, facets_matched },
  token_validation: { ...validationResult },
  metrics: { ...all 8 metrics }
}
```

6. Write to `.planning/baseline.json` using `atomicWriteJson(path.join(resolveWorktreeRoot(cwd), '.planning', 'baseline.json'), baseline)`.
7. Output success message with session counts and file path. In `--raw` mode, output the full baseline JSON.

**Key constraints:**
- Token validation MUST run before baseline is written (per locked decision).
- baseline.json annotates which token source was used and limitations.
- Facets-derived metrics computed on facets-matched subset ONLY, with n reported separately.
- Every facets-derived field has `_annotation: "AI-generated estimate with unknown accuracy"`.
- Report matched vs filtered session counts (e.g., "42 of 268 sessions matched project filter").
- If session-meta fields are missing for a metric (e.g., no tool_counts), handle gracefully by computing on available data and reporting n.
- Some session-meta fields may vary across Claude Code versions. Use defensive property access (`session.field || 0`).
  </action>

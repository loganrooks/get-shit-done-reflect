**Bugs in scope:**
- **#2005**: `phase complete` silently skips ROADMAP updates and mis-reports `is_last_phase` when the in-progress milestone is wrapped in `<details>`
- **#1972**: `atomicWriteFileSync` not extended to milestone.cjs, phase.cjs, frontmatter.cjs — file writes in these modules are crash-unsafe
- **#1981**: worktree branch check uses `reset --soft` which fails to correct file tree — causes silent data loss on merge-back

### Patch sourcing strategy [grounded]
- All three issues are OPEN in upstream with no merged fix code. Upstream has a branch `fix/2005-phase-complete-silently-skips-roadmap-up` but it contains zero fix commits beyond v1.34.2 baseline.

- Fixes should be structured so they can be offered upstream as PRs later.

### Bug #2005: details-wrapped ROADMAP corruption [grounded]
- **Root cause identified:** `stripShippedMilestones()` at `core.cjs:1103` uses `/<details>[\s\S]*?<\/details>/gi` which strips ALL `<details>` blocks — including the current milestone if an agent wraps it in `<details>`.
- **Primary path:** `extractCurrentMilestone()` at `core.cjs:1123` does positive lookup by milestone version from STATE.md — this is the correct approach but falls back to `stripShippedMilestones()` on failure.
- **Affected operations:** `replaceInCurrentMilestone()` at `core.cjs:1197` silently returns unchanged content if milestone extraction fails/strips the target section. `cmdPhaseComplete` in `phase.cjs` calls roadmap update functions that depend on correct milestone extraction.
- **Our ROADMAP.md uses `<details>` for shipped milestones** — we're directly affected.
- Fix approach: make the `<details>` parsing smarter — only strip blocks that contain `SHIPPED` milestones, or improve `extractCurrentMilestone` to handle the edge case where the current milestone is inside a `<details>` block.

### Bug #1972: atomicWriteFileSync extension [grounded]
- **Current state:** `atomicWriteFileSync` exists in `core.cjs:1535` and is used by `core.cjs` internal callers and `atomicWriteJson`. Other modules use raw `fs.writeFileSync`.
- **Write sites to convert:**
  - `milestone.cjs`: 7 `writeFileSync` calls (lines 77, 157, 164, 181, 188, 191, 195)
  - `phase.cjs`: 7 `writeFileSync` calls (lines 363, 378, 455, 479, 593, 762, 795)
  - `frontmatter.cjs`: 2 `writeFileSync` calls (lines 364, 378)
- All three modules already `require('./core.cjs')` — just need to destructure `atomicWriteFileSync` from the import and replace calls.
- **Note:** `.gitkeep` writes (phase.cjs:363, 455) are debatable — these create empty marker files, not critical data. Include for consistency but low-priority if it complicates the diff.

### Bug #1981: worktree reset --soft data loss [open]
- **Issue:** The `worktree_branch_check` mitigation detects when `EnterWorktree` creates a branch from `main` instead of the feature branch, but corrects it with `git reset --soft` which only moves the commit pointer — the working tree still contains `main`'s files.
- **Consequence:** Executor works on `main`'s file tree, commits those files, and on merge-back the feature branch is silently overwritten.
- **Open question:** The fix could be `reset --hard` (destructive — loses any worktree-local changes), `checkout HEAD -- .` (restores working tree from HEAD), or restructuring EnterWorktree to create the branch correctly in the first place. Researcher should investigate the `EnterWorktree` flow and `worktree_branch_check` implementation to determine the safest fix.

### Test strategy [grounded]
- Each bug fix should include at least one regression test demonstrating the bug is fixed.
- #2005: Test that `replaceInCurrentMilestone` works when milestone sections have `<details>`-wrapped siblings.
- #1972: Existing tests for atomic writes in `gsd-tools.test.js` cover `atomicWriteFileSync` core behavior; new tests should verify the modules call the atomic version.
- #1981: Test requires mocking git worktree operations — assess feasibility during research.

### Commit strategy [grounded]
- One commit per bug fix for traceability: `fix(roadmap): #2005 ...`, `fix(core): #1972 ...`, `fix(worktree): #1981 ...`
- Each commit should pass tests independently.
- Reference upstream issue numbers in commit messages for traceability.

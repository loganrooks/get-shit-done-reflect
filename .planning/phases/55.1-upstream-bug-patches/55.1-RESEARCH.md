**Primary recommendation:** Implement in order: #2005 (highest severity, code-level fix in core.cjs), then #1972 (mechanical, low risk), then #1981 (workflow file additions with correct `reset --hard` pattern).

## Architecture Patterns

### Bug #2005: Two Independent Sub-Bugs, One Root Cause

The ROADMAP corruption has two distinct failure modes that must both be fixed:

**Sub-bug A: `replaceInCurrentMilestone` write scoping (core.cjs:1197-1206)**

Current code:
```javascript
// Source: core.cjs:1197-1206 (local codebase)
function replaceInCurrentMilestone(content, pattern, replacement) {
  const lastDetailsClose = content.lastIndexOf('</details>');
  if (lastDetailsClose === -1) {
    return content.replace(pattern, replacement);
  }
  const offset = lastDetailsClose + '</details>'.length;
  const before = content.slice(0, offset);
  const after = content.slice(offset);
  return before + after.replace(pattern, replacement);
}
```

**Problem:** Uses `lastIndexOf('</details>')` as a proxy for "after shipped milestones." When the current milestone is itself wrapped in `<details>`, the phase checkbox lives BEFORE `lastDetailsClose`, so `after.replace()` operates on post-details tail and finds nothing. The replacement silently no-ops.

**Fix strategy:** Rewrite `replaceInCurrentMilestone` to use `extractCurrentMilestone`'s range computation instead of the `lastIndexOf` heuristic. This unifies the read and write paths so they cannot disagree. The function should:
1. Call `extractCurrentMilestone` logic to find the current milestone's start/end offsets
2. Apply the replacement only within that range
3. Return the reconstructed content

Since `extractCurrentMilestone` needs `cwd` and `replaceInCurrentMilestone` currently doesn't take it, the function signature must be extended: `replaceInCurrentMilestone(content, pattern, replacement, cwd)`. All callers in `phase.cjs` already have `cwd` available.

**Sub-bug B: `extractCurrentMilestone` positive lookup misses `<summary>` layouts (core.cjs:1150-1159)**

Current code:
```javascript
// Source: core.cjs:1150-1159 (local codebase)
const sectionPattern = new RegExp(
  `(^#{1,3}\\s+.*${escapedVersion}[^\\n]*)`,
  'mi'
);
const sectionMatch = content.match(sectionPattern);
if (!sectionMatch) return stripShippedMilestones(content);
```

**Problem:** The positive lookup only matches markdown headings (`## v1.20 ...`). The standard `/gsd-new-project` layout uses `<summary>` tags and bullet entries for milestones, not headings. When the heading pattern fails, it falls back to `stripShippedMilestones` which strips ALL `<details>` blocks including the current milestone.

**Fix strategy:** Extend the positive lookup to also match:
1. `<summary>` tags containing the version: `<summary>[^<]*v1.20[^<]*</summary>`
2. Bullet entries with the version: `- ... v1.20 ...`

When the match is a `<summary>`, use the enclosing `<details>...</details>` as the section boundary. The upstream issue #2005 proposes this exact approach.

**Our ROADMAP layout observation:** Our fork's ROADMAP.md uses `## v1.20 ...` headings for the current milestone, so the existing positive lookup DOES work for us. But shipped milestones use `<details><summary>...</summary>...</details>` on single lines. The fix is still needed for general correctness and upstream compatibility.

### Bug #1981: Worktree Branch Check -- Workflow File Patches

**Critical finding:** The `worktree_branch_check` sections do NOT exist in our fork's workflow files. They exist ONLY in upstream (v1.34.2). This means #1981 is an ADD operation, not a MODIFY operation.

**Upstream affected files (from issue #1981):**

| File | Status in fork | What to do |
|------|----------------|------------|
| `workflows/execute-phase.md` | Missing worktree_branch_check entirely | Add corrected version |
| `workflows/quick.md` | Missing worktree_branch_check entirely | Add corrected version |
| `workflows/execute-plan.md` | Missing worktree_branch_check entirely | Add corrected version (inline reference) |
| `workflows/diagnose-issues.md` | Missing worktree_branch_check entirely | Add corrected version |

**Root cause confirmed from issue:** `EnterWorktree` (a Claude Code internal mechanism, not our code) creates worktree branches from `main` instead of the current feature branch. The `worktree_branch_check` block was added as a mitigation but uses `git reset --soft` which only moves the commit pointer, leaving the working tree with `main`'s files.

**Why `--hard` is safe:** The check runs as "FIRST ACTION before any other work." At this point the agent has made zero uncommitted changes and zero work. The entire working tree is disposable. `git reset --hard` safely corrects both commit pointer AND file tree.

**Fix pattern (from issue #1981, confirmed correct):**

```bash
# Worktree was created from wrong base (e.g., main instead of feature branch)
# Safe to hard reset -- this check runs before any agent work
git reset --hard {EXPECTED_BASE}

# Verify correction succeeded
if [ "$(git rev-parse HEAD)" != "{EXPECTED_BASE}" ]; then
  echo "ERROR: Could not correct worktree base -- aborting to prevent data loss"
  exit 1
fi
```

**Additional fix for execute-phase.md:** The upstream has a broken `git rebase --onto` attempt before the `reset --soft` that should be removed entirely:
```bash
# REMOVE THIS (upstream execute-phase.md:357-359):
git rebase --onto {EXPECTED_BASE} $(git rev-parse --abbrev-ref HEAD~1 2>/dev/null || git rev-parse HEAD^) HEAD 2>/dev/null || true
git reset --soft {EXPECTED_BASE}
```

**Also fix:** All 4 files document this as "known issue on Windows" but it affects ALL platforms (confirmed macOS in upstream #1957, presumed Linux). Update documentation references accordingly.

### Anti-Patterns to Avoid
- **Silent no-op writes:** `replaceInCurrentMilestone` currently returns unchanged content without any signal when extraction fails. The fix should preserve this behavior (no error thrown) but the REPLACEMENT logic must actually reach the target content.
- **Heuristic duplication:** Don't add a THIRD heuristic for milestone scoping. Unify on `extractCurrentMilestone`'s positive lookup approach. The `replaceInCurrentMilestone` rewrite should delegate to the same logic.
- **Regex over-specificity:** The `<summary>` matching pattern must be loose enough to handle `<summary>v2.0 Bar (Phases 6-10) -- IN PROGRESS</summary>` and tight enough to not match non-milestone content.

## Common Pitfalls

### Pitfall 1: replaceInCurrentMilestone signature change breaks callers
**What goes wrong:** Adding `cwd` parameter to `replaceInCurrentMilestone` without updating all call sites.
**Why it happens:** The function is called from phase.cjs (lines 715, 744) and potentially from external callers via exports.
**How to avoid:** Make `cwd` optional with fallback to current behavior (`lastIndexOf` heuristic) when omitted. This preserves backward compatibility. All internal callers in phase.cjs should pass `cwd`.
**Warning signs:** Tests passing but real usage failing because `cwd` is undefined.

### Pitfall 2: extractCurrentMilestone fallback chain masks errors
**What goes wrong:** The function has 4 fallback paths (STATE.md version -> in-progress marker -> section heading match -> stripShippedMilestones). Each fallback silently degrades. If the positive lookup is improved but still fails on some edge case, the `stripShippedMilestones` fallback will silently corrupt again.
**Why it happens:** The fallback design prioritizes "something works" over "correct behavior."
**How to avoid:** Tests should cover ALL fallback paths, not just the happy path. Consider logging (to stderr, non-blocking) when falling back to `stripShippedMilestones` so the degradation is visible in debug output.
**Warning signs:** Tests that only verify the `<summary>` match path but don't exercise the heading match path.

### Pitfall 3: Workflow file patches drift from upstream structure
**What goes wrong:** Adding `worktree_branch_check` blocks to our fork's workflow files but placing them at the wrong location relative to the `Task()` spawn prompt, or missing one of the 4 affected files.
**Why it happens:** Our fork's workflow files have diverged from upstream in structure. The upstream's line numbers from issue #1981 don't apply directly.
**How to avoid:** Use the upstream file as reference for WHERE the block goes (inside the Task prompt, before `<execution_context>` or `<files_to_read>`), then adapt to our fork's specific structure.
**Warning signs:** The `worktree_branch_check` block appearing outside the Task() prompt string.

### Pitfall 4: atomicWriteFileSync catch block swallows real errors
**What goes wrong:** The existing `atomicWriteFileSync` catch block (core.cjs:1540-1544) catches ALL rename errors and falls back to direct write. If the real issue is a permissions error on the directory, this falls through to `writeFileSync` which also fails -- but now the error is from the fallback, not the original atomic attempt.
**Why it happens:** The catch is intentionally broad for robustness.
**How to avoid:** For the #1972 fix, this is NOT in scope (per CONTEXT.md deferred items). Just convert the calls. Don't modify the catch block.
**Warning signs:** Resisting the urge to "improve" error handling while doing the mechanical adoption.

## Code Examples

### Fix for replaceInCurrentMilestone (recommended approach)

```javascript
// Source: proposed fix based on issue #2005 analysis
function replaceInCurrentMilestone(content, pattern, replacement, cwd) {
  // When cwd is provided, use extractCurrentMilestone's range logic
  if (cwd) {
    // Get the current milestone section boundaries
    const milestoneContent = extractCurrentMilestone(content, cwd);
    if (milestoneContent && milestoneContent !== content) {
      // Find where the milestone content starts in the original
      // (extractCurrentMilestone returns preamble + currentSection)
      // Apply replacement to the milestone portion only
      const milestoneIdx = content.indexOf(milestoneContent.slice(-200));
      // ... range-based replacement
    }
  }

  // Fallback: original heuristic (backward compat when cwd not provided)
  const lastDetailsClose = content.lastIndexOf('</details>');
  if (lastDetailsClose === -1) {
    return content.replace(pattern, replacement);
  }
  const offset = lastDetailsClose + '</details>'.length;
  const before = content.slice(0, offset);
  const after = content.slice(offset);
  return before + after.replace(pattern, replacement);
}
```

**Note:** The exact implementation needs careful range tracking. The pseudocode above shows the approach. A cleaner implementation would refactor `extractCurrentMilestone` to return `{ start, end, content }` and have `replaceInCurrentMilestone` use the same offsets. This is the planner's discretion to structure.

## Open Questions

### Resolved

- **What is the EnterWorktree flow?** EnterWorktree is a Claude Code internal mechanism (not GSD code) that creates worktree branches. The `worktree_branch_check` is a PROMPT-LEVEL mitigation in workflow markdown files, not a code-level function. It instructs executor agents to verify their branch base before starting work. The worktree_branch_check sections are entirely absent from our fork -- they exist only in upstream v1.34.2 workflow files.

- **Should `stripShippedMilestones` be deprecated?** No -- it should remain as the last-resort fallback in `extractCurrentMilestone`. But `replaceInCurrentMilestone` should stop using its own independent heuristic and delegate to `extractCurrentMilestone` instead.

- **Does `atomicWriteFileSync` error handling need improvement?** Not in scope for this phase (CONTEXT.md deferred). The catch block's fallback-to-direct-write behavior is acceptable for now.

- **What about `.gitkeep` writes?** Recommend converting them too for consistency. The overhead is negligible and eliminates reasoning about which writes are "safe enough."

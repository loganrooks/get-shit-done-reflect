**Current state:** `write_context` step (discuss-phase.md lines 886-1002) produces `<domain>`, `<decisions>`, `<canonical_refs>`, `<code_context>`, `<specifics>`, `<deferred>`. - [grounded] Rename `<decisions>` / "Implementation Decisions" to a framing that does not bias toward closure — deliberation concluded this is a root cause of exploration suppression. The quality regression audit found 0 guardrails, 0 working assumptions, 0 constraints across Phases 55-57 vs 4-9 guardrails in Phases 52-54
- [grounded] Add mandatory exploratory-mode sections: Working Assumptions, Derived Constraints, Epistemic Guardrails, Structured Open Questions — these are the sections the context_model already names but that have no template enforcement
- [grounded] Open Questions section should use generative format (DISC-07): research program, downstream decisions affected, reversibility — not binary yes/no. Deliberation audit of rich-era questions (Phase 52-54) showed they specify methodology; thin-era questions (55-57) ask yes/no
- [working assumption] The new section name replacing "Implementation Decisions" should signal research-guiding intent rather than decision-locking — options include "Working Model & Assumptions", "Phase Steering", "Research-Guiding Context". Researcher should examine what framing produces the right downstream behavior in the researcher and planner agents
- [working assumption] Whether exploratory and discuss modes use structurally different templates (separate XML blocks per mode) or the same template with mode-conditional sections. Currently there's one template — the question is whether the added sections appear in both modes or only exploratory

### Typed claim system (DISC-02, DISC-10)

**Current state:** Claims in CONTEXT.md use bare `[grounded]` / `[open]` / `[working assumption]` inline markers (see 55.2-CONTEXT.md as exemplar). No formal type vocabulary. The `[grounded]` marker does double duty as epistemic status AND `--auto` progression gate — under `--auto`, the agent is incentivized to mark everything `[grounded]` to allow forward motion (Goodhart-like perverse incentive identified in quality regression deliberation).

**7 claim types** (from claim-type-ontology deliberation, concluded 2026-04-09, 6 audits across 12 projects):

| Type | Meaning | Researcher action | Justificatory demand |
|------|---------|-------------------|---------------------|
| evidenced | Traceable to specific artifact/measurement/ADR with citation | Verify citation is current; build on it | Must cite artifact |
| decided | Explicitly chosen through deliberation with rationale | Honor it; research implementation, not alternatives | Must record alternatives considered |
| assumed | Working assumption, not yet validated | Investigate, test, potentially revise — primary research target | Must specify challenge protocol |
| open | Genuinely unresolved named uncertainty | Research, propose options, surface tradeoffs | Must define research program |
| projected | Current design justified by future needs that don't yet exist | Check whether projected need is real; flag if contradicted | Must name the future phase/need |
| stipulated | Threshold or criterion chosen, not derived from evidence | Note as choice, look for calibration evidence | Must acknowledge as choice |
| governing | Normative/philosophical/process commitment, framework-level | Respect as framework; do not investigate as empirical | Must distinguish from empirical |

**3-level verification dimension** (cuts across all types):

| Level | Meaning |
|-------|---------|
| cited | Names specific file, line, measurement command, or artifact |
| reasoned | Has stated rationale but no empirical citation |
| bare | No verification path recorded |

- [grounded] 7 claim types adopted as-is from concluded ontology deliberation — this was derived from 6 independent audits across 12 projects and represents convergence of ~40 named groupings
- [grounded] 3-level verification dimension adopted as-is — orthogonal to types, allowing claims to carry independent epistemic confidence
- [grounded] Shared reference doc `references/claim-types.md` to be created — defines types, assignment criteria, researcher actions, justificatory expectations. Agent specs reference this doc rather than embedding the ontology inline
- [grounded] Decouple epistemic status from `--auto` progression eligibility — the `[grounded]` marker currently serves both purposes, creating perverse incentive. Type + verification should be pure epistemic labels; auto-progression should be gated on a separate mechanism
- [grounded] Claim dependency section in CONTEXT.md (DISC-10) records which claims depend on which other claims. A decided claim resting on an assumed claim is flagged as a vulnerability. The context-checker traces these dependency chains
- [working assumption] Claim notation syntax: `[decided:cited]` inline markers vs structured blocks vs YAML-like syntax. The current inline marker approach (`[grounded]`) works for simple binary but may not scale to type+verification. Researcher should evaluate readability and agent-parseability tradeoffs
- [working assumption] Whether claim dependency notation uses directed graph syntax (`A → B`), indented lists, or a dedicated table. The deliberation specified the concept (inferential webs, Brandom) but not the format. Researcher should propose a notation that is both human-readable and context-checker-parseable

### Context-checker agent design (DISC-04)

**Current state:** No context-checker agent exists. No file matching `context-checker` found anywhere in agents/ or .claude/agents/. The DISC-04 requirement specifies: runs after discuss-phase write_context, before plan-phase; verifies type assignments, surfaces untyped claims, checks citations, traces dependency chains, fixes CONTEXT.md in-place, appends verification log to DISCUSSION-LOG.md.

- [grounded] The agent runs post-discuss, pre-plan — this is the pipeline position specified by DISC-04 and consistent with the quality regression root cause (validation-poor: no mechanism to challenge claims before they reach the planner)
- [grounded] Two jobs: (1) verify typed claims (type assignment, citation integrity, justification meeting type-specific demand), (2) surface untyped claims doing significant epistemic work and propose types. The audits identified "invisible load-bearing assumptions" as the #1 blind spot
- [grounded] Output: fixes CONTEXT.md in-place (adding types to untyped claims, flagging phantom citations), appends verification log to DISCUSSION-LOG.md
- [grounded] Agent file: `agents/gsdr-context-checker.md` — new file to create
- [working assumption] Whether the context-checker blocks `--chain` progression on failures, or allows progression with warnings. If it blocks, `--auto --chain` could stall on a trivially fixable issue. If it allows, the planner may receive unchecked claims. Researcher should propose a severity-tiered approach (e.g., block on unresolvable citations, warn on bare verification)
- [working assumption] Whether the context-checker should be an inline step in discuss-phase.md (like scout_codebase) or a separately invokable agent (like the researcher). DISC-04 says "agent" but integration point affects whether `--chain` can orchestrate it automatically
- [working assumption] Model assignment for the context-checker agent. Citation verification and dependency tracing are structured analytical tasks, not creative synthesis — likely appropriate for Sonnet rather than Opus. Researcher should consider model requirements

### DISCUSSION-LOG.md justificatory sidecar (DISC-03, DISC-08)

**Current state:** DISCUSSION-LOG.md is generated in the `git_commit` step (discuss-phase.md lines 1047-1095). Format: header with date/phase/mode, per-area option tables with "Selected" column, user's choice, notes. Marked "Audit trail only — do not use as input to planning, research, or execution agents." The 57.1-DISCUSSION-LOG.md confirms this format is live.

- [grounded] DISCUSSION-LOG.md repurposed from audit-trail-only to justificatory sidecar consumed by the context-checker — this is the DISC-08 requirement. Avoids creating a new document while giving provenance a home outside CONTEXT.md
- [grounded] Per-claim justification meeting type-specific demands: evidenced claims cite artifacts, decided claims record alternatives considered, assumed claims specify challenge protocols, stipulated claims acknowledge the choice
- [grounded] Claim dependency chains recorded in DISCUSSION-LOG.md — which claims support which, forming the inferential web that the context-checker traces
- [grounded] Context-checker verification log appended to DISCUSSION-LOG.md — after the checker runs, it records what it found, what it fixed, and what remains flagged
- [working assumption] Whether the entire DISCUSSION-LOG.md format changes (replacing the current Q&A tables) or the justificatory content is appended as new sections after the existing Q&A audit trail. The audit trail has value for human review — researcher should determine if it can coexist with the sidecar content
- [working assumption] Level of justification granularity per claim. Full natural-language justification paragraphs vs structured key-value pairs (e.g., `alternatives_considered: [list]` for decided claims). Needs to balance thoroughness with agent-parseability

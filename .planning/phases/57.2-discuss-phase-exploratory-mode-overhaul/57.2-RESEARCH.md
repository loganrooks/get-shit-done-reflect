## Summary

This phase modifies five markdown-based agent specification files (no runtime code) to overhaul how the discuss-phase workflow produces CONTEXT.md in exploratory mode. The core problem is well-documented: the write_context template (discuss-phase.md lines 886-1002) lacks structural sections for assumptions, constraints, guardrails, and open questions, so exploratory mode produces the same shallow locked-down output as discuss mode. The quality regression is quantified across 32 CONTEXT.md files -- peak-era output (Phases 52-54) had 6-9 sections with typed assumptions and generative questions; current-era output (Phases 55-57) has 4-5 sections with everything locked as `[grounded]`.

The solution involves three coordinated changes: (1) restructure the CONTEXT.md template with mandatory exploratory sections and a 7-type claim vocabulary replacing the bare `[grounded]/[open]` binary, (2) create a new context-checker agent that verifies claim integrity and surfaces invisible load-bearing assumptions, and (3) update the researcher agent to consume the richer typed-claim vocabulary with distinct actions per type. The claim type ontology is grounded in convergent evidence from 6 independent audits across 12 projects and ~85 CONTEXT.md files -- this is not speculative design but empirically derived categorization.

All modifications target markdown workflow specifications and agent specs -- no JavaScript, no gsd-tools.cjs changes. The upstream cherry-pick (commit 5e88db95, --chain flag) and CONTEXT.md commit fix are small, independent wiring fixes that can be done in any plan order.

**Primary recommendation:** Structure plans around file boundaries: one plan for the reference doc + template restructuring (the foundation), one for the context-checker agent + DISCUSSION-LOG sidecar format, one for the researcher update + upstream cherry-pick. The reference doc must land first since both the template and the context-checker reference it.

### Pattern 1: Claim Notation Syntax -- Use `[type:verification]` Inline Markers

**Recommendation: `[type:verification]` inline markers.**

**What:** Each claim carries its type and verification level as an inline marker, e.g., `[decided:cited]`, `[assumed:bare]`, `[governing:reasoned]`. When verification is omitted, it defaults to `bare`: `[assumed]` = `[assumed:bare]`.

**Why this over alternatives:**

| Format | Readability | Agent parseability | Scales to 7 types | Verdict |
|--------|------------|-------------------|-------------------|---------|
| `[decided:cited]` inline | HIGH -- natural English flow, same position as current `[grounded]` | HIGH -- regex `\[(evidenced\|decided\|assumed\|open\|projected\|stipulated\|governing)(:(cited\|reasoned\|bare))?\]` | YES -- compact, no noise at 7 types | **RECOMMENDED** |
| Structured YAML blocks | LOW -- breaks markdown flow, verbose | HIGH -- standard parsing | Scales but adds ~5 lines per claim | Over-engineered for inline claims |
| Table per claim | MEDIUM -- structured but pulls claims out of narrative context | HIGH | Scales but forces tabular layout | Loses narrative context |
| `{type=decided, verification=cited}` | LOW -- verbose, code-like | HIGH | Scales but noisy | Too syntactically heavy |

**Evidence supporting this choice:** The current `[grounded]` and `[working assumption]` inline markers work well for readability (55.2-CONTEXT.md, 57.2-CONTEXT.md demonstrate this). Extending the same pattern to `[type:verification]` preserves the established reading experience while adding the type dimension. The peak-era CONTEXT.md files (52-54) used bold-prefixed assumptions (`**A1:**`, `**G1:**`) -- the new markers can coexist: `**A1:** [assumed:reasoned] The bridge file format is stable.`

**Syntax rules:**
- Full form: `[decided:cited]` -- type and verification explicit
- Short form: `[assumed]` -- type only, verification defaults to `bare`
- Split claims (one proposition, two statuses): `[evidenced/assumed]` -- "mechanism is evidenced, sufficiency is assumed"
- Placement: at the start of the claim line, immediately after any label prefix (e.g., `- [decided:cited] Use card layout`)

**Regex for agent parsing:**
```
\[(evidenced|decided|assumed|open|projected|stipulated|governing)(?:\/(evidenced|decided|assumed|open|projected|stipulated|governing))?(?::(cited|reasoned|bare))?\]
```

### Pattern 2: Claim Dependency Notation -- Use Labeled Table

**Recommendation: Dedicated dependency table at the end of the Working Model section.**

**What:** A table recording which claims depend on which, with the dependent claim's vulnerability highlighted.

```markdown
## Claim Dependencies

| Claim | Depends On | Vulnerability |
|-------|-----------|---------------|
| [decided] Use wrapper pattern for error handling | [evidenced:cited] 22 exception handlers in src/ | Low -- if evidence changes, decision basis changes |
| [decided] Exclude Windows support | [assumed:bare] Users are Mac/Linux developers | HIGH -- if assumption is wrong, scope decision is undermined |
| [stipulated] 3 retry threshold | [assumed:reasoned] Most transient failures resolve within 3 | MEDIUM -- threshold is only reasonable under this assumption |
```

**Why table over alternatives:**

| Format | Readability | Parseability | Expresses directionality | Verdict |
|--------|------------|-------------|-------------------------|---------|
| Table with Claim/Depends On/Vulnerability | HIGH -- scannable, clear | HIGH -- row-level parsing | YES -- left column depends on right | **RECOMMENDED** |
| Directed graph `A -> B` notation | LOW -- requires mental graph parsing | MEDIUM -- needs custom parser | YES | Over-complex for typically 5-15 dependencies |
| Indented lists | MEDIUM -- readable | LOW -- indentation-sensitive parsing is fragile | Implicit via nesting | Fragile |

**Evidence:** The claim-type-ontology deliberation (lines 148-161) specifies the concept of inferential webs (Brandom) but not the notation. The table format aligns with existing CONTEXT.md conventions -- the 57.2-CONTEXT.md already uses tables for the Open Questions section. The context-checker can parse this table to trace dependency chains: a `[decided]` claim depending on an `[assumed:bare]` claim is a vulnerability the checker should flag.

### Pattern 3: Exploratory-Mode Template -- Use Mode-Conditional Sections in One Template

**Recommendation: Single template with mode-conditional sections, not separate templates.**

**What:** The write_context step uses one template where certain sections are mode-conditional. In exploratory mode, the template produces `<working_model>` (with typed claims), `<constraints>`, `<guardrails>`, `<questions>`, and `<dependencies>`. In discuss mode, it produces `<decisions>` (current behavior, unchanged).

**Why single template:**
- The current architecture has one template. Splitting introduces maintenance burden (two templates that must stay synchronized on shared sections like `<domain>`, `<canonical_refs>`, `<deferred>`).
- The mode routing step (lines 239-250) already sets behavioral flags. Extending this to set template-variant flags is the natural pattern.
- Peak-era CONTEXT.md files (52-54) used the single template -- the rich sections were produced by the agent's behavior, not by a separate template. Structural enforcement means adding the sections to the existing template with mode guards, not creating a parallel template.

### Pattern 4: Section Name for Exploratory Mode -- Use "Working Model & Assumptions"

**Recommendation: "Working Model & Assumptions"**

**Evidence:** All three peak-era CONTEXT.md files (52, 53, 54) already use `## Working Model & Assumptions` as the section name for their assumptions section. This is the established, proven framing that produced the highest-quality output. The XML tag should be `<working_model>` rather than `<decisions>`.

"Phase Steering" and "Research-Guiding Context" are more abstract and less actionable. "Working Model & Assumptions" directly communicates that claims are working hypotheses, not settled decisions.

### Pattern 5: Auto-Progression Gate Replacement

**Recommendation: Type-based rules with verification floor.**

**What:** Decouple `--auto` progression eligibility from epistemic type entirely. Instead, use a new `[auto-ready]` marker that the discuss-phase agent assigns based on separate criteria:

| Claim Type | Auto-Progression Rule |
|-----------|----------------------|
| evidenced | Auto-ready if verification >= `cited` |
| decided | Auto-ready (decisions are by definition resolved) |
| assumed | Auto-ready ONLY with explicit rationale (verification >= `reasoned`) |
| open | NEVER auto-ready (open questions require research) |
| projected | Auto-ready if the projected phase exists in ROADMAP.md |
| stipulated | Auto-ready (choices are by definition resolved) |
| governing | Auto-ready (framework commitments don't need investigation) |

**Implementation:** The `--auto` progression logic in mode_routing and the auto-select behavior in discuss_areas should check the claim TYPE, not a `[grounded]` marker. In `--auto` mode, the agent:
1. Types each claim honestly (using the ontology)
2. Auto-selects based on type rules above
3. Claims that cannot auto-progress become `[open]` entries in the Open Questions section

This eliminates the perverse incentive: the agent can type a claim `[assumed]` without penalty, because `assumed:reasoned` is auto-ready. The incentive shifts from "mark everything grounded" to "provide honest rationale for assumptions."

**Critical insight from the quality regression deliberation:** The Goodhart problem was that `[grounded]` served as both epistemic label AND automation control token. By making auto-progression depend on type + verification level instead of a single label, the epistemic labels become honest again.

### Anti-Patterns to Avoid

- **Modifying `.claude/` directly:** Per CLAUDE.md rules, always edit `agents/`, `get-shit-done/`, `commands/` source directories. Run `node bin/install.js --local` after to sync to `.claude/`.
- **Breaking discuss mode:** The template changes are for exploratory mode. Discuss mode's `<decisions>` template must remain unchanged and functional. Plans must verify both modes work.
- **Embedding the full ontology in agent specs:** Agent specs should reference `references/claim-types.md`, not duplicate the 7-type table inline. This prevents drift between the reference doc and each consumer.
- **Creating a blocking context-checker that stalls `--chain`:** The checker should use severity tiers, not a binary block/pass gate. See context-checker design below.

## Common Pitfalls

### Pitfall 1: Silent Feature Drop (Precedent: sig-2026-04-03)

**What goes wrong:** Discuss_mode routing was silently dropped once before (sig-2026-04-03-discuss-mode-adoption-gap-silent-feature-drop). Template changes get overwritten or ignored by future upstream syncs.
**Why it happens:** The discuss-phase.md is a large file (1095 lines). Changes to internal sections can be lost in wholesale upstream merges.
**How to avoid:** Verification must check feature completeness, not just file existence. Specifically: grep for the new XML tags (`<working_model>`, `<constraints>`, `<guardrails>`, `<questions>`, `<dependencies>`) in the installed file after every upstream sync.
**Warning signs:** CONTEXT.md output missing Working Model section; all claims using old `[grounded]` markers.

### Pitfall 2: Phantom Citation Regression

**What goes wrong:** Claims marked `[evidenced:cited]` with citations that don't resolve (e.g., "Pitfall C3" in Phase 57 CONTEXT.md that was not traceable to any document).
**Why it happens:** The `cited` verification level creates an obligation to provide a real citation. Without the context-checker verifying citations, phantom citations accumulate.
**How to avoid:** The context-checker's first job is citation verification. Every `cited` claim should have its citation resolved to an actual file/line/artifact.
**Warning signs:** Cited claims with vague references ("as discussed", "per the spec") rather than specific paths.

### Pitfall 3: Template Sections Filled with Boilerplate

**What goes wrong:** The quality regression deliberation warns (Rebuttal to Option A): "Template sections can be filled with boilerplate. An Epistemic Landscape section can be filled with generic text that doesn't actually do epistemic work."
**Why it happens:** Structural sections create an obligation to fill them, but don't guarantee quality. The agent may produce "Working Model: this phase follows standard patterns" -- technically filling the section but doing no epistemic work.
**How to avoid:** (1) The template should include per-section guidance with minimum expectations (e.g., "Each assumption must specify what evidence would change it"). (2) The context-checker should flag sections that are under-specified (fewer than N typed claims, no falsification criteria on assumptions). (3) Prediction P4 from the deliberation explicitly tests this: if template alone doesn't match Phase-52 quality, the exploration subagent becomes urgent.
**Warning signs:** All claims in a section share the same type; fewer than 3 typed claims per section; assumptions without challenge protocols.

### Pitfall 4: Cherry-Pick Conflict with Fork Customizations

**What goes wrong:** Upstream commit 5e88db95 modifies discuss-phase.md (17 insertions, 9 deletions). The fork has its own modifications to discuss-phase.md. Cherry-picking may produce merge conflicts.
**Why it happens:** Both fork and upstream modify the same file, potentially the same regions (mode routing, auto_advance step).
**How to avoid:** Cherry-pick the upstream commit BEFORE making template changes. Apply the --chain flag changes to the baseline, then layer template modifications on top. The upstream diff is small (22 net lines) and touches auto_advance (lines 1025-1095) and success_criteria, which are mostly below the write_context changes.
**Warning signs:** Merge conflict markers in discuss-phase.md after cherry-pick.

### Pitfall 5: Context-Checker as Compliance Theater

**What goes wrong:** The checker runs but becomes a checkbox -- it reports "all clear" on obviously thin CONTEXT.md files because its checks are syntactic (types present) rather than semantic (types correct, justification adequate).
**Why it happens:** Type verification is easier than semantic verification. Checking "does every claim have a `[type]` marker" is trivial. Checking "is this claim correctly typed as `decided` when it should be `assumed`" requires judgment.
**How to avoid:** The checker's second job (surfacing untyped claims) is more valuable than the first (verifying typed claims). The spec should prioritize invisible assumption detection over type syntax checking. The checker should ask: "What significant claims are doing epistemic work WITHOUT type markers?" -- this is the intervention the audits identified as most needed.
**Warning signs:** Checker reports zero untyped claims (contradicts audit finding that invisible assumptions are the #1 blind spot -- prediction P3).

### Pitfall 6: Researcher Ignores Types

**What goes wrong:** After updating the researcher spec, the researcher treats all types identically -- investigating everything the same way regardless of type.
**Why it happens:** The 3-bucket model (Decisions=locked, Discretion=flexible, Deferred=ignore) is simpler than the 7-type model. Agents may fall back to the simpler pattern if the type-to-action mapping is not clear enough.
**How to avoid:** The researcher spec must include the full type-to-action table prominently (not buried in a reference doc footnote). Each type must have a concrete behavioral difference. Prediction P2 from the deliberation tests this.
**Warning signs:** RESEARCH.md investigates `decided` claims (should honor them) or ignores `assumed` claims (should be primary research targets).

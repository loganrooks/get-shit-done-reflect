Create `agents/gsdr-context-checker.md` (~200-300 lines). Follow the established agent spec pattern from `agents/gsd-verifier.md` and `agents/gsd-plan-checker.md`: YAML frontmatter + XML sections + numbered steps.

**Frontmatter:**

name: gsdr-context-checker
description: Verifies claim integrity in CONTEXT.md before planning. Checks typed claims, surfaces untyped load-bearing assumptions, traces dependency chains.
tools: Read, Bash, Grep, Glob
model: sonnet
color: yellow

Model is `sonnet` per MEMORY.md convention (sensors/verifiers use Sonnet, not Opus) and RESEARCH.md confirmation that citation verification and dependency tracing are structured analytical tasks.

**`<role>` section:**
State two jobs clearly:
1. **Verify typed claims:** Check type assignments against criteria in `references/claim-types.md`. Verify `cited` citations resolve to actual files/lines. Check justification in DISCUSSION-LOG.md meets type-specific demands per the reference doc.
2. **Surface untyped claims:** Find claims doing significant epistemic work without type markers. Propose types. This is the HIGHER-VALUE job -- invisible load-bearing assumptions are the #1 blind spot (from 6 audits across 12 projects).

Critical mindset: "You are not a rubber stamp. The discuss-phase agent typed its own claims -- you are the external check. If everything looks perfect, be suspicious."

Spawned by: discuss-phase workflow (check_context step) or standalone invocation.

**`<upstream_input>` section:**

List three inputs:
- **CONTEXT.md** -- The document to verify. Path provided at invocation: `${phase_dir}/${padded_phase}-CONTEXT.md`
- **@get-shit-done/references/claim-types.md** -- Reference doc for type definitions, assignment criteria, justificatory expectations
- **DISCUSSION-LOG.md** -- Justificatory sidecar with per-claim provenance. Path provided at invocation: `${phase_dir}/${padded_phase}-DISCUSSION-LOG.md`

**`<severity_tiers>` section:**
Define three severity levels:

FAIL -- Claim integrity broken; cannot be trusted downstream. Examples: phantom citation (evidenced:cited but file/line does not exist), circular dependency chain, evidenced claim contradicts cited artifact. Effect: blocks --chain/--auto progression.

WARN -- Claim integrity weak but not broken; attention needed. Examples: bare verification on assumed claims that could be cited, thin justification (decided claim with no alternatives), dependency chain has assumed:bare as foundation. Effect: logged, does NOT block progression.

INFO -- Suggestions for improvement; no integrity issue. Examples: claim could be upgraded from bare to reasoned, untyped claim doing minor epistemic work, style suggestions. Effect: advisory only.

**`<verification_steps>` section:**
Numbered steps (following verifier/plan-checker pattern):

**Step 1: Load Documents**
Read CONTEXT.md and DISCUSSION-LOG.md. Parse all `[type:verification]` markers using regex from claim-types.md. Build a list of all typed claims with their location (section, line).

**Step 2: Verify Typed Claims**
For each typed claim:
a. **Type assignment check:** Does the type match the claim's content? (e.g., a claim with "we decided" should be `decided`, not `assumed`; a claim citing a file should be `evidenced`, not `assumed`)
b. **Citation integrity** (for `cited` verification level): Resolve the citation. If it names a file path, check the file exists (`test -f`). If it names a grep pattern, run it. If it names a line number, verify content at that line. Mark FAIL if citation does not resolve.
c. **Justification check:** Look up the claim in DISCUSSION-LOG.md Claim Justifications section. Check whether the justification meets the type-specific demand per claim-types.md:
   - evidenced: has citation?
   - decided: has alternatives considered?
   - assumed: has challenge protocol?
   - open: has research program?
   - projected: names future phase?
   - stipulated: acknowledges as choice?
   - governing: names source?
   Mark WARN if justification is missing or thin.

**Step 3: Surface Untyped Claims**
Scan CONTEXT.md for claims doing significant epistemic work WITHOUT type markers. Look for:
- Statements that constrain the solution space ("we follow X pattern", "the API uses Y")
- Assumptions presented as facts ("users are...", "the system can...")
- Implicit dependencies ("since Phase N established...", "because the codebase uses...")
- Thresholds or numbers without derivation ("3 retries", "50ms timeout")

For each untyped claim found: propose a type, explain why, and note the location. Severity: WARN if the claim is load-bearing (affects downstream decisions), INFO if minor.

**Step 4: Trace Dependency Chains**
If CONTEXT.md has a `<dependencies>` section, trace each chain:
- Does the supporting claim actually exist in the document?
- Is the vulnerability assessment accurate? (a decided claim on an assumed:bare foundation IS high vulnerability)
- Are there dependency chains NOT recorded? (decided claims that implicitly depend on assumed claims but the dependency is not listed)

Mark FAIL for circular dependencies. Mark WARN for unrecorded high-vulnerability chains.

**Step 5: Fix and Report**
a. **Fix CONTEXT.md in-place:** Add type markers to untyped claims (WARN-level ones). Update phantom citations if the correct value can be determined (e.g., grep count was wrong). Do NOT change types the discuss-phase agent assigned unless clearly wrong (FAIL-level).
b. **Append verification log to DISCUSSION-LOG.md** with this structure:

Header: `## Context-Checker Verification Log` with Checked date and Agent name.

Typed Claim Verification table: Claim / Type / Verification / Status / Issue columns.

Untyped Claims Surfaced table: Claim Text / Proposed Type / Location / Rationale columns.

Dependency Chain Audit table: Chain / Verdict columns.

Summary: counts of typed claims checked, pass/warn/fail breakdown, untyped claims surfaced, dependency vulnerabilities, overall severity verdict.

**`<success_criteria>` section:**
- Every `cited` claim has its citation resolved or marked FAIL
- Every claim with justification in DISCUSSION-LOG.md has been cross-checked
- At least 1 untyped claim surfaced if CONTEXT.md has >10 claims (prediction P3 from deliberation: invisible assumptions are the #1 blind spot)
- Dependency chains traced and vulnerability assessments verified
- Verification log appended to DISCUSSION-LOG.md
- Overall severity verdict returned

After creating the file, run `node bin/install.js --local` to install to `.claude/agents/`.

**TMPL-04: Reflection-to-requirement linkage**

This requires updates to BOTH the reflector agent spec and the reflect workflow (per Pitfall 6 in research -- both must be updated in tandem).

**In `agents/gsd-reflector.md`:**

1. In the `<output_format>` section, add a new `### Requirement Linkage` section between `### Spike Candidates` and `### Lessons Suggested`:

```markdown
### Requirement Linkage

Maps reflection findings to requirement IDs where applicable. Not all findings map to requirements -- only include linkages where a clear relationship exists.

| Finding | Type | Requirement ID | Relationship |
|---------|------|----------------|--------------|
| {pattern or triage cluster name} | {pattern|triage|spike} | {REQ-ID from REQUIREMENTS.md} | {how the finding relates: "validates", "contradicts", "extends", "blocks"} |

**Unmapped findings:** {list of patterns/triage proposals with no clear requirement mapping -- this is expected for novel issues}
```

2. In Step 10 (Report Results), add: "Include Requirement Linkage section mapping qualifying patterns and triage proposals to requirement IDs from `.planning/REQUIREMENTS.md`. Read REQUIREMENTS.md to get valid requirement IDs. Not all findings will map -- only link where a clear relationship exists."

3. In the `<execution_flow>` section, add a **Step 9.5: Map Findings to Requirements** between Step 9 (Semantic Drift) and Step 10 (Report Results):
   - Read `.planning/REQUIREMENTS.md` (if exists)
   - For each qualifying pattern and triage proposal, check if it relates to a requirement
   - Relationship types: "validates" (finding confirms requirement is working), "contradicts" (finding shows requirement is failing), "extends" (finding suggests requirement should be broader), "blocks" (finding prevents requirement completion)
   - Record linkages for inclusion in the report
   - If REQUIREMENTS.md does not exist, skip this step and note "No REQUIREMENTS.md found -- requirement linkage skipped"

1. In the `spawn_reflector` step's Task() prompt, add to the numbered instruction list (after item 8, before "Return the structured Reflection Report"):

   8.5. Map findings to requirement IDs (read .planning/REQUIREMENTS.md if it exists, link patterns/triage to requirement IDs where clear relationships exist)
   ```

2. In the `receive_report` step, add "Requirement linkage (findings mapped to requirement IDs)" to the bulleted list of what the report contains.

3. In the `present_results` step, add a `### Requirement Linkage` section in the display output (between Remediation Suggestions and the closing separator):

```

2. In the `<guidelines>` section at the bottom of the template file, add guidance for the Internal Tensions section:

```markdown
**Internal Tensions:**
- Only include for architecturally significant features (not wiring or infrastructure)
- A tension is a contradiction the feature introduces to the system's design principles
- Constraint mechanism documents what guardrail addresses the tension
- Residual risk documents what remains unresolved despite the constraint
- If you can't articulate a tension, the feature may not be architecturally significant enough to warrant this section
```

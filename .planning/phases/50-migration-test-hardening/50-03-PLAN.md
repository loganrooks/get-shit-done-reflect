  <action>
Add a new `describe('TST-05: crash recovery for interrupted KB migration')` block after the TST-04 block.

The migrateKB function (bin/install.js line 292) uses a sequence of fs operations: mkdirSync (create destination), cpSync (copy data), renameSync/unlinkSync (swap old to new), symlinkSync (create backward-compat symlink). Each can fail independently.

For crash simulation, temporarily replace the relevant fsSync/fs method to throw mid-operation. Use the pattern from the research (function-level mocking):

```javascript
const origMethod = fsSync.cpSync
fsSync.cpSync = (...args) => { throw new Error('Simulated disk failure') }
try { ... } finally { fsSync.cpSync = origMethod }
```

Create these tests:

1. **`tmpdirTest('cpSync failure: original KB data preserved, no partial destination')`**
   - Create old KB at `knowledge/signals/test/sig-001.md` with known content.
   - Mock `fsSync.cpSync` to throw.
   - Call `migrateKB(tmpdir, ['claude'])` (wrapped in try/catch since it may rethrow).
   - Verify: Original `knowledge/signals/test/sig-001.md` still exists with original content. No partial new KB directory was left behind (check that no `knowledge/` replacement exists that is different from original). No dangling symlinks.

2. **`tmpdirTest('mkdirSync failure: no partial state created')`**
   - Mock the specific `fsSync.mkdirSync` call for the new KB destination to throw.
   - Call `migrateKB(tmpdir, ['claude'])`.
   - Verify: Original data untouched. No new directories created.

3. **`tmpdirTest('renameSync failure: original data preserved, copy may exist but is not authoritative')`**
   - Let cpSync succeed (data copies to new location), then mock `fsSync.renameSync` to throw.
   - Call `migrateKB(tmpdir, ['claude'])`.
   - Verify: Original KB data still exists (the rename that would have removed it failed). Even if a copy exists at the new path, the original is the authority.

4. **`tmpdirTest('successful migration creates backward-compat symlink')`**
   - Create old KB with entries at legacy path structure.
   - Run `migrateKB(tmpdir, ['claude'])` WITHOUT mocking (normal execution).
   - Verify: Migration completed. If the function creates a symlink for backward compatibility, verify the symlink target is correct. This is the "happy path" baseline that the crash tests contrast against.

Important: The mocking approach must be careful about WHICH calls to mock. migrateKB may call mkdirSync/cpSync for multiple purposes. Study the actual migrateKB implementation (bin/install.js line 292) to understand the call sequence before writing mocks. If the function catches errors internally and cleans up, the tests should verify that cleanup happened. If it does NOT catch errors, the tests should verify that the thrown error does not leave partial state.

Note: If migrateKB already has internal error handling that prevents partial state, these tests verify that the error handling works. If it does NOT have internal error handling, these tests document the current behavior and at minimum verify original data is not destroyed.

<release_workflow>
**User explicitly asked: "release workflow (which I always forget how that works, do we tag the branch? the commit? using Github release?)"**

The release workflow is `gsdr:release` — the skill description is "Bump version, update changelog, tag, and create GitHub Release". It handles everything in sequence so you don't have to remember the moving parts.

**What it does (in order):**
1. **Bumps version** in `package.json` per semver. Determines bump type (patch/minor/major) from commit message prefixes since last release — `fix:` and `chore:`/`docs:` = patch bump; `feat:` = minor bump; `BREAKING CHANGE` footer = major bump.
2. **Updates CHANGELOG.md** with the new version header and the commit messages grouped by type.
3. **Tags the COMMIT** (not the branch — tags are always commit-pinned). Tag format is `reflect-v1.20.0` (fork convention with `reflect-` prefix to avoid upstream tag collision; upstream uses `v*` without the prefix).
4. **Pushes the tag** to origin.
5. **Creates a GitHub Release** from the tag, using the CHANGELOG entry as the release notes body. The release targets the tag, which targets the commit — so the release is not "the branch", it's "this specific state of main post-merge".

**Prerequisite: the PR must be MERGED first.** `gsdr:release` runs on `main`, not on the feature branch. The flow is always:
1. Merge PR (main advances)
2. `git checkout main && git pull`
3. `gsdr:release` (bumps, tags the new main HEAD, pushes tag, creates GitHub Release)

**CRITICAL per memory `feedback_patch_release_commit_prefix.md`:** For a PATCH release, the merge commit (or the commits being released) must use the `fix:` prefix. If the merge contains any `feat:` commits, semver bumps MINOR not patch. v1.19.0 got burned by this before. Check the commits in PR #45 — if there are `feat(57.4-*)` commits (Plan 04 used `feat:`), this may trigger a MINOR bump, not patch. This phase is not really a "fix" — it's a substantial new feature (audit skill) — so a MINOR bump (v1.19.x → v1.20.0) may actually be correct. Confirm with the user before releasing whether this is a patch or minor release, because the user said "patch" in the pause message but the commits contain `feat:` prefixes.

**Answer to the literal question "tag the branch or the commit":** Always the commit. Tags in git are pointers to commits, never to branches. The branch is the development vehicle; the tag + release is the permanent reference point on main.

**Answer to "using GitHub Release":** Yes — GitHub Release is the final artifact. The flow produces: commit on main → git tag → `git push origin <tag>` → GitHub Release created from tag (with CHANGELOG body). Users install via `npm install` which resolves the latest tag.
</release_workflow>

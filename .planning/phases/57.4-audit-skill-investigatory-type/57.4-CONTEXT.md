# Phase 57.4: Audit Skill & Investigatory Type - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning
**Mode:** Exploratory --auto -- preserving uncertainty, auto-progression by type rules

<domain>
## Phase Boundary

Build the `/gsdr:audit` command and `gsdr-auditor` agent that operationalize the audit infrastructure established in Phase 57.3. Add the `investigatory` audit type with ground rules I1-I4. Add a frame-reflexivity closing step to ALL audit types. Update `audit-conventions.md` and `audit-ground-rules.md`.

This phase holds two concerns together without collapsing one into the other:

1. **Operational** — the invocable audit skill that Phase 57.3 deferred. The command + agent pair, session directory creation, ground rule embedding, type-aware templates. Concrete engineering following established GSD command patterns.

2. **Epistemic** — the investigatory audit type and frame-reflexivity practice. The REVIEW.md (pre-phase-archive) argues that audit types smuggle assumptions about the form of failure, that the investigatory type should "hold diagnosis open" rather than closing on a failure form early, and that ALL audit types should include a frame-reflexivity closing step. This is not decoration — it comes from a real investigation where the existing taxonomy had no type for "what happened and how did we get here?"

Does NOT include: gsd-tools audit subcommand (deferred per trial-before-formalize), changes to existing agent implementations (gsd-verifier, gsd-integration-checker, etc.), workflow rewrites for audit-milestone or validate-phase, the adaptive harness design from backlog 999.x.

</domain>

## Working Model & Assumptions

### Command architecture (operational)

**Current state:** No `/gsdr:audit` command exists. The audit-milestone command (`commands/gsd/audit-milestone.md`) is the closest pattern — "this command IS the orchestrator." Other reference patterns: `deliberate.md` (context inference, mode detection, free-text arguments), `signal.md` (lightweight KB integration), `validate-phase.md` (phase-targeting).

- [decided:reasoned] Two-file architecture: `commands/gsd/audit.md` (orchestrator) + `agents/gsdr-auditor.md` (executor). Every GSD command dispatches to an agent — audit-milestone → gsd-integration-checker, validate-phase → gsdr-nyquist-auditor, execute-phase → gsdr-executor. The pattern is universal.
- [decided:reasoned] Orchestrator writes full task spec, agent executes — spike-runner pattern. The command does discovery (reads conversation context, infers type/scope), creates session directory, selects ground rules, writes task spec, dispatches agent. The agent receives a fully-formed spec and runs the audit. Task spec preservation is the strongest empirical finding from 57.3 research.
- [decided:reasoned] Type inference from context, not forced selection. The command reads conversation history for context clues and proposes an audit type. If context is sufficient, proceeds directly. If ambiguous, 0-2 clarifying questions. User typically invokes `/gsdr:audit` mid-situation ("something went wrong with Phase 57") — the skill should be able to infer scope and type from that context.
- [decided:reasoned] Session directory follows audit-conventions.md: `.planning/audits/YYYY-MM-DD-{slug}/` with task spec + output files, YAML frontmatter per Section 2 schema, date-first naming convention.
- [decided:reasoned] Ground rules auto-selected by type family, copied into task spec (not referenced by path). Per audit-ground-rules.md Section 4: "the agent needs the rules in its context window, not a pointer to them."
- [assumed:reasoned] The `--auto` flag controls interactivity consistent with other GSD commands. Structural audits can run fully autonomously (checklist-like). Epistemic and investigatory audits may benefit from conversational mode (findings presented incrementally). The flag and audit type together determine the execution mode.
- [assumed:reasoned] Model escalation is an orchestrator decision. The command can decide to request a higher model tier based on audit type/complexity. Design for platform-agnostic "levels" (Claude Code + Codex) rather than specific model names. The model-profiles.cjs entry provides the default; the orchestrator can override.

### The investigatory type (epistemic)

**Current state:** 9 audit types in 3 families (structural, epistemic, compliance) + exploratory escape hatch. No type for "something went wrong and I want to understand what happened before deciding what to fix." The Phase 57 investigation revealed this gap — the existing taxonomy had no type for cross-artifact investigation of discrepancies.

- [decided:reasoned] Add `investigatory` as a named type in audit-conventions.md. The enum-with-escape-hatch pattern (9 named + exploratory) established in 57.3 extends naturally to 10 named + exploratory. This provides type inference, ground rule selection, and body template without removing the escape hatch.
- [decided:reasoned] Four investigatory-specific ground rules (I1-I4) from REVIEW.md, added as Section 3.4 in audit-ground-rules.md:
  - **I1: Start from the situation, not from a theory.** Describe what was expected, what was delivered, and why those expectations are treated as the standard.
  - **I2: Let the investigation guide artifact selection.** Don't mandate which artifacts to read in advance. Follow the evidence.
  - **I3: Present competing explanations.** For each finding, offer at least two interpretations. Don't collapse to one.
  - **I4: Name the position of the investigation.** What this way of looking was oriented toward; what it might not have been oriented toward.
- [decided:reasoned] Investigatory body template from REVIEW.md Part 3: The Situation → What Was Examined → What Was Found → Competing Explanations → What Distinguishes Them → What Remains Unknown → Position of This Investigation. This template starts from discrepancy, not from theory.
- [open] Whether `investigatory` is best understood as a type (10th member alongside others), a family (4th alongside structural/epistemic/compliance), or a mode (modifier applicable to any type — "run this phase_verification in investigatory mode"). REVIEW.md Part 5 says this should be resolved by use, not by theory. For v1 implementation: add as a named type. Whether it evolves into a mode or family depends on how it gets used.
- [assumed:reasoned] The distinction between `investigatory` and `exploratory` is one of occasion, not ontology: investigatory is occasioned by breakdown (something went wrong, expectations were violated), exploratory by possibility (a question beckons, a space opens for thinking). Both are hermeneutically circular. The body templates differ because they start from different places (discrepancy vs. question), but both should tolerate the investigation leading somewhere the starting orientation didn't anticipate.
- [assumed:reasoned] Type inference for investigatory uses context clues: "what happened", "how did we end up here", "something went wrong", "doesn't match what I expected", "I want to understand before fixing." These join the existing inference table in audit-conventions.md.

### Frame-reflexivity (epistemic — applies to ALL types)

**Current state:** Rule 4 (escape hatch) exists for all types: "What did you encounter that these ground rules didn't prepare you for?" This catches excess within the frame. No mechanism catches whether the frame itself was appropriate.

- [decided:reasoned] Add a frame-reflexivity closing step to ALL audit types, distinct from Rule 4. REVIEW.md Part 4: "Did the choice of audit type shape what you found? What would a differently-typed audit have looked for?" Rule 4 catches excess within the frame; frame-reflexivity catches whether the frame was the right one.
- [assumed:reasoned] Frame-reflexivity is lightweight for non-investigatory types (1-2 sentences as a closing reflection) and a full section for investigatory type ("Position of This Investigation" is already in the body template). The question is the same; the expected depth of engagement differs.
- [assumed:reasoned] Frame-reflexivity should be a Rule 5 in audit-ground-rules.md Section 2, alongside Rule 4 (escape hatch). Both are always-included closing rules. This keeps the structure clean — core rules (1-3), closing rules (4-5), type-family extensions (S1-S2, E1-E3, C1, I1-I4).

### Agent design

**Current state:** No `gsdr-auditor` agent exists. The closest agents are `gsdr-context-checker` (has explicit ground rules, verifies claims), `gsd-verifier` (structural verification), and `gsdr-spike-runner` (receives fully-formed spec, executes).

- [decided:reasoned] Tool access: Read, Bash, Grep, Glob, Write. Same as verifier/checker agents. The auditor needs to read artifacts, search codebase, run verification commands, and write output.
- [decided:reasoned] Agent spec embodies the epistemic stance from audit-ground-rules.md — not just lists the rules but explains why they matter, similar to how gsdr-context-checker's spec explains the claim type vocabulary.
- [assumed:reasoned] The agent handles Rule 4 (escape hatch) and Rule 5 (frame-reflexivity) structurally — they appear as named sections in the output, not as optional afterthoughts. The agent is instructed to actually engage with these, not just add boilerplate.
- [open] What model tier should gsdr-auditor use by default? Structural audits (checklist-like) could use sonnet. Epistemic and investigatory audits (judgment-heavy) may warrant opus. The orchestrator has override capability, but the default in model-profiles.cjs needs to be set. This depends on cost/quality tradeoffs that research should investigate.

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] The REVIEW.md (pre-phase-archive) must be taken seriously — it subjects its own proposals to the same critique it applies to the existing taxonomy. Implementation should continue that practice: at each design step, ask what the design makes easy to see and what it might make hard to see.
- **G-2:** [governing:cited] "Don't just naively add a type — investigate the situation in which the audit is demanded" (user direction, verbatim from handoff context). The investigatory type should not be a mechanical addition to a list.
- **G-3:** [governing:reasoned] Frame-reflexivity must be genuine, not performative. If the agent writes "I considered my biases" without actually doing so, the rule has become compliance theater. The agent spec should ground frame-reflexivity in specific questions about what this audit type prepared the agent to notice vs. what it might have missed.
- **G-4:** [assumed:reasoned] The investigatory type's I1-I4 ground rules are a pragmatic approximation of the Heideggerian insight about prejudgment, not a full philosophical response. They should be treated as a strong starting position, not as settled methodology. REVIEW.md Part 6 acknowledges this explicitly.
- **G-5:** [governing:reasoned] The relationship between investigatory and exploratory is deliberately unresolved — they may merge, stay distinct, or evolve differently. Implementation should not force a premature resolution. Add investigatory as a type; let usage data determine if the classification needs revision.

</guardrails>

<questions>
## Open Questions

### Q1: Is investigatory a type, a family, or a mode?
**Research program:** Examine how other GSD type taxonomies handle cross-cutting concerns (e.g., signal_type in knowledge-store.md). Look at how the existing 3-family clustering works in practice — does adding a 4th family make sense, or is investigatory orthogonal to structural/epistemic/compliance? Consider whether "investigatory mode" (applicable to any type) would require different infrastructure than "investigatory type" (standalone).
**Downstream decisions affected:** Family-to-ground-rule mapping in audit-ground-rules.md Section 4 table. Type inference table in audit-conventions.md. Body template selection logic in the command. Whether the agent needs to handle mode-modified types.
**Reversibility:** LOW — the v1 implementation adds investigatory as a named type. If later usage reveals it should be a mode, the change is: update type inference to allow mode modifiers, update ground rule selection to compose rather than switch. The named type continues to work regardless.

### Q2: What model tier should gsdr-auditor default to?
**Research program:** Survey existing agent model assignments in model-profiles.cjs — what determines whether an agent gets sonnet vs. inherit (opus on quality profile). Consider the cost/quality tradeoff: structural audits are cheaper but investigatory audits require deeper reasoning. Look at whether the orchestrator's type-conditional escalation makes the default less important.
**Downstream decisions affected:** model-profiles.cjs entry. Whether the orchestrator needs explicit escalation logic or just uses the profile default.
**Reversibility:** HIGH — model-profiles.cjs entries are trivially changeable post-implementation.

</questions>

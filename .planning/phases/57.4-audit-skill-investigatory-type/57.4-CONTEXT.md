# Phase 57.4: Audit Skill & Investigatory Type - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning
**Mode:** Exploratory --auto -- preserving uncertainty, auto-progression by type rules
**Iteration:** v2 — v1 archived in `pre-phase-archive/57.4-CONTEXT-v1.md`. This version builds on, corrects, and responds to the failures of the prior iteration: the flat taxonomy conflated three orthogonal concerns (subject, orientation, delegation), the template model couldn't compose obligations across axes, and the design lacked a mechanism for cross-model dispatch. A deliberation (`audit-taxonomy-three-axis-obligations.md`, open — fed forward, not concluded) restructured the design. A retrospective (`audit-taxonomy-retrospective-analysis.md`) validated the restructuring against all 13 audit sessions.

<domain>
## Phase Boundary

Build the `/gsdr:audit` command and `gsdr-auditor` agent. Restructure the audit type taxonomy from a flat 8-type list to a 3-axis model (subject × orientation × delegation). Implement an obligations-based output paradigm for investigatory and exploratory orientations while preserving templates for standard routine audits. Add investigatory orientation with ground rules I1-I4 and frame-reflexivity Rule 5. Update `audit-conventions.md` and `audit-ground-rules.md`. Formalize cross-model delegation patterns (WF-01, pulled from Phase 62).

This phase holds two concerns together without collapsing one into the other:

1. **Operational** — the invocable audit skill that Phase 57.3 deferred. The command + agent pair, session directory creation, ground rule embedding, obligation composition, and cross-model dispatch. Concrete engineering following established GSD command patterns.

2. **Epistemic** — the 3-axis taxonomy and obligations model. The deliberation revealed that the flat taxonomy smuggled assumptions about which axes matter. The obligations model replaces templates for non-standard orientations, composing demands from subject and orientation that the auditor must address — not sections to write, but things to engage with. The hermeneutic composition principle (name the tension, navigate situationally) governs how conflicting obligations interact.

Does NOT include: gsd-tools audit subcommand (deferred per trial-before-formalize), changes to existing agent implementations (gsd-verifier, gsd-integration-checker, etc.), workflow rewrites for audit-milestone or validate-phase, the adaptive harness design from backlog 999.x, mid-audit type switching, frame-reflexivity for non-audit workflows.

</domain>

<working_model>
## Working Model & Assumptions

### Three-axis taxonomy (the central restructuring)

**Current state:** Flat 8-type taxonomy in audit-conventions.md Section 3 conflates three orthogonal concerns: what you're auditing (subject), from what stance (orientation), and who does it (delegation). `cross_model_review` is a delegation strategy, not a subject type. `exploratory` is an orientation, not a subject type. The user's observation during discuss-phase v1 triggered the restructuring.

**Axis 1: Subject** (what are you auditing?)

- [decided:reasoned] 9 subject types replacing the original 8: the 7 genuine subjects from the old taxonomy (`phase_verification`, `milestone`, `codebase_forensics`, `requirements_review`, `comparative_quality`, `claim_integrity`, `adoption_compliance`) plus 2 new subjects discovered by testing the decomposition against real audit cases: `process_review` (methodology soundness) and `artifact_analysis` (patterns in a corpus). `cross_model_review` exits the subject enum — it was always a delegation strategy.
- [decided:reasoned] Subject is optional for investigatory and exploratory orientations. When you don't know what you're investigating yet (the Phase 57 case), the investigation discovers its subject.
- [decided:reasoned] Each subject carries an epistemic profile: `oriented_toward`, `assumes`, `might_miss`. The orchestrator reads these when matching situation to subject. If the situation has features that every subject's "might miss" column would miss, that's a signal for investigatory-without-subject. Profiles documented in the deliberation's Axis 1 table.

**Axis 2: Orientation** (from what stance?)

- [decided:reasoned] Three orientations: `standard` (default — close on findings), `investigatory` (something went wrong; hold diagnosis open, competing explanations), `exploratory` (a question beckons; follow it, don't force closure). Orientation is orthogonal to subject — you can investigate, explore, or verify any domain.
- [decided:reasoned] The distinction between investigatory and exploratory is one of occasion: investigatory is occasioned by breakdown (expectations violated), exploratory by possibility (a question opens). Both are hermeneutically circular. The body structures differ because they start from different places (discrepancy vs. question).

**Axis 3: Delegation** (who does it?)

- [decided:reasoned] Two delegation modes: `self` (default — local gsdr-auditor agent) and `cross_model` (dispatch to another model's CLI). Delegation is orthogonal to both subject and orientation.

- [decided:reasoned] The `exploratory` escape hatch is preserved for the *subject* axis only — `audit_subject: exploratory` for audits that don't match any named subject. The orientation axis does not need an escape hatch because its 3 values (standard, investigatory, exploratory) already span the epistemic posture space.

### Obligations model

**Current state:** audit-conventions.md Section 4 has body templates per type. The deliberation revealed that templates can't compose across axes — "which template wins?" is the wrong question when two axes both contribute structure.

- [decided:reasoned] Replace templates with composable obligation sets for non-standard orientations. Each axis contributes obligations — things the audit must *address*, not sections it must *write*. The difference: "write a Competing Explanations section" (template) vs. "you must present competing explanations" (obligation). The second lets the auditor weave competing explanations into the narrative wherever they emerge naturally.
- [decided:reasoned] Three populations confirmed by retrospective: (1) Standard routine audits use templates (VERIFICATION.md, MILESTONE-AUDIT.md — consistency matters), (2) Complex/investigatory/exploratory use obligations (flexibility matters), (3) Multi-agent sessions need additional mechanisms (not fully designed — see open questions).

**Core obligations (every audit):**

- [decided:reasoned] Ground rules 1-3 (cite evidence, test disconfirmation, distinguish measure from measured) — already established in audit-ground-rules.md.
- [decided:reasoned] Rule 4: What exceeded these obligations? (escape hatch — already established)
- [decided:reasoned] Rule 5: Did the framing shape what you found? (frame-reflexivity — new, lightweight for standard, full section for investigatory). This is distinct from Rule 4: Rule 4 catches excess within the frame; Rule 5 catches whether the frame was the right one.

**Orientation obligations:**

- [decided:reasoned] Standard: close on findings with evidence, produce a clear verdict, address all items in scope.
- [decided:reasoned] Investigatory (I1-I4, confirmed from REVIEW.md Part 3): (I1) start from the discrepancy, not a theory; (I2) let the investigation guide artifact selection; (I3) present competing explanations for each major finding; (I4) name the position of the investigation. Plus: show what remains unknown, show how you navigated any tensions between obligations.
- [decided:reasoned] Exploratory: state the initiating question/curiosity; follow the question wherever it leads; name what you found that you weren't looking for; name what the exploration opened (new questions, possibilities); name what you didn't look at (acknowledged partiality, not failure); "I don't know yet" is a valid conclusion.

**Subject obligations:**

- [decided:reasoned] Each subject type contributes its own obligation set. These compose with orientation obligations into a flat list. See deliberation Section "Subject obligations" for the full table.

**Three new obligations from retrospective:**

- [decided:reasoned] Chain integrity: when audit B uses audit A's finding as evidence, and audit A had a quality failure, audit B inherits the failure invisibly. Obligation: for each finding that depends on a predecessor audit's claim, re-verify that claim independently before incorporating it.
- [decided:reasoned] Dispatch hygiene: when running parallel agents, verify that prompts don't contain framing that could systematically bias findings (comparative framing, target counts, desired conclusions).
- [decided:reasoned] Framework invisibility: name what your audit framework cannot see — not what you chose not to look at, but what the structure of the audit makes invisible.

**Composition principle:**

- [decided:reasoned] When obligations from different axes tension against each other, the auditor must: (1) name the tension, (2) name what about the situation creates it, (3) show how they navigated it — responsive to both demands. This is a hermeneutic principle, not an algorithmic one: "If you find yourself cleanly ignoring one obligation in favor of another, you've likely stopped engaging with the tension."
- [assumed:reasoned] The composition principle is written but untested. First real investigatory audits will reveal whether agents can actually engage with it or whether they collapse to ignoring one side.

**Template excess section:**

- [decided:reasoned] All audit output includes "What the Obligations Didn't Capture" — a named section for findings that don't fit any obligation. Consistently substantial content in this section for a particular kind of audit is evidence for new subject types or new obligations.

**Suggested structures:**

- [decided:reasoned] For agents that need scaffolding, suggested output structures remain available per orientation. But they are suggestions, not mandates. The obligations are what's checked. Templates are genuinely useful for standard orientation (consistency, scannability). The orchestrator's complexity assessment (subject well-defined? scope bounded? established precedent?) determines whether to offer template or obligations mode.

### Command architecture

**Current state:** No `/gsdr:audit` command exists. The audit-milestone command is the closest pattern.

- [decided:reasoned] Two-file architecture: `commands/gsd/audit.md` (orchestrator) + `agents/gsdr-auditor.md` (executor). The pattern is followed by commands that orchestrate long-running work (audit-milestone, validate-phase, execute-phase). Lightweight commands (help, signal) operate inline.
- [decided:reasoned] Orchestrator writes full task spec, agent executes — spike-runner pattern. The command does discovery (reads conversation context, infers type/scope), creates session directory, selects obligations and ground rules, writes task spec, dispatches agent. The agent receives a fully-formed spec and runs the audit.
- [decided:reasoned] Type inference from context, not forced selection. The command reads conversation history for context clues and proposes an audit classification (subject × orientation × delegation). If context is sufficient, proceeds directly. If ambiguous, 0-2 clarifying questions.
- [decided:reasoned] Free-text topic as primary invocation, following deliberate.md pattern: `/gsdr:audit "something went wrong with Phase 57"`. Context inference proven in deliberate.md.
- [decided:reasoned] Session directory follows audit-conventions.md: `.planning/audits/YYYY-MM-DD-{slug}/` with task spec + output files, YAML frontmatter per Section 2 schema.
- [decided:reasoned] Ground rules and obligations auto-composed by the orchestrator based on 3-axis classification, copied into task spec (not referenced by path). Per audit-ground-rules.md Section 4.
- [decided:reasoned] Qualified type assignment: for situations that don't fit neatly, the orchestrator writes a fit assessment in the task spec explaining why this classification was chosen and where the auditor should expand.
- [assumed:reasoned] The `--auto` flag controls interactivity consistent with other GSD commands. Structural standard audits can run fully autonomously. Investigatory audits may benefit from conversational mode. The flag and classification together determine execution mode.

### Cross-model delegation (WF-01, pulled from Phase 62)

**Current state:** Cross-model dispatch works but is fragile. Known problems: environment setup issues, agents finishing early, instructions not properly conveyed, lots of hand-holding.

- [decided:reasoned] WF-01 pulled from Phase 62 into 57.4. The audit command is the natural home for cross-model dispatch since audits are the primary cross-model use case.
- [decided:reasoned] The command generates the task spec (ground rules, obligations, scope, fit assessment), then dispatches to the target model's CLI with proper environment setup and instruction framing.
- [open] Specific cross-model dispatch mechanics: robust CLI invocation patterns, environment detection, instruction framing, output collection. Failure modes (agents finishing early, env misconfig, instruction degradation) need investigation during research.

### Agent design

**Current state:** No `gsdr-auditor` agent exists. Closest agents: `gsdr-context-checker` (explicit ground rules), `gsd-verifier` (structural verification), `gsdr-spike-runner` (receives fully-formed spec).

- [decided:reasoned] Tool access: Read, Bash, Grep, Glob, Write. Same as verifier/checker agents.
- [decided:reasoned] Agent spec embodies the epistemic stance from audit-ground-rules.md — not just lists the rules but explains why they matter.
- [decided:reasoned] The agent handles Rule 4 (escape hatch) and Rule 5 (frame-reflexivity) structurally — they appear as named sections in the output, not optional afterthoughts.
- [decided:reasoned] gsdr-auditor model: sonnet in frontmatter (consistent with gsdr- agent pattern). Orchestrator escalates for investigatory/epistemic audits when higher reasoning is warranted. This matches the gsdr-context-checker pattern.
- [assumed:reasoned] The "What the Obligations Didn't Capture" section and the frame-reflexivity closing step must be genuinely engaged with, not perfunctory. The agent spec should ground these in specific questions rather than generic prompts.

### Frontmatter schema changes

**Current state:** audit-conventions.md Section 2 has `audit_type` as a single required field.

- [decided:reasoned] Three new fields replacing the semantic content of `audit_type`: `audit_subject` (optional for investigatory/exploratory), `audit_orientation` (standard | investigatory | exploratory), `audit_delegation` (self | cross_model:{model_id}).
- [decided:reasoned] Backward compatibility: existing artifacts have `audit_type`. Migration adds the new 3 fields; `audit_type` maps to `audit_subject` with `audit_orientation: standard` and `audit_delegation: self` as defaults. Legacy field preserved.
- [assumed:reasoned] The frontmatter field count stays within the ~12-field budget established in Phase 57.3 (3 required + 5 recommended + 4 optional). The 3 new fields replace `audit_type` semantically (1 becomes 3), but the migration keeps `audit_type` for backward compat, which adds 2 net fields. This is acceptable at 14 total but should be monitored.

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] The REVIEW.md (pre-phase-archive) and the deliberation must be taken seriously — both subject their proposals to the same critique they apply to the existing taxonomy. The deliberation's self-reflexive warnings (especially about performative humility) are design constraints, not decoration.
- **G-2:** [governing:bare] "Don't just naively add a type — investigate the situation in which the audit is demanded" (user direction during v1 discuss-phase). The 3-axis restructuring is the response to this directive — it investigates the taxonomic assumptions, not just the missing type.
- **G-3:** [governing:reasoned] Frame-reflexivity (Rule 5) and the "What the Obligations Didn't Capture" section must be genuine, not performative. If the agent writes "I considered my biases" without actually doing so, the rule has become compliance theater. The agent spec should ground these in specific questions.
- **G-4:** [governing:reasoned] The hermeneutic composition principle is a practice, not an algorithm. If agents consistently resolve obligation tensions by cleanly picking one side, the principle has failed. Monitor the first investigatory audits for this collapse pattern.
- **G-5:** [governing:reasoned] The relationship between investigatory and exploratory orientations is deliberately left as a design choice, not ontologically settled. They may merge, stay distinct, or evolve differently. Implementation should not foreclose this evolution.
- **G-6:** [governing:reasoned] The aporia of epistemic humility is a condition of the framework's existence, not a problem to solve. The framework creates structural openings (Rule 4, Rule 5, framework invisibility obligation, excess section) where limits are encountered in practice. Whether these function as genuine openings or performative gestures depends on the practice of the agents who use them — which is not something the framework can guarantee. No "Known Limits" section.

</guardrails>

<questions>
## Open Questions

### Q1: Can agents engage with the hermeneutic composition principle?
**Research program:** No research can answer this prior to implementation — it requires running real investigatory audits with composed obligation sets and examining whether the agent names tensions and navigates them or collapses to ignoring one side. The implementation should include explicit instruction in the agent spec and the first 2-3 investigatory audits should be reviewed for composition quality.
**Downstream decisions affected:** Whether the composition principle needs algorithmic support (e.g., explicit tension-detection prompting), or whether agent instruction is sufficient.
**Reversibility:** MEDIUM — if agents consistently collapse, adding algorithmic support is additive work, not a rewrite. But the philosophical character of the framework depends on hermeneutic resolution, not algorithmic resolution.

### Q2: Cross-model delegation mechanics
**Research program:** Investigate robust CLI invocation patterns for dispatching to Codex CLI / other model CLIs from within a Claude Code session. Document specific failure modes (agents finishing early, env misconfig, instruction degradation). Look at existing cross-model dispatch patterns in the codebase (the codex-drift-audit, the discuss-phase cross-model review).
**Downstream decisions affected:** Whether the audit command can reliably dispatch cross-model in --auto mode, or whether cross-model always requires interactive oversight. Whether the task spec format needs modification for cross-model consumption.
**Reversibility:** HIGH — cross-model delegation is an additive axis. If it doesn't work reliably, `audit_delegation: self` remains the default and cross-model becomes a documented manual workflow.

### Q3: Multi-agent obligation enforcement
**Research program:** The retrospective identified that when multiple agents run in parallel (6+ agents in the session-log audit), obligation enforcement at the individual agent level is lost — obligations are checked only at synthesis. Investigate whether each sub-agent should receive orientation obligations in its task spec, and whether the synthesizer should have its own obligations that include checking sub-agent compliance.
**Downstream decisions affected:** Whether the orchestrator generates per-agent task specs with obligation subsets, or a single task spec for a single auditor agent.
**Reversibility:** MEDIUM — adding per-agent obligation distribution later requires restructuring the orchestrator's task spec generation, but doesn't invalidate the single-agent path.

</questions>

**Process body skeleton (write all 8 subsections in full):**

## Mode Detection

Parse `$ARGUMENTS`:
- `--continue <session-slug>`: resume an existing session. Find it at `.planning/audits/YYYY-MM-DD-{slug}/`, read the task spec, and either dispatch again or report status.
- Free-text topic (e.g., `"something went wrong with Phase 57"`): new audit, use topic as starting point.
- No arguments: check the current conversation for context clues. Summarize what you understand the audit topic to be and confirm with the user (0-2 questions).
- Flags: `--auto` (non-interactive), `--subject X`, `--orientation Y`, `--delegation Z` (explicit classification overrides, skip Step 2).

Pattern source: `commands/gsd/deliberate.md` Step 1 lines 55-62 (context-inference on no-arguments).

## Step 1: Context Inference

If mode is "new audit" and no explicit classification flags, read the current conversation history for clues. Look for phrases matching the table in RESEARCH.md §"Pattern 5: Context inference" (e.g., "something went wrong" → investigatory orientation; "I'm curious" → exploratory; "did Phase N achieve its goal" → standard phase_verification).

Read STATE.md to anchor the situation (current phase, current milestone, any blockers).

Produce a draft classification as a tuple: `(subject, orientation, delegation)`. Subject may be `null` (omitted) for investigatory or exploratory orientations when no named subject fits.

## Step 2: Classify with 0-2 Clarifying Questions

If the draft classification is confident (all three axes clear from context), proceed directly.

If ambiguous, ask 0-2 clarifying questions via `AskUserQuestion`. Do NOT ask more than 2 — per CONTEXT.md DC-6 "Claude's Discretion: how many clarifying questions (0-2 based on context sufficiency)".

Produce a final classification. If the classification doesn't cleanly map to the 9 subjects, consider omitting subject and marking `audit_orientation: investigatory` or `exploratory` (the investigation discovers its subject).

Write a **fit assessment** — a brief paragraph explaining why this classification was chosen and where the auditor should feel free to expand. This becomes part of the task spec.

## Step 3: Complexity Assessment (template territory vs obligations territory)

Read the classification and assess:
- **Population 1** (standard + well-known subject + no chain dependencies): template territory. Suggested scaffolds from `audit-conventions.md` Section 4 are available and appropriate.
- **Population 2** (investigatory, exploratory, cross-model, or multi-subject tensions): obligations territory. Task spec says "address these obligations" not "fill these sections."
- **Population 3** (multi-agent parallel dispatch): deferred per CONTEXT.md Q3. If the user's situation looks like Population 3, warn them that the obligation enforcement at sub-agent level is not yet implemented and offer to proceed with self-delegation single-agent.

This assessment is light. Do NOT build a state machine. Use judgment.

## Step 4: Compose Obligations

Read `audit-ground-rules.md` (already rewritten by Plan 02). Compose the obligation set for this audit:
- **Core obligations (always):** Rules 1-5 (cite evidence, test disconfirmation, distinguish measure from measured, escape hatch, frame-reflexivity)
- **Orientation obligations:** pick standard / investigatory (I1-I4 + unknowns + tensions) / exploratory per Step 2's classification
- **Subject obligations:** pick the row from the 9-subject table matching the audit_subject. If subject is omitted, skip.
- **Cross-cutting obligations:**
  - Chain integrity: include if `predecessor_audits:` is non-empty OR if the user references prior audit findings
  - Dispatch hygiene: include if delegation is `cross_model`
  - Framework invisibility: include if orientation is investigatory or exploratory (encouraged for standard but not required)

The obligation set is a flat list. When it lands in the task spec (Step 5), include the composition principle too — the auditor may encounter tensions and must navigate them per the hermeneutic principle.

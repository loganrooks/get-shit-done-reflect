**Epistemic profile table (Axis 1 table, deliberation lines 69–79):**

| Subject | Oriented toward | Assumes | Might miss |
|---|---|---|---|
| `phase_verification` | Structural completeness | Phase goal is the right standard | Goal itself being wrong; quality within passing criteria |
| `requirements_review` | Coverage and feasibility | Requirements are the authoritative scope | Requirements that should exist but don't |
| `process_review` | Methodology soundness | Process has a spec or expected behavior | Process working as designed but design is wrong |
| `artifact_analysis` | Patterns in data | Corpus is representative; patterns are meaningful | What the corpus excludes; patterns that only appear across corpora |
| `claim_integrity` | Claim warrant | Typed claim vocabulary captures the relevant distinctions | Claims that don't fit the vocabulary |
| `codebase_forensics` | Code structure | Code is the ground truth | Documentation that should override code; design intent behind structure |
| `comparative_quality` | Quality comparison | Comparison axis is meaningful | What the axis makes invisible; confounding variables |
| `milestone` | Integration completeness | Phases should connect | Whether they should connect differently |
| `adoption_compliance` | Practice matches intent | Intent is documented and correct | Intent that should change; undocumented reasonable deviations |

**Subject is optional** for `investigatory` and `exploratory` orientations — when you don't know what you're investigating yet, the investigation discovers its subject.

### Core Obligations (every audit)

Source: `audit-taxonomy-three-axis-obligations.md` lines 114–118 + CONTEXT.md working model + retrospective.

1. **Rule 1:** Every factual claim cites file:line and quotes the relevant passage. (carried from `audit-ground-rules.md` §1, survives rewrite)
2. **Rule 2:** For every finding, BEFORE writing it, ask "What would disconfirm this?" and CHECK. (carried from §1, survives)
3. **Rule 3:** Distinguish what you measured from what the measure captures. (carried from §1, survives)
4. **Rule 4 (escape hatch):** What did you encounter that these ground rules didn't prepare you for? (carried from §2, survives — but now joined by Rule 5 for frame-level excess)
5. **Rule 5 (frame-reflexivity, NEW in v2):** Did the framing shape what you found? (added by this phase; authority: REVIEW.md Part 4, lines 103–116)

### Rule 5 — exact wording from REVIEW.md Part 4

Source: `pre-phase-archive/REVIEW.md` lines 103–116.

> "every audit type smuggles in assumptions about the form of failure, and those assumptions shape what the auditor can find... This is not the same as Rule 4 (escape hatch), which asks 'what didn't fit the template.' This is more fundamental: 'was this the right template to begin with?' Rule 4 catches excess within the frame. This catches whether the frame itself was appropriate."

Lightweight closing step for standard; full section for investigatory. From REVIEW.md line 150: "A lightweight version of I4 as a closing step: 'Did the choice of audit type shape what you found? What would a differently-typed audit have looked for?'"

### Orientation Obligations

Source: `audit-taxonomy-three-axis-obligations.md` lines 119–140.

**Standard:**
- Close on findings with evidence
- Produce a clear verdict or assessment
- Address all items in scope

**Investigatory (I1–I4, origin REVIEW.md lines 70–74):**
- **I1** — Start from the discrepancy, not a theory. Describe what was expected, what was delivered, and why those expectations are being treated as the standard — the choice of comparison point is already an interpretive act.
- **I2** — Let the investigation guide artifact selection. Don't mandate which artifacts to read in advance. The artifact chain is a finding, not an input.
- **I3** — Present competing explanations. For each finding, offer at least two interpretations. Don't collapse to one.
- **I4** — Name the position of the investigation. Every investigation is conducted from somewhere — with particular attunements, particular things it's prepared to notice, particular things it isn't.
- **Plus** — Show what remains unknown; show how you navigated any tensions between obligations.

**Exploratory:**
- State the question/curiosity that initiated the exploration
- Follow the question wherever it leads — permission to change direction
- Name what you found that you weren't looking for
- Name what the exploration opened (new questions, possibilities)
- Name what you didn't look at (acknowledged partiality, not failure)
- "I don't know yet" is a valid conclusion

### Subject Obligations

| Subject | Obligations |
|---|---|
| `phase_verification` | Verify execution not just existence; check wiring definition→use; compare against success criteria |
| `requirements_review` | Read requirement text, assess specificity; check for missing requirements (negative space); assess feasibility |
| `process_review` | Compare execution against process spec/intent; examine methodology assumptions; check if process worked-as-designed vs. design is wrong |
| `artifact_analysis` | State corpus and why it's representative; look for patterns AND anti-patterns; note what the corpus excludes |
| `claim_integrity` | Verify claims against citations; surface untyped load-bearing assumptions; trace dependency chains |
| `codebase_forensics` | Examine actual code not documentation; trace data flow; identify structural patterns |
| `adoption_compliance` | Verify against actual practice not documented practice; survey multiple instances; classify deviations |
| `milestone` | Check cross-phase integration; verify E2E flows; check interface mismatches between phases |
| `comparative_quality` | Define comparison axis explicitly; compare like with like; note what axis makes invisible |

### Three new obligations (origin: retrospective, exact wording required in ground-rules.md rewrite)

**Chain integrity** — source: retrospective Gap 4, line 194–197.

> "When audit B uses audit A's finding as evidence, and audit A had a quality failure, audit B inherits the failure invisibly. This is the single most consequential failure mode in the discuss-phase session chain. **Obligation:** For each finding that depends on a predecessor audit's claim, re-verify that claim independently before incorporating it. State the re-verification or state why it was not done."

**Dispatch hygiene** — source: retrospective Gap 5, lines 199–203.

> "When running model comparison experiments or multi-agent parallel dispatch, the prompts themselves can contaminate the results (the contamination experiment in the session-log audit found 50% GPT inflation from framing effects). **Obligation:** For `cross_model` delegation, verify that the delegation prompt does not include framing that could systematically bias findings (comparative framing, target counts, desired conclusions). If the comparison is intentional, note it explicitly in the audit frontmatter as a confound."

**Framework invisibility** — source: retrospective Gap 7, lines 211–215.

> "The obligations have I4 ('name the position of the investigation') but nothing that specifically asks: 'What does this audit's framework make invisible? What kinds of findings would not appear in this audit no matter how rigorously it was conducted?' **Obligation:** Name what your audit framework cannot see — not what you chose not to look at, but what the structure of the audit makes invisible."

### Composition Principle — hermeneutic, not algorithmic

Source: `audit-taxonomy-three-axis-obligations.md` lines 156–164.

> "Obligations from different axes compose into a flat list. When obligations tension against each other (e.g., I2 'follow the evidence freely' vs. S1 'run the test'), the auditor must:
> 1. Name the tension
> 2. Name what about the situation creates it
> 3. Show how they navigated it — responsive to both demands
> 4. The resolution emerges from engagement with the situation, not from a precedence rule
>
> This is a hermeneutic principle, not an algorithmic one: **'If you find yourself cleanly ignoring one obligation in favor of another, you've likely stopped engaging with the tension.'**"

**Untested — explicitly flagged as Q1 in CONTEXT.md** (deliberation's open question 5, lines 298–300). The composition principle is written but cannot be validated until real investigatory audits run. First 2–3 investigatory audits should be reviewed for collapse-to-one-side behavior.

### Template Excess Section — mandatory in every output

> "All audit output includes: **'What the Obligations Didn't Capture'** — a named section for findings that don't fit any obligation. If this section is consistently substantial for a particular kind of audit, that's evidence for a new subject type or new obligations." (deliberation lines 174–176)

### The 8 concrete examples — load-bearing for obligation composition

Source: `audit-taxonomy-three-axis-obligations.md` lines 197–265. Reproduced verbatim or tightly summarized because the agent spec must ground the composition principle in these examples rather than in abstract prose.

**Exploratory combinations:**

1. **"I'm curious about how our signal corpus has evolved"** → `artifact_analysis × exploratory × self`. Obligations: core + exploratory (follow curiosity, name what wasn't looked at, don't force closure) + artifact_analysis (state corpus, look for patterns, note exclusions). "I noticed X but I'm not sure what it means yet" is fine.

2. **"What patterns exist in how we write CONTEXT.md files?"** → `process_review × exploratory × self`. Obligations: core + exploratory + process_review. Named what they didn't look at ("I only read 10 of 85 files — this is a sample, not a census").

3. **"Something about the way we handle phases feels off but I can't say what"** → **no subject × `exploratory` × `self`**. Obligations: core + exploratory only. No domain obligations. Maximum freedom. "Name what you found that you weren't looking for" is the whole point.

**Investigatory combinations:**

4. **The Phase 57 case: "The implementation doesn't match the vision"** → **no subject × `investigatory` × `self`**. Core + investigatory (I1–I4, competing explanations, what remains unknown). No subject obligations — investigation discovers the subject. I3 (competing explanations) is what distinguishes this from a standard audit that would close on the first plausible story.

5. **"Why do our plans keep missing requirements?"** → `process_review × investigatory × self`. Core + investigatory + process_review. The process_review obligation "compare against spec" may tension with investigatory "competing explanations" — the spec itself may be the problem. The auditor navigates this per the composition principle.

6. **"Get GPT 5.4 to investigate what happened with Phase 57"** → **no subject × `investigatory` × `cross_model`**. All obligations embedded in task spec for the external model. I4 is especially rich — "I'm approaching this without having been part of the decisions that led here."

7. **"Something went wrong with the cross-model audit last time — have GPT 5.4 investigate"** → `process_review × investigatory × cross_model`. All three axes active. I4 especially rich — GPT naming its own position as external model investigating a process it wasn't part of.

**Standard combinations:**

8. **"Did Phase 57.4 achieve its goal?"** → `phase_verification × standard × self`. Core + standard + phase_verification. The routine case. Suggested template scaffolding available (Goal → Artifacts Checked → Findings → Verdict).

9. **"Are our requirements well-specified?"** → `requirements_review × standard × self`. The "check for missing requirements" obligation is the most valuable — forces the auditor to look at negative space.

10. **"Have GPT 5.4 check our claim types"** → `claim_integrity × standard × cross_model`. External perspective adds independence, not investigatory openness.

**Tension and transition cases:**

11. **"Check the codebase and investigate anything surprising"** → `codebase_forensics × starts standard, may shift to investigatory`. This is where the obligations model shines over templates. The auditor accumulates obligations as the situation demands. No "switching templates."

12. **"Explore whether our audit taxonomy is doing what we think"** → `process_review × exploratory × self`. A meta-audit. Most important finding may be something the audit framework itself makes invisible.

### Anti-Patterns to Avoid

- **Treating reference files as authority for v2 taxonomy/obligations.** The #1 known failure for this phase — see `sig-2026-04-10-discuss-phase-authority-weighting-gap.md`. The planner must cite deliberations, not the reference files, for v2 content. The reference files are deliverables of this phase, not inputs.
- **Building elaborate classifier logic into the command.** Per CONTEXT.md's "Claude's Discretion" section, exact argument parsing and complexity classification are flexible. Don't over-engineer. The orchestrator's Step 1 ("context inference") and Step 2 ("classify with 0–2 questions") should be lightweight LLM reading, not a state machine.
- **Making obligations-as-sections.** The whole point is "you must present competing explanations" (obligation) not "write a Competing Explanations section" (template). Task specs should say "address these obligations" not "fill these sections."
- **Omitting the "What the Obligations Didn't Capture" section.** It is mandatory in every audit output. Per deliberation line 174: "a named section for findings that don't fit any obligation."
- **Collapsing the composition principle to a precedence rule.** If the command or agent says "if obligations conflict, X wins," the principle has failed. The task spec must surface tensions to the auditor and ask them to name/navigate, not resolve.
- **Performative Rule 5 / framework invisibility.** If the agent writes "I considered my biases" without actually doing so, the rule is compliance theater. The agent spec must ground these in specific questions.
- **Editing `.claude/` directly.** Per CLAUDE.md — v1.15 Phase 22 shipped without agent-protocol.md because of this. Always edit `commands/gsd/audit.md` and `agents/gsdr-auditor.md` in source; run `node bin/install.js --local` after.
- **Scheduling reference file rewrites as polishing.** DC-4 was explicitly retyped to prevent this. The rewrites are primary deliverables, not post-implementation cleanup.
- **Generating a strict per-axis precedence for ground rules.** The obligations model replaces the type-family selection matrix. The rewritten `audit-ground-rules.md` should NOT have a replacement selection table; it should have composition rules ("subject contributes, orientation contributes, they compose into a flat list").

## Common Pitfalls

### Pitfall 1: Authority weighting failure — treating reference files as v2 ground truth

**What goes wrong:** The planner reads `audit-conventions.md` Section 3 (flat 8-type taxonomy), sees it as authoritative, and writes plans against the flat taxonomy. The superseded sections look normal because the supersession banner is at the top of the file and easy to skim past.

**Why it happens:** Trained default — reference docs are usually authoritative. The planner assumes "this is what I cite" without checking freshness against deliberations.

**How to avoid:**
1. Always read the supersession banner FIRST (top 20 lines of `audit-conventions.md` and `audit-ground-rules.md`)
2. The banner explicitly names what survives and what is to-be-rewritten
3. For anything not named as surviving, cite the deliberations
4. Use CONTEXT.md's `<user_constraints>` block as a checklist — DC-4's retyping to `[decided:reasoned]` is the canary for this failure mode

**Warning signs:** Plan text citing `audit-conventions.md` Section 3 or Section 4 as authority. Plan text assuming the 3-axis frontmatter fields already exist.

**Evidence:** `sig-2026-04-10-discuss-phase-authority-weighting-gap.md` — the Phase 57.4 v2 discuss-phase hit this exact failure mode and had to be corrected post-context-checker.

### Pitfall 2: Compliance-theater Rule 5 and framework invisibility

**What goes wrong:** The agent spec embeds Rule 5 and framework invisibility but grounds them in generic prompts ("consider your biases"). The agent writes "I considered my biases" without doing so. The obligation becomes performative.

**Why it happens:** Abstract philosophical framing doesn't automatically translate to agent behavior. Generic prompts produce generic compliance.

**How to avoid:** Ground Rule 5 and framework invisibility in *specific questions*, not generic prompts:
- Rule 5 specific: "If this audit had been classified as `{alternative orientation or subject}`, what would it have looked for that you didn't? What findings would that audit have produced that yours doesn't?"
- Framework invisibility specific: "Name a concrete finding that would not appear no matter how rigorously this audit was conducted, because of how this audit's scope was framed. If you can't name one, that's suspicious — the framework is probably hiding something from you that it's also hiding from itself."

**Warning signs:** Agent output containing phrases like "I considered biases" or "I was aware of my position" without concrete consequence in the findings.

**Evidence:** REVIEW.md line 166: "The frame-reflexivity step (Part 4) is a suggestion, not a tested practice. It could become performative (agents writing 'I considered my biases' without actually doing so) if not grounded in specific questions about specific blind spots."

### Pitfall 3: Composition principle collapse

**What goes wrong:** When orientation obligations and subject obligations tension, the agent cleanly picks one side instead of naming the tension and navigating it. The hermeneutic principle becomes an algorithmic precedence.

**Why it happens:** LLMs default to clean resolution. "Present competing explanations AND compare against spec" is uncomfortable — cleanly resolving the tension feels more rigorous than dwelling in it.

**How to avoid:**
1. Task spec must surface anticipated tensions explicitly: "The following obligations may tension in your situation: [list]. Your job is not to resolve them before starting — it's to engage with them as you go and report how you navigated them."
2. The task spec must include the composition principle verbatim as text, not as a rule the agent is expected to abstract from
3. The output must contain a "How I navigated tensions" section whenever an investigatory or multi-subject audit runs — and if this section says "there were no tensions," that itself is a red flag to surface
4. First 2–3 investigatory audits must be reviewed for collapse — this is Q1's research program

**Warning signs:** Outputs where every tension is cleanly resolved; outputs that never mention tensions; outputs where the "How I navigated tensions" section is empty.

**Evidence:** Deliberation line 164: "If you find yourself cleanly ignoring one obligation in favor of another, you've likely stopped engaging with the tension." Confirmed as Q1 — cannot be validated pre-implementation.

### Pitfall 4: Cross-model dispatch failure — instruction degradation, env misconfig, early exit

**What goes wrong:** The command invokes `codex exec` to dispatch to GPT-5.4. The Codex CLI session: finishes early with nothing written; misconfigures environment (wrong working directory, missing config); misinterprets the task spec (treats it as a conversation not a task); dispatches under Codex's internal rule "spawn_agent only when user explicitly asks" (per `sig-2026-03-24-codex-delegation-policy-hidden-from-config.md`) and drops delegation silently.

**Why it happens:** Codex CLI has different dispatch conventions, different environment assumptions, and different instruction-following patterns than Claude Code. The instruction "read X, write Y" that works perfectly in Claude Code may be interpreted as a conversation opener in Codex.

**How to avoid:**
1. Use `codex exec "<full task spec contents>"` with the **entire task spec as the prompt** — not a path to a file the Codex session must read. The task spec file is still written for traceability, but the prompt passed to `codex exec` contains the full content.
2. Be explicit about output file path in the prompt: "Write your audit output to `<absolute path>`. Do not ask for confirmation. Proceed immediately."
3. Be explicit about working directory in the prompt and/or `cd` before the invocation.
4. Use `workspace-write` sandbox mode (as `codex-review-task-spec.md` line 5 documents).
5. Capture stdout with `tee` to a log file (as `spike-008/run-codex-otel.sh` line 38 does).
6. Check for output file existence after `codex exec` returns; report failure explicitly if missing.
7. Do NOT expect `codex exec` to produce anything other than text on stdout — file-writing happens inside the Codex session, not via return values.
8. Test dispatch in `--auto` mode on a trivially-scoped audit before shipping. If it's unreliable, fall back to `self` delegation as the default and document `cross_model` as user-interactive only.

**Warning signs:** `codex exec` completes in <5 seconds with no output file; stdout contains conversational openers ("I'd be happy to help..."); output file exists but is empty or contains a meta-response; env errors in stdout about missing config.

**Evidence:** `sig-2026-03-24-codex-delegation-policy-hidden-from-config.md` (Codex delegation rule hidden from config); deliberation line 100 ("currently fragile" — environment setup issues, agents finishing early, instructions not properly conveyed, lots of hand-holding); CONTEXT.md Q2 listed as [open]. **This is the phase's highest technical risk.**

### Pitfall 5: Ghost agent entries in model-profiles.cjs

**What goes wrong:** The planner adds `gsdr-auditor` to model-profiles.cjs but the agent file never lands (or lands at a different name), creating a new ghost entry alongside the existing `gsd-ui-auditor` and `gsd-doc-verifier` ghosts.

**Why it happens:** No automated integrity check between model-profiles.cjs entries and agent spec files.

**How to avoid:**
1. Add the model-profiles.cjs entry **after** the agent file has been written and verified to exist
2. Use the exact key `gsd-auditor` in MODEL_PROFILES (NOT `gsdr-auditor`) because `resolveModelInternal()` normalizes `gsdr-` → `gsd-` at line 1416 before lookup
3. Optionally add a cleanup note in the commit message referencing `sig-2026-04-09-ghost-agent-entries-in-model-profiles-no-spec.md` and noting that the existing ghosts (`gsd-ui-auditor`, `gsd-doc-verifier`) remain as tech debt

**Warning signs:** Plan that adds model-profiles entry without a dependency on the agent file being in place first.

**Evidence:** `sig-2026-04-09-ghost-agent-entries-in-model-profiles-no-spec.md`.

### Pitfall 6: Backward-compat frontmatter budget drift

**What goes wrong:** Adding `audit_subject`, `audit_orientation`, `audit_delegation` while preserving `audit_type` for backward compat pushes the frontmatter field count from 12 to 14. CONTEXT.md notes "acceptable but should be monitored." If further additions happen, the Phase 57.3 "no more fields than signal KB schema" pitfall returns.

**How to avoid:** Document the 14-field budget explicitly in the rewritten `audit-conventions.md` Section 2 with a clear "this is the ceiling for 57.4; any further additions require deliberation." Consider moving `ground_rules` field to an optional-status if the rewrite allows it (it was Recommended in 57.3).

**Evidence:** CONTEXT.md `<working_model>` "Frontmatter schema changes" section, [assumed:reasoned] claim on budget monitoring.

## Cross-model Dispatch — Focused Research Findings (Q2)

Because Q2 is the phase's highest technical risk, this section gathers the focused findings in one place.

### What already works (evidenced)

- [evidenced:cited] `codex exec "<prompt>"` as a shell invocation produces output and can tee to a log — verified by `.planning/spikes/008-cross-runtime-otel-and-bridge-validation/run-codex-otel.sh` line 38
- [evidenced:cited] Codex can read files, run git commands, and perform structured review tasks when given sufficient prompt detail — verified by `.planning/audits/2026-04-09-discuss-phase-exploration-quality/codex-gpt54-review.md` (6 numbered sections, substantive content, follows the prompt structure from `codex-review-task-spec.md`)
- [evidenced:cited] Codex uses `~/.codex/config.toml` for configuration; `[otel]` section is supported — verified in the spike 008 script

### What is known-broken or fragile (evidenced)

- [evidenced:cited] Codex operates under an internal rule "spawn_agent only when user explicitly asks for subagents" that is hidden from `.planning/config.json`, project roadmap, and runtime capability matrix — `sig-2026-03-24-codex-delegation-policy-hidden-from-config.md`. **Implication:** Codex may refuse to dispatch sub-agents even when the task spec tells it to. Design the task spec so the cross-model audit is executed by the top-level Codex session, not delegated further.
- [evidenced:reasoned per deliberation line 100] "cross-model dispatch works but is fragile... known problems: environment setup issues, agents finishing early, instructions not properly conveyed, lots of hand-holding." Source: `audit-taxonomy-three-axis-obligations.md` line 100.
- [evidenced:cited per retrospective §3.1] The session-log audit found 50% GPT inflation from framing effects in parallel dispatch prompts — dispatch hygiene is a real and measured concern, not theoretical.

### What does NOT exist in the codebase (negative findings)

- [evidenced:verified] No `codex exec` invocation in any command file under `commands/gsd/`. Verified via `Grep pattern='codex exec' path=commands/gsd`.
- [evidenced:verified] No `codex exec` invocation in any workflow file under `get-shit-done/workflows/`. Verified via `Grep pattern='codex exec' path=get-shit-done`.
- [evidenced:verified] No Node.js wrapper or library for Codex dispatch anywhere in `bin/` or `get-shit-done/bin/`. Verified via Grep across the bin hierarchy.
- [evidenced:verified] No existing `/gsdr:cross-model-review` command (WF-01 was deferred to Phase 62 and is now being pulled into 57.4 per CONTEXT.md).
- [evidenced:verified] The `codex-drift-audit` and `codex-gpt54-review` artifacts exist but document their dispatch only in prose — they were manually invoked by a human typing `codex exec`, not by an automated command.

### Concrete recommendation for the planner

1. **Scope cross_model dispatch tightly in 57.4**: ship it as a Bash-invocation pattern that the `/gsdr:audit` orchestrator executes when `audit_delegation: cross_model:{model_id}` is classified. Do NOT build abstraction or CLI adapter infrastructure.
2. **The task spec is the prompt**: inline the full task spec contents into the `codex exec` prompt (not a path to the file). This isolates dispatch from any "did Codex read the file correctly?" failure mode.
3. **Explicit output file path in the prompt**: the task spec tells the model where to write. The command verifies the file exists after dispatch returns.
4. **Logging is mandatory**: always `tee` the Codex stdout to a log file in the session directory. Failures must be diagnosable.
5. **Disable auto-mode for cross_model by default**: CONTEXT.md lists `--auto` as governing interactivity; for cross-model, recommend `--auto` produces a warning and requires explicit opt-in (e.g., `--auto --trust-cross-model`) or remains interactive-by-default until a spike validates auto-mode reliability.
6. **Frame Q2 as spike-worthy**: this is not an "investigate during planning" question — it is an empirical reliability question that only running a dispatch will answer. The planner should recommend a spike whose output determines whether cross_model ships as locked, experimental, or deferred in 57.4.

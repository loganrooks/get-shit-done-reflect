## Claim Justifications

### Command architecture (operational)

**[decided:reasoned] Two-file architecture: commands/gsd/audit.md + agents/gsdr-auditor.md**
- **Alternatives considered:** Single-file command (no agent), three-file (command + agent + workflow)
- **Why rejected:** Single-file can't be reused by other workflows. Three-file adds unnecessary indirection — "command IS the orchestrator" pattern works with command + agent.
- **User said:** Pre-phase archive establishes two-file architecture as a decision.

**[decided:reasoned] Orchestrator writes full task spec, agent executes**
- **Alternatives considered:** Agent infers its own scope, shared config file
- **Why rejected:** Task spec preservation is the strongest empirical finding from 57.3 research. Agent inferring scope loses the audit trail. Follows spike-runner pattern.
- **User said:** Pre-phase archive: "spike-runner pattern. The command does discovery, type inference, ground rule selection, task spec generation."

**[decided:reasoned] Type inference from context**
- **Alternatives considered:** Forced type selection menu, always ask
- **Why rejected:** User invokes mid-situation with natural language. Forced selection adds friction to a moment where the user wants help, not menus.
- **User said:** Pre-phase archive: "infer from context, not force selection"

### The investigatory type (epistemic)

**[decided:reasoned] Add investigatory as 10th named type**
- **Alternatives considered:** 4th family, mode modifier, fold into exploratory
- **Why rejected:** Family reinforces family taxonomy as primary (part of the problem REVIEW.md identifies). Mode requires infrastructure changes beyond v1. Folding into exploratory loses the breakdown-vs-possibility distinction.
- **User said:** "Don't just naively add a type — investigate the situation in which the audit is demanded."

**[decided:reasoned] Ground rules I1-I4**
- **Alternatives considered:** Core rules only, custom rules per investigation
- **Why rejected:** Core rules alone don't capture the investigatory stance (holding diagnosis open, competing explanations, naming position). Custom-per-investigation is too unstructured for an agent spec.
- **User said:** Pre-phase archive REVIEW.md Part 3 proposes I1-I4 with full philosophical justification.

**[open] Whether investigatory is type, family, or mode**
- **What's been tried:** REVIEW.md Part 5 analyzes all three options
- **Why unresolved:** The right answer depends on usage patterns that don't exist yet
- **Research delegation:** Examine how other GSD taxonomies handle cross-cutting concerns. Consider whether mode infrastructure would be complex to build.

**[assumed:reasoned] Investigatory vs. exploratory distinction is one of occasion**
- **Challenge protocol:** Would falsify if usage shows investigatory and exploratory being invoked interchangeably with no difference in starting condition
- **Evidence checked:** REVIEW.md Part 5 provides Heideggerian analysis (breakdown vs. possibility)
- **Why reasonable:** The body templates differ because starting conditions differ; distinction may soften with use

### Frame-reflexivity

**[decided:reasoned] Frame-reflexivity for ALL audit types**
- **Alternatives considered:** Investigatory only, optional per type
- **Why rejected:** REVIEW.md Part 4 argues the concern (types smuggle assumptions) applies beyond investigatory — every type presupposes a form of failure
- **User said:** "We need a newfound critical eye towards our framing, our perhaps implicit assumptions that we need to expose."

**[assumed:reasoned] Rule 5 alongside Rule 4**
- **Challenge protocol:** Would falsify if frame-reflexivity and escape hatch are consistently conflated in practice (agents treat them identically)
- **Evidence checked:** REVIEW.md Part 4 distinguishes them: Rule 4 catches excess within frame, frame-reflexivity catches whether frame is right
- **Why reasonable:** The distinction is clear in principle; whether agents maintain it in practice is the concern (guardrail G-3)

**[assumed:reasoned] Lightweight for non-investigatory, full section for investigatory**
- **Challenge protocol:** Would falsify if lightweight treatment (1-2 sentences) consistently produces empty boilerplate rather than genuine reflection
- **Evidence checked:** No direct evidence — REVIEW.md acknowledges performativity risk
- **Why reasonable:** Starting lightweight is conservative; can deepen if shallow treatment proves insufficient

### Agent design

**[decided:reasoned] Tool access: Read, Bash, Grep, Glob, Write**
- **Alternatives considered:** Read-only (no Write), full tool access (including Agent/Task)
- **Why rejected:** Read-only can't produce output files. Full access with Agent/Task is unnecessary — auditor is the leaf agent.

**[open] Model tier for gsdr-auditor**
- **What's been tried:** Surveyed that existing checker agents use various tiers
- **Why unresolved:** Cost/quality tradeoff depends on audit type, and orchestrator can override
- **Research delegation:** Check model-profiles.cjs patterns for similar judgment-heavy agents

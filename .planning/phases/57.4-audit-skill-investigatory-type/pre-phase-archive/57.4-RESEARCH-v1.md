## Summary

Phase 57.4 has two interlocked deliverables: an invocable `/gsdr:audit` command + `gsdr-auditor` agent pair (operational), and an updated audit type taxonomy with the `investigatory` type and frame-reflexivity practice (epistemic). The operational work follows established GSD patterns precisely — every decision is already made in CONTEXT.md and the reference infrastructure exists in audit-conventions.md and audit-ground-rules.md. The epistemic work touches those reference files directly: adding Section 3.4 (I1-I4), adding Rule 5 (frame-reflexivity), adding the `investigatory` type and body template.

Research investigated two open questions: (1) whether `investigatory` is better implemented as a type, family, or mode, and (2) what model tier `gsdr-auditor` should default to. The first question finds that a named type is the right v1 choice — the infrastructure cost of mode modifiers is not justified before there is usage evidence. The second question finds that `sonnet` is the appropriate quality-profile default for `gsdr-auditor`, consistent with all other audit/verification agents in the system, with the orchestrator retaining option to escalate for investigatory audits.

There is also a critical implementation detail: `gsdr-auditor` is a **fork-specific agent** (`gsdr-` prefix), not a upstream `gsd-` agent. The only existing `gsdr-` agent (`gsdr-context-checker`) does NOT have a `model-profiles.cjs` entry — it declares `model: sonnet` in its spec frontmatter. The same approach is correct for `gsdr-auditor`. The command resolves the agent with `Task(agent="gsdr-auditor", ...)` without `resolve-model`.

**Primary recommendation:** Build command + agent as a two-file pair. Update audit-conventions.md and audit-ground-rules.md in the same phase. Keep model declaration in agent spec frontmatter (`model: sonnet`), not in model-profiles.cjs. The orchestrator handles type escalation via task spec content, not model switching.

## Architecture Patterns

### Pattern 1: Command IS the Orchestrator

[evidenced:cited] `commands/gsd/audit-milestone.md` line 16: "This command IS the orchestrator." The command reads context, makes decisions, and dispatches to an agent. The agent executes a fully-formed spec.

**How the `/gsdr:audit` command flows:**

```
1. Parse $ARGUMENTS (optional scope/type hints, --auto flag)
2. Read conversation context for type inference clues
3. If context ambiguous: ask 0-2 clarifying questions
4. Infer audit_type from context clues (type inference table)
5. Create session directory: .planning/audits/YYYY-MM-DD-{slug}/
6. Build task spec (ground rules + body template + scope + target)
7. Write task spec to {slug}-task-spec.md
8. Dispatch: Task(agent="gsdr-auditor", model=..., input=task_spec_content)
9. Write output to {slug}-output.md
10. Commit per commit_docs setting
```

**Key principle:** The command writes ground rules into the task spec. Per audit-ground-rules.md Section 4: "the agent needs the rules in its context window, not a pointer to them."

### Pattern 2: Agent Receives Fully-Formed Spec

[evidenced:cited] `agents/gsd-spike-runner.md` role: "You do NOT handle the Design phase. The orchestrator that spawns you has already created DESIGN.md... Your job starts with a completed DESIGN.md."

For `gsdr-auditor`, this means: the agent receives ground rules + body template + scope in the task spec. It does NOT read audit-conventions.md or audit-ground-rules.md itself. The orchestrator has already selected and embedded everything. The agent focuses entirely on executing the audit.

### Pattern 3: Agent Spec Embeds Epistemic Stance

[evidenced:cited] `agents/gsdr-context-checker.md` role section: the agent spec explains WHY claim types matter, not just WHAT to do. It says "Critical mindset: You are not a rubber stamp."

For `gsdr-auditor`, the agent spec should explain WHY ground rules matter (citing the evidence from audit-ground-rules.md Section 1: same agent class + different task spec = dramatically different output quality). The Rules 4 and 5 sections should appear as named output sections, not optional afterthoughts.

### Pattern 4: gsdr- Agents Declare Model in Frontmatter

[evidenced:cited] `agents/gsdr-context-checker.md` frontmatter: `model: sonnet`. This agent is NOT in `model-profiles.cjs` (verified: grep returns no results for `gsdr-context-checker` in model-profiles.cjs). The `discuss-phase.md` workflow dispatches it with `Task(agent="gsdr-context-checker", ...)` without a `resolve-model` call.

[assumed:reasoned] The `gsdr-` prefix signals a fork-specific agent that lives outside the upstream model-profiles machinery. The model is declared in the spec frontmatter. This is the correct pattern for `gsdr-auditor`.

[evidenced:cited] `commands/gsd/deliberate.md` Step 1 (New mode): "If no arguments: check the current conversation for context first. The user may have been discussing something that naturally led here." This is the canonical GSD pattern for context-first inference.

[evidenced:cited] `audit-conventions.md` Section 1: `.planning/audits/YYYY-MM-DD-{slug}/` with `{slug}-task-spec.md` and `{slug}-output.md`.

### Anti-Patterns to Avoid

- **Ground rules by reference:** Never write "follow the rules in audit-ground-rules.md" in a task spec. Copy them.
- **Agent does type selection:** The command selects type; the agent executes a typed spec.
- **model-profiles.cjs entry for gsdr- agents:** The gsdr-context-checker precedent shows the correct pattern is frontmatter declaration.
- **Frame-reflexivity as boilerplate:** The agent spec must ground Rule 5 in specific questions about specific blind spots. "I considered my framing" without actual engagement is compliance theater per REVIEW.md Part 6 and guardrail G-3.
- **Investigatory as type creep:** Adding investigatory requires updating the type inference table AND the ground rule selection table in Section 4. Both must be updated or type inference and ground rule selection will be inconsistent.

## Open Questions — Resolved

### Q1: Is investigatory a type, family, or mode?

**Research finding:** [assumed:reasoned] Add as a named type (v1). Do not add a 4th family. Do not implement mode modifier infrastructure.

**Evidence chain:**

1. The existing 3-family clustering (structural/epistemic/compliance) is itself "interpretive, not observed" per audit-ground-rules.md Section 3 header comment. Adding a 4th family would be as provisional as the first 3.

2. The infrastructure cost of a mode modifier is non-trivial: type inference would need to produce `{base_type}+investigatory` compounds, ground rule selection would need to compose rather than switch, the body template would need to blend two families. All of this before a single real investigatory audit exists.

3. REVIEW.md Part 5 is explicit: "Whether it's a type, a family, or a mode is a design question that should be resolved by use, not by theory." There is one incident grounding the investigatory type. One incident does not constitute "use data."

4. CONTEXT.md Q1 reversibility assessment: LOW — "If later usage reveals it should be a mode, the change is: update type inference to allow mode modifiers, update ground rule selection to compose rather than switch. The named type continues to work regardless."

5. [assumed:reasoned] The "4th family" framing is seductive because structural/epistemic/compliance form a tidy set. But investigatory is better understood as orthogonal to that axis: it can investigate structural failures, epistemic failures, or compliance failures. This is evidence FOR mode/modifier eventually — but that evolution should happen after usage reveals cross-family investigatory audits, not as v1 speculation.

**Decision:** Add as the 9th named type alongside `exploratory`. Family field: use `investigatory` as a new single-member family for now. This preserves the family field's schema without forcing a premature taxonomy consolidation.

**Ground rule selection table entry:**

| Audit Type | Ground Rule Set | Extensions |
|------------|-----------------|------------|
| `investigatory` | `core+investigatory` | Rules I1, I2, I3, I4, Rule 5 (frame-reflexivity is mandatory for investigatory) |

### Q2: What model tier should gsdr-auditor default to?

**Research finding:** [assumed:reasoned] `model: sonnet` in agent spec frontmatter, NOT in model-profiles.cjs.

**Evidence chain:**

1. [evidenced:cited] `gsdr-context-checker.md` frontmatter declares `model: sonnet` and has no model-profiles.cjs entry. This is the only existing `gsdr-` agent and is the direct pattern to follow.

2. [evidenced:cited] All audit/verification agents in model-profiles.cjs use sonnet at quality profile: `gsd-verifier`, `gsd-nyquist-auditor`, `gsd-integration-checker`, `gsd-plan-checker`, `gsd-ui-auditor`, `gsd-doc-verifier` — all return `sonnet` at quality profile. Only `gsd-planner`, `gsd-debugger`, and `gsd-advisor` get `inherit` (opus at quality).

3. [assumed:reasoned] The distinction in (2) is: planners and debuggers make architectural decisions that affect the codebase trajectory. Auditors/verifiers make assessment decisions that flag problems but don't decide solutions. Investigatory audits are judgment-heavy compared to structural audits, but the agent is still an assessor, not a decision-maker. The judgment lives in the task spec design (which the orchestrator controls), not in the agent's reasoning capacity.

4. [assumed:reasoned] The CONTEXT.md working model says "the orchestrator can decide whether to escalate to a higher model tier based on audit type/complexity." If the command passes `model=inherit` to the Task call for investigatory audits, this achieves type-conditional escalation without adding per-audit-type entries to model-profiles.cjs. The spec's `model: sonnet` is the floor; the command can override upward.

**Decision:** `model: sonnet` in agent spec frontmatter. Command dispatches with `model=inherit` for investigatory/epistemic audits (gets opus at quality profile), `model=sonnet` explicit for structural audits. No model-profiles.cjs entry required. This is cleaner than two entries and consistent with the gsdr- agent pattern.

## Common Pitfalls

### Pitfall 1: Writing Rule 5 as Decoration

**What goes wrong:** The agent writes "I considered whether this audit type was appropriate" as a boilerplate paragraph without actually doing so.

**Why it happens:** Rule 5 asks a self-referential question. LLMs are trained to produce helpful-sounding text. Without grounding in specific questions, the output will be generic.

**How to avoid:** The agent spec should provide concrete prompts for Rule 5:
- "What assumptions did this audit type make about the form of the problem?"
- "What would a `[different_type]` audit have looked for instead?"
- "Did the body template force any findings into a shape they didn't actually have?"

**Warning signs:** Rule 5 output is two sentences and contains phrases like "I attempted to remain objective" or "I considered multiple perspectives" without naming any specific type-induced blind spot.

### Pitfall 2: Investigatory Audit Collapses to a Single Explanation

**What goes wrong:** Ground rule I3 requires competing explanations. If the investigation finds strong evidence for one explanation, the auditor may skip presenting alternatives.

**Why it happens:** LLMs prefer coherent narratives. Presenting a competing explanation that is later dismissed feels redundant.

**How to avoid:** The agent spec should be explicit: I3 is not satisfied by "the obvious explanation is X, though one could imagine Y." It requires genuinely presenting Y with its supporting evidence before distinguishing them.

**Warning signs:** The "Competing Explanations" section has one real explanation and one strawman, or the explanations are presented asymmetrically (evidence for A, handwave for B).

### Pitfall 3: Type Inference Closes Too Early

**What goes wrong:** The command reads the first context clue and selects a type without checking for investigatory clues that appear later in the conversation.

**Why it happens:** Context inference scans are typically first-match.

**How to avoid:** Investigatory cues ("something went wrong", "doesn't match what I expected") should override structural/epistemic/compliance cues if they appear in the same conversation. The user's last expression of intent is more relevant than earlier framing.

### Pitfall 4: Task Spec Missing the Scope Target

**What goes wrong:** The command writes ground rules and body template but fails to include a specific description of what to audit.

**Why it happens:** The spec template is generated mechanically; the scope description requires reading the conversation.

**How to avoid:** The task spec structure should always include a "## What to Audit" or "## Scope and Target" section that is conversation-specific, not template-derived.

### Pitfall 5: Installer Not Run After Adding New Command/Agent Files

**What goes wrong:** New files in `commands/gsd/` and `agents/` are not reflected in `.claude/` until `node bin/install.js --local` is run.

**Why it happens:** The dual-directory architecture (CLAUDE.md) requires manual reinstall. New contributors and agents forget this step.

**How to avoid:** Verification must include `node bin/install.js --local` and confirm the new files appear under `.claude/`.

**Warning signs:** `audit.md` and `gsdr-auditor.md` exist in `commands/gsd/` and `agents/` but NOT in `.claude/get-shit-done-reflect/commands/gsd/` and `.claude/get-shit-done-reflect/agents/`.

### New Section 3.4 in audit-ground-rules.md

```markdown
### 3.4 Investigatory Audits

**Applies to:** `investigatory`

**Rule I1: Start from the situation, not from a theory.**
Describe what was expected, what was delivered, and why those expectations are being treated as the standard. The choice of comparison point is already an interpretive act. The gap between expectation and delivery is a starting orientation, not a neutral fact.

**Rule I2: Let the investigation guide artifact selection.**
Don't mandate which artifacts to read in advance. Follow the evidence. If the discrepancy points to the planning stage, read planning artifacts. If it points to requirements, read requirements. The artifact chain is a finding, not an input.

**Rule I3: Present competing explanations.**
For each finding, offer at least two interpretations. Do not collapse to one. "The research narrowed scope" could mean the researcher made an error, OR the researcher correctly applied constraints the investigator hasn't understood yet.

**Rule I4: Name the position of the investigation.**
Every investigation is conducted from somewhere — with particular attunements, particular things it's prepared to notice, particular things it isn't. What would a differently-situated investigation attend to?
```

### Rule 5 Addition to audit-ground-rules.md Section 2

After Rule 4 (escape hatch), add:

```markdown
### Rule 5: Frame-Reflexivity (Always Included)

Did the choice of audit type shape what you found? What would a differently-typed audit have looked for? What does this type make easy to see, and what might it make hard to see?

This is distinct from Rule 4 (which catches excess within the frame). Rule 5 catches whether the frame itself was appropriate.

For **investigatory audits**: this is the "Position of This Investigation" section — a full section is expected.

For **all other audit types**: a sentence or two as a closing reflection. The question is genuine, not rhetorical. If the answer is "no, this type was correct" — say why.
```

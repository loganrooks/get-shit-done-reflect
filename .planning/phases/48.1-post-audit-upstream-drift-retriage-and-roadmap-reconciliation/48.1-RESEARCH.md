## Summary

This research investigated the live upstream drift between the fork's audited baseline (`v1.22.4`, tag `a143ba0`) and the current `upstream/main` head (`60fda20`). The delta is now **372 commits** spanning 6 released tags (v1.23.0 through v1.28.0) plus 31 unreleased commits on main. Of the 372 total commits, **228 touch current-milestone-relevant file surfaces** (bin/install.js, get-shit-done/bin/lib/, hooks/, tests/).

The core finding is that **Assumption A2 from CONTEXT.md holds**: the majority of must-consider post-baseline changes cluster naturally into Phases 49 (config migration), 50 (migration test hardening), 51 (update system hardening), and 54 (infrastructure). No wholesale roadmap replacement is needed. However, upstream's `core.cjs` has grown from 492 lines to 1,168 lines since v1.22.4, and the fork's adopted copy is frozen at 530 lines (v1.22.4 plus fork additions). Several upstream additions to `core.cjs` -- worktree-aware planning resolution, file locking, findProjectRoot bug fix, planningPaths helper, commit_docs gitignore auto-detection -- represent infrastructure improvements that should be adopted during the config migration and test hardening phases rather than ignored.

**Assumption A3 also holds**: post-baseline changes touch some Phase 45-48 surfaces (core.cjs, init.cjs, frontmatter.cjs) but do NOT invalidate the modular architecture decisions. The fork's additions (parseIncludeFlag, loadManifest, loadProjectConfig, atomicWriteJson, fork module extensions) have zero overlap with the upstream additions. The correct response is targeted module update during Phase 49-51, not Phase 45-48 reopening.

**Primary recommendation:** Produce a dated classification ledger as a governance artifact inside this phase directory, update ROADMAP.md and STATE.md to reflect routing decisions, and let Phase 49 planning consume the ledger as its upstream-awareness input. No additional inserted phases are needed.

### Anti-Patterns to Avoid
- **Silent absorption:** Adopting upstream changes without recording them in the ledger. Every adopt/fold must be traceable.
- **Anxiety-driven reopening:** Reopening Phase 45-48 because upstream changed the same files. Reopening requires evidence of actual conflict or regression, not just file overlap.
- **Scope creep through triage:** Using "must-integrate-now" as a back door to expand v1.18 scope. The bar should be: does this change materially block or degrade an open phase's planned work?

## Common Pitfalls

### Pitfall 1: Conflating File Overlap with Architectural Conflict
**What goes wrong:** Seeing that upstream changed core.cjs (which our fork also modified) and concluding that Phase 46-48 work is invalidated.
**Why it happens:** File-level overlap is easy to spot; semantic conflict analysis requires reading the actual changes.
**How to avoid:** For each overlapping file, check whether the upstream changes touch the same functions or code regions as the fork's modifications. In this case, fork additions (parseIncludeFlag, loadManifest, etc.) are appended at the end of core.cjs and do not overlap with upstream additions (resolveWorktreeRoot, withPlanningLock, planningPaths, findProjectRoot fixes).
**Warning signs:** A triage decision that says "reopen Phase 46" without citing a specific function-level conflict.

### Pitfall 2: Treating Unreleased Main-Only Changes as Released
**What goes wrong:** Planning around upstream changes that may be reverted or amended before the next tag.
**Why it happens:** `upstream/main` includes 31 commits past the latest release (v1.28.0).
**How to avoid:** Mark each classification entry as "released in vX.Y.Z" or "main-only (unreleased)". Weight released changes higher in routing decisions.
**Warning signs:** A "must-integrate-now" classification citing only main-only commits without acknowledging their pre-release status.

### Pitfall 3: Scope Creep Via Feature Adoption
**What goes wrong:** The triage phase expands v1.18 scope by classifying new upstream features (workstreams, forensics, milestone-summary, Windsurf) as "must-integrate-now".
**Why it happens:** New features look valuable, and the proximity of the triage exercise creates urgency that does not exist.
**How to avoid:** Apply the bar strictly: does the change materially block or degrade an already-open v1.18 phase? New features that do not affect open-phase work are `candidate-next-milestone`, not `must-integrate-now`.
**Warning signs:** More than 2-3 items in the "must-integrate-now" bucket.

### Pitfall 4: Omission as Implicit Deferral
**What goes wrong:** Upstream changes not mentioned in the ledger are silently deferred without recorded rationale.
**Why it happens:** The commit volume (372 commits) makes exhaustive treatment feel impractical.
**How to avoid:** Group changes by surface area or feature cluster, not individual commits. Classify the cluster, not each commit. The ledger should have rows for all material clusters, including those deferred.
**Warning signs:** A downstream planner encountering an upstream change they were not warned about.

### Phase 45-48 Impact Assessment (Q3 from CONTEXT.md)

**Finding: No reopening required.**

Evidence examined:
1. **core.cjs overlap:** Fork added 4 helper functions (parseIncludeFlag, loadManifest, loadProjectConfig, atomicWriteJson) appended after the upstream exports block. Upstream added resolveWorktreeRoot, withPlanningLock, planningPaths, and findProjectRoot improvements. These are in different code regions with no function-name or semantic overlap. The fork's modular architecture from Phase 46 is not affected.

2. **init.cjs overlap:** Fork extended init.cjs in Phase 48 with --include flag support and fork-specific init function modifications. Upstream added workspace commands, text_mode support, reviews flag, and other features. Fork extensions are appended using the `module.exports.funcName` pattern. No conflict.

3. **frontmatter.cjs overlap:** Fork extended with signal schema validation in Phase 48. Upstream had 4 commits changing frontmatter.cjs but these affect different functions (status preservation, one-liner extraction, CRLF recovery). No conflict.

4. **gsd-tools.cjs router:** Fork reduced gsd-tools.cjs to a pure CLI router in Phase 48. Upstream's gsd-tools.cjs has grown with worktree wiring. Since the fork's router pattern does not inline functions, the worktree-root wiring from upstream can be adopted as a small addition to the router without architectural impact.

**Conclusion:** The modular extraction and extension work from Phase 45-48 remains architecturally sound. The upstream additions are orthogonal, not contradictory. They should be folded into Phase 49-51 module updates, not treated as Phase 45-48 rework.

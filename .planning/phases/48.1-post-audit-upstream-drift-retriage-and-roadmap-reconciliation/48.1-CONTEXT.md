# Phase 48.1: Post-audit upstream drift retriage and roadmap reconciliation - Context

**Gathered:** 2026-03-24
**Status:** Ready for research

<domain>
## Phase Boundary

Produce the steering artifacts and routing decisions that make live upstream
drift explicit after the `v1.22.4` audit baseline. This phase exists to record:

- what changed upstream since the audit baseline
- which of those changes materially affect still-open v1.18 phases or already
  landed Phase 45-48 surfaces
- what is in-scope now, what should be folded into existing open phases, what
  becomes next-milestone candidate work, and what is explicitly deferred

This phase updates project memory and roadmap authority. It does **not** itself
implement the post-baseline upstream changes unless the retriage concludes a new
inserted phase or explicit routing change is required.

**What this phase does NOT include:**
- Wholesale retargeting v1.18 to latest `upstream/main`
- Silent deferral of all post-baseline upstream work to the next milestone
- Implementing config migration, update hardening, or feature adoption work
  directly (those remain Phase 49+ concerns unless explicitly rerouted)
- Reopening Phase 45-48 implementation without concrete evidence that a
  post-baseline upstream change materially alters already-landed surfaces

</domain>

<assumptions>
## Working Model & Assumptions

**A1. A bounded retriage is sufficient; a full new fork audit is probably not.**
The existing 10-report fork audit already established the baseline up through
`v1.22.4`. Phase 48.1 likely only needs a targeted post-baseline analysis on
surfaces relevant to open v1.18 work and recently landed modular-runtime work.
- *Validate by:* Checking whether a filtered upstream diff/log across
  `bin/install.js`, `get-shit-done/bin/lib/`, `get-shit-done/workflows/`,
  `hooks/`, and tests covers the meaningful current-milestone drift.
- *Falsified if:* Important current-phase implications are found outside that
  filtered surface and require a broader re-audit.

**A2. Most relevant post-baseline changes will route into Phases 49-51 or 54, not require wholesale roadmap replacement.**
The strongest live drift cluster is around install authority, project-root and
worktree resolution, non-Claude runtime behavior, config preservation, and hook
safety. Those map naturally onto the already-open migration/update/governance
phases.
- *Validate by:* Classifying the live upstream deltas and checking whether the
  majority fit existing open phase boundaries.
- *Falsified if:* Several critical deltas form a coherent new work package that
  does not fit 49-51/54 and demands additional inserted phases.

**A3. Some post-baseline changes may touch Phase 45-48 surfaces without requiring those phases to be reopened.**
Path, worktree, config, or runtime fixes in newer upstream code may affect the
same modules that were reconciled in Phases 45-48, but the correct response may
be targeted follow-on routing rather than revisiting completed modularization.
- *Validate by:* Comparing the relevant post-baseline upstream deltas against
  the modular-runtime surfaces landed in Phase 45-48 and checking whether the
  issue is architectural, behavioral, or merely additive.
- *Falsified if:* A post-baseline upstream change directly invalidates a
  completed Phase 45-48 decision or reveals a current regression in landed code.

</assumptions>

<decisions>
## Implementation Decisions

### Triage frame
- Keep `v1.22.4` as the v1.18 audit baseline while analyzing newer upstream
  movement against it.
- Treat this phase as a routing-and-authority phase, not a stealth "sync to
  latest" implementation phase.
- Use explicit classification buckets for newer upstream changes:
  `must-integrate-now`, `fold-into-open-phase`, `candidate-next-milestone`, and
  `defer`.

### Priority order
- Triage post-baseline upstream changes in this order:
  1. Changes that materially affect still-open Phase 49-51 or 54 work
  2. Changes that may invalidate or pressure already-landed Phase 45-48 runtime
     surfaces
  3. Broader feature additions and UX expansions that appear valuable but are
     not clearly current-milestone-critical

### Output expectation
- The result of this phase must give downstream planners a canonical, dated view
  of live upstream drift and its routing decisions before Phase 49 planning
  begins.
- If a newer upstream change belongs in current v1.18 scope, this phase must
  record where it lands in the roadmap rather than leaving it implicit.

### Claude's Discretion
- Exact artifact shape for the canonical drift snapshot and classification
  matrix, provided it is dated, auditable, and easy for downstream planners to
  consume
- Whether to organize upstream changes by commit clusters, affected files, or
  affected phases, provided the final output still supports explicit routing
  decisions
- Whether the canonical classification lives primarily in a new 48.1
  `RESEARCH.md` or a closely linked governance-style note, provided ROADMAP.md
  and STATE.md end up reflecting the outcome clearly

</decisions>

<questions>
## Open Questions

### Q1. Which post-baseline upstream changes are `must-integrate-now` for current v1.18?
- **Type:** material
- **Why it matters:** This determines whether Phase 49-51 or 54 should change
  before planning proceeds.
- **Downstream decision affected:** Whether existing open phases are sufficient
  as written or need immediate scope adjustment.
- **Reversibility:** Medium. Misclassifying a change as defer/candidate can
  poison later planning; misclassifying too much as current work can bloat the
  milestone.
- **Research should investigate:** Recent upstream changes around config
  preservation, project-root/worktree resolution, `$HOME` path handling,
  non-Claude model resolution, hook field validation, and Windows shell/stdin
  safety.

### Q2. Do any post-baseline changes require additional roadmap surgery beyond the current 48.1 insertion?
- **Type:** efficient
- **Why it matters:** Some changes may fit neatly into existing phases, while
  others may form a coherent inserted work package of their own.
- **Downstream decision affected:** Whether 48.1 only patches routing or also
  causes another inserted phase or stronger phase-boundary rewrite.
- **Reversibility:** Medium.
- **Research should investigate:** Whether the must-integrate-now set clusters
  naturally into Phase 49, 50, 51, or 54, or whether there is a cross-cutting
  slice that cannot be responsibly absorbed there.

### Q3. Do any post-baseline upstream changes require reopening or patching already-landed Phase 45-48 surfaces?
- **Type:** formal
- **Why it matters:** The milestone currently treats Phase 45-48 as complete and
  verified; reopening them should require evidence, not anxiety.
- **Downstream decision affected:** Whether remediation belongs in existing open
  phases, a new inserted phase, or not at all.
- **Reversibility:** Medium.
- **Research should investigate:** Post-`v1.22.4` upstream changes in
  `bin/install.js`, `bin/lib/*.cjs`, hooks, and workflows that overlap the
  modular-runtime surfaces recently landed here.

### Q4. What should be the canonical artifact for the dated drift snapshot and classification ledger?
- **Type:** formal
- **Why it matters:** Future planners need one place to look, not a scattered
  set of notes across chat, roadmap lines, and git commands.
- **Downstream decision affected:** How 48.1 outputs are consumed by later
  `discuss-phase`, `plan-phase`, and governance updates.
- **Reversibility:** High.
- **Research should investigate:** Whether the canonical record should live in
  `48.1-RESEARCH.md`, a dedicated note under `.planning/research/` or
  `.planning/governance/`, or a hybrid with ROADMAP/STATE pointers.

### Q5. What is the right comparison target for routing decisions: latest upstream tag, live `upstream/main`, or both?
- **Type:** formal
- **Why it matters:** The roadmap currently records both `v1.28.0` and
  `upstream/main` positions. Routing should distinguish released upstream
  behavior from unreleased head changes.
- **Downstream decision affected:** Which changes are treated as current
  candidates versus watchlist material.
- **Reversibility:** High.
- **Research should investigate:** Whether classification should explicitly mark
  items as `released-upstream` vs `main-only`.

</questions>

<guardrails>
## Epistemic Guardrails

- Verify live upstream state from git during research; the commit counts in the
  roadmap are point-in-time and may drift again before planning finishes.
- Do not infer relevance from commit count alone. A large upstream delta does
  not justify automatic current-milestone expansion.
- Distinguish between:
  - released upstream changes
  - unreleased `upstream/main` changes
  - changes that are relevant to open v1.18 phase surfaces
  - changes that are merely interesting future capabilities
- Every `must-integrate-now` or `fold-into-open-phase` classification should
  cite specific upstream commits, files, or artifact evidence.
- Do not reopen Phase 45-48 simply because newer upstream code exists. Reopening
  requires concrete evidence of conflict, regression risk, or changed substrate
  assumptions.
- Do not let 48.1 collapse into a second full fork audit. This phase is about
  targeted retriage and explicit routing.
- If a change is deferred, record the reason for deferment explicitly; omission
  is not a valid defer strategy.

</guardrails>

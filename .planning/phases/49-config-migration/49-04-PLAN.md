**Tests to add (new describe block: "manifest apply-migration rename migrations"):**

1. **"renames depth to granularity with value mapping"**
   - Config: `{ mode: 'yolo', depth: 'comprehensive', manifest_version: 1 }` with manifest having the rename migration + health_check feature
   - Run `manifest apply-migration --raw`
   - Assert: output.changes includes `{ type: 'field_renamed', from: 'depth', to: 'granularity', old_value: 'comprehensive', new_value: 'fine' }`
   - Assert: config on disk has `granularity: 'fine'` and NO `depth` key

2. **"renames depth to granularity with standard value (passthrough)"**
   - Config: `{ mode: 'yolo', depth: 'standard', manifest_version: 1 }`
   - Run apply-migration
   - Assert: config has `granularity: 'standard'`, no `depth`

3. **"passes through unknown depth value without destroying it"**
   - Config: `{ mode: 'yolo', depth: 'custom_value', manifest_version: 1 }`
   - Run apply-migration
   - Assert: config has `granularity: 'custom_value'`, no `depth`
   - (This tests Pitfall 5 from research: unknown values pass through)

4. **"cleans up depth when both depth and granularity exist"**
   - Config: `{ mode: 'yolo', depth: 'comprehensive', granularity: 'fine', manifest_version: 1 }`
   - Run apply-migration
   - Assert: config has `granularity: 'fine'` (preserved), NO `depth` key
   - Assert: changes array does NOT include a field_renamed entry (rename skipped, only cleanup)

5. **"skips rename when granularity already set and depth absent"**
   - Config: `{ mode: 'yolo', granularity: 'fine', manifest_version: 1 }`
   - Run apply-migration
   - Assert: no field_renamed change in output
   - Assert: config still has `granularity: 'fine'`

6. **"preserves unknown config fields through migration" (CFG-06)**
   - Config: `{ mode: 'yolo', depth: 'standard', my_custom_field: 'preserved', another_field: { nested: true }, manifest_version: 1 }`
   - Run apply-migration
   - Assert: config has `my_custom_field: 'preserved'` and `another_field: { nested: true }` after migration
   - Assert: depth was renamed to granularity AND custom fields survived

7. **"is idempotent: second run produces zero changes"**
   - Config: `{ mode: 'yolo', depth: 'comprehensive', manifest_version: 1 }` with full manifest (health_check feature + rename migration)
   - Run apply-migration once (produces changes)
   - Run apply-migration AGAIN on the same config
   - Assert: second run output has `total_changes: 0`

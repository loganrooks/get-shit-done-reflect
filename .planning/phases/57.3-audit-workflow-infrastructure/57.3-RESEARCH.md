## Summary

Phase 57.3 establishes conventions, templates, and metadata standards for audit workflows. The domain is primarily organizational (directory structure, frontmatter schemas, template design, ground rule reference documents) rather than library/framework-oriented. Research investigated five areas: (1) how the existing signal KB schema can be adapted for audit frontmatter, (2) what the current audit artifact landscape looks like for migration planning, (3) whether the audit taxonomy should be an enum or open tags, (4) what type-specific ground rules are worth formalizing, and (5) how workflow-produced audits should relate to the audit directory.

The codebase already has strong precedent for the conventions needed. The signal KB schema (YAML frontmatter with `signal_type` enum + `custom` escape hatch) provides a tested model for audit type classification. The `gsdr-context-checker.md` agent spec demonstrates what ground rules look like in practice. The `verification-report.md` template shows the pattern for structured audit output templates. The gold-standard audit session (`2026-04-09-discuss-phase-exploration-quality/`) demonstrates the directory pattern: dated directory containing task specs alongside outputs.

The existing audit landscape is messy but well-characterized. There are approximately 43 audit artifacts across 4 locations: 18 bare files + 2 dated directories + 7 claim-audit files in `.planning/audits/`, 10 fork-audit files in `.planning/fork-audit/`, 5 root-level audit files + 4 sensor trial files in `.planning/` root, plus 63 VERIFICATION.md files and 7 MILESTONE-AUDIT.md files produced by workflows in phase directories and `.planning/milestones/`.

**Primary recommendation:** Model the audit frontmatter schema after the signal KB schema (Section 4 of `knowledge-store.md`), use an enum-with-escape-hatch pattern for `audit_type` (matching the `signal_type` pattern that has proven effective with 1.9% custom usage), create a ground rules reference document (modeled on `claim-types.md`), and migrate existing artifacts conservatively with frontmatter addition only.

## Architecture Patterns

### Recommended Audit Directory Structure
```
.planning/audits/
├── YYYY-MM-DD-{slug}/              # Multi-file audit sessions
│   ├── {slug}-task-spec.md         # Task spec given to agent
│   ├── {slug}-output.md            # Agent output
│   ├── {slug}-2-task-spec.md       # Follow-up task spec (if chain)
│   ├── {slug}-2-output.md          # Follow-up output
│   └── ...
├── YYYY-MM-DD-{slug}.md            # Single-file audits (lightweight)
└── ...
```

### Pattern 1: Three-Layer Audit Schema
**What:** Shared metadata (strict) + type-aware body (structured) + escape hatch (open)
**When to use:** Every audit artifact

**Layer 1 -- Shared frontmatter (required for all audits):**
```yaml
---
date: 2026-04-09
audit_type: comparative_quality   # enum or "exploratory" escape hatch
auditor_model: claude-sonnet-4-6
scope: "discuss-phase CONTEXT.md quality across Phases 52-57"
triggered_by: "deliberation: exploratory-discuss-phase-quality-regression.md"
task_spec: rigorous-comparative-audit-task-spec.md  # relative path within session dir
ground_rules: core  # which ground rule set was applied: core | core+structural | core+epistemic | custom
output_files:
  - rigorous-comparative-audit.md
predecessor_audits: []  # chain of prior audits on same question
tags: [discuss-phase, context-quality, epistemic-rigor]
---
```

**Layer 2 -- Type-aware body templates (per audit type):**
Different audit types get different suggested body structures. Templates are guides, not straitjackets.

**Layer 3 -- Escape hatch:**
`audit_type: exploratory` for audits that don't match any existing type. These are legitimate. Recurring use of `exploratory` signals a new type to formalize.

### Pattern 2: Ground Rules Reference Document
**What:** A shared reference document containing core ground rules + type-specific extensions
**When to use:** Referenced from every audit task spec

**Structure (modeled on `claim-types.md`):**
```markdown
# Audit Ground Rules Reference

## Core Rules (apply to ALL audit types)

### 1. Every factual claim cites file:line
### 2. Test disconfirming evidence before writing findings
### 3. Distinguish what was measured from what the measure captures

## Escape Hatch Rule (always included)
### 4. What did you encounter that these ground rules didn't prepare you for?

## Type-Specific Extensions

### Structural Audits (phase_verification, milestone, codebase_forensics)
### Epistemic Audits (cross_model_review, requirements_review, comparative_quality, claim_integrity)
### Compliance Audits (adoption_compliance)
```

### Pattern 3: Audit Session Index Pattern
**What:** The task-spec frontmatter IS the index. No separate index file needed.
**When to use:** For discoverability across audit sessions

[evidenced:cited] The signal KB uses `index.md` as a derived artifact regenerated from individual files. For audits, the same pattern works: each audit's frontmatter contains all the metadata needed for search/filter via grep. A separate index can be generated later if needed but is not required for Phase 57.3. Per `knowledge-store.md` Section 1: "Files are source of truth -- indexes and caches are derived, never authoritative."

### Anti-Patterns to Avoid

- **Single universal template:** [evidenced:cited] The user explicitly rejected this in discuss-phase: "not one size fits all, there are different types of audits." Per DISCUSSION-LOG.md Area 1, options A (single template) and D (no taxonomy) were rejected.

- **Closed enum without escape hatch:** [evidenced:cited] The signal KB's `signal_type` enum includes `custom` and it has been used 4 times in 216 signals (1.9%) for genuinely novel observations that don't fit the 8 standard categories. This confirms the escape hatch is not a dumping ground -- it serves its intended purpose of accommodating genuine novelty.

- **Ground rules as enforcement checklists:** [governing:cited] Per the forms-excess deliberation: "the reviewer's checklist becomes the definition of rigor, foreclosing what rigor might require beyond it." Ground rules must be framed as practices that enable rigor, with the explicit escape hatch question included.

- **Rewriting audit content during migration:** [stipulated:bare] G-5 from CONTEXT.md: artifacts are historical evidence. Their deficiencies are informative. Add frontmatter, don't rewrite bodies.

## Common Pitfalls

### Pitfall 1: Over-Formalization (The Forms-Excess Trap)
**What goes wrong:** Creating so many required fields, templates, and rules that audits become compliance exercises rather than genuine investigations.
**Why it happens:** The natural instinct when standardizing is to add structure. Each field seems useful in isolation, but collectively they create friction that crowds out the actual thinking.
**How to avoid:** Keep the shared frontmatter to ~8-10 fields. Make body templates "suggested" not "required." Include the escape hatch question. Count: if the frontmatter has more fields than the signal KB schema (which has ~15 fields total, most optional), something is wrong.
**Warning signs:** Audit outputs that perfectly fill templates but contain no surprising findings. Ground rules followed as checklist items rather than genuine epistemic practices.

### Pitfall 2: Migration Scope Creep
**What goes wrong:** Starting to "improve" or reorganize audit content during migration, turning a conservative move-and-tag operation into a rewrite.
**Why it happens:** Reading old audit files reveals obvious improvements. The temptation to fix them is strong.
**How to avoid:** Migration tasks should be explicitly scoped: add frontmatter, move to correct location, done. Content improvements are a separate (deferred) activity.
**Warning signs:** Migration pull request with large diffs in file bodies, not just frontmatter additions and file moves.

### Pitfall 3: Workflow Integration Too Early
**What goes wrong:** Trying to modify the audit-milestone, verify-phase, or other workflows to adopt the new conventions in the same phase.
**Why it happens:** G-3 from CONTEXT.md explicitly scopes this as conventions/templates, not workflow rewrites. But once you have conventions, the urge to wire them in is strong.
**How to avoid:** Phase 57.3 creates the reference documents and templates. Future phases (58, 62) integrate them into workflows. The boundary is clear: if you're editing `workflows/*.md` or `agents/gsd-verifier.md`, you've crossed it.
**Warning signs:** Tasks that involve editing workflow files or agent specs (beyond the new ground rules reference document).

### Pitfall 4: Conflating Workflow Audits with Standalone Audits
**What goes wrong:** Trying to force VERIFICATION.md files (which live in phase directories and are produced by `verify-phase` workflow) into the `.planning/audits/` directory, creating duplication or broken references.
**Why it happens:** VERIFICATION.md IS an audit artifact. The desire for a single "find all audits" index is real.
**How to avoid:** [assumed:reasoned] Keep VERIFICATION.md and MILESTONE-AUDIT.md in their current locations (phase directories and `.planning/milestones/`). They are workflow-produced artifacts with their own conventions. The `.planning/audits/` directory is for standalone audits not produced by standard phase/milestone workflows. A future cross-reference index can span both locations without moving files. See Q3 resolution below.
**Warning signs:** Creating symlinks, duplicating files, or trying to centralize workflow-produced artifacts.

### Pitfall 5: Phantom Citations in Audit Infrastructure Itself
**What goes wrong:** [evidenced:cited] Per the context-checker's verification of CONTEXT.md, 5 `[evidenced:cited]` claims cite agent process descriptions ("confirmed by infrastructure survey agent") rather than resolvable file paths. The `cited` verification level requires a durable artifact. If the audit infrastructure itself relies on agent-process citations, it reproduces the problem it was designed to solve.
**How to avoid:** Every citation in the ground rules reference, templates, and metadata schema should point to a file that exists in the repo, not to an agent run that produced no surviving output.
**Warning signs:** Claims citing "per research agent" or "per audit agent" without a file path.

## Open Questions

### Resolved

- **Q1: Closed enum vs open tags for audit taxonomy?** [evidenced:cited] Use enum-with-escape-hatch, matching the `signal_type` pattern. Evidence: `signal_type` in the KB has 9 named types + `custom`. In 216 signals, `custom` was used 4 times (1.9%) -- all for genuinely novel observations, not as a dumping ground. This validates the pattern: most signals fit the taxonomy, the escape hatch handles the rest without degrading classification quality. For audits, use the 8 observed types + `exploratory` escape hatch.

- **Q3: How should workflow-produced audits relate to the audit directory?** [assumed:reasoned] Keep VERIFICATION.md in phase directories and MILESTONE-AUDIT.md in `.planning/milestones/` -- they are workflow-produced artifacts with established conventions and 63+7=70 existing instances. The `.planning/audits/` directory is for standalone audits not produced by standard phase/milestone workflows. Rationale: (1) moving 70 files is high-risk for low benefit; (2) workflow-produced audits are already discoverable via their well-known locations; (3) a future cross-reference index can span both locations if "search all audits" becomes a real need. The first integration point for the new conventions is the audit-milestone workflow adopting the frontmatter schema, not moving its output.

- **Q5: Cross-project audit organization?** [assumed:reasoned] Cross-project audits live in the harness project's `.planning/audits/` directory (since they audit the harness's effect on other projects). The frontmatter includes a `scope` field that names the projects involved, plus `tags` for each project. This matches how the signal KB handles cross-project signals: they live in the project that generated them, with tags for cross-referencing. No special directory structure needed.

### Still Open

- **Whether the `audit-milestone` workflow should produce its output in `.planning/audits/` or remain in `.planning/milestones/`:** This is a workflow change question that exceeds Phase 57.3 scope (G-3). The conventions established here should be compatible with either answer. For now, the recommendation is: remain in `.planning/milestones/` but adopt the new frontmatter schema.

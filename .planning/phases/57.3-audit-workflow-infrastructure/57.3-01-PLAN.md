   - Rule 1: Every factual claim cites file:line and quotes the relevant passage. Include the bad/good example from the gold-standard task spec (lines 26-27).
   - Rule 2: For every finding, BEFORE writing it, ask "What would disconfirm this?" and CHECK. Not a rhetorical question -- actually look for counter-evidence.

3. **Escape Hatch Rule (always included)** -- Rule 4: "What did you encounter that these ground rules didn't prepare you for?" Frame as an invitation, not enforcement. Per the forms-excess deliberation: "the reviewer's checklist becomes the definition of rigor, foreclosing what rigor might require beyond it." This rule ensures the ground rules don't become a ceiling on rigor.

4. **Type-Family Extensions** -- Additional rules for audit families, derived from the 9 failure patterns in CONTEXT.md:
   - **Structural audits** (phase_verification, milestone, codebase_forensics): "Run the test, don't just read it" -- verification must execute, not inspect. "Check wiring, not just existence" -- a file that exists but isn't connected is not complete.
   - **Epistemic audits** (cross_model_review, requirements_review, comparative_quality, claim_integrity): "Read content, not metadata" -- do not count signals, read 5-10 and describe content (from gold-standard rule 4). "When you don't know, say 'I don't know'" -- do not hedge with qualifications as substitute for investigation (from gold-standard rule 5). "Check whether cited artifacts still exist and support the claim."
   - **Compliance audits** (adoption_compliance): "Verify against the actual practice, not the documented practice" -- check what the code does, not what the README says.

5. **How to Reference in Task Specs** -- Show the exact markdown snippet to include in audit task specs. Reference format: "Per `get-shit-done/references/audit-ground-rules.md`, core rules:" followed by the 3+1 rules. Show how to add type-specific extensions.

6. **Governing Principle** -- Close with the framing: ground rules are practices that enable rigor, not checklists that define it. They should be understood as starting conditions for epistemic discipline, not exhaustive descriptions of what rigor requires. Per the forms-excess deliberation (open, not concluded): "a formal system encounters something that exceeds its categories, and the system's response to the excess reveals something about the system itself."

Use the claim-types.md notation throughout where appropriate (e.g., `[evidenced:cited]` for claims drawn from the gold-standard audit, `[governing:reasoned]` for the forms-excess framing).

Do NOT include rules about agent implementation changes -- this phase creates the reference document; future phases wire it into agents (per deferred: "Audit agent epistemic upgrades -- adding ground rules to existing agents is a future phase").

**Part A: Create `get-shit-done/references/audit-conventions.md`**

Create a conventions reference document defining the full audit infrastructure. Structure:

1. **Header** with consuming agents/workflows list and design authority citation (same as ground rules doc).

2. **Directory Structure** -- The canonical directory convention:
   ```
   .planning/audits/
   +-- YYYY-MM-DD-{slug}/           # Multi-file audit sessions
   |   +-- {slug}-task-spec.md      # Task spec given to agent
   |   +-- {slug}-output.md         # Agent output
   |   +-- {slug}-2-task-spec.md    # Follow-up task spec (if chain)
   |   +-- {slug}-2-output.md       # Follow-up output
   +-- YYYY-MM-DD-{slug}.md         # Single-file audits (lightweight)
   ```
   - Date-first (`YYYY-MM-DD`) for chronological sort
   - Slug is kebab-case descriptive name
   - Multi-file sessions get directories; single-file audits can be bare files
   - Task spec files use `-task-spec.md` suffix (per Claude's discretion)
   - Workflow-produced audits (VERIFICATION.md, MILESTONE-AUDIT.md) stay in their current locations (phase dirs, milestones/) -- `.planning/audits/` is for standalone audits

3. **Frontmatter Schema** -- Model after knowledge-store.md Section 3 (Common Base Schema). Required and optional fields:

   **Required fields:**
   | Field | Type | Description |
   |-------|------|-------------|
   | `date` | YYYY-MM-DD | Date the audit was conducted (NOT migration date) |
   | `audit_type` | enum | See type taxonomy below |
   | `scope` | string | What was audited -- concise description of the target |

   **Recommended fields:**
   | Field | Type | Description |
   |-------|------|-------------|
   | `auditor_model` | string | Model identifier (e.g., `claude-sonnet-4-6`, `gpt-5.4`) |
   | `triggered_by` | string | What launched this audit (e.g., `deliberation: X`, `manual: user request`, `workflow: audit-milestone`) |
   | `task_spec` | string | Relative path to the task spec file within the session directory |
   | `ground_rules` | string | Which ground rule set was applied: `core`, `core+structural`, `core+epistemic`, `core+compliance`, `core+custom`, or `none` (for pre-infrastructure audits) |
   | `tags` | array | Searchable tags for cross-referencing |

   **Optional fields:**
   | Field | Type | Description |
   |-------|------|-------------|
   | `output_files` | array | List of output file names in the session directory |
   | `predecessor_audits` | array | Prior audits on the same question (chain of inquiry) |
   | `migrated_from` | string | Original path before migration (only for migrated artifacts) |
   | `migrated_date` | YYYY-MM-DD | Date of migration (only for migrated artifacts) |

   Keep total to ~10-12 fields (per Pitfall 1: if frontmatter has more fields than the signal KB schema, something is wrong).

4. **Audit Type Taxonomy** -- Enum with escape hatch (matching `signal_type` pattern from knowledge-store.md):

   | Type | Family | Description |
   |------|--------|-------------|
   | `phase_verification` | structural | Checking code exists and works |
   | `milestone` | structural | Cross-phase integration |
   | `codebase_forensics` | structural | Structural understanding of codebase |
   | `cross_model_review` | epistemic | External perspective from different model |
   | `requirements_review` | epistemic | Coverage, feasibility, tensions |
   | `comparative_quality` | epistemic | Investigating a specific quality question |
   | `claim_integrity` | epistemic | Typed claim verification |
   | `adoption_compliance` | compliance | How X is used across projects |
   | `exploratory` | (escape hatch) | Does not match any existing type |

   Include note: "This taxonomy is provisional and empirically derived from ~10 projects over ~2 months. New types WILL emerge. Recurring use of `exploratory` signals a new type to formalize." Per governing constraint G-1 from CONTEXT.md.

5. **Type-Aware Body Templates** -- Suggested (not required) body structures for each family:
   - **Structural audits:** Goal/scope, artifacts checked, wiring verification, findings table (file, line, issue, severity), gaps summary
   - **Epistemic audits:** Question being investigated, methodology, evidence examined (with citations), findings with counter-evidence tested, limitations of this audit
   - **Compliance audits:** Standard being checked, projects/files surveyed, compliance matrix, deviations with context, recommendations
   - **Exploratory audits:** Question/motivation, what was examined, what was found, what this might mean, what follow-up is needed
   
   Frame as "suggested structure" -- the escape hatch for body format is simply: write whatever the audit demands. The frontmatter is the strict layer; the body is the flexible layer.

6. **Three-Layer Model Summary** -- Explicit statement of the design:
   - Layer 1 (strict): Shared frontmatter -- required for ALL audit artifacts
   - Layer 2 (structured but provisional): Type-aware body templates -- guides, not straitjackets
   - Layer 3 (open): Escape hatch -- `audit_type: exploratory` + freeform body for types that don't exist yet
   
   Per stipulated constraint from CONTEXT.md.

7. **Relationship to Other Audit Locations** -- Clarify:
   - `.planning/audits/` = standalone audits (this convention)
   - Phase directories (`VERIFICATION.md`) = workflow-produced phase verification (unchanged)
   - `.planning/milestones/` (`MILESTONE-AUDIT.md`) = workflow-produced milestone audits (unchanged)
   - Future: audit-milestone workflow adopts frontmatter schema; cross-reference index spans all locations if needed

## Gray Areas (Audit Trail)

### Area 1: Audit Taxonomy & Type-Specific Handling

**Gray area:** What types of audits exist and should the infrastructure handle them differently?

**Options presented:**
- (A) Single universal template for all audit types — maximum regularity, minimum flexibility
- (B) Closed enum of audit types, each with its own template — structured but risks over-formalization
- (C) Open tagging system with shared metadata + type-aware body suggestions — balance of regularity and openness
- (D) No taxonomy — just shared frontmatter and freeform bodies — maximum flexibility, minimum structure

**Resolution:** --auto selected (C) as the grounded default, supported by the forms-excess deliberation's analysis and the user's explicit direction that "there shouldn't be just one template" but also "there needs to be some regularity across, like frontmatter or metadata."

**User interventions:**
- User explicitly stated "not one size fits all, there are different types of audits" — rejecting options A and D
- User referenced `.planning/deliberations/forms-excess-and-framework-becoming.md` as a governing constraint — the taxonomy must remain provisional
- User stated "there needs to be some regularity across, like frontmatter or metadata" — the shared layer is decided, the type-specific layer is open

**Why this resolution:** The empirical evidence (8 types observed across ~10 projects) supports structure. The forms-excess deliberation supports provisionality. The user's interventions require both regularity (shared metadata) and type-awareness (different bodies). Option C is the only option that satisfies all three constraints.

### Area 2: Existing Audit Mess — Cleanup Scope

**Gray area:** Should this phase clean up the existing scattered audit artifacts or just establish conventions for future audits?

**Options presented:**
- (A) Forward-only — establish conventions, don't touch existing artifacts
- (B) Full migration — move all ~25 scattered artifacts into new conventions with frontmatter
- (C) Conservative migration — move files, add frontmatter, don't rewrite content

**Resolution:** --auto selected (C) based on user direction. The user explicitly stated this should be a requirement of the phase: "part of the auto-discussion here needs to look at the existing mess of the existing .planning folder as well, and try to propose a fix." Conservative migration preserves historical evidence while making it findable.

**User interventions:**
- User explicitly expanded scope: the existing mess must be addressed, not just future conventions
- User directed adding AUDIT-03 as a new requirement for migration/cleanup

**Why this resolution:** The artifacts are historical evidence — their deficiencies are informative (e.g., the shallow Audit 1 vs rigorous Audit 3 comparison is itself an audit finding). Rewriting content would destroy that evidence. Adding frontmatter and moving to consistent locations preserves content while enabling searchability.

### Area 3: Epistemic Ground Rules — Universal vs Type-Specific

**Gray area:** Should ground rules be universal across all audit types or vary per type?

**Options presented:**
- (A) Universal core set only — 3 rules from AUDIT-02 apply to everything
- (B) Universal core + type-specific extensions — shared foundation with per-type supplements
- (C) Fully type-specific — each audit type has its own ground rules
- (D) No prescribed rules — just the "what didn't these rules prepare you for?" prompt

**Resolution:** --auto selected (B) as the grounded default. The 3-audit progression provides strong evidence that a core set works. The forms-excess deliberation warns that checklists become the definition of rigor. The escape hatch ("what did you encounter that these rules didn't prepare you for?") addresses the deliberation's concern.

**Why this resolution:** Core rules address 5 of 9 failure patterns. Type-specific rules address domain needs (code audits need "run the test"; epistemic audits need "check whether cited artifacts exist"). The escape hatch prompt is the forms-excess deliberation's recommendation operationalized.

### Area 4: Regularity vs Openness Tension

**Gray area:** How to balance searchable regularity with the openness that audit findings require?

**Options presented:**
- This was presented as an explicit tension, not a multiple-choice question

**Resolution:** --auto captured as a governing tension with a stipulated resolution: shared metadata layer (strict) + type-aware body templates (structured but provisional) + escape hatch (for types that don't exist yet). The resolution is explicitly provisional — the user stated "these are open questions, open tensions."

**User interventions:**
- User explicitly named this as the central design challenge
- User stated the tension itself should be preserved, not resolved: "these are open questions, open tensions"

**Why this resolution:** Not resolved — captured as a living tension. The stipulated resolution (three-layer approach) is a working bet, not a settled answer. The CONTEXT.md marks it as `[stipulated:bare]` to signal that it's a choice, not derived from evidence.

### Audit taxonomy (empirically observed)

**Current state:** Research across ~10 projects identified at least 8 distinct audit types. These were discovered, not prescribed — the taxonomy is provisional.

- [assumed:reasoned] At least 8 audit types exist in practice, each serving different epistemic purposes: (1) phase verification — checking code exists and works, (2) milestone audit — cross-phase integration, (3) cross-model epistemic review — external perspective on decisions, (4) requirements review — coverage, feasibility, tensions, (5) comparative quality audit — investigating a specific question, (6) claim integrity audit — typed claim verification, (7) codebase forensics — structural understanding, (8) adoption/compliance audit — how X is used across projects. This list is empirical, not exhaustive — future audit types will emerge
- [assumed:reasoned] These 8 types cluster into roughly 3 families by what they verify: **structural** (does X exist and wire correctly — types 1, 2, 7), **epistemic** (are claims warranted and traceable — types 3, 4, 5, 6), **compliance** (does practice match intent — type 8). The families suggest different ground-rule emphasis but may be wrong
- [open] Whether the taxonomy should be formalized as a closed enum (e.g., `audit_type: phase_verification | milestone | cross_model | ...`) or as an open tagging system that allows new types. The forms-excess deliberation argues against closed enums; practical searchability argues for some regularity
- [governing:reasoned] The taxonomy is provisional and must remain open to types that don't yet exist — per `.planning/deliberations/forms-excess-and-framework-becoming.md`: "a formal system encounters something that exceeds its categories, and the system's response to the excess reveals something about the system itself"

### Audit failure modes (empirically cataloged)

**Current state:** 9 distinct failure patterns identified from signals, deliberations, and audit artifacts. These are the problems the infrastructure must address.

- [assumed:reasoned] 9 audit failure patterns cataloged from KB signals and deliberation evidence: (1) shallow counting without reading, (2) fabricated/phantom citations, (3) severity downgrade bias, (4) facts claimed without codebase verification, (5) invented automated workflow narratives, (6) tech debt dismissed via empty YAML arrays, (7) verification checking structure not function, (8) plan-checker structural blindness, (9) epistemic quality regression under --auto — sources include `sig-2026-02-23-audit-tech-debt-dismissal.md`, `sig-2026-02-22-protocol-section-13-fabricated-provenance.md`, `sig-2026-03-30-audit-severity-downgrade-bias.md`, `sig-2026-03-04-deliberation-skill-lacks-epistemic-verification.md`, and deliberation lines 318-326. Research agent outputs not persisted; researcher should re-verify individual patterns against cited signals
- [assumed:reasoned] 5 of 9 failure patterns are mitigated by task spec ground rules (patterns 1, 2, 4, 5, 9). The remaining 4 are structural scope gaps in existing agents (patterns 3, 6, 7, 8) — reasoning: patterns 1/2/4/5/9 all have "absence of task spec ground rules" as a root cause factor per the deliberation's meta-observation; researcher should verify by checking each signal's root cause analysis
- [evidenced:cited] The empirical proof that ground rules work: Audit 1 (no ground rules) produced shallow work with phantom claims. Audit 3 (7 explicit ground rules) corrected Audit 1's errors and produced traceable findings. Same agent class, same evidence base, different task spec — per `rigorous-comparative-audit-task-spec.md` and its output vs `outcome-comparison-task-spec.md` and its output in `.planning/audits/2026-04-09-discuss-phase-exploration-quality/`

### Shared metadata and regularity

**Current state:** No shared metadata standard across audit types. Provenance is a signal-KB concept (`detection_method`, `origin`), not yet applied to audit outputs.

- [assumed:reasoned] All audit types should share a common frontmatter schema for provenance and searchability — at minimum: `date`, `audit_type`, `auditor_model`, `scope`, `triggered_by`, `output_files`. This enables `grep`/`gsd-tools` to find and filter audits without understanding each type's body format
- [assumed:reasoned] The shared frontmatter should be YAML (consistent with signal files, VERIFICATION.md, and all other `.planning/` artifacts) rather than inline markdown headers
- [open] What additional metadata fields are needed beyond the minimum set. Candidates: `session_id`, `launch_context` (which workflow spawned it), `epistemic_ground_rules` (reference to which rule set applied), `predecessor_audits` (chain of prior audits on same question), `severity_verdict` (PASS/WARN/FAIL). More fields = more searchability but more friction
- [decided:reasoned] Task specs must be preserved alongside outputs — this is the single strongest empirical finding from the audit-of-audits research. The quality delta between Audit 1 (no spec) and Audit 3 (preserved spec with ground rules) is the anchor evidence for AUDIT-01

### Directory conventions

**Current state:** Mixed conventions. Dated directories exist for 2 of ~20 audit sessions. Bare files at `.planning/audits/` root use 3 different naming schemes.

- [assumed:reasoned] Date-first directory convention (`YYYY-MM-DD-{slug}/`) is correct for audit sessions that produce multiple files — this is already working for the 2 directories that use it (`session-log-audit-2026-04-07/`, `2026-04-09-discuss-phase-exploration-quality/`). Single-file audits may not need directories
- [open] Whether ALL audits should use dated directories (consistent but heavier) or whether single-file audits can remain as dated bare files (lighter but two conventions). The regularity argument favors "always directories"; the practical argument is that a one-file code review doesn't need a directory
- [assumed:reasoned] The `.planning/fork-audit/` directory and the 5 root-level audit files from 2026-04-02 should be migrated into `.planning/audits/` under the new convention. The sensor trial files are a different artifact type (experimental protocol, not audit) and should be separately addressed
- [open] Whether audit artifacts that live in phase directories (e.g., VERIFICATION.md, REVIEW.md produced by workflows) should be symlinked or indexed from `.planning/audits/`, or whether `.planning/audits/` only holds standalone audits not produced by standard phase workflows

### Epistemic ground rules

**Current state:** Only `gsdr-context-checker` has explicit ground rules. The deliberation proved that ground rules in task specs dramatically improve audit quality.

- [decided:reasoned] Ground rules must be part of the infrastructure, not optional — the empirical evidence from the 3-audit progression is unambiguous: ground rules are the highest-leverage intervention for audit quality
- [assumed:reasoned] A core set of ground rules should apply across all audit types: (1) every factual claim cites file:line, (2) test disconfirming evidence before writing findings, (3) distinguish what was measured from what the measure captures. These are the 3 rules from AUDIT-02 and they proved effective in the rigorous audit
- [open] Whether type-specific ground rules should supplement the core set. Epistemic audits may need "check whether cited artifacts still exist and support the claim." Code audits may need "run the test, don't just read it." Cross-model reviews may need "name your own model's blind spots." The question is which type-specific rules are worth formalizing now vs which emerge through practice
- [governing:reasoned] Ground rules must be understood as practices that enable rigor, not checklists that define it — per the forms-excess deliberation: "the reviewer's checklist becomes the definition of rigor, foreclosing what rigor might require beyond it." The ground rules should include something like "what did you encounter that these rules didn't prepare you for?"

### Integration with existing workflows

**Current state:** The audit-milestone workflow, verify-phase workflow, validate-phase workflow, and code-review workflow all produce audit-like output but don't use the audit infrastructure being designed here.

- [assumed:reasoned] The new audit conventions should be adopted by existing workflows incrementally — not a big-bang rewrite. The first integration point is the audit-milestone workflow since it's the most visible and already produces structured output (MILESTONE-AUDIT.md)
- [open] How deeply should the audit infrastructure integrate with the workflow system. Options range from "just a directory convention and metadata standard that workflows adopt gradually" to "a shared audit-dispatch mechanism that any workflow can call with type-specific templates." The former is Phase 57.3 scope; the latter might be a future phase
- [projected:reasoned] Phase 58 (structural enforcement) and Phase 62 (workflow commands including /gsdr:cross-model-review) will be the first consumers of this infrastructure beyond manual ad-hoc audits. The conventions we establish now must serve those phases without requiring rework

### The regularity-vs-openness tension

- [governing:reasoned] The infrastructure must balance two competing goods: enough regularity for audits to be findable, searchable, and comparable (shared frontmatter, directory conventions, provenance) vs enough openness for audit types that don't fit the taxonomy and findings that don't fit any template — per `.planning/deliberations/forms-excess-and-framework-becoming.md`
- [stipulated:bare] The resolution is: **shared metadata layer** (frontmatter + directory convention — strict) + **type-aware body templates** (different per type — structured but provisional) + **escape hatch** (a way to create an audit that doesn't match any existing type, with the expectation that recurring escape-hatch uses reveal new types to formalize)
- [open] What the escape hatch actually looks like. Candidates: an `audit_type: exploratory` catch-all, a freeform body with only frontmatter required, or no special mechanism (just create the directory with frontmatter and write whatever). The question is whether the escape hatch needs explicit support or whether "just follow the metadata convention" is enough

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] Do not treat the 8-type taxonomy as exhaustive or final. It was discovered from ~10 projects over ~2 months of practice. Future audit types WILL emerge. The infrastructure must accommodate types that don't exist yet, or it will reproduce the rigidity it was designed to overcome
- **G-2:** [governing:reasoned] The empirical evidence for ground rules (3-audit progression) is strong but narrow — it comes from one deliberation session, one agent class (Sonnet), one question domain (CONTEXT.md quality). Do not over-generalize. The researcher should look for counter-evidence: cases where ground rules didn't help, or where they produced compliance without substance
- **G-3:** [governing:reasoned] "Infrastructure" in this phase means conventions, templates, and metadata — not new agent implementations or workflow rewrites. The temptation to build a `gsd-audit-dispatcher` agent or a comprehensive `gsd-tools audit` command suite is scope creep. Phase 57.3 establishes the conventions; future phases build tooling on top
- **G-4:** [governing:cited] Per the forms-excess deliberation: "the most important insights came from conversation — from one participant pushing back on another's claims." No audit infrastructure replaces genuine dialogue. The ground rules enable rigor; they don't guarantee it. Include "what happened during this audit that these ground rules didn't prepare you for?" as a standard prompt — not enforced, inviting
- **G-5:** [stipulated:bare] Migration of existing artifacts should be conservative — move files, add frontmatter, but do NOT rewrite or "improve" audit content. The artifacts are historical evidence; their current state (including their deficiencies) is informative

</guardrails>

<questions>
## Open Questions

### Q1: Should the audit taxonomy be a closed enum or an open tagging system?
**Research program:** Compare how the signal KB handles `signal_type` (a closed enum with `custom` escape hatch) vs how the claim-types system handles types (7 types, extensible by design). Check whether the signal system's `custom` type has been used in practice and whether it's been useful or a dumping ground. Look at how other knowledge management systems handle evolving taxonomies.
**Downstream decisions affected:** Frontmatter schema (enum validation vs freeform string), template selection logic, search/filter capabilities
**Reversibility:** MEDIUM — changing from enum to tags is easy; changing from tags to enum requires migrating existing frontmatter

### Q2: What type-specific ground rules are worth formalizing now?
**Research program:** For each of the 8 observed audit types, review the existing audit outputs in this project and identify which specific ground rules would have prevented the failures observed. Cross-reference with the 9 failure patterns. Identify which rules are universal (apply to all types) vs type-specific. Pay attention to the forms-excess warning: rules that become checklists lose their epistemic force.
**Downstream decisions affected:** Whether ground rules are embedded in type-specific templates, referenced from a shared rules document, or passed as task spec preamble
**Reversibility:** HIGH — ground rules are text in templates, easily revised. But getting them wrong means the first audits under the new system reproduce old failures

### Q3: How should workflow-produced audits (VERIFICATION.md, MILESTONE-AUDIT.md) relate to the audit directory?
**Research program:** Inventory all workflow-produced audit artifacts and where they currently live. Determine whether phase-directory placement (current) is adequate for discoverability or whether a cross-reference in `.planning/audits/` is needed. Consider: will anyone ever want to search "all audits across all phases"? If so, the current scattered placement fails.
**Downstream decisions affected:** Whether `.planning/audits/` is the canonical index for ALL audits or just standalone ones, migration scope, workflow changes needed
**Reversibility:** LOW — if we decide later that workflow audits should be indexed in `.planning/audits/`, we need to retroactively add entries for all completed phases

### Q4: What is the right migration strategy for the existing ~25 scattered audit artifacts?
**Research program:** Catalog all existing audit artifacts (already partially done by research agents). For each, determine: can it be moved as-is with just frontmatter added? Does it need a dated directory? Are there dependency chains (review → review-of-review → decisions) that must be preserved? What about the sensor trial files — are they audits or a different artifact type?
**Downstream decisions affected:** Migration plan scope, whether migration is automated or manual, how to handle artifacts that don't fit any type
**Reversibility:** HIGH — file moves are trivially reversible with git

### Q5: How should cross-project audits be organized?
**Research program:** The claim taxonomy audit spanned 12 projects and ~85 CONTEXT.md files. The adoption audit covered 6 projects. These don't belong to any single project's `.planning/` directory. Where do they live? Options: in the harness project's audits (since it's auditing the harness's effect), in a shared location, or duplicated with project-specific views. Check how the signal KB handles cross-project signals.
**Downstream decisions affected:** Directory convention, whether audit frontmatter needs a `projects` array field, cross-project searchability
**Reversibility:** MEDIUM — moving cross-project audits later requires updating references

</questions>

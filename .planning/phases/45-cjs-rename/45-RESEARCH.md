## Summary

This phase is a pure rename operation: `get-shit-done/bin/gsd-tools.js` becomes `get-shit-done/bin/gsd-tools.cjs`, with all references across 66 non-planning source files updated accordingly. The rename is the prerequisite gate for Phases 46-48 (modular migration). Node.js treats `.cjs` files identically to `.js` files in CJS mode -- `__dirname`, `__filename`, `require()`, and shebang execution all work unchanged.

The codebase investigation reveals the prior "56 files" count is now **66 source files** (excluding `.planning/` and `.claude/` directories) containing `gsd-tools.js` references, with **1,491 total occurrences across 255 files** (including planning artifacts). However, only **66 source files** need direct editing. The `.planning/` files are historical artifacts that should NOT be modified (they document what was true at the time). The `.claude/` installed copies are regenerated by `node bin/install.js --local` and will automatically pick up the rename.

The highest-risk component is the installer's `replacePathsInContent()` function (line 1334, `bin/install.js`). Research found that the regex does NOT match on `.js` extension broadly -- it matches on `gsd-` prefix with a `(?!tools)` negative lookahead (line 1373: `\bgsd-(?!tools)/g`). The lookahead preserves the `gsd-tools` filename during GSDR namespace rewriting. After the rename, this regex continues to work correctly because it matches on `gsd-` prefix, not on the `.js` extension. The critical finding is: **the installer regex needs NO changes for the rename itself**. Only the installer's comment at line 1361 needs updating to reflect `gsd-tools.cjs` instead of `gsd-tools.js`.

**Primary recommendation:** Rename the file first (`git mv`), then update all 65 remaining source files in a single atomic commit. The installer regex is safe as-is. Test files should keep their `.js` extension.

### Anti-Patterns to Avoid

- **Blind find-and-replace across all 255 files:** Would corrupt `.planning/` historical artifacts and `.claude/` installed copies (which will be regenerated). Only the 66 source files should be touched.
- **Modifying the installer regex itself:** The `\bgsd-(?!tools)/g` regex works correctly for both `.js` and `.cjs` because it matches on prefix, not extension. Changing it introduces risk for zero benefit.
- **Renaming test files to `.cjs`:** The test files use `require('node:test')` and CJS syntax, which works in `.js` files. Renaming to `.cjs` would require updating `package.json` scripts and offers no benefit since they invoke the target via `execSync`, not `require()`.
- **Modifying `.planning/` files:** These are historical records. Introducing `.cjs` references into plans, summaries, and signals from months ago falsifies the record.

## Common Pitfalls

### Pitfall 1: Corrupting Historical Planning Artifacts
**What goes wrong:** A global find-and-replace changes `gsd-tools.js` to `gsd-tools.cjs` in `.planning/` files, falsifying historical records.
**Why it happens:** 189 of the 255 matching files are in `.planning/`. A blanket replacement seems efficient.
**How to avoid:** Only edit the 66 source files. Explicitly scope grep verification to `get-shit-done/ agents/ commands/ bin/ tests/ CHANGELOG.md`.
**Warning signs:** Git diff shows changes in `.planning/phases/08-core-merge/` or similar historical directories.

### Pitfall 2: Breaking the Installer Regex
**What goes wrong:** Someone modifies the `\bgsd-(?!tools)/g` regex to add `.cjs` awareness, introducing a subtle bug.
**Why it happens:** The comment says "preserves gsd-tools.js" which after the rename might seem incorrect.
**How to avoid:** Update only the COMMENT (line 1361). The regex itself is extension-agnostic.
**Warning signs:** Installer tests fail on namespace rewriting.

### Pitfall 3: Test File Rename Creates npm Script Breakage
**What goes wrong:** Renaming `gsd-tools.test.js` to `gsd-tools.test.cjs` breaks `npm run test:upstream` because package.json still references `.test.js`.
**Why it happens:** Assuming all `gsd-tools.js` references need updating without checking what they point to.
**How to avoid:** Test files keep `.js` extension. Only the TOOLS_PATH constant inside them changes.
**Warning signs:** `npm run test:upstream` fails with "file not found".

### Pitfall 4: Missing the install.test.js Integration Test
**What goes wrong:** The integration test at line 2518-2537 (`installed files preserve gsd-tools.js filename`) checks that installed workflows reference `gsd-tools`. After the rename, installed workflows will reference `gsd-tools.cjs`. The assertion message and content check need updating.
**Why it happens:** This test is 2500 lines into the file, easy to miss.
**How to avoid:** Search for all 10 occurrences in `install.test.js` and analyze each individually per the Category 4 analysis above.
**Warning signs:** `npm test` fails on "installed files preserve gsd-tools" test.

### Pitfall 5: Forgetting the Shell Script
**What goes wrong:** `reconcile-signal-lifecycle.sh` at line 24 has `GSD_TOOLS="$HOME/.claude/get-shit-done/bin/gsd-tools.js"`. If missed, signal lifecycle reconciliation breaks.
**Why it happens:** It's the only `.sh` file with a reference, easy to overlook among 27 workflows.
**How to avoid:** Include it in the explicit file list.
**Warning signs:** Signal reconciliation errors in production use.

### Pitfall 6: Overlooking the multi-runtime.test.js Regex Exemption
**What goes wrong:** The exemption comment at line 142 says `gsd-tools.js` but the actual regex uses `(?!tools)`. If someone later reads the comment and sees `.js` while the actual file is `.cjs`, they might incorrectly modify the regex.
**Why it happens:** Comment/code drift.
**How to avoid:** Update the comment to `.cjs` for accuracy, but leave the regex unchanged.
**Warning signs:** Future maintainer confusion.

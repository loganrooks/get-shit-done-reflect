# Phase 45: CJS Rename - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Rename `gsd-tools.js` to `gsd-tools.cjs` with all shell references updated across agents, workflows, commands, references, and tests. Zero functional changes. This is the prerequisite gate for all subsequent modularization (Phases 46-48). The rename enables CJS `require()` semantics that upstream's modular dispatcher depends on.

**What this phase does NOT include:**
- Creating the `lib/` directory or any modules (Phase 46)
- Modifying the dispatcher logic (Phase 46)
- Extracting fork functions into modules (Phase 47)
- Extending upstream modules with fork capabilities (Phase 48)

</domain>

## Implementation Decisions

### Reference Completeness

The modular migration research (2026-03-10) identified 56 files containing `gsd-tools.js` references. However, this count was a snapshot. The researcher must re-enumerate at execution time because:

- QT22-32 (2026-03-17 to 2026-03-19) modified agents, workflows, and references after the research was conducted
- The installed `.claude/` copies are derived from source and will be regenerated by `node bin/install.js --local`, so they are NOT direct rename targets — the installer handles the source-to-runtime path conversion
- The `bin/install.js` file itself contains regex patterns that match `gsd-tools.js` — these are code, not references, and need semantic analysis not find-and-replace

**Reference categories (researcher must distinguish):**

| Category | Action | Example |
|----------|--------|---------|
| Shell invocations | Rename to `.cjs` | `node ~/.claude/get-shit-done/bin/gsd-tools.js` in workflows |
| Test path constants | Rename to `.cjs` | `const GSD_TOOLS = path.resolve(...)` pointing to the binary |
| Installer regex patterns | Semantic update required | `replacePathsInContent()` regex matching `.js` extension |
| Test fixture/assertion data | Case-by-case analysis | Tests that assert the installer correctly handles `gsd-tools` preservation |
| Documentation references | Rename to `.cjs` | CHANGELOG, FORK-DIVERGENCES, references/*.md |
| Package.json entries | Verify inclusion | `files` array, script definitions |

**Critical edge case:** Some `install.test.js` assertions test that the namespace rewriting regex preserves `gsd-tools.js` via the `(?!tools)` lookahead. After the rename, these tests need to verify `gsd-tools.cjs` preservation instead. The distinction between "test data that should change" and "test data that exercises backward compatibility" must be resolved during research.

### Installer Transition Safety

The installer is the highest-risk component in this rename. Three concerns:

**1. `replacePathsInContent()` regex update:**
The installer rewrites `~/.claude/get-shit-done/` to `./.claude/get-shit-done/` (global-to-local path conversion). The regex currently handles `.js` extensions. After the rename, source files will contain `.cjs` references. The regex must handle `.cjs` correctly without introducing false positive matches on other `.cjs` files that may appear in Phase 46+.

**2. Stale file cleanup on upgrade:**
When a user upgrades from v1.17 (which has `gsd-tools.js`) to v1.18 (which has `gsd-tools.cjs`), the old `gsd-tools.js` file will be left behind in `.claude/get-shit-done/bin/` unless the installer explicitly removes it. A stale `gsd-tools.js` alongside `gsd-tools.cjs` could cause confusion or, worse, shell scripts that haven't been updated could silently execute the old version.

*Signal reference:* `sig-2026-02-23-installer-clobbers-force-tracked-files` — The installer's `fs.rmSync` + `copyWithPathReplacement` pattern deletes and recreates directories. This actually HELPS with stale file cleanup (the old .js is deleted when the directory is recreated), but the signal warns that this same mechanism can clobber force-tracked files. Verify that no force-tracked files exist in `get-shit-done/bin/`.

*Signal reference:* `sig-2026-03-06-agent-bypassed-quick-cycle-for-coinstall-fix` — The co-installation fix added `isLegacyReflectInstall()` guards. Verify these guards are not sensitive to the filename extension.

**3. Dual-installation detection:**
The `dual-installation.md` reference documents co-installation with upstream GSD. The detection logic may reference `gsd-tools.js` by name. Researcher must check whether dual-install detection depends on the filename or the directory structure.

### Verification Rigor Standard

Three signals establish that "tests pass" is insufficient verification for this project:

1. **`sig-2026-03-06-plan-verification-misses-architectural-gaps`** — Plan checker verified Phase 41 plans but missed that no plan registered a new hook in `bin/install.js`. The checker validates plans against requirements, not against the full codebase integration surface.

2. **`sig-2026-03-01-plan-checker-misses-second-order-effects`** — Manual review found 6 gaps, 3 unvetted assumptions, and 3 second-order effects in Phase 35 plans that the checker didn't catch.

3. **`sig-2026-02-23-installer-clobbers-force-tracked-files`** — Plan 29-02 asserted "the installer does NOT touch `.claude/agents/`" without verifying against actual code.

**Verification standard for Phase 45:**

- **Exhaustive grep check:** `grep -r 'gsd-tools\.js' get-shit-done/ agents/ commands/ bin/ tests/` must return zero hits (excluding intentional test fixture data for backward compat testing, if any)
- **All three test suites pass:** `npm test` (vitest), `npm run test:upstream`, `npm run test:upstream:fork`
- **Installer smoke test:** `node bin/install.js --local` completes successfully, and the installed `.claude/get-shit-done/bin/gsd-tools.cjs` exists (not `.js`)
- **Behavioral equivalence spot check:** Run 3-5 representative CLI commands before and after the rename, diff the output. Must be identical.
- **No stale files:** After `install.js --local`, verify `.claude/get-shit-done/bin/gsd-tools.js` does NOT exist

**From modular migration research (MODULAR-MIGRATION.md):**

- The plan checker will NOT catch installer integration issues — explicit verification steps must be in the plan

## Epistemic Guardrails

Lessons from signal history that apply to Phase 45 planning and execution:

**1. Verify assumptions against code, not descriptions** (from `sig-2026-02-23-installer-clobbers-force-tracked-files`, `sig-2026-03-01-plan-checker-misses-second-order-effects`):
- Every plan assertion about installer behavior must cite the specific function and line number
- "The installer does X" claims require evidence from code reading, not from documentation

**2. Plan for what the plan-checker can't see** (from `sig-2026-03-06-plan-verification-misses-architectural-gaps`):
- The plan-checker validates requirement coverage and dependency ordering, not codebase integration
- The plan must include explicit verification tasks for: installer regex correctness, stale file absence, hook registration integrity

**3. Process discipline over momentum** (from `sig-2026-03-06-agent-bypassed-quick-cycle-for-coinstall-fix`):
- If issues are discovered during execution that fall outside phase scope, they must be routed through proper workflow (quick task or deferred), not fixed inline

**4. Test fixture vs live reference distinction** (novel to this phase):
- A find-and-replace approach to updating references will corrupt test fixture data
- The plan must distinguish between references that should change (invocations) and references that are test data (assertions about installer behavior)

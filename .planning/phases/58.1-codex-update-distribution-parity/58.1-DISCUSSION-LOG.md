## Claim Justifications

### Update command and workflow remain Claude-shaped

**[evidenced:cited] Source update command/workflow still encode Claude-first paths**
- **Citation:** `commands/gsd/update.md`, `get-shit-done/workflows/update.md`
- **Verification:** Both files were inspected directly. The command points to `~/.claude/.../update.md`, and the workflow checks only `./.claude/.../VERSION` / `~/.claude/.../VERSION`, clears `.claude/cache/gsd-update-check.json`, and runs update commands without `--codex`.

**[decided:reasoned] 58.1 must patch source, not generated `.codex/` mirrors**
- **Alternatives considered:** Patch installed Codex mirror only; special-case source repo behavior
- **Why rejected:** This repo’s AGENTS contract makes source authoritative and mirrors outputs; installed-only edits regress on reinstall
- **Why chosen:** The user reported a product gap, not a one-off local mirror corruption

**[governing:reasoned] Runtime path policy should not be duplicated across hand-maintained markdown and JS**
- **Source:** Repo-local operating rule plus direct comparison of source workflow paths with `bin/install.js`
- **Scope of governance:** Command/workflow/runtime-detection changes in this phase

### Codex version authority is already solved elsewhere, but update flow is not using it

**[evidenced:cited] Installer already resolves Codex global location and config-dir precedence**
- **Citation:** `bin/install.js` `getGlobalDir('codex')`
- **Verification:** The function explicitly uses `--config-dir`, then `CODEX_CONFIG_DIR`, then `~/.codex`

**[evidenced:cited] Integration test already proves installed Codex VERSION outranks stale repo-local mirror**
- **Citation:** `tests/integration/multi-runtime.test.js` fixture `Codex writer provenance prefers installed harness VERSION over repo mirror`
- **Verification:** The test sets `project/.codex/get-shit-done-reflect/VERSION` to stale data and asserts the installed global Codex VERSION wins

**[decided:reasoned] Update flow should adopt the same authority rule**
- **Alternatives considered:** local-first lookup; config-only precedence; ignore source-repo divergence
- **Why rejected:** local-first is the observed bug; config-only loses active install authority; ignoring divergence leaves the user issue unresolved
- **Why chosen:** The updater should not contradict the authority rule already encoded in the runtime provenance surface

**[decided:reasoned] Divergent local/global Codex installs should target the older stale scope**
- **Alternatives considered:** always update both; keep first-match; surface divergence but do nothing
- **Why rejected:** always updating both broadens semantics; first-match hides the issue; no-op leaves the gap
- **Why chosen:** Narrowest operational fix that still resolves the reported masking behavior

**[assumed:reasoned] Update flow can reuse installer/runtime directory facts cleanly**
- **Challenge protocol:** If direct reuse causes coupling problems, introduce a small shared helper with tests rather than duplicating path logic again
- **Evidence checked:** `bin/install.js` already centralizes the needed Codex directory precedence
- **Why reasonable:** The missing behavior is policy reuse, not missing runtime facts

### Hook-adjacent update indicators are real, but Codex should not inherit fictional hook substrate

**[evidenced:cited] Codex install currently has no installer-wired hook or statusline surface**
- **Citation:** `bin/install.js`
- **Verification:** The Codex install path returns before hook/statusline configuration; no `settings.json` or hook registration is written for Codex

- **Why chosen:** Honest narrowness with room to fix adjacent command-related cache bugs if they are real

- **Citation:** `.planning/ROADMAP.md` Phase 58 Success Criteria 1 and 8 (`XRT-01` per-runtime substrate / waiver discipline)

**Researched:** 2026-04-20

## User Constraints

### Locked Decisions

[decided:reasoned] Phase 58.1 is a narrow Codex runtime-parity repair inserted after Phase 58. Its job is not to redesign the whole update UX or solve the cross-project distribution gap. Its job is to ensure the Codex update command and adjacent update-indicator surfaces stop missing stale Codex installs because they still think in Claude-only paths, cache files, and local-first detection rules.

In scope for 58.1:
- Codex-aware installed-version detection for the update command and any shared update-indicator surfaces it owns
- Resolution rules when project-local and global Codex installs diverge, especially in the source repo where repo-local `.codex/` mirrors are outputs rather than authority
- Runtime-correct update execution: `--codex`, local/global targeting, and non-default Codex config-dir resolution when applicable
- Cleanup/alignment of any update cache or indicator surface touched by the same flow so Codex does not inherit Claude-only cache semantics by accident
- Focused regression coverage for divergent Codex install scopes and runtime-correct update routing

Explicitly out of scope for 58.1:
- A full multi-runtime redesign of `gsdr-update` for Claude, Gemini, OpenCode, and Codex simultaneously
- The broader cross-project distribution gap for non-GSDR projects running global upstream GSD; that remains a later milestone concern already called out in parity research
- Introducing Codex hook/statusline substrate parity; Codex install still has no installed hook or statusline surface in `bin/install.js`
- General patch-sensor or cross-runtime artifact drift work already scoped to Phase 60 and later

[decided:reasoned] 58.1 must patch the source command/workflow surface, not the generated `.codex/` mirror, because the source repo is authoritative and the mirror is an output.

[decided:reasoned] The update flow should adopt the same authority rule: active installed Codex scope first, repo-local mirror last, and never let a repo-local mirror in the source repo silently mask a stale global Codex install.

[decided:reasoned] When project-local and global Codex installs both exist and diverge, 58.1 should detect the divergence and target the older stale scope rather than blindly updating whichever path is checked first.

[governing:reasoned] A missing Codex hook/statusline substrate should be represented honestly as absence or `does-not-apply-with-reason`, not papered over by writing Claude cache paths into Codex-facing logic.

[governing:reasoned] If runtime or config-dir cannot be resolved confidently, the update flow should degrade explicitly rather than guessing Claude defaults.

## Summary

[evidenced:cited] The current Codex update path is wrong in four concrete ways. First, the source workflow still chooses a single scope by local-first shell checks and never enumerates both Codex scopes (`get-shit-done/workflows/update.md:11-30`). Second, the actual update command is runtime-implicit: it runs `npx get-shit-done-reflect-cc --local` or `--global`, while the installer defaults such invocations to Claude when no runtime flag is supplied (`get-shit-done/workflows/update.md:144-169`; `bin/install.js:3336-3343`). Third, the existing path-rewrite layer only rewrites `~/.claude/` and `$HOME/.claude/`, not `./.claude/`, so installed Codex update workflows still contain Claude-local relative paths (`bin/install.js:1348-1375`). Fourth, Codex has no installed hook/statusline substrate, so clearing a Codex update-hook cache is a fictional behavior unless the phase explicitly models it as a no-op (`bin/install.js:2846-2908`; `hooks/gsd-check-update.js:10-17`; `hooks/gsd-statusline.js:71-104`).

[evidenced:cited] The strongest existing authority rule is already present elsewhere in the codebase: `resolveWriterGsdVersion()` prefers the installed harness version over config and repo mirror, and `tests/integration/multi-runtime.test.js` already verifies that a stale repo-local `.codex` mirror must not outrank the installed global Codex harness (`get-shit-done/bin/lib/provenance.cjs:122-177`; `tests/integration/multi-runtime.test.js:769-813`). Phase 58.1 should reuse that authority model, but not by importing `bin/install.js` directly. `bin/install.js` is a hotspot with side-effectful CLI defaults, and AGENTS explicitly says to keep source/runtime distinctions sharp and pair installer changes with focused tests (`AGENTS.md:39-44`, `AGENTS.md:94-109`).

**Primary recommendation:** Make Phase 58.1 a thin-workflow, shared-resolver repair: add one small Node-level Codex update-target resolver under `get-shit-done/bin/lib/`, expose it through installed `gsd-tools.cjs` (or a sibling script), rewrite the update workflow to consume that JSON and always emit explicit `--codex` / `--local|--global` / `--config-dir` arguments, and treat Codex update-cache clearing as an explicit no-op rather than a fake hook parity surface.

## Phase Shape

[assumed:reasoned] Recommended phase shape:

1. Add one shared resolver that returns:
   - detected runtime
   - local and global Codex install paths plus versions
   - selected target scope
   - explicit install args
   - resolved global config dir
   - cache file to clear, or `null`
   - explanatory reason string for user messaging and tests
2. Keep `commands/gsd/update.md` and `get-shit-done/workflows/update.md` thin:
   - the command remains an entrypoint
   - the workflow owns UX and confirmation

4. Keep Codex hook/statusline work out of scope; model Codex update cache behavior as `does-not-apply-with-reason`.
5. Add focused unit coverage for the resolver and one integration-style temp-install fixture proving Codex local/global/custom-dir routing.

## Architecture Patterns

### Recommended Project Structure

```text
commands/gsd/update.md                    # Thin command entrypoint
get-shit-done/workflows/update.md         # UX + confirmation only
get-shit-done/bin/lib/update-target.cjs   # New shared resolver (recommended seam)

tests/integration/multi-runtime.test.js   # Divergent-scope authority fixture
```

### Pattern 1: Thin Workflow Over JSON Resolver

**What:** The workflow shells out once to a shared helper that returns structured update-target data, then uses that data for messaging and execution.

**When to use:** Any update skill that needs runtime- and scope-aware behavior without duplicating installer rules in markdown.

**Example:**
```javascript
// Recommended contract shape, not existing code.
{
  runtime: 'codex',
  active_scope: 'global',
  local: { path: './.codex/get-shit-done-reflect/VERSION', version: '1.19.6+dev' },
  global: { path: '/home/user/.codex/get-shit-done-reflect/VERSION', version: '1.18.0' },
  selected_target: {
    scope: 'global',
    install_args: ['--codex', '--global'],
    config_dir: null,
    reason: 'active_global_stale_while_repo_mirror_is_newer'
  },
  cache_to_clear: null
}
```

### Pattern 2: Installed-Harness Authority With Repo-Mirror Fallback

**What:** Prefer the installed harness version over config and repo mirror; only use the repo mirror when no installed harness version exists.

**When to use:** Codex version reporting and stale-target selection, especially inside the source repo.

**Example:**
```javascript
if (installedVersion) return { value: installedVersion, source: 'installed_harness' };
if (configVersion) return { value: configVersion, source: 'config' };
if (repoMirrorVersion) return { value: repoMirrorVersion, source: 'repo_mirror' };
```
Source: `get-shit-done/bin/lib/provenance.cjs:147-169`

### Pattern 3: Explicit Nondefault Config-Dir Propagation

**What:** If the resolved global Codex dir is not `~/.codex`, carry that exact directory into the update command and user messaging.

**When to use:** Any global Codex update originating from `--config-dir` installs or `CODEX_CONFIG_DIR` environments.

**Example:**
```javascript
if (explicitDir) return expandTilde(explicitDir);
if (process.env.CODEX_CONFIG_DIR) return expandTilde(process.env.CODEX_CONFIG_DIR);
return path.join(os.homedir(), '.codex');
```
Source: `bin/install.js:163-171`

### Anti-Patterns to Avoid

- **Local-first shell snippets in markdown:** They hide stale global installs and are already inconsistent with the provenance authority rule.
- **Runtime-implicit `npx` installs:** `--global` or `--local` alone falls back to Claude (`bin/install.js:3336-3343`).
- **Importing `bin/install.js` directly for detection:** It is the right source of facts but the wrong long-term seam.
- **Pretending Codex has a live update-check cache:** Codex currently has no installed update-check hook or statusline surface in the installer path (`bin/install.js:2902-2908`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Codex global-dir resolution | Ad-hoc shell checks for `~/.codex`, env vars, and custom dirs | Shared helper extracted from `bin/install.js` precedence | Custom-dir and env precedence are easy to get wrong |
| Scope authority | Local-first `if [ -f ... ]` logic in markdown | Shared resolver using installed-harness > config > repo mirror | The source repo's `.codex/` mirror is an output, not authority |
| Cache semantics | Fake `.codex/cache/gsdr-update-check.json` cleanup | Resolver-provided `cache_to_clear`, which is `null` for Codex | Codex has no installed update-check hook today |
| Runtime selection | Implicit `--local` / `--global` install commands | Explicit `--codex` plus scope flag and optional `--config-dir` | The installer otherwise defaults to Claude |

**Key insight:** The dangerous complexity here is not npm or changelog fetching. It is scope authority. Once that policy lives in one pure helper, the workflow becomes simple again.

## Common Pitfalls

### Pitfall 1: Relative `.claude` Paths Survive Codex Installation

**What goes wrong:** Installed Codex update workflows still read `./.claude/...` for the "local" scope.

**Why it happens:** `replacePathsInContent()` rewrites `~/.claude/` and `$HOME/.claude/`, but not `./.claude/` (`bin/install.js:1348-1375`). Temp installs from current source confirm the resulting Codex workflow still contains `./.claude/get-shit-done-reflect/VERSION`.

**How to avoid:** Move scope detection out of markdown, or add direct regression coverage for relative-path conversion if any markdown path logic remains.

**Warning signs:** A Codex-installed local update workflow prints `./.claude/` as the local install path.

### Pitfall 2: Codex Update Commands Quietly Target Claude

**What goes wrong:** A Codex skill runs `npx get-shit-done-reflect-cc --global` and updates Claude instead of Codex.

**Why it happens:** `install.js` defaults runtime-less global/local installs to Claude (`bin/install.js:3336-3343`), while the workflow currently emits runtime-less commands (`get-shit-done/workflows/update.md:144-155`).

**How to avoid:** Always emit `--codex` from the workflow's selected target.

**Warning signs:** After a Codex-triggered update, `.claude/` changes but `.codex/` remains stale.

### Pitfall 3: Repo Mirrors Mask the Real Stale Target

**What goes wrong:** In the source repo, a newer repo-local `.codex/` mirror makes the updater think everything is current while the active global Codex install is stale.

**Why it happens:** The workflow checks one scope first and stops. The test suite already proves this is the wrong authority model for Codex (`tests/integration/multi-runtime.test.js:769-813`).

**How to avoid:** Enumerate both scopes and select the stale target with explicit reasoning.

**Warning signs:** Global Codex help/skills still show the older version after an "up-to-date" message.

### Pitfall 4: Fictional Codex Update Cache Behavior

**What goes wrong:** The workflow clears a Codex update-hook cache path even though no installed Codex hook creates that cache.

**Why it happens:** The current source hook files are Claude-oriented (`hooks/gsd-check-update.js:10-17`; `hooks/gsd-statusline.js:71-104`) and Codex installs skip hooks entirely (`bin/install.js:2846-2908`).

**How to avoid:** Treat Codex cache cleanup as `does-not-apply-with-reason` unless a later phase actually ships a Codex update-check substrate.

**Warning signs:** The only reference to a Codex update cache is in workflow text, not installed runtime behavior.

## Open Questions

### Resolved

- **Q1: When both project-local and global Codex installs exist and diverge, should one `gsdr-update` invocation update only the stale target or update both scopes?**  
  **Answer:** Update one scope only. Select the stale target explicitly and report the other scope if it still diverges.  
  **Why:** Existing install semantics are scope-specific, `saveLocalPatches()` runs per target, and auto-updating both expands blast radius unnecessarily (`bin/install.js:2603-2614`, `bin/install.js:2640-2644`).  
  **Tradeoff:** User may need a second command if both scopes are stale, but the behavior stays narrow and inspectable.

- **Q2: Where should runtime-aware update detection live so the command/workflow, any hook-adjacent indicator surface, and tests all share one authority rule?**  
  **Answer:** In a new pure helper under `get-shit-done/bin/lib/`, exposed through installed `gsd-tools.cjs` or a small sibling script.  
  **Why:** The repo architecture expects runtime truth in Node code, not markdown; `provenance.cjs` already provides the closest reusable authority rule; `bin/install.js` is the wrong long-term owner because it is a CLI hotspot (`.planning/PROJECT.md:199-200`; `get-shit-done/bin/lib/provenance.cjs:122-177`; `AGENTS.md:107-109`).  
  **Tradeoff:** Slightly more initial plumbing than inline shell, but it prevents immediate re-drift.

- **Q3: What shared cache behavior, if any, actually matters for Codex once we acknowledge that Codex installs do not currently have hooks or a statusline?**  
  **Answer:** For 58.1, Codex update-hook cache behavior is `does-not-apply-with-reason`. Do not retrofit `hooks/gsd-check-update.js` or `hooks/gsd-statusline.js` for Codex in this phase. Only clear caches for runtimes that actually have the update-check hook surface.  
  **Why:** Codex installs still skip hooks/statusline entirely (`bin/install.js:2846-2908`), so the live bug is command-side routing, not shared hook infrastructure.  
  **Tradeoff:** Codex keeps no background update indicator until a later hook-parity phase, but this phase stays within the inserted narrow boundary.

- **Q4: How should non-default Codex config dirs participate in local/global divergence detection?**  
  **Answer:** Treat the resolved global config dir as part of the identity of the global scope. The resolver should emit it explicitly, and the update command should pass `--config-dir <resolvedDir>` whenever the global dir is not the default `~/.codex`.  
  **Why:** Current installed workflows already know the custom global path for reading VERSION and running `gsd-tools`, but they still emit `npx ... --global` with no `--config-dir`, which can reinstall the wrong location. Temp-install inspection from current source confirmed this behavior on both explicit `--config-dir` and `CODEX_CONFIG_DIR` installs.  
  **Tradeoff:** The command line gets longer, but it stops the updater from silently targeting `~/.codex`.

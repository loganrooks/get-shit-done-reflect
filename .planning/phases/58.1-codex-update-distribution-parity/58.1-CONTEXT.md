# Phase 58.1: Codex Update Distribution Parity - Context

**Gathered:** 2026-04-20
**Status:** Ready for planning
**Mode:** Exploratory --auto -- user-reported runtime gap synthesised from roadmap state, cross-runtime research, and direct inspection of the source update surfaces

<domain>
## Phase Boundary

[decided:reasoned] Phase 58.1 is a narrow Codex runtime-parity repair inserted after Phase 58. Its job is not to redesign the whole update UX or solve the cross-project distribution gap. Its job is to ensure the Codex update command and adjacent update-indicator surfaces stop missing stale Codex installs because they still think in Claude-only paths, cache files, and local-first detection rules.

In scope for 58.1:
- Codex-aware installed-version detection for the update command and any shared update-indicator surfaces it owns
- Resolution rules when project-local and global Codex installs diverge, especially in the source repo where repo-local `.codex/` mirrors are outputs rather than authority
- Runtime-correct update execution: `--codex`, local/global targeting, and non-default Codex config-dir resolution when applicable
- Cleanup/alignment of any update cache or indicator surface touched by the same flow so Codex does not inherit Claude-only cache semantics by accident
- Focused regression coverage for divergent Codex install scopes and runtime-correct update routing

Explicitly out of scope for 58.1:
- A full multi-runtime redesign of `gsdr-update` for Claude, Gemini, OpenCode, and Codex simultaneously
- The broader cross-project distribution gap for non-GSDR projects running global upstream GSD; that remains a later milestone concern already called out in parity research
- Introducing Codex hook/statusline substrate parity; Codex install still has no installed hook or statusline surface in `bin/install.js`
- General patch-sensor or cross-runtime artifact drift work already scoped to Phase 60 and later

**Milestone anchor:** v1.20 target feature "Cross-runtime distribution gap: ensure GSDR fixes reach projects running global GSD" in `.planning/PROJECT.md`
**Depends on:** Phase 58 (for the governing XRT-01/per-runtime substrate contract recorded in ROADMAP.md; this dependency is on the declarative contract, not on Phase 58 already being complete)

</domain>

<working_model>
## Working Model & Assumptions

### Update command and workflow remain Claude-shaped

**Current state:** [evidenced:cited] The source update command and workflow still describe update detection and execution in Claude-oriented terms. `commands/gsd/update.md` points at `~/.claude/get-shit-done/workflows/update.md`, and `get-shit-done/workflows/update.md` checks only `./.claude/get-shit-done/VERSION` and `~/.claude/get-shit-done/VERSION`, runs `npx get-shit-done-reflect-cc --local/--global` without a runtime flag, and clears only `.claude/cache/gsd-update-check.json`.

- [evidenced:cited] `commands/gsd/update.md` and `get-shit-done/workflows/update.md` still encode Claude-first path assumptions in the source repo.
- [decided:reasoned] 58.1 must patch the source command/workflow surface, not the generated `.codex/` mirror, because the source repo is authoritative and the mirror is an output.
- [governing:reasoned] Runtime-specific path or cache behavior should not be encoded in hand-maintained markdown in multiple places when the installer/runtime layer already owns the directory rules.

### Codex version authority is already solved elsewhere, but update flow is not using it

**Current state:** [evidenced:cited] `bin/install.js` already knows Codex runtime directories and precedence (`--config-dir` -> `CODEX_CONFIG_DIR` -> `~/.codex`). The integration suite already contains a concrete regression for a similar authority problem: `tests/integration/multi-runtime.test.js` proves Codex artifact provenance must prefer the active installed harness VERSION over a stale repo-local `.codex` mirror.

- [evidenced:cited] `bin/install.js` `getGlobalDir('codex')` already resolves Codex global location correctly, including explicit and env-based config dirs.
- [evidenced:cited] `tests/integration/multi-runtime.test.js` already encodes the principle that installed Codex harness VERSION outranks a repo-local `.codex/get-shit-done-reflect/VERSION` mirror.
- [decided:reasoned] The update flow should adopt the same authority rule: active installed Codex scope first, repo-local mirror last, and never let a repo-local mirror in the source repo silently mask a stale global Codex install.
- [decided:reasoned] When project-local and global Codex installs both exist and diverge, 58.1 should detect the divergence and target the older stale scope rather than blindly updating whichever path is checked first.
- [assumed:reasoned] The same runtime/config-dir facts used by the installer can be surfaced to the update flow without inventing a second independent path-resolution system.

### Hook-adjacent update indicators are real, but Codex should not inherit fictional hook substrate

**Current state:** [evidenced:cited] The source repo contains `hooks/gsd-check-update.js` and `hooks/gsd-statusline.js`, and both still carry Claude-scoped update-cache semantics. `gsd-check-update.js` hard-codes `~/.claude/cache/gsd-update-check.json`, while `gsd-statusline.js` reads the same cache filename from `CLAUDE_CONFIG_DIR || ~/.claude`. At the same time, `bin/install.js` explicitly does not install hooks or statusline for Codex.

- [evidenced:cited] `hooks/gsd-check-update.js` hard-codes `.claude/cache/gsd-update-check.json`, while `hooks/gsd-statusline.js` still reads the same cache filename from a Claude-scoped config root.
- [evidenced:cited] `bin/install.js` returns early for Codex install before writing `settings.json`, hook registrations, or statusline configuration.
- [decided:reasoned] 58.1 may clean up shared update-indicator semantics if the command/update flow depends on them, but it must not silently expand Codex into a new hook/statusline runtime.
- [governing:reasoned] A missing Codex hook/statusline substrate should be represented honestly as absence or `does-not-apply-with-reason`, not papered over by writing Claude cache paths into Codex-facing logic.

### Codex behavior

- [evidenced:cited] Phase 58's roadmap contract requires explicit per-runtime substrate and Codex degradation / waiver paths for capability-touching work (`XRT-01` and Success Criterion 8).
- [decided:reasoned] `does-not-apply-with-reason`: per that Phase 58 contract, Codex installed runtime still has no installer-wired SessionStart hook or statusline surface; 58.1 addresses Codex through the explicit `gsdr-update` command / installed skill and shared source logic, not by introducing Codex hook parity.
- [decided:reasoned] `applies-via-installer`: if Claude/Gemini hook files or cache semantics are touched as shared source surfaces, they should continue to work through the existing installer-wired hook/statusline path without becoming the authority for Codex install targeting.

### Regression surface is present but missing update-specific coverage

**Current state:** [evidenced:cited] The repo already has strong runtime installation tests (`tests/unit/install.test.js`, `tests/integration/multi-runtime.test.js`) covering Codex layout, VERSION files, and adjacent source-vs-installed precedence behavior. What is missing is update-specific coverage for Codex divergence and update-target selection.

- [evidenced:cited] `tests/unit/install.test.js` already validates runtime-specific path rewriting, Codex install layout, and namespace transforms for installed hook files.
- [evidenced:cited] `tests/integration/multi-runtime.test.js` already validates Codex layout and one authoritative VERSION-precedence rule.
- [decided:reasoned] 58.1 must add dedicated regression coverage for update detection/routing rather than relying on unrelated install tests to catch this class of bug indirectly.
- [assumed:reasoned] Focused unit coverage plus one integration-style runtime-routing check should be sufficient; a full live npm update round-trip is unnecessary for this phase.

### Claude's Discretion

- Exact helper placement for runtime-aware update detection if a shared helper is warranted
- Whether cache alignment lands as runtime-tagged filenames, runtime-aware directory lookup, or a documented `does-not-apply` simplification
- The narrowest test split between unit and integration surfaces

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] A repo-local `.codex/get-shit-done-reflect/VERSION` file inside the source repo is not evidence of the user’s active global Codex install.
- **G-2:** [governing:reasoned] Project-local and global installs are separate scopes. Checking one and assuming the other is current is the bug class this phase exists to close.
- **G-3:** [governing:reasoned] Do not fix the gap by editing generated `.codex/` outputs directly. Any mirror repair that does not change source will regress on the next install.
- **G-4:** [governing:reasoned] Do not let cache/statusline cleanup drive scope. Clearing the wrong cache is a real bug, but inventing new cross-runtime notification infrastructure is not the goal.
- **G-5:** [governing:reasoned] If a shared helper is introduced, it should make runtime authority clearer, not hide more policy in another ambiguous fallback chain.
- **G-6:** [governing:reasoned] Cross-project update distribution beyond GSDR-managed installs remains out of scope. This phase fixes the in-project/in-runtime detection and routing gap that the user hit now.

</guardrails>

<questions>
## Open Questions

All four questions below were answered in `58.1-RESEARCH.md`. They remain here as the research agenda that phase research closed, so planner/verifier can trace what was open at discuss time and how it was resolved before execution planning.

### Q1: When both project-local and global Codex installs exist and diverge, should one `gsdr-update` invocation update only the stale target or update both scopes?
**Research program:** Inspect current update semantics, local-patch backup behavior, and user expectation implicit in existing `--local` / `--global` flows. Determine whether dual-update would be surprising or whether targeting the older stale scope preserves better visibility and lower blast radius.
**Downstream decisions affected:** Update-detection helper behavior, user messaging, workflow wording, and regression test shape.
**Reversibility:** MEDIUM — either choice is implementable, but switching later changes user-visible semantics.

### Q2: Where should runtime-aware update detection live so the command/workflow, any hook-adjacent indicator surface, and tests all share one authority rule?
**Research program:** Compare a thin shared library/helper against keeping the logic inline in workflow markdown plus hook JS. Evaluate testability, source-of-truth clarity, and how much the existing installer/runtime layer can be reused without circular coupling.
**Downstream decisions affected:** File ownership, test placement, and how much logic remains in markdown versus JS.
**Reversibility:** LOW — helper placement is refactorable, but duplicating detection logic now will likely recreate drift immediately.

### Q3: What shared cache behavior, if any, actually matters for Codex once we acknowledge that Codex installs do not currently have hooks or a statusline?
**Research program:** Trace whether the update command itself needs cache cleanup for mixed-runtime worktrees, or whether the only real bug is command-side runtime targeting. Distinguish live shared behavior from dead Claude-only leftovers.
**Downstream decisions affected:** Whether 58.1 edits the hook/statusline files at all, whether cache naming changes are warranted, and how broad the regression surface needs to be.
**Reversibility:** MEDIUM — unnecessary cache changes expand blast radius; missing a live shared cache surface leaves stale indicators behind.

### Q4: How should non-default Codex config dirs participate in local/global divergence detection?
**Research program:** Follow `bin/install.js` precedence, explicit `--config-dir` handling, and env-based configuration to determine whether update detection should expose the resolved config dir in messages and tests.
**Downstream decisions affected:** Update command invocation, helper inputs, and integration-test fixtures.
**Reversibility:** LOW — getting config-dir precedence wrong risks updating the wrong installation entirely.

</questions>

## Summary

This phase migrates the knowledge base primary location from `~/.gsd/knowledge/` (user-global, outside version control) to `.planning/knowledge/` (project-local, version-controlled). The migration touches 19 agent/workflow source files and 11 workflow/reference source files that reference `~/.gsd/knowledge/` paths, plus 3 test files, 2 shell scripts, and the installer. No changes to `gsd-tools.js` are needed -- all KB path logic is in agent specs and workflows.

The core technical challenge is implementing a consistent fallback pattern across 20+ files: check `.planning/knowledge/` first, fall back to `~/.gsd/knowledge/` when it does not exist. Each file uses KB paths differently -- some in bash code blocks, some in prose instructions, some in `cat`/`grep` commands -- so the fallback pattern must be adapted per context. Additionally, the lessons layer is deprecated (no `lessons/` directory in the new structure), and the signal schema gets optional environment enrichment fields.

A prior lesson [les-2026-02-16-dynamic-path-resolution-for-install-context] established that KB path references must resolve dynamically based on install context. This phase implements the recommended approach: check project-local first, fall back to user-global. The lesson's recommendation to use `GSD_HOME` environment variable is superseded by the simpler `.planning/knowledge/` directory existence check.

**Primary recommendation:** Implement a standard KB path resolution snippet used consistently across all agent specs and workflows, then apply it file-by-file with context-appropriate adaptation.

### Pattern 1: KB Path Resolution in Agent Specs (Bash Context)

**What:** Standard fallback pattern for bash code blocks in agent specs
**When to use:** Any agent spec or workflow that reads/writes KB in bash
**Recommended pattern:**

```bash
# KB path resolution -- project-local primary, user-global fallback
if [ -d ".planning/knowledge" ]; then
  KB_DIR=".planning/knowledge"
else
  KB_DIR="$HOME/.gsd/knowledge"
fi
```

This is the simplest detection heuristic. It checks directory existence, not index.md existence, because:
- A newly initialized project has the directory but may not yet have index.md
- Directory existence is the committed decision from CONTEXT.md

### Pattern 2: KB Path Resolution in Agent Specs (Prose/Instruction Context)

**What:** How to express the fallback in agent prose instructions (not executable code)
**When to use:** Agent specs that tell agents to "read `~/.gsd/knowledge/index.md`" in English prose

```markdown
Read the KB index:
- If `.planning/knowledge/` exists, read `.planning/knowledge/index.md`
- Otherwise, fall back to `~/.gsd/knowledge/index.md`
```

For simpler references that just name the path:

```markdown
... at the project knowledge base (`.planning/knowledge/` or `~/.gsd/knowledge/` fallback) ...
```

### Pattern 3: KB Path Resolution in Shell Scripts

**What:** How kb-rebuild-index.sh and reconcile-signal-lifecycle.sh resolve KB path
**When to use:** Standalone shell scripts that operate on KB

```bash
# KB path resolution: project-local primary, user-global fallback
if [ -d ".planning/knowledge" ]; then
  KB_DIR=".planning/knowledge"
elif [ -n "${GSD_HOME:-}" ]; then
  KB_DIR="$GSD_HOME/knowledge"
else
  KB_DIR="$HOME/.gsd/knowledge"
fi
```

The `GSD_HOME` check is preserved as a middle tier for backward compatibility with the existing scripts.

### Pattern 4: Migration Script

**What:** One-time migration of KB content
**When to use:** Run once after creating `.planning/knowledge/`

```bash
#!/usr/bin/env bash
# migrate-kb.sh -- One-time migration from ~/.gsd/knowledge/ to .planning/knowledge/
set -euo pipefail

SRC="$HOME/.gsd/knowledge"
DEST=".planning/knowledge"

if [ ! -d "$SRC" ]; then
  echo "No source KB at $SRC -- nothing to migrate"
  exit 0
fi

mkdir -p "$DEST/signals" "$DEST/reflections" "$DEST/spikes"

# Copy signals (preserve directory structure)
if [ -d "$SRC/signals" ]; then
  cp -r "$SRC/signals/"* "$DEST/signals/" 2>/dev/null || true
fi

# Copy reflections
if [ -d "$SRC/reflections" ]; then
  cp -r "$SRC/reflections/"* "$DEST/reflections/" 2>/dev/null || true
fi

# Copy spikes
if [ -d "$SRC/spikes" ]; then
  cp -r "$SRC/spikes/"* "$DEST/spikes/" 2>/dev/null || true
fi

# Rebuild index at new location
# (kb-rebuild-index.sh must be updated first to support project-local)
echo "Migration complete. Run kb-rebuild-index.sh to generate index."
```

**Lessons NOT migrated** -- deliberate omission per locked decision. The 7 existing lessons remain as historical artifacts at `~/.gsd/knowledge/lessons/`.

### Anti-Patterns to Avoid

- **Dual-write:** Writing to both locations simultaneously. Rejected in deliberation (Option C) due to complexity and inconsistency risk.
- **Symlink from `.planning/knowledge/` to `~/.gsd/knowledge/`:** Defeats the purpose of version control. The data must actually live in `.planning/`.
- **Hardcoding either path without fallback:** Must always check project-local first, fall back to user-global. Never assume one location.
- **Using `replacePathsInContent()` for KB migration:** The installer's path replacement is for `~/.claude/` to runtime-specific paths. KB paths (`~/.gsd/knowledge/`) are explicitly excluded by the two-pass system. The new `.planning/knowledge/` paths should NOT go through `replacePathsInContent()` at all -- they are relative paths, not runtime-specific.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Index generation | Custom index builder | Existing `kb-rebuild-index.sh` (updated for new path) | Already handles signals, spikes, lessons; atomic write; handles empty KB |
| Signal dedup | Custom dedup logic | Existing synthesizer dedup (unchanged) | Dedup logic is in agent spec, not path-dependent |
| Path replacement in installer | New path conversion logic | NO installer path conversion for `.planning/knowledge/` | Paths are relative, not runtime-specific; `replacePathsInContent()` should NOT touch them |

**Key insight:** This phase changes WHERE agents read/write, not HOW they read/write. The existing KB infrastructure (index builder, synthesizer, reflector) works unchanged once paths are updated. Don't redesign the KB -- just move it.

## Common Pitfalls

### Pitfall 1: Forgetting to Update Both Tilde and $HOME Variants

**What goes wrong:** Agent specs use both `~/.gsd/knowledge/` and `$HOME/.gsd/knowledge` (in bash variable assignments like `KB_DIR="$HOME/.gsd/knowledge"`). Missing one variant leaves stale references.
**Why it happens:** grep for `~/.gsd/knowledge` misses `$HOME/.gsd/knowledge` (no tilde).
**How to avoid:** Search for BOTH patterns: `~/.gsd/knowledge` AND `$HOME/.gsd/knowledge` AND `.gsd/knowledge` (catches both).
**Warning signs:** Tests pass but agents still write to user-global in some workflows.

### Pitfall 2: Installer replacePathsInContent() Interference

**What goes wrong:** The installer's two-pass path replacement has explicit handling for `~/.gsd/knowledge` (Pass 1 preserves it). If `.planning/knowledge/` references are added to source files, they must NOT be processed by `replacePathsInContent()`.
**Why it happens:** Source files (`agents/`, `get-shit-done/`) go through `replacePathsInContent()` during install. `.planning/knowledge/` is a relative path that should pass through unchanged.
**How to avoid:** Verify that `.planning/knowledge/` references are not affected by either pass of `replacePathsInContent()`. The function only matches `~/.claude/` and `$HOME/.claude/` patterns, so `.planning/knowledge/` is safe -- but verify this.
**Warning signs:** Installed agent specs at `.claude/agents/` have mangled `.planning/knowledge/` paths.

### Pitfall 3: kb-rebuild-index.sh Lessons Section

**What goes wrong:** The existing `kb-rebuild-index.sh` has a `build_lesson_rows()` function and always generates a `## Lessons` section in the index. If the project-local KB has no `lessons/` directory, this section will show 0 entries but still appear.
**Why it happens:** Script assumes all three types exist.
**How to avoid:** Either (a) keep the lessons section showing 0 entries (harmless), or (b) conditionally skip the section when no lessons directory exists. Option (a) is simpler and adequate -- the section being empty is self-documenting.
**Warning signs:** None critical -- this is cosmetic.

### Pitfall 4: reconcile-signal-lifecycle.sh Hardcoded Paths

**What goes wrong:** `reconcile-signal-lifecycle.sh` hardcodes `$HOME/.gsd/knowledge/signals` and `$HOME/.gsd/bin/kb-rebuild-index.sh`. These must be updated with the same fallback pattern.
**Why it happens:** Script was written before project-local KB was conceived.
**How to avoid:** Apply Pattern 3 (shell script path resolution) to this script.
**Warning signs:** Signal lifecycle reconciliation silently writes to wrong KB location.

### Pitfall 5: Test Assertions That Enforce `~/.gsd/knowledge/`

**What goes wrong:** Tests in `multi-runtime.test.js` and `cross-runtime-kb.test.js` explicitly assert that files reference `~/.gsd/knowledge/`. After migration, source files will reference `.planning/knowledge/` as primary. Tests must be updated to accept both.
**Why it happens:** Tests were designed to enforce the old single-location model.
**How to avoid:** Update test assertions to accept `.planning/knowledge/` as primary while still allowing `~/.gsd/knowledge/` references (fallback path mentions).
**Warning signs:** `npm test` fails with "uses old path" assertions.

### Pitfall 6: Reflector Spec Still Producing Lessons

**What goes wrong:** If the reflector spec is not updated to stop writing lesson files, new reflections will attempt to write lessons to a nonexistent `lessons/` directory.
**Why it happens:** Reflector spec at `agents/gsd-reflector.md` line 334 instructs writing to `~/.gsd/knowledge/lessons/{category}/`.
**How to avoid:** Update reflector spec to remove lesson-writing instructions. Reflection reports remain; only individual lesson file generation is removed.
**Warning signs:** Reflector errors on missing lessons directory, or writes lessons to user-global location.

### Pitfall 7: Partial Source/Runtime Sync

**What goes wrong:** Editing source files (`agents/`, `get-shit-done/`) but forgetting to run `node bin/install.js --local` to sync to `.claude/`. CLAUDE.md explicitly warns about this (Phase 22 incident).
**Why it happens:** Dual-directory architecture of this project.
**How to avoid:** Each plan's verification step should include `node bin/install.js --local` and verify `.claude/` files match.
**Warning signs:** Source files updated but runtime agents still use old paths.

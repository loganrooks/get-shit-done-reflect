# Phase 38.1: Project-Local Knowledge Base - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning
**Deliberation:** `.planning/deliberations/project-local-knowledge-base.md`

<domain>
## Phase Boundary

Migrate KB primary location from `~/.gsd/knowledge/` to `.planning/knowledge/` for version control and remote execution access. Deprecate the lessons layer. Enrich signal schema for cross-project readiness. Graceful fallback to `~/.gsd/knowledge/` for environments without project-local KB.

</domain>

<decisions>
## Implementation Decisions

### MUST: Location Migration

- **Structure:** `.planning/knowledge/` with subdirectories: `signals/`, `reflections/`, `spikes/` — NO `lessons/`
- **Agent files:** All 19 source files updated in one sweep — no partial migration
- **Tests:** All 5 test files updated; tests cover BOTH project-local primary AND user-global fallback
- **No gsd-tools.js changes** — KB path logic stays in agent specs (gsd-tools.js has zero KB references)
- **kb-rebuild-index:** Update shell script to work on `.planning/knowledge/`

### Fallback behavior

- When `.planning/knowledge/` exists, agents use it exclusively
- When `.planning/knowledge/` doesn't exist, agents fall back to `~/.gsd/knowledge/`
- Detection heuristic: `.planning/knowledge/` directory existence (simple, direct)
- Writes go to project-local if it exists, otherwise to user-global
- Silent operation — no logging of which KB location is active

### Content migration

- One-time migration of existing signals, reflections, spikes from `~/.gsd/knowledge/`
- Lessons are NOT migrated (deprecated — see structural decisions below)
- Migration automated via script
- Index.md rebuilt from scratch at new location
- `~/.gsd/knowledge/` left in place after migration (fallback for non-project contexts)

### Installer changes

- `.planning/knowledge/` created during `gsd:new-project` (not general install)
- If `.planning/` exists when installer runs, ensure `.planning/knowledge/` exists too
- Agent specs use relative path `.planning/knowledge/` directly — no installer path conversion needed
- All KB content fully committed to git — knowledge evolution is auditable

### SHOULD: Structural Clarity — Lessons Deprecation

- **No `lessons/` directory** in new `.planning/knowledge/` structure
- **Existing 7 lessons** remain at `~/.gsd/knowledge/lessons/` as historical artifacts (not migrated)
- **Follow-up task** (not this phase): promote lesson content to agent specs / workflow docs where it belongs:
  - "Exhaustive KEEP lists" → executor agent spec constraint
  - "Runtime verification" → verifier agent spec instruction
  - "Verify before assuming" → planner agent spec checklist
  - "Context bloat" → agent protocol principle
  - "Plan checker gaps" → plan checker code enhancement
  - "Zero deviation = spec phases" → observation, no permanent home needed
  - "Dynamic path resolution" → already in CLAUDE.md
- **Reflector spec:** Update to stop producing lesson files; reflection reports (patterns, triage proposals, remediation suggestions) all remain — that's where the real analytical value lives
- **KB becomes three types:** signals + reflections + spikes

### COULD: Future-Proofing — Signal Schema Enrichment

- **Add environment fields** to signal frontmatter:
  - `environment.os` — auto-populated from system (e.g., "darwin", "linux")
  - `environment.node_version` — auto-populated
  - `environment.config_profile` — from project config (quality/balanced/budget)
- **Add `source: local` field** — default value; enables future `external` signals from other projects
- **Reserve fields** (documented in deliberation, NOT implemented):
  - `upstream_ref` — link to shared version when signal-sharing is built
  - `upstream_status` — none|reported|acknowledged|fixed
- **Update signal-producing workflows** (signal command, collect-signals sensors) to include new fields

### Claude's Discretion

- Exact fallback detection pattern in agent specs (bash conditional, ls check, etc.)
- Migration script location (standalone script vs gsd-tools.js subcommand)
- Order of agent file updates within each plan
- How to update existing signal files (add new fields retroactively vs only new signals)

</decisions>

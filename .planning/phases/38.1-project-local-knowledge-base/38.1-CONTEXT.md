# Phase 38.1: Project-Local Knowledge Base - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate KB primary location from `~/.gsd/knowledge/` to `.planning/knowledge/` so the KB is version-controlled and available in remote execution environments. Graceful fallback to `~/.gsd/knowledge/` for environments without project-local KB. No new KB capabilities — this is a location migration with fallback semantics.

</domain>

<decisions>
## Implementation Decisions

### Migration scope
- All 19 agent files updated in one sweep — no partial migration
- All 5 test files updated to assert project-local primary
- Tests cover both project-local primary AND user-global fallback behavior
- No new `gsd-tools.js kb` subcommand — keep KB path logic in agent specs (agents resolve paths themselves)
- kb-rebuild-index: update the shell script to work on `.planning/knowledge/` (keep as shell script for now)

### Fallback behavior
- When `.planning/knowledge/` exists, agents use it exclusively — ignore `~/.gsd/knowledge/`
- When `.planning/knowledge/` doesn't exist, agents fall back to `~/.gsd/knowledge/`
- Detection heuristic: `.planning/knowledge/` directory existence (simple, direct)
- Writes go to project-local if it exists, otherwise to user-global
- Silent operation — no logging of which KB location is active (agents just use the right path)

### Content migration
- One-time migration of existing `~/.gsd/knowledge/` content into `.planning/knowledge/`
- Migration automated via script (not manual steps)
- Index.md rebuilt from scratch at new location (ensures index matches actual content)
- `~/.gsd/knowledge/` left in place after migration — still serves as fallback for non-project contexts

### Installer changes
- `.planning/knowledge/` created during `gsd:new-project` only (not during general install)
- If `.planning/` already exists when installer runs, ensure `.planning/knowledge/` exists too
- Agent specs use relative path `.planning/knowledge/` directly — no installer path conversion needed
- Directory structure mirrors `~/.gsd/knowledge/` exactly (signals/, lessons/, spikes/, reflections/, index.md)
- All KB content fully committed to git — knowledge evolution is auditable

### Claude's Discretion
- Exact fallback detection pattern in agent specs (bash conditional, ls check, etc.)
- Migration script location (standalone script vs gsd-tools.js subcommand)
- Order of agent file updates within each plan
- Whether to add .planning/knowledge/ to .gitignore template for non-GSD-Reflect projects

</decisions>

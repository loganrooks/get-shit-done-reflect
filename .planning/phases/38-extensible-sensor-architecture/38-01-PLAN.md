<task type="auto">
  <name>Task 1: Define sensor contract and rewrite collect-signals for auto-discovery</name>
  <files>get-shit-done/workflows/collect-signals.md</files>
  <action>
Rewrite the collect-signals.md workflow to replace hardcoded sensor spawning with dynamic auto-discovery. The key changes:

**1. Add `discover_sensors` step** (new step, before `load_sensor_config`):
- Scan `.claude/agents/gsd-*-sensor.md` files (runtime path; the npm source uses `~/.claude/agents/` which the installer converts to `./.claude/agents/`)
- For each file: extract sensor name from filename pattern `gsd-{name}-sensor.md` -> `{name}`
- Parse frontmatter for contract fields: `sensor_name`, `timeout_seconds`, `config_schema`
- Handle frontmatter parse failures gracefully: log warning, skip malformed sensor, continue
- Build `DISCOVERED_SENSORS` list with: name, spec_path, timeout_seconds (default 45s if not declared)
- Report: "Discovered {N} sensors: {names}"

**2. Modify `load_sensor_config` step:**
- Remove the hardcoded fallback default `{artifact: {...}, git: {...}, log: {...}}`
- For each discovered sensor, check `config.signal_collection.sensors[name]`
- If config entry exists: use its `enabled` and `model` values
- If no config entry exists: default to `{enabled: true, model: "auto"}` -- this is the critical "drop a file" design decision per user decision
- To disable a sensor, user adds explicit `"sensor_name": {"enabled": false}` to config
- Build `ENABLED_SENSORS` and `DISABLED_SENSORS` lists from cross-referencing discovery with config
- Report: "Enabled: {names}. Disabled: {names}."

**3. Rewrite `spawn_sensors` step:**
- Remove the three hardcoded Task() blocks (artifact, git, log)
- Replace with a dynamic FOR EACH loop over ENABLED_SENSORS
- Each spawn: `Task(subagent_type="gsd-{name}-sensor", model="{resolved_model}", run_in_background=true, ...)`
- Pass the same context as before: phase directory, project name, model profile
- Track spawned sensor names for output collection

**4. Modify `collect_sensor_outputs` step:**
- Add per-sensor timeout tracking: record spawn timestamp, compare against sensor's declared `timeout_seconds` when collecting output
- If a sensor's output is not received within its timeout: treat as returning empty array, log inline warning "WARNING: {name}-sensor timed out ({timeout}s), signals may be incomplete"
- After each sensor output is collected (success or failure), track stats via Phase 37 mechanism:
  - Success: `node ./.claude/get-shit-done/bin/gsd-tools.js automation track-event "sensor_{name}" fire`
  - Failure/timeout: `node ./.claude/get-shit-done/bin/gsd-tools.js automation track-event "sensor_{name}" skip "{reason}"` where reason is "timeout", "parse-error", or "agent-error"

**5. Modify `present_results` step:**
- Add standing caveat at end of collection output: "Sensors: {enabled_list}. For sensor limitations, see agent specs or run `gsd sensors blind-spots`."
- Include inline warnings for any sensors that failed or timed out (per user decision)

**6. Document the sensor contract** in a comment block at the top of collect-signals.md (inside the existing architecture notes comment). The contract specifies:
- **Input:** Sensor receives phase number, phase directory path, project name, model profile
- **Output:** JSON wrapped in `## SENSOR OUTPUT` / `## END SENSOR OUTPUT` delimiters with structure `{sensor: string, phase: number, signals: array}`
- **Error handling:** On failure, return empty signals array -- never crash or return non-JSON
- **Timeout:** Declared via `timeout_seconds` frontmatter field; orchestrator enforces; on timeout, treated as empty array
- **Config:** `config_schema` frontmatter field (optional, null for current sensors; Phase 39 CI sensor expected to use it)
- **Blind spots:** `<blind_spots>` section in agent spec body (prose documentation, not structured data)

**Important anti-patterns to avoid:**
- Do NOT hardcode any sensor name as a literal string in the workflow logic (only in comments/documentation). The whole point of auto-discovery is eliminating this.
- Do NOT modify config.json during discovery. Discovery is read-only.
- Do NOT parse agent spec bodies for contract data. Contract metadata is in frontmatter only.
- The workflow uses `~/` prefix paths (installer converts to `./` during install). Use `~/.claude/agents/gsd-*-sensor.md` as the discovery path.

<task type="auto">
  <name>Task 2: Update feature-manifest sensor defaults for auto-discovery compatibility</name>
  <files>get-shit-done/feature-manifest.json</files>
  <action>
Update the `signal_collection` feature in feature-manifest.json to be compatible with auto-discovery:

**1. Update the `sensors` schema field:**
- Change the `default` from the hardcoded `{artifact: {...}, git: {...}, log: {...}}` to an empty object `{}`
- Update the `description` to: "Per-sensor overrides (enabled/disabled, model). Discovered sensors not listed here default to enabled with model 'auto'."
- This is the key change: the manifest no longer needs to know which sensors exist. The file system is the source of truth for which sensors exist; config is only for overrides.

**2. Keep the log sensor's disabled default as a known behavior note:**
- The log sensor is currently disabled by default via the hardcoded manifest. After this change, if a user has no `sensors` config at all, the log sensor would become enabled by default (since all discovered sensors default to enabled).
- To preserve backward compatibility: the collect-signals workflow's auto-discovery handles this correctly because the log sensor's agent spec frontmatter can indicate it's a disabled stub. However, per the architecture decision, the config is what controls enable/disable, not the agent spec.
- Solution: Add a comment in the manifest description noting that projects upgrading from pre-38 should add `"log": {"enabled": false}` to their sensors config if they want to keep the log sensor disabled. The version migration system (Phase 6) can handle this for existing projects.

**3. Verify no other manifest references need updating:**
- Check that `init_prompts` for signal_collection still make sense (the _gate prompt asking to configure sensors is still valid)
- No changes needed to other features

Run `npm test` after changes to verify nothing breaks.
  </action>

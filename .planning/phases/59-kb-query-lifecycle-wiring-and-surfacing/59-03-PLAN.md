<objective>
Implement the four-check `kb health` contract locked by ROADMAP SC-5 and audit §7.1 #7. Ships in the same wave as the query/search/link-show read surface (Plan 02) because it shares no files with that plan and has no mutable-state concerns — both plans read from the kb.db and the signal files without writing. Health check is the watchdog that makes the dual-write invariant testable without running a real transition, and makes the lifecycle-vs-plan drift (the audit's central B1 concern) observable.

Purpose: Without `kb health`, there is no user-facing way to detect: (a) re-emergence of malformed edges after Plan 01 repairs, (b) divergence between file and SQL `lifecycle_state` (dual-write bugs would go silent), (c) plans with `resolves_signals` referring to signals that never transitioned to `remediated`, or (d) advisory-layer awareness of how many `depends_on` references exist. The four checks are the minimum needed to close the loops Phase 59 opens.
Output: A new `lib/kb-health.cjs` module, router wiring for `kb health [--all] [--format json]`, and unit coverage for each check on both clean and planted-failure fixtures.
</objective>

  <action>Create `get-shit-done/bin/lib/kb-health.cjs` exporting `cmdKbHealth(cwd, options, raw)`. Reuse existing helpers — research Pitfall 8 is explicit that `kb health` must NOT duplicate walkers/parsers. Import (or re-export) `discoverSignalFiles`, `getKbDir`, `getDbPath` from `./kb.cjs`; import `extractFrontmatter` from `./frontmatter.cjs`. Required behavior per research Pattern 5:

**Check 1 — Edge integrity (reuses Plan 01 logic):**
- Query: `SELECT link_type, COUNT(*) FROM signal_links GROUP BY link_type` then per type compute total/resolves/orphaned/malformed (same shape as Plan 01 rebuild integrity report).
- PASS iff malformed=0 AND orphan rate (% of total) < 5% (configurable threshold, keep simple).
- On FAIL: print `FAIL: edge integrity — <N> malformed, <M> orphaned. Run 'kb repair --malformed-targets' or investigate source signal files.`
- On PASS: print `PASS: edge integrity — 0 malformed, <M> orphaned (<P%>)`.

**Check 2 — Lifecycle-vs-plan consistency:**
- Walk `.planning/phases/**/NN-PLAN.md` (pattern: `.planning/phases/*/*-PLAN.md` plus decimal-phase directories like `58.1-...`). For each plan file, `extractFrontmatter()` and read `resolves_signals` (expected to be an array of signal IDs or absent).
- Only consider plans where a matching `-SUMMARY.md` file exists in the same directory — that signals the plan is complete (matches existing execute-plan workflow pattern).
- For each resolves_signals id in a completed plan, query `SELECT lifecycle_state FROM signals WHERE id = ?`. Expected lifecycle_state: `remediated` or `verified`. Any other value (or signal not found) is a drift.
- PASS iff all resolves_signals references of completed plans resolve to remediated/verified state.
- On FAIL: print `FAIL: lifecycle-vs-plan consistency — <N> drift(s): <plan>.resolves_signals[<id>] is in state '<state>', expected remediated/verified. Run 'kb transition' or 'reconcile-signal-lifecycle.sh' (deprecated).`
- (This is Option B from the 2026-03-04 deliberation, per audit §7.1 #7.)

**Check 3 — Dual-write verification (KB-05 invariant):**
- Select signal sample: default 20 random signals (`options.all ? all : sample(20)`). Use a deterministic seed-from-cwd + timestamp hash for reproducibility in tests, OR `Math.random()` with a `--seed N` flag. Pick whatever fits existing test style; fixed-seed-in-tests via an env var is fine.
- For each sampled signal: (a) query `SELECT lifecycle_state FROM signals WHERE id = ?` → `sqlState`; (b) find the .md file via `discoverSignalFiles()` + basename match, read it, extract frontmatter → `fileState = fm.lifecycle_state || 'detected'`; (c) assert `sqlState === fileState`.
- PASS iff all sampled signals match.
- On FAIL: print `FAIL: dual-write invariant — <N>/<total sampled> signals diverge: <first-3-divergences>. Run 'kb rebuild' to re-sync from files (files are source of truth per KB-05).`
- With `--all`, `total sampled` = all signals (278 today). Without, = 20.

**Check 4 — `depends_on` freshness summary (advisory):**
- Scan all signal + spike rows for populated `depends_on` fields (not null, not empty array).
- For each referenced path (depends_on fields are paths like `package.json` per knowledge-store.md §4), check `fs.existsSync(path.join(cwd, <ref>))`.
- This check is ALWAYS `PASS` (it's a summary, not a gate — per research Pattern 5 and Pitfall C4/D2). The output is `SUMMARY: depends_on freshness — <N> KB entries carry depends_on; <M> references resolve to extant paths; <K> references dangling.`
- Dangling references are listed if `options.format === 'json'` or `--verbose` is set, but they don't trip exit code.

**Overall exit code:** 0 if all four checks pass (dual-write, edge integrity, lifecycle-vs-plan all PASS; depends_on is always SUMMARY not FAIL). Non-zero otherwise; exit code encodes which check failed (bitmask: 1=edge, 2=lifecycle, 4=dual-write; 0b111 = 7 if all three hard-checks fail simultaneously). This lets callers like CI distinguish failure classes without re-parsing text.

Output formats:
- Default text: four sections labeled `Check 1`..`Check 4`, each with `PASS`/`FAIL`/`SUMMARY` header + one-sentence remediation.
- `--format json` or `--raw`: `{ exit_code, checks: { edge_integrity: {status, ...}, lifecycle_vs_plan: {status, ...}, dual_write: {status, sample_size, ...}, depends_on_freshness: {status, counts, ...} } }`.

Wire into `gsd-tools.cjs` `case 'kb'` router: `subcommand === 'health'` → dispatch to `kbHealth.cmdKbHealth(cwd, parseHealthOptions(args), raw)`. Usage string extended: `gsd-tools kb <rebuild|stats|migrate|repair|query|search|link show|health|...>` (Plan 04 adds `transition` and `link create|delete`).

No file writes in any check. No SQL writes. Read-only; shares the kb.db connection safely with concurrent Plan 02 readers.</action>

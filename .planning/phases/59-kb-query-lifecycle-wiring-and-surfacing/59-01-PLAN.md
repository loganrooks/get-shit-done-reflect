  <action>Edit `get-shit-done/bin/lib/kb.cjs` in three coordinated changes:

Per research R12, this is the cheap insurance that prevents the `[object Object]` bug class from re-emerging on `qualified_by` edges once they start being written. For array-valued fields (`qualified_by`, `related_signals`), validate each element with the same `typeof === 'string'` check before pushing. The current guard `if (fm.recurrence_of && String(fm.recurrence_of).trim())` is exactly what produced `"[object Object]"` for empty-object YAML values — that pattern must disappear from all four spots.

(b) `initSchema(db)` (lines ~59-103): add `CREATE INDEX IF NOT EXISTS idx_signal_links_target ON signal_links(target_id, link_type)`. Idempotent, no migration drama. Per research R2, this index is the load-bearing fix for KB-04c inbound-edge lookups — without it `SELECT source_id FROM signal_links WHERE target_id = ?` plans as SCAN, which blows up at larger corpora. ALSO extend the `CREATE TABLE IF NOT EXISTS signal_links (...)` statement (lines ~98-103) with two new columns: `created_at TEXT DEFAULT ''` and `source_content_hash TEXT DEFAULT ''` — these implement audit §7.1 #8 "Edge provenance minimum" (enables future forensic work cheaply). Because `kb.db` is derived and gitignored per KB-05, and the schema_version bump 2→3 in (c) below already forces a full rebuild, no ALTER TABLE is needed — the rebuild recreates the table with the new shape. Corresponding `INSERT OR IGNORE INTO signal_links` statement (line ~579) MUST be updated to `INSERT OR IGNORE INTO signal_links (source_id, target_id, link_type, created_at, source_content_hash) VALUES (?, ?, ?, ?, ?)` and populated on insert during `extractLinks()` / rebuild loop: `created_at = new Date().toISOString()` (ISO timestamp at insert time), `source_content_hash = crypto.createHash('sha256').update(<source signal .md file body contents>).digest('hex')` (reuse the content hash already computed for `signals.content_hash` if available at that point in the rebuild loop — avoids a second file read).

(c) FTS5 wiring (new block in `initSchema()`, after signal_links setup): follow research Pattern 1 exactly — create `signal_fts` as `CREATE VIRTUAL TABLE IF NOT EXISTS signal_fts USING fts5(id UNINDEXED, title, body, content='signals', content_rowid='rowid', tokenize='porter unicode61')` PLUS three AFTER triggers (signals_ai / signals_ad / signals_au) that mirror INSERT/DELETE/UPDATE. Then bump `meta.schema_version` from 2 to 3, and during version-bump run `INSERT INTO signal_fts(signal_fts) VALUES('rebuild')` so existing 278 signal rows get indexed. Do NOT revive the canonical-row FTS shape that Phase 57.7 dropped (see kb.cjs:172-185 comment and Pitfall 1 in RESEARCH.md) — the content='signals' external-content form is explicitly the correct rewrite.

Note: the current `signals` table does not have `title` / `body` columns. Options per research: (i) ALTER TABLE signals ADD COLUMN title TEXT DEFAULT '' / ADD COLUMN body TEXT DEFAULT '' and populate these during the rebuild loop by deriving `title` from the first H2 of the markdown body and `body` as the post-frontmatter body text; (ii) keep FTS5 contentless-with-triggers where the triggers themselves derive title/body inline. Pick option (i) because it keeps the column shape declarative and the triggers simple; make sure ALTER TABLE runs via `CREATE TABLE IF NOT EXISTS` + ALTER-idempotence (check pragma table_info first). Rebuild must populate `signals.title` and `signals.body` on every INSERT OR REPLACE so FTS stays coherent.

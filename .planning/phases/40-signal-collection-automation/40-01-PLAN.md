  <action>
Add three functions and CLI routing to gsd-tools.js for reentrancy lockfile management:

1. `cmdAutomationLock(cwd, feature, options, raw)`:
   - Lock path: `.planning/.{normalized_feature}.lock` (project-local, uses underscore-normalized feature name)
   - First check for existing lock. If exists:
     - Compare mtime against TTL (from `--ttl N` option, default 300 seconds)
     - If age > TTL: stale lock. Remove it, then proceed to acquire (stale info included in final output)
     - If age <= TTL: active lock. Output `{ locked: true, holder: {parsed lock content}, age_seconds }` and return (do NOT acquire)
   - If no lock or stale removed: write lock file with JSON content:
     ```json
     { "pid": process.pid, "timestamp": ISO, "trigger_source": options.source || "unknown", "ttl_seconds": ttl }
     ```
   - Output a SINGLE JSON object (never two -- consumers parse stdout as one JSON):
     - Normal acquire: `{ locked: false, acquired: true }`
     - Stale-then-acquire: `{ locked: false, acquired: true, stale_removed: true, stale_age_seconds: N }`

2. `cmdAutomationUnlock(cwd, feature, raw)`:
   - Lock path same convention as lock
   - If file exists: unlink, output `{ released: true }`
   - If not: output `{ released: false, reason: 'no_lock_found' }`

3. `cmdAutomationCheckLock(cwd, feature, options, raw)`:
   - Read-only check (does NOT acquire). Same stale detection as lock.
   - Output `{ locked: true/false, stale: true/false, age_seconds, holder }` as appropriate

Add CLI routing in the `automation` case block (after existing `track-event` else-if):
- `lock`: parse `--source` and `--ttl` options, call `cmdAutomationLock`
- `unlock`: call `cmdAutomationUnlock`
- `check-lock`: parse `--ttl` option, call `cmdAutomationCheckLock`
- Update the error message to include new subcommands: "Available: resolve-level, track-event, lock, unlock, check-lock, regime-change"

Also add `auto_collect` field to feature-manifest.json under `signal_collection.schema`:

Also add `.planning/.*.lock` to `.gitignore` to prevent stale lockfiles from appearing in `git status`.

Important: Place new functions near the existing automation functions (after `cmdAutomationTrackEvent`, around line 5255). Follow the same patterns: use `output()` for JSON, `error()` for failures, normalize feature names with `.replace(/-/g, '_')`.

Critical: The `cmdAutomationLock` function must emit exactly ONE `output()` call per invocation. When a stale lock is removed and a new lock acquired, combine into a single output object (`{ locked: false, acquired: true, stale_removed: true, stale_age_seconds: N }`). Two `output()` calls would produce invalid JSON on stdout, breaking all consumers that parse the result. See sig-2026-03-05-phase40-plan-gaps-pre-execution-review.

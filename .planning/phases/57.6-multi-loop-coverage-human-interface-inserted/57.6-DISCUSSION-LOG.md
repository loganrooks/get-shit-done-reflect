## Gray Areas (Audit Trail)

### Loop extension strategy and extractor sequencing

| Option | Description | Selected |
|--------|-------------|----------|
| All net-new extractors per loop | Write four fresh extractor sets (one per remaining loop) and leave existing tags alone | |
| Layered: tag first, extend second | Extend `serves_loop` tagging on existing extractors where defensible, then add net-new extractors for gaps | ✓ |
| Tag-only (no new extractors) | Rely entirely on re-tagging the 17-extractor registry to populate remaining loops | |

**Selected option:** Layered: tag first, extend second.
**Rationale (auto):** 57.5-VERIFICATION.md explicitly flags `serves_loop` concentration as residual technical debt. The 17-extractor registry already contains loop-relevant features (`runtime_session_identity`, `claude_settings_at_start`, `kb_signal_stats`, `codex_runtime_metadata`) that are simply not tagged for the four remaining loops. Layered coverage closes the tagging gap cheaply before committing net-new extractors, and the requirement text (MEAS-RUNTIME-01/06/07/08, MEAS-DERIVED-02/03/04, MEAS-GSDR-03) still requires net-new work that tagging alone cannot deliver. Tag-only would fail the roadmap's "extractors shipped" success criterion; all-net-new would duplicate data already in the registry.

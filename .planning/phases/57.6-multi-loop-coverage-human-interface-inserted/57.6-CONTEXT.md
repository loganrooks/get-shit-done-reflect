## Phase Boundary

Phase 57.6 extends the three-layer measurement substrate 57.5 landed so it actually serves the four self-improvement loops the deliberation already named but 57.5 only touched at the taxonomy layer (agent-performance, signal-quality, cross-session-patterns, cross-runtime-comparison), adds a human-first text-rendered report surface on top of the JSON contract, and produces an end-to-end diagnostic demonstration that exercises the epistemic machinery (competing interpretations, distinguishing features, anomaly register) on a real observed pattern rather than a toy case.

In scope for 57.6:
- additional `serves_loop` tagging on existing extractors so the six-loop taxonomy is not cosmetic — every loop declared in `LOOP_DEFINITIONS` must have substantive features flowing through it
- net-new extractors specifically for the four remaining loops: thinking-composite (`MEAS-RUNTIME-01`), marker-density revised (`MEAS-RUNTIME-06`), compaction cross-runtime (`MEAS-RUNTIME-07`), clear-invocation (`MEAS-RUNTIME-08`), facets first-class (`MEAS-DERIVED-02`), write-path provenance (`MEAS-DERIVED-03`), `/insights` mass-rewrite boundary (`MEAS-DERIVED-04`), skip_reason canonicalization (`MEAS-GSDR-03`)
- a text-first human report surface (`measurement report {loop}` or equivalent) that renders interpretation objects, anomaly registers, and distinguishing-feature tables via markdown tables and ASCII charts, reusing the JSON query contract rather than diverging from it
- expanded Codex coverage beyond the single proof-of-architecture extractor, with compaction as the natural cross-runtime bridge
- a committed diagnostic artifact demonstrating end-to-end use of the measurement system on at least one real observed pattern, written as an interpretation object (not a dashboard screenshot)
- mandatory coverage-stratification enforcement on facets aggregates per MEAS-DERIVED-02

Explicitly out of scope for 57.6:
- reasoning-quality measurement mechanism (`MEAS-RUNTIME-11`) and spike C5 -- remains 57.7
- transcript-content structural extractors, privacy/storage model, session-content analysis -- 57.7
- automated distinguishing-feature suggestion, intervention-outcome tracking, and progressive/degenerating revision classification (`MEAS-ARCH-09` deepening) -- 57.7
- phantom-thinking-token reconciler requiring real tokenizer (`MEAS-RUNTIME-05`) and FTS5 resolution (`MEAS-GSDR-06`) -- 57.7
- hook-derived live collection and broader Codex parity beyond the compaction+metadata slice -- Phase 60
- rich visualization (web dashboard, charting libraries, GUI) -- revisit after 57.6 per deliberation §6 deferral ledger

**Requirements anchor:** MEAS-ARCH-08 (dual interface), MEAS-RUNTIME-01, MEAS-RUNTIME-06, MEAS-RUNTIME-07, MEAS-RUNTIME-08, MEAS-DERIVED-02, MEAS-DERIVED-03, MEAS-DERIVED-04, MEAS-GSDR-03

## Working Model & Assumptions

### Loop extension strategy and extractor sequencing

**Superseded 57.5 state:** 57.5 shipped 17 extractors tagged almost entirely with `serves_loop: ['pipeline_integrity']` or `['intervention_lifecycle']`. The six-loop taxonomy is declared in `registry.cjs` `LOOP_DEFINITIONS` and surfaced via `contract.loop_catalog`, but `classifyQuestion("signal quality")`, `classifyQuestion("agent performance")`, `classifyQuestion("cross session patterns")`, and `classifyQuestion("cross runtime comparison")` all classify correctly yet return zero filtered features. The 57.5 VERIFICATION.md anti-patterns section flags this explicitly as residual technical debt, not a blocker for 57.5 closure.

- [governing:reasoned] Every loop declared in `LOOP_DEFINITIONS` must end 57.6 with substantive features flowing through it -- a declared loop with no implementing extractors is the technical debt the verifier flagged, not progress.
- [decided:reasoned] Loop coverage should be layered rather than all net-new: first, additional `serves_loop` tagging on existing extractors where semantically defensible (e.g., `runtime_session_identity` → `cross_session_patterns`; `claude_settings_at_start` → `agent_performance`; `kb_signal_stats` → `signal_quality`; `codex_runtime_metadata` → `agent_performance` + `cross_runtime_comparison`); then net-new extractors fill the remaining loop-specific metric gap.
- [decided:reasoned] Net-new extractor slate for 57.6 is exactly the requirement set specified by the roadmap: `MEAS-RUNTIME-01` (thinking composite, agent_performance), `MEAS-RUNTIME-06` (marker density revised, agent_performance — effort only, not quality), `MEAS-RUNTIME-07` (compaction, cross_runtime_comparison + agent_performance), `MEAS-RUNTIME-08` (clear-invocation, cross_session_patterns), `MEAS-DERIVED-02` (facets first-class, signal_quality + agent_performance), `MEAS-DERIVED-03` (write-path provenance — cross-cutting), `MEAS-DERIVED-04` (`/insights` mass-rewrite boundary — cross-cutting), `MEAS-GSDR-03` (skip_reason canonicalization — pipeline_integrity/signal_quality).
- [assumed:reasoned] "Substantive coverage" for 57.6's success bar is "at least one implemented named-metric per loop from `LOOP_DEFINITIONS`," not full parity across every named_metric. Named_metrics are currently aspirational labels, not a delivery contract; tightening them to per-metric delivery expands scope beyond what the roadmap authorized.
- [projected:reasoned] 57.7 will deepen content-structural extractors and reasoning-quality measurement, so 57.6's coverage needs to be wide enough that 57.7 can plug into each loop, but not so deep that 57.7 has no room to move.

### Human interface shape and report surface

**Current state:** The query layer is JSON-first and already produces interpretation objects with competing readings, distinguishing features, anomaly registers, runtime dimension summaries, and provenance. No text renderer exists. `gsd-tools measurement query` returns JSON via `--raw` or a minimal human summary otherwise. MEAS-ARCH-08 is the dual-interface requirement.

- [decided:reasoned] The human interface ships as a new `gsd-tools measurement report {loop}` subcommand (naming subject to router conventions) that renders the JSON contract returned by `queryMeasurement()` into text-first primitives -- not a parallel renderer that reimplements the query pipeline.
- [decided:reasoned] Text-first primitives in 57.6: (a) markdown tables as the default rendering for feature tables, interpretations, symmetry markers, and distinguishing features; (b) ASCII bar charts for small-cardinality numerical distributions (e.g., runtime coverage, loop feature counts); (c) a structured header block per report exposing provenance, coverage, reliability tiers, and anomaly counts -- so a pipe-to-file output is self-describing without needing the JSON.
- [decided:reasoned] Reports are pre-built per loop (six templates covering the six `LOOP_DEFINITIONS` entries) because the ROADMAP success criterion says "pre-built common reports per loop" and this is less ambitious and more reversible than a generic report DSL.
- [decided:reasoned] Terminal color is out of scope for 57.6. Output must remain pipe-friendly, diff-readable, and valid markdown when redirected to a file so diagnostic artifacts can be committed.
- [assumed:reasoned] REPL-style exploration is deferred past 57.6. The deliberation says "potentially REPL-style," the pre-built-report path is the smaller reversible commitment, and ad-hoc exploration via `measurement query --question "..."` already exists from the shell. If the 57.6 verifier judges this insufficient, a decision point lands in 57.7 or later — it does not block this phase.
- [decided:reasoned] The report layer surfaces the same provenance/coverage/reliability the JSON contract does -- a human-sanitized view that hides coverage gaps recreates the exact naive-reader trap the deliberation warns against.

### Codex coverage expansion

**Current state:** 57.5 shipped one Codex extractor (`codex_runtime_metadata` in `extractors/codex.cjs`) as proof of cross-runtime architecture, plus a Codex raw loader at `sources/codex.cjs`. No Codex-specific compaction, metadata stratification, or cross-runtime symmetry beyond presence markers exists yet.

- [decided:reasoned] Compaction (`MEAS-RUNTIME-07`) is the primary Codex expansion in 57.6 because the requirement is natively cross-runtime by spec (Claude scans JSONL for `compact_boundary` + `isCompactSummary:true`; Codex scans for `event_msg/context_compacted` + `replacement_history`), and because it directly populates the Cross-Runtime Comparison loop's `named_metrics` slot.
- [decided:reasoned] A second Codex-facing work item extends `codex_runtime_metadata` with agent_performance fields (sandbox mode, effort level breakdown, reasoning-effort stratification where Codex exposes them) rather than writing a parallel extractor -- keeps Codex surface balanced with Claude's agent_performance inputs without inflating the extractor count.
- [governing:reasoned] Cross-runtime coverage does NOT mean "match Claude everywhere." Asymmetry-as-data requires each Codex feature to either produce meaningful data or emit a runtime-specific availability marker; silent padding with `not_available` stubs is noise, not symmetry.
- [decided:reasoned] Full Codex parity across all loops stays deferred to Phase 60 per the deliberation §6 deferral ledger. 57.6 expands Codex along the cross-runtime loop and one agent-performance extension only.
- [assumed:reasoned] Clear-invocation (`MEAS-RUNTIME-08`) is declared Claude-only for 57.6 (runtimes: `['claude-code']`) because the project's current Codex corpus does not support a symmetric `/clear` pattern and forcing runtime breadth here adds noise. The feature lands in `cross_session_patterns` as asymmetric-only; Codex can be added later without refactoring.

### Practical diagnostic demonstration

**Current state:** The project carries a fully-documented observed pattern (Phase 57 vision-drop cascade) with audit artifacts, a manual signal, a concluded deliberation, and a named remediation (GATE-09 + 57.5/57.6/57.7). 57.5 built the interpretation object shape but did not commit a real diagnostic as an artifact.

- [decided:reasoned] The primary 57.6 demonstration diagnoses the Phase 57 vision-drop cascade using the measurement substrate, because (a) the audit chain gives it the richest documented evidence of any pattern the project currently holds, (b) it exercises the intervention_lifecycle loop 57.5 populated plus the pipeline_integrity loop 57.6 strengthens, and (c) the remediation already landed so intervention-outcome framing is available without pulling 57.7 forward.
- [decided:reasoned] The diagnostic output is a committed interpretation-object artifact (proposed location: `.planning/measurement/diagnostics/`) carrying competing readings ("scope-narrowing cascade" vs "requirements-anchoring trap" vs "governance-file absence from planner context"), distinguishing features mapped onto computed/uncomputed status, and an anomaly register -- not a one-shot shell invocation that disappears when the terminal closes.
- [assumed:reasoned] A second, lighter diagnostic exercising cross_session_patterns or signal_quality (candidate: "does facets coverage asymmetry explain apparent agent-performance drift?" which forces MEAS-DERIVED-02 stratification) lands alongside the primary — enough to prove two loops work end-to-end without turning this phase into a diagnostics-writing exercise.
- [governing:reasoned] The demonstration must not be success theater. If the measurement system cannot support the chosen diagnostic, the phase documents the gap and revises the plan, rather than tuning the diagnostic to match whatever the substrate happens to render cleanly.

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] Do not let "multi-loop coverage" collapse into a tagging exercise. Every `serves_loop` addition must be defensible against the loop's `theory_of_change` in `LOOP_DEFINITIONS`, and every net-new extractor must produce features that would discriminate between at least one pair of competing interpretations relevant to the loop.
- **G-2:** [governing:reasoned] Do not ship an aggregate report that averages across runtimes without symmetry markers, or across `/insights` coverage classes without stratification. Silent aggregation is the exact failure mode the E5.8 stress test identified.
- **G-3:** [governing:reasoned] Human reports must expose the same epistemic provenance that the JSON contract does -- coverage gaps, reliability tiers, `not_available`/`not_emitted` counts, anomaly register entries. A sanitised view recreates the naive-reader trap.
- **G-4:** [governing:reasoned] Summary length, marker density, facets content, and clear-invocation counts are NOT reasoning-quality measures. Each must be labelled as effort / semantic-summary / operator-habit feature, never as a quality proxy, regardless of how tempting the downstream rendering makes it look.
- **G-5:** [governing:reasoned] Codex expansion does not mean "match Claude everywhere." Asymmetry-as-data requires that each Codex feature either produces meaningful data or emits a runtime-specific availability marker; silent `not_available` padding is noise.
- **G-6:** [governing:reasoned] The end-to-end diagnostic demonstration is not a showcase. If the substrate cannot support the chosen diagnostic, the plan records the gap and revises -- it does not tune the diagnostic to fit the substrate's current shape.
- **G-7:** [governing:reasoned] Do not pre-aggregate into the measurement store during rebuild to make reports faster. Resolution-on-demand (MEAS-ARCH-05) is a governing principle; performance pressure on the report layer must be solved without discarding fine-grained features.

</guardrails>

## Open Questions

### Q1: What facets stratification mechanism is enforceable without making the report layer unusable?
**Research program:** Prototype one facets-backed aggregate (e.g., friction counts, goal_categories distribution) under three stratification strategies: (a) refuse-to-aggregate unless stratification axis is supplied; (b) auto-stratify by the MEAS-DERIVED-03 write-path-and-coverage class by default; (c) warn-and-render as aggregate with a stratified breakdown beneath. Compare on information value, rendering legibility, and implementation cost. Check how E5.8 already stratifies and whether that logic can lift into a helper.
**Downstream decisions affected:** API shape of all facets-backed extractors, report-layer contract for aggregates, the enforceability of MEAS-DERIVED-02, and whether stratification becomes a registry-level concept or an extractor-local one.
**Reversibility:** MEDIUM -- baking the wrong stratification contract into extractors is load-bearing; strategy within a chosen contract can evolve.

### Q2: Should any REPL-style exploration ship in 57.6, even as a thin loop-over-queries shell?
**Research program:** Check whether the `gsd-tools` router already has any REPL infrastructure or session-state helpers. Scope a minimum-viable REPL (node `readline` + saved history + slash commands mapped to measurement subcommands) and a "no-REPL" path (relying on shell history + one-shot `measurement query` + `measurement report`). Prototype both on the vision-drop diagnostic session. Measure information value vs. maintenance burden, then decide.
**Downstream decisions affected:** 57.6 scope boundary (reports-only vs reports + explore), CLI surface growth, the shape of the 57.7 deepening work.
**Reversibility:** HIGH -- deferring REPL does not foreclose shipping later; shipping a rotting REPL is the worse failure mode.

### Q3: What counts as "diagnostic demonstration success" for phase close?
**Research program:** Read MEAS-ARCH-09 full text on intervention-outcome grounding and check how the Phase 57 audit framed its own "the fix is GATE-09" closure. Decide whether 57.6's demonstration is (a) an interpretation object pointed at existing interventions (GATE-09, 57.5 architecture) where the intervention-outcome pair is inherited, (b) a live diagnostic with a new not-yet-acted-on interpretation, or (c) both. Check whether the 57.6 verifier needs a truth that the diagnostic changed something measurable, or whether producing the artifact with full provenance is sufficient.
**Downstream decisions affected:** Whether 57.6 produces one or two persistent diagnostic artifacts, whether diagnostics cross into 57.7 intervention-outcome tracking territory, and how verification checks the demonstration.
**Reversibility:** MEDIUM -- too ambitious risks pulling 57.7 scope forward; too minimal risks verifier judging the demonstration insufficient.

### Q4: Which loops will remain thinly covered at 57.6 close, and is that acceptable?
**Research program:** Map each of the four remaining loops against its `LOOP_DEFINITIONS.named_metrics` entries and count how many of those metrics are implemented (vs tagged-only vs missing) after the proposed 57.6 extractor wave. Compare against ROADMAP success criterion 1 ("shipped for agent-performance, signal-quality, cross-session-patterns, and cross-runtime-comparison loops") -- does "shipped" mean every `named_metric` implemented, or at least one per loop? Resolve this explicitly before plan-phase so the plan does not silently overshoot or undershoot.
**Downstream decisions affected:** 57.6 task breakdown, which requirements close as full vs partial, what 57.7 still owes, whether the named_metrics field in `LOOP_DEFINITIONS` becomes aspirational or contractual.
**Reversibility:** LOW -- misreading the success bar at discuss-phase cascades into plan-phase and execution scope. Getting this wrong is how the 57.5 phase inherited its "residual technical debt" anti-pattern in the first place.

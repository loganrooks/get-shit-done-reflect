## Summary

Phase 57.6 extends the three-layer measurement substrate that 57.5 landed (17-extractor registry + JSON query contract + `measurement.db` store + `extractor_registry` parity) along three axes: (1) close the four under-served loops (`agent_performance`, `signal_quality`, `cross_session_patterns`, `cross_runtime_comparison`) via layered tagging + nine net-new extractors; (2) add a text-first human report surface that wraps `queryMeasurement()` rather than reimplementing it; (3) commit two interpretation-object diagnostic artifacts that exercise the post-Popperian machinery on real observed patterns. All extractors plug into the existing `REQUIRED_EXTRACTOR_FIELDS` contract (`registry.cjs:106–115`) without schema churn. All raw reads route through the existing source loaders (`sources/claude.cjs`, `sources/codex.cjs`, `sources/gsdr.cjs`).

The research resolves the four open questions from discuss-phase. **Q4 (HIGH vulnerability)**: the success bar is "at least one named_metric implemented per loop" — the per-loop coverage matrix below shows that after the proposed 57.6 work, every loop has ≥ 2 named_metrics implemented; tightening to "every named_metric" would expand scope by 8 additional extractors with no requirement anchor. **Q1 (MEDIUM)**: facets stratification lands as a helper in `measurement/stratify.cjs` invoked by extractors at feature-build time (not at report layer), with the stratification axis (`write_path` + `facets_coverage_class`) emitted as a required field on every facets-backed feature. **Q2 (MEDIUM)**: the existing shell-invocable `measurement query` + the new `measurement report {loop}` pipe-friendly subcommand together satisfy ROADMAP SC-3; REPL is deferred because the no-REPL path has zero open questions and the REPL path has unresolved scope (slash-command grammar, history persistence, state model). **Q3 (MEDIUM)**: demonstration success = two committed interpretation-object artifacts with full provenance, competing readings, distinguishing-features-with-computed/uncomputed status, and anomaly registers. The verifier does NOT require measurable behavioral change — that would pull 57.7 intervention-outcome tracking forward.

**Primary recommendation:** Plan 57.6 as a three-wave sequence — Wave 1 (low-risk, parallelisable): tag edits on existing extractors + MEAS-GSDR-03 skip_reason vocabulary + MEAS-DERIVED-03 write-path helper; Wave 2 (core extractors): MEAS-RUNTIME-01/06/07/08 + MEAS-DERIVED-02/04 + codex_runtime_metadata breadth extension; Wave 3 (integration): report subcommand + six templates + two diagnostic artifacts. Report-layer work cannot start until Wave 2 has shipped features into the store.

### Pattern 1: The Extractor Contract (inherited from 57.5)

**What:** Every new extractor declares the full 8-field contract and returns an array of feature records built via `buildFeatureRecord()`.

**When to use:** Every extractor in 57.6.

**Example (inherited from 57.5 — `runtime.cjs:164–200`):**
```javascript
// Source: get-shit-done/bin/lib/measurement/extractors/runtime.cjs (57.5)
const runtimeSessionIdentityExtractor = defineExtractor({
  name: 'runtime_session_identity',
  source_family: 'RUNTIME',
  raw_sources: ['claude_session_meta', 'claude_jsonl_projects'],
  runtimes: ['claude-code'],
  reliability_tier: 'artifact_derived',
  features_produced: ['runtime_session_identity'],
  serves_loop: ['pipeline_integrity', 'intervention_lifecycle'],  // ← 57.6 extends these
  distinguishes: ['runtime_identity', 'era_boundary'],
  status_semantics: ['exposed', 'derived', 'not_available'],
  extract(extractor, context) {
    const claude = getClaudeDataset(context);  // memoised loader
    return claude.sessions.map(session => buildFeatureRecord(extractor, {
      feature_name: `runtime_session_identity:${session.session_id}`,
      runtime: 'claude-code',
      availability_status: identityAvailability(session),
      symmetry_marker: 'asymmetric_only',
      reliability_tier: identityReliability(session),
      value: buildSessionIdentityValue(claude, session),
      coverage: buildCoverage(session),
      provenance: { /* ... */ },
      freshness: combineFreshness(context.observed_at, [/* ... */]),
      notes: [/* ... */],
    }));
  },
});
```

### Pattern 2: Raw Loader vs Extractor Boundary

**What:** Raw loaders expose session/thread data with parsed fields; extractors do the semantic interpretation. Scanning for new markers (compaction events, clear invocations, thinking blocks) can live EITHER in the loader (adds to `session.{field}`) OR in the extractor itself (iterates `session.parent_jsonl.records`).

**Recommendation (from reading `sources/claude.cjs`):** the loader already parses JSONL records into `jsonl.records` and exposes helpers (`extractEffortOverride`, `aggregateAssistantUsage`, `countHumanTurns`, `extractSessionVersion`). **For 57.6, put new scanning helpers in the loader** (co-located with `extractEffortOverride` etc.), and let the extractor only aggregate/tag/wrap. Rationale: (a) existing 57.5 extractors already follow this pattern; (b) parsing JSONL records once per session (in the loader) and exposing derived fields is cheaper than re-parsing in each extractor; (c) loader changes are tested in `tests/unit/measurement-claude-source-*.test.js` which already exists.

**Exception:** for `MEAS-DERIVED-02` (facets first-class), facet records are already loaded via `buildFacetIndex()` (`sources/claude.cjs:638–662`) and attached as `session.facets.record`. The extractor reads `session.facets.record.{underlying_goal, goal_categories, outcome, friction_counts, ...}` directly — no loader change needed.

### Pattern 3: Availability Status Must Be Explicit (Four-Class + Availability-Marker)

**What:** Every feature record carries `availability_status ∈ {exposed, derived, not_available, not_applicable, not_emitted}` and `symmetry_marker ∈ {symmetric_available, symmetric_unavailable, asymmetric_derived, asymmetric_only}`.

**When to use:** Every feature record, every extractor.

**Critical examples:**
- `MEAS-RUNTIME-01` thinking composite: `exposed` when thinking blocks present; `not_emitted` when model emitted zero blocks but is thinking-capable; `not_applicable` for Haiku; `not_available` for subagent dispatches (per spike 009 DECISION.md). This is the **three-level gate** the spike decision specifies.
- `MEAS-RUNTIME-08` clear invocation: `exposed` for Claude sessions; `not_applicable` for Codex (no `/clear` symmetry). `symmetry_marker: asymmetric_only`.
- `MEAS-RUNTIME-07` compaction: when zero `compact_boundary` records in 211-session corpus (per E6.2 Layer 1), `not_emitted` (mechanism exists but didn't fire). When source not available, `not_available`.

### Pattern 4: Resolution-on-Demand (MEAS-ARCH-05, G-7)

**What:** Never pre-aggregate during rebuild. Fine-grained feature rows get written; aggregation happens in query or report layer.

**When to use:** All new extractors. If an extractor needs N-session rolling stats (e.g., facets coverage class), emit per-session rows with stratification axes attached, then let `query.cjs` filter/group.

**Anti-pattern violation to catch in verification:** an extractor that pre-computes "mean goal_category distribution across matched facets" at rebuild time would collapse fine-grained data. Instead, emit one `facets_semantic_summary:{session_id}` row per facet-matched session with the full facets record + write_path + coverage_class, and let the report layer aggregate/stratify via `stratify.cjs` helpers.

### Anti-Patterns to Avoid

- **Silent loop aggregation without symmetry markers:** already blocked at feature-record level by `computeSymmetryMarker()` (`registry.cjs:260–277`) which returns `asymmetric_*` when runtimes mix availability. Verification must confirm new Codex + Claude compaction features expose this correctly.
- **Hiding coverage gaps in the report layer (G-3 violation):** the text renderer MUST print `anomaly_count` and `by_source_family.{missing_sources, stale_sources, unknown_sources}` from the JSON contract in the structured header. A sanitised summary view recreates the verificationism trap the deliberation §2.2 identifies. Enforce by including header-block field coverage in the report test fixture.
- **Marker density rendered as "reasoning quality" (G-4 violation):** headers and column labels on `marker_self_correction_density` and `marker_uncertainty_density` MUST use the string "effort" (not "quality"). The spike 010 qualitative_comparison.md finding #6 is explicit: summary length is NOT a quality proxy. Enforce via literal-string test check in the agent_performance template.
- **Re-reading raw files inside extractors:** `context.claude` / `context.codex` / `context.gsdrRaw` are memoised for a reason. Re-reading `~/.claude/projects/*.jsonl` per extractor is the dominant cost — already 211 files × N records at rebuild. Pattern 2 above locks in reuse.
- **Pre-aggregated rows in `measurement.db`:** 57.5's `store.cjs` writes one row per feature produced by `extractor.extract()`. Any new extractor returning a single "aggregate of everything" row for a loop violates MEAS-ARCH-05 and collapses diagnostics. Verification must inspect row counts per extractor against expected session/thread cardinality.

## Common Pitfalls

### Pitfall 1: Tag-and-forget "serves_loop" expansion

**What goes wrong:** Plan adds `signal_quality` to `kb_signal_stats` without re-running the `theory_of_change` check, producing a loop that technically has an extractor but whose extractor doesn't discriminate any competing interpretation for that loop.

**Why it happens:** Tagging is cheap; justification is not. The audit 57.5 already flagged this as an anti-pattern ("Info" level in `57.5-VERIFICATION.md`).

**How to avoid:** G-1 enforcement — for each tag edit, the plan must cite the loop's `theory_of_change` text (`registry.cjs:20, 38, 52, 67, 82, 97`) and argue why the existing extractor serves it. The plan-phase task must include a per-tag justification matrix.

**Warning signs:** A `serves_loop: [...]` list grows to 3+ entries without corresponding `distinguishes` field expansion.

### Pitfall 2: Marker density rendered/labeled as "reasoning quality"

**What goes wrong:** A report column header or interpretation sentence labels `marker_self_correction_density` as "reasoning quality metric," undoing the G-4 commitment.

**Why it happens:** The numbers are tempting. The surface is a number per 1k chars; the default instinct is "higher = better."

**How to avoid:**
- Hard-code a `caveat: "effort_tracking_not_quality_proxy"` field in the feature value. (Already in the shape spec above.)
- Template must render this caveat in the header for `agent_performance.cjs`.
- Add a verification test: `grep -r "reasoning quality" measurement/report/loops/agent_performance.cjs` must return zero hits.

**Warning signs:** Any report text near marker density uses the words "quality," "better," or "good."

### Pitfall 3: Silent Codex asymmetry padding

**What goes wrong:** Writing `codex_compaction_events` with runtime-asymmetric coverage that emits `not_available` stubs for sessions where no rollout JSONL was found, rather than emitting meaningful `not_emitted` when the thread exists but has zero compactions.

**Why it happens:** Defaulting to `not_available` is the easy path.

**How to avoid:** Follow MEAS-ARCH-04 four-class semantics: `not_available` ONLY when the source is missing; `not_emitted` when source present but feature not triggered; `not_applicable` when runtime doesn't support feature; `exposed/derived` when feature fired. The `availabilityForArtifacts` helper in `gsdr.cjs:29-34` is already the pattern.

**Warning signs:** `availability_status: 'not_available'` appears on a feature whose raw_sources all exist.

### Pitfall 4: Stratification bypass in the report layer

**What goes wrong:** Report template renders `facets_semantic_summary` with an aggregate "mean goal_categories per session" without the `size_bucket` stratification, silently biasing toward longer sessions (E5.8 Finding C: mean user_msg 20.1 with facets vs 5.4 without; 25.6% new vs 40.7% historical coverage).

**Why it happens:** Aggregate views look cleaner. The stratified table is 3x as tall.

**How to avoid:**
- Stratification is enforced at extractor output level (Q1 resolver): every `facets_semantic_summary` row carries `stratification.{write_path, facets_coverage_class, size_bucket}`. No aggregate without these fields present is expressible.
- Report-layer default: show stratified breakdown. `--no-stratification` flag required to collapse, AND prints warning in header.
- Verification test: the `signal_quality.cjs` template includes the `size_bucket` column unless explicitly flagged.

**Warning signs:** A report renders a summary statistic across sessions without any group-by axis visible.

### Pitfall 5: Running compaction extractors against real corpus and flagging zero-result as a bug

**What goes wrong:** `claude_compaction_events` emits zero compactions for 211 sessions; a reviewer interprets this as the extractor being broken.

**Why it happens:** Absence-of-events looks like a bug when it's the data (per E6.2 Layer 4: 1M-context model + 49 `/clear` invocations preempt auto-compact).

**How to avoid:**
- Feature value must carry a `notes` entry: "Claude compaction mechanism verified present via binary introspection (E6.2 Layer 2) and official docs (E6.2 Layer 3); observed zero firings in local corpus per operator workflow pattern (E6.2 Layer 4)."
- `availability_status: 'not_emitted'` is correct; do not downgrade to `not_available`.
- The report template for `cross_runtime_comparison` includes this context in the interpretation narrative.

**Warning signs:** Test assertions expect non-zero compaction counts from real corpus.

### Pitfall 6: Facets mid-rebuild data drift

**What goes wrong:** The rebuild runs while `/insights` is active; some session-meta files are in bulk-write cluster, others are waiting. The mtime clustering classifier produces inconsistent `write_path` classifications across the same rebuild.

**Why it happens:** The source is inherently asynchronous (per E5.8 Finding B — two-path write model).

**How to avoid:** `derived_write_path_provenance` includes `observed_at` in provenance AND re-reads mtime per invocation. Stratification classifications are a snapshot — they may shift between runs, which is data, not a defect. The report layer MUST surface `observed_at` in the header block.

**Warning signs:** Retroactive classification instability between consecutive rebuilds.

### Pitfall 7: Reloading raw sources per extractor

**What goes wrong:** A new extractor calls `loadClaude(context.cwd, ...)` instead of `context.claude`, re-parsing 211 JSONLs.

**Why it happens:** Copy-paste from an earlier template.

**How to avoid:** The helper `getClaudeDataset(context)` at `extractors/runtime.cjs:8-10` is the contract. New extractors MUST use this pattern. Code review checklist.

**Warning signs:** rebuild time increases non-linearly with extractor count.

const clearInvocationExtractor = defineExtractor({
  name: 'clear_invocation',
  source_family: 'RUNTIME',
  raw_sources: ['claude_jsonl_projects'],
  runtimes: ['claude-code'],
  reliability_tier: 'direct_observation',
  features_produced: ['clear_invocation'],

    return claude.sessions.map(session => {
      const available = session.parent_jsonl.status === 'matched';

      return buildFeatureRecord(extractor, {
        feature_name: `clear_invocation:${session.session_id}`,
        runtime: 'claude-code',

          session_id: session.session_id,
          clear_invocation_count: count,
          operator_habit: count > 0 ? 'preemptive_reset' : 'no_reset_observed',
          caveat: 'operator_habit_not_reasoning_quality',
        },
        coverage: buildCoverage(session),
        provenance: {
          session_id: session.session_id,
          parent_jsonl_path: session.parent_jsonl.path || null,
          scan_rule: '<command-name>/clear user-record match',
        },
        freshness: combineFreshness(context.observed_at, [session.parent_jsonl.path]),
        notes: [
          'Clear-invocation is an operator-habit signal, not a reasoning-quality signal (G-4).',
          'In this project: 49 /clear vs 0 /compact across 211 sessions (E6.2 Layer 1).',
        ],
      });
    });
  },
});

// Helper in sources/claude.cjs (new):
function countClearInvocations(records) {
  let count = 0;
  for (const record of records) {
    if (record.type !== 'user') continue;
    const content = record.message && record.message.content;
    const text = typeof content === 'string' ? content : (Array.isArray(content) && content[0] && typeof content[0].text === 'string' ? content[0].text : '');
    if (text.includes('<command-name>/clear</command-name>')) count++;
  }
  return count;
}

exports.render = function(queryResult, opts = {}) {

    out.push(`**Reliability tier:** ${interp.reliability_tier}`);
  }

  return out.join('\n');
};

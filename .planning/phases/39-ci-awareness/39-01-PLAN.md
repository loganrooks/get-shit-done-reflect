Create `agents/gsd-ci-sensor.md` following the exact structure of `agents/gsd-artifact-sensor.md` and `agents/gsd-git-sensor.md`. Use those two files as structural templates -- match section order, XML tag names, and formatting conventions exactly.

**Frontmatter (sensor contract):**
```yaml
---
name: gsd-ci-sensor
description: Detects failed GitHub Actions runs, branch protection bypasses, and test regressions via gh CLI
tools: Read, Bash, Grep
color: red
# === Sensor Contract (EXT-02) ===
sensor_name: ci
timeout_seconds: 30
config_schema:
  repo:
    type: string
    description: "GitHub repo in OWNER/REPO format (default: auto-detect from git remote)"
    default: null
  workflow:
    type: string
    description: "Filter to specific workflow name (default: all workflows)"
    default: null
---
```

This is the FIRST sensor with a non-null `config_schema`, validating that Phase 38's config extensibility works.

**Role section:** Sensor agent that checks GitHub Actions CI status via the `gh` CLI and returns structured signal candidates. Emphasize: does NOT write to the KB -- that is the synthesizer's job. Same framing as artifact sensor.

**References section:** Same pattern as artifact/git sensors -- reference signal-detection.md (`@~/.claude/get-shit-done/references/signal-detection.md`) and knowledge-store.md (`@~/.claude/agents/knowledge-store.md`).

**Inputs section:** Receives phase number, phase directory path, and project name from the orchestrator (same as git sensor).

**Execution flow -- 6 steps:**

**Step 1: Pre-Flight Check (CI-04 -- CRITICAL)**
This step MUST run before ANY gh command. Check two things in order:
1. `command -v gh` -- is the gh CLI installed? If not: print WARNING, return degraded output, exit.
2. `gh auth status` -- is the user authenticated? If not: print WARNING with "Run: gh auth login", return degraded output, exit.

If either check fails, the sensor MUST:
- Print a human-readable WARNING to stdout explaining what happened and how to fix it
- Return the sensor output with `"signals": [], "degraded": true, "degradation_reason": "<specific reason>"` and a `"warning"` field that says this is NOT a clean bill of health
- NEVER return a clean `{"signals": []}` that could be misinterpreted as "CI is passing"

Include this exact language in the spec: "The distinction between 'checked and found nothing' vs 'unable to check' is the core CI-04 requirement."

**Step 2: Detect Repository and Branch**
- Auto-detect repo: `gh repo view --json nameWithOwner --jq '.nameWithOwner'` with timeout 5000ms
- If config_schema provides `repo` override via config, use that instead
- Get current branch: `git branch --show-current` (fallback to "main")
- If repo detection fails, return degraded output (similar to auth failure)

**Step 3: Runtime and Model Detection**
Same pattern as artifact sensor Step 3.0 and git sensor Step 5:
- Runtime detection from path prefix (~/.claude/ -> claude-code, etc.)
- Model detection from self-knowledge
- Store for inclusion in signal candidates

**Step 4: Query Recent CI Runs and Detect Failures (CI-03)**
```bash
gh run list --branch "$BRANCH" --limit 5 \
  --json conclusion,displayTitle,headSha,createdAt,name,status,event
```
- If config_schema provides `workflow` filter: add `--workflow "$WORKFLOW"`
- Use `--json` flag (NEVER parse plain text output -- research anti-pattern)
- Use node inline script to parse JSON and filter for `conclusion: "failure"`
- Also check main branch if current branch is not main (main failures affect everyone)

For each failed run, create a signal candidate:
- `summary`: "CI run '{name}' failed on branch {branch} (commit {headSha[0:7]})"
- `signal_type`: "deviation"
- `signal_category`: "negative"
- `severity`: "critical" if on main branch, "notable" if on feature branch
- `tags`: ["ci", "workflow-failure"] (add "main-branch" tag if on main)
- `evidence.supporting`: ["gh run list shows conclusion=failure for run '{displayTitle}'"]
- `evidence.counter`: ["Failure may be transient (infrastructure issue, flaky test)"]
- `confidence`: "high"
- `confidence_basis`: "Direct API query of GitHub Actions run status"
- `context`: include phase, source: "github-actions", runtime, model

For multiple consecutive failures on the same branch, note the pattern in the summary and escalate severity.

**Step 5: Branch Protection Bypass Detection (CI-05)**
Check recent commits on main for missing or failed required status checks:

1. Query required checks: `gh api "repos/$REPO/branches/main/protection/required_status_checks" --jq '.contexts[]' 2>/dev/null`
   - If this returns error (no branch protection), skip bypass detection -- this is expected for repos without branch protection
2. Get 5 most recent main commits: `gh api "repos/$REPO/commits?sha=main&per_page=5" --jq '.[].sha' 2>/dev/null`
3. For each commit, check if required checks passed: `gh api "repos/$REPO/commits/$SHA/check-runs" --jq '[.check_runs[] | {name: .name, conclusion: .conclusion}]' 2>/dev/null`
4. If any required check is missing or conclusion != "success", create a bypass signal:
   - `summary`: "Branch protection bypassed: commit {sha[0:7]} on main missing/failed required check '{check_name}'"
   - `severity`: "critical" (bypassed protection is always critical)
   - `tags`: ["ci", "branch-protection", "bypass"]
   - `confidence`: "high"

Limit to 5 commits to avoid rate limiting (research Pitfall 2). Use `2>/dev/null` on all gh api calls.

**Step 6: Test Regression Detection (CI-06)**
Compare test counts between the 2 most recent completed CI runs:

1. Get run IDs: `gh run list --workflow "CI" --limit 2 --status completed --json databaseId --jq '.[].databaseId'`
2. For each run, extract test count: `gh run view "$RUN_ID" --log 2>/dev/null | grep -oE '[0-9]+ passed' | head -1 | grep -oE '[0-9]+'`
3. If both counts extracted and newer < older, create a regression signal:
   - `summary`: "Test regression detected: count dropped from {old} to {new}"
   - `severity`: "notable"
   - `tags`: ["ci", "test-regression"]
   - `confidence`: "low"
   - `confidence_basis`: "Log parsing of vitest output -- format-dependent, may miss non-vitest runners"
4. If parsing fails for either run, skip silently -- do NOT report false data

**Step 7: Return Results**
Use exact delimited output format from sensor contract:
```
## SENSOR OUTPUT
{json block}
## END SENSOR OUTPUT
```

Include a note: If no failures found, return `"signals": []` with NO degraded flag (this IS a genuine clean result). If pre-flight failed, return `"signals": [], "degraded": true`.

**Guidelines section:** Match format of other sensors. Key guidelines:
- Read signal-detection.md before every collection run
- Return ALL candidates regardless of severity
- Never write to the KB
- Use `--json` flag on all gh commands (never parse plain text)
- Always use timeout on execSync calls
- Cap at 10 signal candidates total to avoid flooding
- All paths use `~/` prefix (installer converts to `./`)

**Blind spots section (EXT-07):**
- Cannot detect CI failures on branches other than current + main
- Cannot detect failures in workflows not tracked by GitHub Actions
- Cannot distinguish transient failures (flaky tests, infrastructure) from genuine regressions
- Cannot run when gh CLI is not installed or not authenticated (degrades gracefully)
- Test count parsing is format-dependent (vitest "N passed" pattern); non-vitest runners may not be detected
- Does not check upstream repo CI status (fork-scoped only)
- Rate limiting: heavy usage could exhaust GitHub API quota (5000 req/hour authenticated)

**Required reading section:** `@~/.claude/get-shit-done/references/agent-protocol.md` (same as other sensors).

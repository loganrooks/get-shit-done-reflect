Create `agents/gsd-ci-sensor.md` following the exact structure of `agents/gsd-artifact-sensor.md` and `agents/gsd-git-sensor.md`.

**Frontmatter (sensor contract):**
```yaml
---
name: gsd-ci-sensor
description: Detects failed GitHub Actions runs via gh CLI, returning signal candidates via the standardized sensor contract
tools: Read, Bash, Grep
color: red
# === Sensor Contract (EXT-02) ===
sensor_name: ci
timeout_seconds: 30
config_schema:
  repo:
    type: string
    description: "GitHub repo in OWNER/REPO format (default: auto-detect from git remote)"
    default: null
  workflow:
    type: string
    description: "Filter to specific workflow name (default: all workflows)"
    default: null
---
```

This is the FIRST sensor with a non-null `config_schema`, validating that Phase 38's config extensibility works.

**Role section:** Sensor agent that checks GitHub Actions CI status via the `gh` CLI and returns structured signal candidates. Does NOT write to the KB.

**References section:** Same pattern as artifact/git sensors -- reference signal-detection.md and knowledge-store.md.

**Inputs section:** Receives phase number, phase directory path, and project name from the orchestrator (same as other sensors).

**Execution flow -- 5 steps:**

**Step 1: Pre-Flight Check (CI-04 -- CRITICAL)**
This step MUST run before ANY gh command. It checks two things:
1. `command -v gh` -- is the gh CLI installed?
2. `gh auth status` -- is the user authenticated?

If either fails, the sensor MUST:
- Print a human-readable WARNING to stderr/stdout explaining what happened
- Return empty signals with `degraded: true` and `degradation_reason` in the sensor output
- NEVER return a clean empty signals array that could be interpreted as "CI is passing"

The distinction between "checked and found nothing" vs "unable to check" is the core CI-04 requirement.

**Step 2: Detect Repository and Branch**
- Auto-detect repo: `gh repo view --json nameWithOwner --jq '.nameWithOwner'`
- If config_schema provides `repo` override, use that instead
- Get current branch: `git branch --show-current` (fallback to "main")

**Step 3: Query Recent CI Runs**
```bash
gh run list --branch "$BRANCH" --limit 5 \
  --json conclusion,displayTitle,headSha,createdAt,name,status,event
```
- If config_schema provides `workflow` filter: add `--workflow "$WORKFLOW"`
- Parse JSON output with node inline script
- Identify runs with `conclusion: "failure"`

**Step 4: Build Signal Candidates**
For each failed run, create a signal candidate with:
- `summary`: "CI run '{name}' failed on branch {branch} (commit {sha})"
- `signal_type`: "deviation"
- `signal_category`: "negative"
- `severity`: "critical" if on main branch, "notable" if on feature branch
- `tags`: ["ci", "workflow-failure"]
- `evidence.supporting`: ["gh run list shows conclusion=failure for run #{id}"]
- `evidence.counter`: ["Failure may be transient (infrastructure issue, flaky test)"]
- `confidence`: "high"
- `confidence_basis`: "Direct API query of GitHub Actions run status"
- `context.source`: "github-actions"
- Include `runtime` and `model` from self-detection (same pattern as artifact sensor Step 3.0)

For multiple consecutive failures on the same branch, escalate severity and note the pattern.

**Step 5: Return Results**
Use exact delimited output format from sensor contract:
```
## SENSOR OUTPUT
```json
{ "sensor": "ci", "phase": N, "signals": [...] }
```
## END SENSOR OUTPUT
```

If no failures found, return `"signals": []` (NOT degraded -- this is a genuine clean result).
If pre-flight failed, return `"signals": [], "degraded": true, "degradation_reason": "..."`.

**Guidelines section:** Same pattern as other sensors -- read signal-detection.md, return ALL candidates, never write to KB, use `--json` flag on all gh commands (never parse plain text), always use timeout on execSync calls.

**Blind spots section (EXT-07):**
- Cannot detect CI failures on other branches (checks current branch + main only)
- Cannot detect failures in workflows not tracked by GitHub Actions
- Cannot distinguish transient failures from genuine regressions
- Cannot run when gh CLI is not installed or not authenticated (degrades gracefully)
- Test count parsing (if attempted) is format-dependent and fragile
- Does not check upstream repo CI status (fork only)

**Required reading section:** Reference agent-protocol.md (same as other sensors).

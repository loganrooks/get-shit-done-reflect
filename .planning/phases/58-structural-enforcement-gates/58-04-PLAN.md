Schema fields (per Research R6):

Top-level frontmatter (required):
- `phase`: string (e.g., `58-structural-enforcement-gates`)
- `ledger_schema`: string, must equal `v1`
- `generated_at`: ISO8601 string
- `generator_role`: enum [`planner` | `executor` | `verifier`]
- `entries`: array (may be empty if phase close with no ledger-able claims; such empty case is rare and a verifier WARN, not FAIL)

Per-entry required:
- `context_claim`: string (quoted claim text or claim-ID)
- `disposition`: enum [`implemented_this_phase` | `explicitly_deferred` | `rejected_with_reason` | `left_open_blocking_planning`]
- `load_bearing`: boolean
- `role_split_provenance`:
  - `written_by`: enum [`planner` | `executor` | `verifier`]
  - `written_at`: ISO8601
  - `session_id`: string

Per-entry conditional:
- if `disposition == implemented_this_phase`: `evidence_paths` (array of strings, min 1 entry) required
- if `disposition == explicitly_deferred`: `target_phase_if_deferred` (string, pattern `^Phase \d+(\.\d+)?$`) required
- if `disposition == rejected_with_reason`: `narrowing_provenance.originating_claim` (string) AND `narrowing_provenance.rationale` (string) required; `narrowing_provenance.narrowing_decision` (string) strongly recommended

Load-bearing classification rule (authoritative, per Research R6):

A CONTEXT claim is `load_bearing: true` if ANY of:
1. Its type is `[decided:*]`, `[stipulated:*]`, or `[governing:*]`
2. Its type is `[evidenced:*]` AND it appears in the `<constraints>` DC-table
3. Its tag set includes `load-bearing`
4. Its type is `[assumed:*]` AND another claim in the dependency table cites it as "Depends On"
5. Its type is `[projected]` AND it cross-references another phase

Validator implementation: The new `ledger` schema entry should use the existing `frontmatter.cjs` validation plumbing (whatever function handles schema checks today). Write conditional-required logic in a small helper if the existing plumbing doesn't support `if X then Y` â€” but prefer extending existing plumbing over a new path.

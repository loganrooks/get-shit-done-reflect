**Part A: GATE-09b planning-gate in `plan-phase.md`**

Add a step near the beginning of `plan-phase.md` (after CONTEXT is loaded, before planner is dispatched) that scans `<phase-dir>/<phase>-CONTEXT.md` for `[open]` claims affecting scope boundaries.

```bash
# GATE-09b: planning-gate check for unresolved [open] scope-boundary claims
CONTEXT_FILE="$phase_dir/${padded_phase}-CONTEXT.md"
if [ -f "$CONTEXT_FILE" ]; then
  # Count [open] claims that are not followed by a resolution marker (e.g., "deferred to Phase X" or "resolved")
  OPEN_UNRESOLVED=$(grep -cE '\[open[:\]]' "$CONTEXT_FILE" || true)
  OPEN_RESOLVED=$(grep -cE '\[open[:\]].*resolved|\[open[:\]].*deferred to Phase' "$CONTEXT_FILE" || true)
  UNRESOLVED=$((OPEN_UNRESOLVED - OPEN_RESOLVED))

  if [ $UNRESOLVED -gt 0 ]; then
    echo "::notice::gate_fired=GATE-09b result=block unresolved_claims=$UNRESOLVED"
    echo "GATE-09b: $UNRESOLVED [open] scope-boundary claims in CONTEXT.md require resolution or deferral before planning proceeds."
    grep -nE '\[open[:\]]' "$CONTEXT_FILE" | head -20
    echo ""
    echo "Each [open] claim must either resolve (e.g., decide) or defer to a named Phase (e.g., 'deferred to Phase 60.1')."
    exit 1
  fi
  echo "::notice::gate_fired=GATE-09b result=pass"
fi
```

This is a coarse heuristic (exact open-claim detection would require a YAML/markdown parser for typed claims; the simple grep catches the common case). Document the heuristic in a HEADNOTE so future improvements are legible.

Create the verifier primitive that closes GATE-09d + the embedded meta-gate (GATE-09e).

**Function contract:**

```javascript
// verifyLedger({ phase, strict = true, includeMetaGate = true })
// Returns: { status: 'pass'|'block', missing_claims: [...], unwired_gates: [...], warnings: [...] }
verifyLedger(options)
```

**Algorithm:**

1. Load `<phase-dir>/NN-CONTEXT.md`. Extract all load-bearing claims per Plan 04's load-bearing rule (types `[decided:*]`, `[stipulated:*]`, `[governing:*]`, `[evidenced]` in constraints, `[assumed]` underpinning decided, `[projected]` cross-phase). Produce a list of claim-identifiers.

2. Load `<phase-dir>/NN-LEDGER.md` (via Plan 04's ledger schema). If missing, return `{status: 'block', missing_claims: ['ledger_not_present']}`.

3. Validate ledger frontmatter via `frontmatter validate --schema ledger` — if invalid, block.

4. **Claim-coverage check (GATE-09d core):** for each load-bearing CONTEXT claim, check if ANY ledger entry has matching `context_claim` (fuzzy match acceptable — substring or claim-ID match). Claims without coverage are `missing_claims[]`.

5. **Evidence check (GATE-09d):** for each ledger entry with `disposition: implemented_this_phase`, confirm `evidence_paths[]` is non-empty AND every path exists on disk.

6. **Meta-gate emission check (GATE-09e embedded):** query the gate_fire_events extractor (from Plan 19) for the phase:
   ```javascript
   const events = await measurement.query('gate_fire_events', { phase });
   const gatesIntroducedByPhase = /* read from feature-manifest.json or ROADMAP.md phase entry */;
   const firedGates = new Set(events.features.gate_fire_by_gate_id.filter(r => r.count > 0).map(r => r.gate_id));
   const unwired = gatesIntroducedByPhase.filter(g => !firedGates.has(g));
   ```
   Record `unwired_gates[]`.

   **Guard:** if `gate_fire_events` extractor is not registered yet (Plan 19 may not have run), emit a warning `meta_gate_extractor_missing` and skip the meta-gate check. This avoids deadlock between plans 17 and 19 at run time. In a second-pass verifier run (after Plan 19 lands), the meta-gate fires correctly.

7. **Emission:**
   - On pass: `console.log("::notice title=GATE-09d::gate_fired=GATE-09d result=pass missing_claims=0 unwired_gates=0")`
   - On block: `console.log("::notice title=GATE-09d::gate_fired=GATE-09d result=block missing_claims=" + missing.length + " unwired_gates=" + unwired.length)` + exit non-zero.

**CLI:** `gsd-tools verify ledger <phase_number> [--strict] [--no-meta-gate]`. Register in `gsd-tools.cjs`. Help text explains the 3 checks (claim-coverage, evidence-paths, meta-gate).

**Phase-introduced-gates source:** the authoritative list of which gates were introduced in a phase lives in:
- `feature-manifest.json` if a `gates_introduced_by_phase` field exists (preferred).
- Otherwise, parse the phase's REQUIREMENTS.md range from ROADMAP.md.

Implement the parse once; cache result. Document the source of truth in a HEADNOTE.

**Researched:** 2026-04-17
**Domain:** Provenance/signature normalization across workflow artifacts, KB indexing, runtime identity, and legacy compatibility in a file-first Node.js workflow system
**Confidence:** HIGH for existing-code-shape findings and immediate migration risks; MEDIUM for the final `harness/platform/vendor` vocabulary choice because the project has adjacent conventions but no single pre-existing authority table

## User Constraints (from CONTEXT.md)

### Locked decisions

- 57.8 is an audit-bounded provenance repair phase, not a general telemetry redesign.
- New signals must split provenance into `about_work[]`, `detected_by`, and `written_by`.
- PLAN, SUMMARY, and VERIFICATION gain a required nested `signature` block.
- `provenance_status` must reuse the existing `exposed` / `derived` / `not_available` vocabulary rather than inventing a new taxonomy.
- `gsd_version` precedence must be `installed_harness` first, `.planning/config.json` second, repo-local mirrors last.
- KB migration must be additive and preserve one-milestone compatibility with legacy flat `runtime` / `model` / `gsd_version`.
- The nine audited 57.6 signals stay substantively intact and are annotated as legacy provenance, not rewritten.

### Governing guardrails

- Missing provenance facts must be persisted as `not_available`, not guessed.
- Source repo paths are canonical; runtime mirrors are outputs.
- Templates, validators, docs, and tests must move together.

## Executive Findings

### Q1: Shared helper seam

**Recommendation:** add a small shared provenance helper under `get-shit-done/bin/lib/` rather than coupling PLAN/SUMMARY/VERIFICATION writers directly to the measurement extractor modules.

Why:

- `get-shit-done/bin/lib/init.cjs` already resolves planner/executor/verifier model *targets*, but not the full artifact signature payload.
- `get-shit-done/bin/lib/measurement/extractors/runtime.cjs` and `get-shit-done/bin/lib/measurement/extractors/codex.cjs` already embody the correct uncertainty discipline (`exposed` / `derived` / `not_available`), but they are measurement-oriented dataset builders, not lightweight writer-time helpers.
- 57.8 needs the same provenance vocabulary in three places at once:
  - artifact frontmatter (`signature`)
  - signal frontmatter (`about_work[]`, `detected_by`, `written_by`)
  - KB migration/query code

**Best fit:** a helper such as `get-shit-done/bin/lib/provenance.cjs` that owns:

- current-runtime detection
- session/thread ID capture from runtime env
- writer-side `gsd_version` resolution with source marker
- signature assembly for planner/executor/verifier artifacts
- split-provenance assembly for signals

This keeps the measurement extractors authoritative as a reference implementation without forcing artifact writers to load the whole measurement substrate.

### Q2: Canonical runtime mapping

The repo has enough adjacent vocabulary to standardize a minimal two-runtime mapping now:

| Canonical field | Claude writer value | Codex writer value | Status discipline |
|---|---|---|---|
| `harness` | `claude-code` | `codex-cli` | `exposed` when the runtime is directly known from the active session path/env |
| `platform` | `claude` | `codex` | `derived` from `harness` |
| `vendor` | `anthropic` | `openai` | `derived` from `harness`; Codex telemetry can corroborate with `model_provider` but should not be required for writer-time signatures |
| `profile` | `.planning/config.json` `model_profile` | same | `derived` |
| `model` | best-available session-native model if exposed; otherwise resolved target or `not_available` | best-available thread model from Codex thread metadata | `exposed` when runtime metadata provides it, otherwise `derived` or `not_available` |
| `reasoning_effort` | usually `not_available` unless runtime/session metadata exposes a per-session value | `threads.reasoning_effort` | `exposed` on Codex when available; `not_available` is legitimate on Claude |

Why this is the least-bad choice:

- The capability matrix and model-profile docs already use runtime names `Claude Code` and `Codex CLI` as the stable cross-runtime surface.
- Measurement extractors already stabilize on `runtime: 'claude-code'` and `runtime: 'codex-cli'`.
- `platform` and `vendor` are useful for future Phase 60.1 consumers only if they are deterministic and boring. Deriving them from `harness` avoids a second runtime-detection problem.

### Q3: Session/thread/agent ID availability

**Key finding:** the repo already has an explicit runtime-session env seam in `get-shit-done/bin/lib/core.cjs`.

`WORKSTREAM_SESSION_ENV_KEYS` includes:

- `CODEX_THREAD_ID`
- `CLAUDE_SESSION_ID`
- `GSD_SESSION_KEY`
- several terminal/session fallbacks

In this live Codex session, `CODEX_THREAD_ID` is present in the shell environment. That means artifact writers do not need to infer session identity from git history or post-hoc log scans when the runtime already exposes it.

**Recommended 57.8 rule:**

- Keep the canonical artifact/signature field name `session_id`.
- On Codex, populate `session_id` from `CODEX_THREAD_ID` and document that this is the runtime-native conversation/thread identifier carried in the cross-runtime `session_id` slot.
- On Claude, populate `session_id` from `CLAUDE_SESSION_ID` when present.
- If neither runtime-native env var is present, emit `not_available`.
- Do **not** require `agent_id` in 57.8. There is no equivalent generic writer-time env contract today; the only nearby evidence is resume/interruption state and measurement-side rollout metadata, which are not reliable writer-time inputs for all artifact paths.

Availability matrix:

| Writer path | `session_id` | `agent_id` | Notes |
|---|---|---|---|
| PLAN writer | available when runtime exports `CODEX_THREAD_ID` or `CLAUDE_SESSION_ID`; else `not_available` | `not_available` | do not reconstruct later |
| SUMMARY writer | same | `not_available` | executor subagents may have their own runtime-native session/thread env; rely on that if present |
| VERIFICATION writer | same | `not_available` | verifier path should use the same helper |
| Signal writer (`written_by`) | same | `not_available` in 57.8 | writer identity is role/runtime/model/version first; agent UUID is future work if the runtime exposes it cleanly |

### Q4: Compatibility bridge and KB shape

The KB layer is still flat:

- `get-shit-done/bin/lib/kb.cjs` stores `runtime`, `model`, and `gsd_version` as flat columns.
- `signalToRow()` maps directly from flat frontmatter into those columns.
- `tests/unit/kb.test.js` already contains additive migration precedent.

**Recommendation:** extend the schema additively with explicit split-provenance payload storage while retaining the flat echoes for one milestone cycle.

The cleanest shape is:

- keep legacy columns: `runtime`, `model`, `gsd_version`
- add new columns:
  - `provenance_schema`
  - `about_work_json`
  - `detected_by_json`
  - `written_by_json`
  - `provenance_status`

Why JSON columns:

- `about_work[]` is structurally richer than the current flat table.
- preserving the file-first frontmatter as source of truth still allows KB to index/search the split payload without inventing a second normalized subtable design in the same phase
- additive JSON TEXT columns match the repo’s existing “source files canonical, SQLite derived” posture

Legacy bridge rule:

- new signals dual-write split provenance plus deprecated flat echoes
- pre-57.8 signals get `provenance_schema: v1_legacy`
- the nine audited 57.6 signals additionally get `provenance_status: legacy_mixed`

## Critical Existing-Code Findings

### 1. Generic frontmatter support is not ready for `about_work[]` arrays of objects

This is the highest-risk implementation seam discovered in local research.

`get-shit-done/bin/lib/frontmatter.cjs` currently:

- parses nested objects reasonably well
- does **not** correctly round-trip arrays of embedded objects in generic frontmatter

Local reproduction against the current parser/serializer showed:

- `extractFrontmatter()` turns
  - `about_work:`
  - `  - role: planner`
  into `["role: planner"]` instead of an object entry
- `reconstructFrontmatter()` serializes an object array element as `- [object Object]`

That means PROV-01 is blocked on parser/serializer work, not just schema text and docs.

**Implication:** 57.8 planning must include a real frontmatter round-trip upgrade before signal files can carry `about_work[]` as embedded signature objects.

### 2. Artifact signature frontmatter is structurally absent today

Current state:

- `plan` schema requires no `signature`
- `summary` schema requires no `signature`
- `verification` schema requires no `signature`
- templates are inconsistent:
  - PLAN: no signature block
  - SUMMARY: lone top-level `model`
  - VERIFICATION: no structured signature at all

**Implication:** template edits, schema edits, and writer changes must land together or the repo will describe a provenance model the tools do not enforce.

### 3. Runtime-identity vocabulary already exists and should be reused

Measurement runtime identity already exposes the exact uncertainty style 57.8 wants:

- Claude side: `buildSessionIdentityValue()` returns `session_id`, `runtime`, `model`, `gsd_version`, `profile`
- Codex side: `buildRuntimeMetadataValue()` returns `thread_id`, `runtime`, `model`, `reasoning_effort`, `gsd_version`, `profile`, and corroborating runtime metadata

**Implication:** reuse the field-status semantics and version/profile derivation posture, but not the measurement extractor surface directly.

### 4. Writer-side version precedence already has a live failure case

This workspace currently has conflicting authorities:

- `.planning/config.json`: `1.19.4+dev`
- repo-local installed mirror `.codex/get-shit-done-reflect/VERSION`: `1.18.2+dev`

**Implication:** mirror-first precedence is not a theoretical bug; it is already demonstrably wrong in this repo. 57.8 should centralize writer-side version resolution rather than leaving it implicit inside individual prompts or scripts.

## Recommended Architecture

### Shared helper contract

Recommended exports for `get-shit-done/bin/lib/provenance.cjs`:

```javascript
resolveWriterRuntimeContext(cwd)
resolveWriterGsdVersion(cwd, runtimeContext)
buildArtifactSignature({ cwd, role, modelHint, generatedAt })
buildSignalProvenance({ cwd, aboutWork, detectedBy, writtenBy })
buildLegacyFlatEcho(splitProvenance)
```

Responsibilities:

- read runtime-native env (`CODEX_THREAD_ID`, `CLAUDE_SESSION_ID`)
- derive `harness` / `platform` / `vendor`
- resolve `profile`
- resolve `gsd_version` with source marker:
  - `installed_harness`
  - `config`
  - `repo_mirror`
  - `not_available`
- assemble per-field `provenance_status`

### Frontmatter work required

57.8 cannot safely ship without extending generic frontmatter round-trip support for:

- nested `signature.provenance_status`
- signal `about_work[]` arrays of embedded objects
- `detected_by` and `written_by` nested objects

If this is not fixed, either:

- new signal/artifact files will serialize incorrectly, or
- the repo will be forced into a weaker JSON-string-in-YAML workaround that contradicts the CONTEXT decision to use embedded signature objects

### Migration posture

Use the existing additive migration pattern already present in `kb.cjs` / `tests/unit/kb.test.js`:

- extend schema, do not break rebuild
- accept both legacy flat and new split provenance during the compatibility window
- annotate, do not rewrite, the nine audited 57.6 signals

## Execution Implications For Planning

The natural plan split is:

1. **Artifact/provenance substrate**
   - shared provenance helper
   - frontmatter round-trip upgrade
   - PLAN/SUMMARY/VERIFICATION template + schema changes

2. **Signal pipeline + KB migration**
   - signal docs/specs/workflow updates
   - split provenance persistence
   - additive KB migration and compatibility bridge
   - legacy annotation

3. **Regression coverage**
   - parser round-trip tests for object arrays
   - artifact validation tests
   - KB old/new coexistence tests
   - targeted runtime/version precedence tests

That split follows the real dependency chain: signal provenance cannot ship honestly until the shared signature vocabulary and frontmatter machinery are real.

  <files>get-shit-done/bin/lib/core.cjs, get-shit-done/bin/lib/provenance.cjs</files>
  <action>
Create a helper module that turns the research findings into reusable code rather than prompt folklore.

The helper must:

- detect the active runtime/harness from writer-time context, not from repo-local mirror paths
- read runtime-native session env where available:
  - `CODEX_THREAD_ID`
  - `CLAUDE_SESSION_ID`
- keep the canonical artifact field name `session_id`; on Codex it may carry the thread id as the runtime-native session token
- centralize writer-side `gsd_version` precedence:
  - installed harness VERSION first
  - `.planning/config.json` `gsd_reflect_version` second
  - repo-local mirror VERSION last
  - `not_available` if none can be justified
- reuse the measurement vocabulary `exposed` / `derived` / `not_available`
- treat `agent_id` as out of scope for 57.8 unless a clean writer-time seam already exists; default to `not_available`, do not invent a pseudo-ID

If the cleanest implementation is to expose a small raw session/env helper from `core.cjs`, do that instead of copying the env key list into multiple files.
  </action>
  <verify>
Use a targeted Node check against the helper in the live Codex environment to confirm:

- `session_id` resolves from `CODEX_THREAD_ID`
- `harness` resolves to `codex-cli`
- `gsd_version` returns both a value and a source marker
- missing fields that cannot be justified (for example `agent_id`) return `not_available`
  </verify>
  <done>
There is exactly one writer-side provenance helper, and it can build an artifact-signature payload with explicit field-status markers without consulting git history or fabricating missing IDs.
  </done>

<task type="auto">
  <name>Task 2: Upgrade frontmatter round-trip and validation for nested provenance structures</name>
  <files>get-shit-done/bin/lib/frontmatter.cjs</files>
  <action>
Fix the real parser/serializer gap discovered in research.

Specifically:

- generic frontmatter parsing must preserve `about_work[]` as an array of embedded objects, not `['role: planner']`
- generic reconstruction must serialize object arrays correctly, not `- [object Object]`
- plan/summary/verification schemas must require `signature` and reject empty or partial signature payloads missing canonical nested keys
- signal schema must accept the new split provenance fields while keeping legacy flat echoes optional during the compatibility window

Do not implement a weaker JSON-string-in-YAML workaround unless you hit a hard parser wall and can prove the embedded-object approach is impossible. The CONTEXT decision was explicit: embedded signature objects, not flattened strings.
  </action>
  <verify>
Add or run a focused Node reproduction that round-trips a sample frontmatter block containing:

- `signature.provenance_status`
- `about_work[]` with at least one object entry
- `detected_by`
- `written_by`

Then run `gsd-tools frontmatter validate` on representative PLAN/SUMMARY/VERIFICATION fixtures to prove:

- `signature` is structurally required
- an empty `signature: {}` or missing nested keys fails validation
- a full signature payload passes
  </verify>
  <done>
The object-array parser bug is gone, and schema validation now makes complete signature-bearing artifact frontmatter enforceable instead of aspirational.
  </done>
</task>

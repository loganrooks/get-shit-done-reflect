# Phase 57.8: Signal Provenance Split & Artifact Signature Blocks - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

[decided:bare] Phase 57.8 is the narrow, audit-bounded provenance repair slice inserted between 57.7 and 58. Its job is not to redesign the whole sensor pipeline or complete telemetry-to-signal integration. Its job is to stop the project from writing ambiguous provenance records that collapse artifact-under-judgment identity, detector identity, and writer identity into one flat `runtime/model/gsd_version` payload. <!-- typed by context-checker -->

In scope for 57.8:
- Role-split signal provenance for new signals: `about_work[]`, `detected_by`, `written_by`
- Uniform `signature` block on PLAN, SUMMARY, and VERIFICATION artifacts with per-field provenance markers
- Writer-side `gsd_version` precedence fix so active harness/config beats stale repo-local mirrors
- Sensor/spec/workflow updates so signal candidates and persisted signals conform to the new provenance contract
- Additive KB/schema migration plus transparent legacy compatibility
- Retroactive annotation of the nine committed Phase 57.6 signals as `legacy_mixed`, without rewriting their substantive observations
- Reference-doc updates so `signal-detection.md`, `knowledge-store.md`, templates, validators, and tests all describe the same provenance model

Explicitly out of scope for 57.8:
- Full telemetry-to-signal integration, reflection stratification, or intervention-outcome lifecycle wiring beyond the provenance slice already named in PROV-01..08
- Provenance graphs or general causal-attribution modeling beyond the array-shaped `about_work[]` seam
- Broader artifact-signing rollout to every workflow artifact; 57.8 is limited to PLAN, SUMMARY, VERIFICATION, and the signal/task-spec surfaces named in PROV-04
- Sensor-pipeline redesign, real-agent E2E chain tests, or wider Codex/Claude parity work already parked in Phase 60 / 60.1
- Re-synthesizing or reinterpreting older signals; 57.8 annotates provenance limitations, it does not rewrite history

**Requirements anchor:** PROV-01, PROV-02, PROV-03, PROV-04, PROV-05, PROV-06, PROV-07, PROV-08
**Depends on:** Phase 57.7 (measurement/runtime identity substrate exists; `resolveModelInternal`, `buildSessionIdentityValue`, and the `exposed` / `derived` / `not_available` provenance discipline already exist and should be reused rather than reinvented)

</domain>

<working_model>
## Working Model & Assumptions

### Role-split signal provenance model

**Current state:** [evidenced:cited] Source-of-truth signal docs and KB schema still define provenance as optional flat `runtime`, `model`, and `gsd_version` fields on signals, while the artifact sensor stamps its own runtime/model on candidates and the signal synthesizer fills writer-side version data. The 2026-04-16 provenance audit shows that this produces a hybrid payload that cannot answer either "who did the work?" or "who wrote this signal?" reliably.

- [evidenced:cited] `get-shit-done/references/signal-detection.md` and `agents/knowledge-store.md` still describe flat `runtime`, `model`, and `gsd_version` provenance on signals rather than explicit role-split provenance.
- [decided:reasoned] 57.8 adopts three explicit provenance roles for newly-created signals: `about_work[]` for the artifact signatures implicated in the signal, `detected_by` for the sensor/runtime/model that detected it, and `written_by` for the synthesizer/runtime/model that persisted it.
- [stipulated:reasoned] `about_work[]` is an array of embedded artifact-signature objects, not a graph. Length is `1` when a single role/artifact is implicated and greater than `1` when a signal spans multiple artifacts or workflow roles. If this shape proves insufficient, provenance graphs are deferred to v1.21 rather than pulled into 57.8.
- [decided:reasoned] Sensors populate detector provenance only. They do not infer work provenance. The synthesizer populates writer provenance only. `about_work[]` comes from signed PLAN/SUMMARY/VERIFICATION artifacts or explicit workflow annotations, never from the sensor's own self-identity.
- [decided:reasoned] Legacy flat `runtime` / `model` / `gsd_version` remain readable and writable for one milestone cycle as compatibility echoes of the new split fields, with explicit deprecation language and provenance-schema markers per PROV-08.
- [governing:reasoned] Provenance must no longer collapse detector identity, writer identity, and artifact-under-judgment identity into one field set. If the system cannot say which role a fact belongs to, it records uncertainty explicitly rather than flattening roles together.

### Artifact signature block semantics and scope

**Current state:** [evidenced:cited] PLAN frontmatter currently requires `phase`, `plan`, `type`, `wave`, `depends_on`, `files_modified`, `autonomous`, and `must_haves` but no signature block. SUMMARY frontmatter has a lone `model:` field in the template but not as a required schema field. VERIFICATION has no signature block at all beyond a prose footer. At the same time, `get-shit-done/bin/lib/init.cjs` already resolves planner/executor/verifier model identifiers, and measurement runtime extractors already use `exposed` / `derived` / `not_available` provenance markers.

- [evidenced:cited] `get-shit-done/bin/lib/frontmatter.cjs`, `get-shit-done/templates/phase-prompt.md`, `get-shit-done/templates/summary-standard.md`, and `get-shit-done/templates/verification-report.md` do not currently enforce a uniform planner/executor/verifier signature block.
- [decided:reasoned] PLAN, SUMMARY, and VERIFICATION each gain a required nested `signature` block with these canonical fields: `role`, `harness`, `platform`, `vendor`, `model`, `reasoning_effort`, `profile`, `gsd_version`, `generated_at`, `session_id` (when available), and per-field `provenance_status`.
- [decided:reasoned] `provenance_status` values reuse the measurement substrate's existing vocabulary: `exposed`, `derived`, and `not_available`. 57.8 does not invent a second uncertainty taxonomy for artifact signatures.
- [decided:reasoned] Signature blocks are populated at artifact-creation time from orchestration/runtime context, not reconstructed later from audits or git history. If a field is unavailable at write time, the signature records `not_available` instead of a guessed value.
- [assumed:reasoned] `resolveModelInternal`, current config state, and the existing runtime-identity helpers already provide enough information to sign `model`, `profile`, and `gsd_version` at orchestration time; the highest-risk unknown is consistent `session_id` / `thread_id` availability across planner, executor, and verifier paths.
- [decided:reasoned] 57.8 scope for artifact signatures is limited to PLAN, SUMMARY, and VERIFICATION plus the signal/task-spec surfaces named in PROV-04. VALIDATION and other artifact types stay out of scope unless they are unavoidably touched by the same provenance-writing seam.
- [projected:cited] Phase 58's GATE-09 scope-translation ledger depends on role-aware artifact signatures so later gate-effectiveness signals can attribute failures to planner, executor, verifier, or harness rather than to an undifferentiated runtime blob.

### Version precedence and runtime-identity vocabulary

**Current state:** [evidenced:cited] This workspace already has conflicting version authorities: `.planning/config.json` reports `1.19.4+dev`, while the repo-local installed mirror `.codex/get-shit-done-reflect/VERSION` reports `1.18.2+dev`. The audit demonstrated that current provenance-writing behavior can therefore emit materially wrong `gsd_version` values when it trusts stale mirror state over active harness/config state.

- [evidenced:cited] `.planning/config.json` and the repo-local `.codex/get-shit-done-reflect/VERSION` file currently disagree about the active GSD Reflect version for this workspace, so provenance writers already have a demonstrated stale-source failure mode.
- [decided:reasoned] Writer-side `gsd_version` precedence in 57.8 is: active installed harness VERSION first, `.planning/config.json` `gsd_reflect_version` second, repo-local mirrors only as a last-resort diagnostic fallback. Repo-local installed copies are not authoritative writer provenance in the source repo.
- [decided:reasoned] Every persisted `gsd_version` in both signals and artifact signatures carries a provenance marker that names the source of the value (`installed_harness`, `config`, `repo_mirror`, or `not_available`).
- [assumed:reasoned] The currently-active runtime/harness is discoverable from the same orchestration/runtime context that already resolves model IDs and measurement provenance. If an artifact writer cannot discover it reliably, 57.8 should emit `not_available` rather than fall back to a guessed mirror path.
- [open] A single canonical mapping for `harness`, `platform`, and `vendor` across Claude and Codex has not been written down yet. The measurement extractors, installed runtime mirrors, and source templates use adjacent but not yet fully identical vocabularies.
- [projected:cited] Phase 60.1 telemetry-signal integration should reuse the 57.8 signature vocabulary instead of inventing a second runtime-identity language. 57.8 therefore needs a mapping table durable enough that later telemetry work can consume it unchanged.

### Retroactive annotation, additive migration, and legacy handling

**Current state:** [evidenced:cited] The audit identified nine committed Phase 57.6 signals whose substantive observations remain useful, but whose flat `runtime/model/gsd_version` fields are not reliable for planner/executor/verifier attribution. The existing KB implementation and tests already contain additive migration precedent from Phase 56 (`source` -> `detection_method` + `origin`) and explicit old/new signal-format compatibility tests.

- [evidenced:cited] The nine committed 2026-04-17 Phase 57.6 signals are already in the KB and must remain substantively intact; the audit's recommendation is to annotate their provenance as mixed, not to regenerate them.
- [decided:reasoned] 57.8 treats provenance migration as additive. Pre-57.8 signals parse as `provenance_schema: v1_legacy`, new signals write the split fields, and KB rebuild/query surfaces must handle both shapes without error.
- [decided:reasoned] The nine audited Phase 57.6 signals are retroactively annotated with `provenance_status: legacy_mixed` (and any companion schema/deprecation markers required by the final field design) only. 57.8 must not fabricate `about_work[]` arrays or rewrite those signals' substantive summaries post hoc.
- [assumed:reasoned] The Phase 56 KB migration pattern in `get-shit-done/bin/lib/kb.cjs` and `tests/unit/kb.test.js` is reusable for the provenance split: additive fields, dual-read behavior, and rebuild-first validation can land without violating the file-first invariant.
- [decided:reasoned] Source docs, task specs, workflow prompts, schema validators, and tests ship in the same phase as the code. A code-only split that leaves `signal-detection.md`, `knowledge-store.md`, `collect-signals.md`, or agent specs describing the old flat model fails the very provenance-drift problem 57.8 is meant to close.
- [projected:cited] Phase 58 gate-evaluation work and Phase 60.1 telemetry-signal integration both depend on 57.8 shipping backward-compatible provenance fields rather than forcing downstream consumers through a breaking schema jump.

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] Detector provenance is not work provenance. A sensor may know who detected a problem; it does not thereby know who authored the implicated plan, summary, or verification artifact.
- **G-2:** [governing:reasoned] Do not use legacy flat `runtime/model/gsd_version` on old signals as evidence about planner/executor/verifier responsibility. Those fields are explicitly non-authoritative for role attribution once 57.8 lands.
- **G-3:** [governing:reasoned] Telemetry is a source, not the whole answer. Reuse exposed/derived runtime identity where the logs actually provide it, but sign artifacts at orchestration time where logs are silent.
- **G-4:** [governing:reasoned] If a provenance field is unavailable, persist `not_available`. Never backfill `session_id`, `vendor`, `reasoning_effort`, or `harness` from shaky inference just to make the signature block look complete.
- **G-5:** [governing:reasoned] 57.8 does not grow into provenance graphs, all-artifact signatures, or full telemetry-signal unification. Those are separate phases because they change the problem, not just the implementation.
- **G-6:** [governing:reasoned] Legacy compatibility is real, not ceremonial. Flat fields stay readable and writable for one milestone cycle, and deprecation language must tell downstream agents and users exactly what is authoritative during that window.
- **G-7:** [governing:reasoned] Retroactive 57.6 annotation preserves substantive signal observations and only marks provenance limitations. "Fixing" provenance by fabricating historical attribution would be worse than keeping the mixed record visible.
- **G-8:** [governing:reasoned] Because this is the source repo, source-vs-installed ambiguity must be named explicitly in code and docs. Any implementation that quietly treats `.codex/` or other mirrors as canonical in the source tree repeats the version-precedence bug under a new name.
- **G-9:** [governing:reasoned] Docs, prompts, validators, and tests are part of the provenance system. Shipping only the writer code while leaving the reference docs or schemas stale is another provenance failure, not a partial success.

</guardrails>

### Q1: What shared helper should compose the artifact `signature` block so PLAN, SUMMARY, VERIFICATION, and signal writers all use one provenance vocabulary and one field-status discipline?
**Research program:** Trace the write paths for PLAN creation, SUMMARY generation, VERIFICATION generation, and signal persistence. Compare the existing orchestration-time model-resolution seam (`resolveModelInternal` + `init.cjs`) with the measurement-runtime identity seam (`buildSessionIdentityValue` and Codex runtime metadata). Decide whether 57.8 needs one small shared provenance helper or whether extending existing helpers is enough.
**Downstream decisions affected:** Where the signature block is assembled, how per-field provenance markers are computed, whether signal and artifact provenance semantics stay aligned, and which unit/integration tests own the new helper.
**Reversibility:** MEDIUM — helper placement is refactorable, but picking the wrong seam can duplicate provenance logic across artifact writers and signal writers immediately.

### Q2: What is the canonical mapping for `harness`, `platform`, `vendor`, and `profile` across Claude and Codex, and which of those fields are `exposed` versus `derived` in each runtime?
**Research program:** Compare the runtime-identity fields already emitted by `get-shit-done/bin/lib/measurement/extractors/runtime.cjs` and `get-shit-done/bin/lib/measurement/extractors/codex.cjs` with current installed-runtime conventions, config-derived model-profile resolution, and the audit's role-attribution needs. Produce a two-runtime mapping table that the templates, docs, and KB code can all share.
**Downstream decisions affected:** Final `signature` block schema, `provenance_status` assignment, doc wording in `signal-detection.md` / `knowledge-store.md`, and compatibility with future Phase 60.1 telemetry-signal integration.
**Reversibility:** MEDIUM — changing names later is possible, but vocabulary drift across runtime, artifact, and signal surfaces is exactly the class of problem 57.8 is trying to eliminate.

### Q3: Which artifact-writing paths can reliably expose `session_id`, `thread_id`, or `agent_id`, and where should 57.8 intentionally emit `not_available` instead?
**Research program:** Trace planner/executor/verifier artifact writers and any runtime state available at write time. Check whether these IDs are available from orchestration context, runtime environment, or current logs without post-hoc reconstruction. Build a role-by-role availability matrix and use it to decide where the required `signature` block allows `not_available`.
**Downstream decisions affected:** Final signature-block required fields, schema validation rules, test fixtures, and whether any writer path needs a small orchestration change to surface IDs that already exist.
**Reversibility:** LOW — treating IDs as always-available when they are not will either force fabricated values or make the schema fail at runtime; both are harder to unwind than being conservative now.

### Q4: How should KB/query surfaces present both legacy flat fields and role-split provenance during the one-milestone compatibility window without duplicating or contradicting attribution?
**Research program:** Inspect the current KB read/write and rebuild behavior in `get-shit-done/bin/lib/kb.cjs`, the existing migration precedent in `tests/unit/kb.test.js`, and cross-runtime readability expectations in `tests/integration/cross-runtime-kb.test.js`. Decide what dual-read/dual-write contract 57.8 can support cleanly while surfacing deprecation clearly.
**Downstream decisions affected:** PROV-05 and PROV-08 implementation shape, query output semantics, how the nine legacy 57.6 signals are annotated, and whether new split fields are mirrored into the old flat fields automatically.
**Reversibility:** MEDIUM — the compatibility bridge is temporary, but a confusing bridge will contaminate downstream interpretation and may force extra migration work in v1.21.

### Reusable Assets
- `resolveModelInternal` and the `init.cjs` role-resolution calls already provide planner/executor/verifier model IDs; 57.8 should build on that seam instead of inventing a separate model lookup path.
- The measurement runtime extractors already encode best-available provenance using `exposed`, `derived`, and `not_available`, plus `gsd_context` provenance notes. That is the closest existing implementation of the uncertainty discipline 57.8 needs for artifact signatures.

### Established Patterns

- Runtime asymmetry is treated as data, not noise. Missing provenance facts should surface as `not_available`, not be normalized away.

# Signal Cross-Reference: Fork Signals vs Upstream Issues

**Analysis date:** 2026-03-28
**Fork signal count:** 139 signals across 7 milestones (v1.12-v1.18)
**Upstream issue count:** 100 issues examined (most recent 100, March 2026 window)

## Methodology

This is a first-of-its-kind analysis for this project. The methodology is documented explicitly so future phases can replicate or improve it.

### 1. Fork Signals: Source and Categorization

Fork signals are sourced from `.planning/knowledge/index.md`. Each signal has a severity (minor/notable/critical), lifecycle stage (detected/triaged/resolved), and one or more tags. Tags are the primary categorization mechanism.

**Aggregation approach:** Tags were grouped into concern themes that represent the domain the signal observes. For example, tags `deviation`, `plan-accuracy`, and `scope-creep` all contribute to the "Process quality / deviations" theme because they observe different facets of the same concern: plans not matching reality.

**Tag-to-theme mapping:**

| Theme | Tags Included | Count |
|-------|--------------|-------|
| Process quality / deviations | deviation, plan-accuracy, scope-creep, plan-fidelity | 38 + 10 = 48 |
| Testing gaps | testing, test-count, tdd, verification-gap | 13 + scattered = ~18 |
| Configuration correctness | config, manifest, migration | 12 + scattered = ~15 |
| CI / build reliability | ci, branch-protection, wiring-validation | 8 + scattered = ~12 |
| Workflow gaps | workflow-gap, workflow, resume-work, checkpoint | 4 + scattered = ~8 |
| Agent behavior | agent-behavior, agent-delegation | ~5 |
| Git workflow | git-workflow, branch-management, post-merge | ~8 |
| Knowledge/epistemic | epistemic-gap, knowledge-surfacing, kb-migration | ~7 |
| Extraction quality | extraction, quality, unauthorized-removal | ~6 |

Note: Signals often have multiple tags, so individual signals may appear in multiple theme counts. The "count" column reflects approximate contribution weight, not unique signal counts.

### 3. Comparison Approach

The comparison operates at the **theme level**, not at the individual entry level. For each concern domain:
- Does the fork have signals about it?
- Does upstream have issues about it?

The intersection reveals shared concerns. The differences reveal blind spots or different priorities.

### 4. Limitations

**Category mismatch:** Fork signals track *process/epistemic* concerns (deviations from plans, test gaps, config mistakes, workflow gaps). Upstream issues track *user-facing* bugs and feature requests. These are fundamentally different levels of observation:

- Fork signals are **proactive** -- generated by the system observing itself during execution
- Upstream issues are **reactive** -- generated by users encountering problems after deployment

Direct comparison between individual entries is meaningless. A fork signal about "plan accuracy" and an upstream issue about "verifier accepts circular tests" may share a concern domain (quality assurance) but observe it through entirely different mechanisms. The analysis accounts for this by comparing **themes** (shared problem domains), not individual entries.

**Temporal asymmetry:** The fork's 139 signals span 7 milestones (~2 months). The upstream's 100 issues span approximately 10 days (March 18-28, 2026). The fork data is longitudinal; the upstream data is a snapshot of a high-activity period.

**Selection bias:** The upstream sample is the 100 most recent issues. Older upstream concerns (pre-March 2026) are not represented. The fork's signal KB captures the entire project history.

## Shared Concerns (Both Sides Caught)

Three concern domains show genuine convergence -- both sides independently identified them as problems:

### 1. Testing / Verification Quality

**Fork observation:** 18 signals tracking test-count mismatches, TDD discipline lapses, verification gaps requiring human testing, and CI test regressions. The fork catches these *during execution* as process deviations.

**Upstream observation:** 11 issues reporting that verification tools accept inadequate evidence (#1457 circular tests, #1459 unverified phase completion, #1418 silent scope reduction). Users catch these *after deployment* when outcomes don't match expectations.

**Convergence meaning:** Both projects recognize that automated verification is unreliable. The fork's signal pipeline catches process-level failures; upstream's user reports catch tool-level failures. Neither side has solved this -- it appears to be a fundamental limitation of current AI-assisted verification.

### 2. Configuration Correctness

**Fork observation:** 15 signals tracking config migration edge cases, manifest self-test failures, model selection mismatches, and config field handling errors.

**Upstream observation:** 6 installer/update issues reporting cache path errors (#1421), silent file deletion on update (#1423), and stale hook detection (#1249).

**Convergence meaning:** Configuration handling is inherently fragile across environments, runtimes, and versions. Both sides encounter it repeatedly. The fork's declarative manifest approach and upstream's installer-driven approach are different strategies for the same problem.

### 3. Workflow Lifecycle Brittleness

**Fork observation:** 8 signals about workflow gaps (scope revision absence, PR workflow skips, checkpoint handling). Process-level observations about how workflows fail to guide execution correctly.

**Upstream observation:** 16 issues about workflow bugs (branch regex failures, missing flags, skipped phases, field destruction on state transitions). Feature-level reports about specific workflow failures.

**Convergence meaning:** Workflow systems are complex state machines that break at transitions. The fork observes this as process gaps; upstream observes this as code bugs. Both perspectives are valid and complementary.

## Fork-Only Catches

These are concerns the fork's epistemic pipeline identified that upstream's reactive process appears to miss:

### 1. Plan Accuracy / Process Quality (48 signals)

The fork's largest signal theme has no direct upstream equivalent. Upstream has no concept of "plan accuracy" -- it does not generate plans with the same specification rigor, so it cannot observe deviations from plan expectations. The fork's deviation tracking (38 signals) represents a unique observation layer: the system watches itself for mismatches between intention and execution.

**Value proposition test:** Does this observation prevent problems? Yes -- plan accuracy signals led to the plan checker (Phase 38), which catches second-order effects and dependency gaps before execution begins. Upstream discovers these problems only when users report execution failures.

### 2. CI Build Reliability as Process Concern (12 signals)

The fork treats CI failures as epistemic signals worthy of recording and analyzing. Signals like "CI failures ignored throughout v1.16" and "branch protection bypass" represent meta-observations about the project's own CI discipline. Upstream's CI concerns are embedded in other bug reports rather than observed as a distinct category.

### 3. Knowledge / Epistemic Quality (7 signals)

Signals about KB migration risks, knowledge surfacing effectiveness, and signal lifecycle representation gaps reflect the fork's unique self-monitoring capability. The fork observes not just *what it builds* but *how well its self-improvement system works*. This meta-level observation has no upstream parallel.

### 4. Extraction Quality (6 signals)

When the fork extracts or refactors agent specifications, it monitors the extraction for quality regressions (unauthorized removal of knowledge surfacing, fabricated provenance, codebase mapper deletion). This catch-the-mistake-as-it-happens pattern is enabled by the signal pipeline's real-time observation.

## Upstream-Only Catches

These are concerns that upstream's user-reported issues surface that the fork's signal system does not capture:

### 1. Cross-Runtime Compatibility (16 issues)

Upstream's second-largest issue theme is completely absent from the fork's signal KB. Every new runtime (Codex, Gemini, Windsurf) generates installation and compatibility issues that real users encounter. The fork's single-runtime focus means it never encounters these problems -- the signal pipeline has nothing to observe.

**Is this a blind spot?** Partially. The fork has cross-runtime converters (v1.14 QT22-28), but it does not test them in production runtime environments. If cross-runtime support is a fork goal, the signal pipeline should add sensors for converter correctness. If it is not a goal, the absence is acceptable.

### 2. Path Resolution Across Environments (9 issues)

Users report tilde expansion failures, ENOENT errors in new Node versions, incorrect project_root detection, and Windows-specific path issues. The fork operates in a single stable environment (Linux, one Node version), so environmental path issues never arise.

**Is this a blind spot?** No -- this is a scope difference. The fork is a single-user, single-environment project. Adding path resolution sensors would generate noise without value.

### 3. Worktree / Multi-Project (6 issues)

Upstream users work with git worktrees and multi-project workspaces that generate unique problems (merge overwrites, edit permissions, root resolution in monorepos). The fork does not use worktrees.

**Is this a blind spot?** No -- this is a conscious scope exclusion. The fork's `.planning/` directory model assumes single-project, single-workspace.

### 4. Agent Quality at Scale (13 issues)

Upstream receives more agent quality reports because their user base is larger and more diverse. Issues like "agents ignore instructions" and "manager bypasses skill pipeline" reflect problems that emerge at scale with varied projects and user expectations.

**Is this a blind spot?** Partially. The fork's agent protocol and plan checker address some of these concerns proactively, but the fork cannot observe how agents behave in environments it has never tested.

### 5. Security Hardening (minimal but present)

Upstream has `security.cjs` for prompt injection detection and path traversal prevention. The fork has zero security-related signals or tooling.

**Is this a blind spot?** Yes -- a genuine one. If the fork's signals and agent specifications can be manipulated through prompt injection, the epistemic pipeline itself is vulnerable. Security hardening of the agent layer is a legitimate gap.

## What This Tells Us

### The Fork's Proactive Pipeline vs Upstream's Reactive Pipeline

The cross-reference reveals a fundamental asymmetry in quality observation:

| Dimension | Fork (Signals) | Upstream (Issues) |
|-----------|---------------|-------------------|
| **Observation timing** | During execution | After deployment |
| **Observer** | The system itself | External users |
| **Granularity** | Process-level (plan accuracy, CI discipline) | Feature-level (bugs, missing capabilities) |
| **Coverage** | Deep but narrow (single project, single runtime) | Broad but shallow (many projects, many runtimes) |
| **Meta-level** | Observes its own observation quality | Does not observe observation quality |

The fork catches problems that upstream *cannot* catch (plan accuracy, epistemic quality, extraction regressions) because upstream does not have the observation infrastructure. Upstream catches problems that the fork *does not encounter* (cross-runtime, multi-environment, scale-dependent) because the fork operates in a narrow environment.

### Does the Fork's Investment Pay Off?

**Yes, for its domain.** The signal pipeline catches a different *category* of problems than user-reported issues. The 48 process-quality signals, 18 testing signals, and 12 CI signals represent observations that would never appear as upstream issues because they describe *how the system builds itself*, not *what it produces for users*.

**No, for breadth concerns.** The fork's signal pipeline is blind to cross-runtime, multi-environment, and security concerns because it cannot observe what it does not encounter. The 16 cross-runtime issues and 9 path-resolution issues in upstream's tracker represent a real-world diversity that the fork's single-environment operation filters out.

### The Complementary Relationship

The two observation systems are complementary, not competing:
- **Proactive signals** excel at catching process regressions, plan accuracy degradation, and epistemic quality issues
- **Reactive issues** excel at catching environmental diversity problems, user-expectation mismatches, and scale-dependent failures

A project that had both -- proactive signal observation AND diverse user-reported feedback -- would have stronger quality assurance than either alone.

## Implications for Fork Policy

### Respond To

1. **Security hardening:** This is a genuine blind spot. The fork should consider adopting upstream's `security.cjs` or implementing minimal prompt-injection protection for agent specifications. The epistemic pipeline is only as trustworthy as its inputs.

2. **Cross-runtime converter testing:** If the fork maintains cross-runtime converters, add at least a smoke test that runs converters against each target runtime's actual configuration format. Currently, converter correctness is not verified against real environments.

### Accept As Out-of-Scope

1. **Multi-environment path resolution:** The fork is a single-user, single-environment project. Adding sensors for Windows paths or alternative Node versions would not produce actionable signals.

2. **Worktree / multi-project support:** This is a conscious architectural decision, not a gap. The `.planning/` model assumes single-project workspace.

3. **i18n / internationalization:** Not relevant for a single-user English-language project.

### Monitor Without Acting

1. **Agent quality at scale:** The fork's agent protocol and plan checker proactively address some agent quality concerns, but the fork cannot test at upstream's user diversity. Watch upstream's agent-quality issues for patterns that might apply to the fork's agent specifications.

2. **SDK / headless direction:** Upstream's move toward headless automation (v1.30.0 GSD SDK) represents a fundamentally different trajectory. Monitor for infrastructure decisions that affect the shared codebase, but do not adopt the SDK direction.

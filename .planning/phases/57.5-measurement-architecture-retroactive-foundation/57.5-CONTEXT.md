<domain>
## Phase Boundary

Phase 57.5 builds the first real measurement substrate the project can plan and reason against: a three-layer architecture (`raw -> extractor registry -> interpretation/query`) that runs retroactively over existing corpora, covers the intervention-lifecycle and pipeline-integrity loops first, and proves the architecture is genuinely cross-runtime with at least one Codex extractor. This phase is the corrective response to the Phase 57 vision-drop audit: it must not collapse back into "another session-meta reader."

In scope for 57.5:
- raw-source adapters and provenance model for the sources this phase actually uses
- extractor registry as the extension seam
- first extractor wave for intervention-lifecycle and pipeline-integrity loops across MEAS-RUNTIME, MEAS-DERIVED, and MEAS-GSDR
- JSON-first query interface with interpretation objects carrying competing readings, distinguishing features, anomaly registers, and reliability/provenance status
- retroactive execution over the existing corpus without requiring new collection
- one low-risk Codex extractor proving the runtime dimension works in practice
- kb freshness repair needed to make MEAS-GSDR stats trustworthy

Explicitly out of scope for 57.5:
- reasoning-quality evaluation mechanism
- transcript-content structural analysis and privacy model
- human-readable report surfaces, dashboards, or REPL-style exploration
- full Codex parity across all loops
- hook-derived live collection and broader sensor expansion

**Requirements anchor:** MEAS-ARCH-01 through MEAS-ARCH-07, MEAS-ARCH-10, MEAS-RUNTIME-02/03/04/09/10, MEAS-DERIVED-01/05/06, MEAS-GSDR-01/02/04/05
**Depends on:** Phase 56 (SQLite foundation), Phase 57 (telemetry baseline), governance commit `5d236d6f`, handoff refresh `2ab12d93`

</domain>

<working_model>
## Working Model & Assumptions

### Three-layer architecture and source families

**Current state:** The codebase already has three relevant but disconnected measurement-adjacent surfaces: `telemetry.cjs` reads Claude `session-meta` and `facets`; `kb.cjs` builds a derived SQLite index over knowledge artifacts; `automation.cjs` and `.planning/config.json` persist automation health counters. What does not exist yet is the shared registry/query architecture that treats these as parts of one measurement system.

- [governing:reasoned] The measurement system exists to make self-improvement judgments more trustworthy, not to maximize metric collection or produce a decorative dashboard.
- [decided:reasoned] 57.5 should implement the measurement system as three explicit layers: raw-source adapters that preserve native artifacts, a registry of pure extractors, and a query/interpretation layer that composes extractor outputs into structured answers.
- [decided:reasoned] Raw-source families must be explicit from day one: `RUNTIME` for transcript-native artifacts, `DERIVED` for `/insights` products, and `GSDR` for our own artifacts/config/state.
- [evidenced:cited] `get-shit-done/bin/lib/telemetry.cjs` already demonstrates a raw-adapter pattern for Claude `session-meta` plus `facets`, but it is phase-57-specific and not registry-driven.
- [evidenced:cited] `get-shit-done/bin/lib/kb.cjs` already demonstrates the project's accepted pattern for a rebuildable derived SQLite cache over file-first truth.

### Loop coverage and extractor priorities

**Current state:** REQUIREMENTS.md now names six loops and 34 MEAS requirements, but the roadmap explicitly stages them across 57.5, 57.6, and 57.7. The handoff and spikes narrowed what 57.5 should lead with.

- [decided:reasoned] 57.5 covers the intervention-lifecycle and pipeline-integrity loops first; agent-performance, signal-quality, cross-session patterns, and richer cross-runtime comparison remain downstream.
- [decided:reasoned] The first extractor wave should favor already-grounded or low-ambiguity surfaces: automation health, kb stats/freshness, JSONL token totals, human-turn counts, settings/era boundaries, session-meta/facets provenance, and coverage audits.
- [decided:reasoned] 57.5 should repair the kb freshness path as part of the measurement substrate rather than treating it as a separate maintenance task; otherwise MEAS-GSDR-04 remains structurally untrustworthy.
- [assumed:reasoned] A single low-risk Codex extractor should be enough to prove the runtime dimension in 57.5 if it focuses on metadata/provenance rather than ambitious feature parity.
- [projected:reasoned] Richer reasoning, compaction, and content-level extractors belong to 57.6/57.7, so 57.5 must preserve their seams without pulling them forward.

### Query and interpretation model

**Current state:** The project already has the right epistemic shape in prose: audits, deliberations, and spikes carry competing readings, challenge logic, and anomaly language. What is missing is a reusable query object that can carry that shape in code.

- [decided:reasoned] The query layer should return JSON first and treat interpretations as structured objects, not bare numeric summaries.
- [decided:reasoned] 57.5's interpretation object should include competing interpretations, distinguishing features, anomaly register, runtime/source provenance, and reliability tier.
- [decided:reasoned] 57.5 should stop short of intervention-history and revision-classification machinery; those are real requirements, but they belong to later phases once the registry/query substrate exists.
- [decided:reasoned] Missingness must be semantic: query results should distinguish `exposed`, `derived`, `not_available`, `not_applicable`, and `not_emitted` where relevant, rather than collapsing everything into null.
- [assumed:reasoned] Existing `gsd-tools telemetry` commands should survive as compatibility views or thin wrappers over the new substrate rather than remaining a divergent architecture.

### Corpus boundaries and reasoning signals

**Current state:** The 2026-04-16 correction chain materially changed the epistemic status of several sources. Session-meta is no longer treated as live runtime telemetry, thinking-summary length is no longer treated as a quality signal, and /insights artifacts now carry explicit provenance problems that the architecture must encode rather than ignore.

- [evidenced:cited] JSONL is the canonical source for token totals and human-turn extraction; session-meta token and `user_message_count` fields are annotated artifacts, not first-class measurement ground truth.
- [decided:reasoned] Reasoning-quality measurement is out of 57.5 scope. This phase may preserve metadata and statuses needed for later work, but it does not choose or ship a quality proxy.
- [decided:reasoned] `/insights` products should be modeled as `DERIVED` sources with explicit write-path and coverage provenance, not treated as continuously updated runtime truth.
- [decided:reasoned] 57.5 should make coverage gaps visible in its outputs rather than hiding them in documentation or notes.
- [projected:reasoned] 57.5 should preserve seams so 57.6 can add human reports and 57.7 can add content-structural extractors without forcing a rewrite of raw adapters or query contracts.

### Claude's Discretion

- Exact module/file naming for the measurement registry and query surface
- Whether derived query-time data is cached in a sibling SQLite store, layered onto `kb.db`, or computed on demand, as long as raw/extractor/query separation remains intact
- Exact shape of extractor registration metadata and interpretation JSON
- Compatibility wrapper strategy for existing telemetry subcommands and tests

- **DC-7:** [governing:reasoned] Reliability tiers and explicit unavailability statuses are output semantics, not comments. Query consumers must be prevented from silently treating derived, missing, or runtime-specific data as direct observation.

<guardrails>
## Epistemic Guardrails

- **G-1:** [governing:reasoned] Do not let "measurement architecture" collapse back into a narrow session-meta reader. This phase exists because that narrowing already happened once.
- **G-2:** [governing:reasoned] Do not overbuild 57.5 into a dashboard, REPL, or transcript-analysis system. 57.5 is substrate work, not the whole measurement programme.
- **G-3:** [governing:reasoned] Distinguish reasoning-load observability from reasoning-quality evaluation. Summary length and marker densities cannot be smuggled in as quality metrics here.
- **G-4:** [governing:reasoned] Coverage gaps, era boundaries, and write-path provenance are part of the measured reality and must appear in outputs, not just docs.
- **G-5:** [governing:reasoned] Keep the extension seam cheap: adding an extractor should remain "write a pure function and register it," not "touch multiple core abstractions."
- **G-6:** [governing:reasoned] No silent cross-runtime averaging. If a result changes meaning by runtime or source family, the query result must say so.

</guardrails>

<questions>
## Open Questions

### Q1: Which Codex extractor is the right 57.5 proof of architecture?
**Research program:** Inspect current Codex raw artifacts and choose the lowest-ambiguity extractor that materially serves intervention-lifecycle, pipeline-integrity, or provenance/rich-metadata requirements. Compare candidate extractors on extraction certainty, testability, and seam value for later phases.
**Downstream decisions affected:** Registry schema, raw-adapter scope, test fixtures, and how aggressively 57.5 touches Codex-specific parsing.
**Reversibility:** MEDIUM -- the wrong choice can waste time or drag 57.6/60 scope forward, but the architecture can still survive if the registry seam is sound.

### Q2: What is the right storage/integration shape for derived measurement outputs?
**Research program:** Compare three concrete implementation strategies: extend `kb.db`, create a sibling measurement cache/store, or compute purely on demand. Evaluate each against retroactive rerunability, zero-dependency constraints, CLI ergonomics, and testability.
**Downstream decisions affected:** File/module layout, CLI surface design, migration burden, and whether existing measurement views can become thin wrappers.
**Reversibility:** HIGH -- if raw/extractor/query separation is preserved, the backing storage choice can evolve later.

### Q3: How should existing telemetry commands relate to the new measurement substrate?
**Research program:** Enumerate the current `telemetry` contract and tests, then decide whether 57.5 re-implements those views on top of the registry or keeps them as a compatibility wrapper with clear ownership boundaries.
**Downstream decisions affected:** User-facing command continuity, test migration, and the risk of duplicated measurement logic.
**Reversibility:** HIGH -- wrapper strategies are additive, but leaving the relationship undefined creates architectural drift immediately.

### Q4: What provenance defaults should query results use for coverage and freshness?
**Research program:** Define how parent-JSONL absence, stale `/insights` state, bulk-vs-single write path, and size-biased facets coverage are surfaced in extractor outputs and aggregate interpretations. Prototype one or two end-to-end query objects before fixing the schema.
**Downstream decisions affected:** Extractor output schema, interpretation reliability rules, and whether aggregate reports can be trusted by default.
**Reversibility:** MEDIUM -- weak provenance modeling will infect downstream loops and be expensive to unwind once queries are built on it.

</questions>

## Future Awareness

### Protected Seams
- Keep raw adapters, extractor registry, and interpretation/query composition separate so later phases can add new extractors without architecture churn.
- Preserve explicit source-family and runtime/provenance semantics in schemas and query outputs.
- Keep existing telemetry user surfaces migratable onto the new substrate rather than baking in a permanent split.
- Choose the first Codex extractor in a way that exercises the shared registry/status model, not a one-off path.

### Explicit Non-Decisions
- No reasoning-quality mechanism selection in 57.5.
- No transcript-content extractor or privacy/storage policy in 57.5.
- No human-first report UI, REPL, or dashboard in 57.5.
- No full Codex parity or hook-derived live collection in 57.5.

### Current Posture
- Zero-dependency Node/Markdown/SQLite project with file-first truth and rebuildable derived caches.
- Retroactive analysis over existing corpora is a core requirement, not an optional migration aid.
- Runtime asymmetry, source derivation, and reliability tier are treated as data, not as noise to wash out.

### Future Shape Notes
- Phase 57.6 adds the remaining loops, text-first human reports, and broader Codex/runtime comparison.
- Phase 57.7 adds transcript-structural extractors, reasoning-quality mechanism, intervention histories, and revision classification.
- Phase 58 depends on 57.7 because structural gates need the measurement substrate to evaluate whether interventions actually helped.
- Phase 60 can add live/hook-derived raw sources into the same registry if 57.5 keeps the source-adapter seam clean.

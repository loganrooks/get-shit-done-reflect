# Phase 57.5: Measurement Architecture & Retroactive Foundation — Research

**Researched:** 2026-04-16
**Domain:** Retroactive measurement architecture over runtime artifacts, derived `/insights` outputs, and GSDR state/artifacts
**Confidence:** HIGH for the planning-critical architecture and sequencing choices; MEDIUM for exact query naming and final storage path names; LOW for any deeper Codex/content-level extractors beyond the recommended proof-of-architecture slice

## Summary

Phase 57.5 should be planned as a substrate phase, not as a report/UI phase and not as a second telemetry utility. The correct architectural move is:

1. Keep raw truth in native artifacts (`~/.claude/usage-data/*`, `~/.codex/state_5.sqlite`, `~/.codex/sessions/*.jsonl`, `.planning/*`, git history).
2. Introduce a rebuildable sibling measurement cache/store dedicated to extractor outputs and queryable feature rows rather than extending `kb.db`.
3. Build a pure extractor registry with explicit source-family, runtime, and reliability metadata.
4. Put a JSON-first query layer on top that returns interpretation objects with provenance, reliability, coverage, freshness, and semantic unavailability markers.
5. Preserve the existing `telemetry` CLI surface as a compatibility view backed by shared loaders and, where feasible, shared extractors.

The lowest-risk Codex proof extractor is **Codex runtime metadata/provenance extraction** from `~/.codex/state_5.sqlite` plus session JSONL/session-meta headers, not an ambitious content or performance feature. The local corpus already exposes `model`, `reasoning_effort`, `sandbox_policy`, `cwd`, `rollout_path`, timestamps, and runtime identity in a stable enough shape to prove cross-runtime architecture without dragging Phase 57.6/57.7 or Phase 60 scope forward.

The phase should be planned around four implementation concerns:

- architecture and cache/query substrate
- first extractor wave for intervention-lifecycle and pipeline-integrity
- telemetry compatibility + Codex proof + retroactive corpus run
- kb freshness repair and measurement validation over the real corpus

## Locked Boundaries From Context

These are planning constraints, not open design space:

- Three layers are fixed: raw adapters -> extractor registry -> interpretation/query.
- Source families are fixed: `RUNTIME`, `DERIVED`, `GSDR`.
- 57.5 leads with the intervention-lifecycle and pipeline-integrity loops.
- 57.5 proves cross-runtime architecture with one low-risk Codex extractor, not broad Codex parity.
- 57.5 is retroactive by design: no new live collection prerequisite.
- Query results must surface provenance/reliability/unavailability semantically.
- Reasoning-quality measurement and transcript-content analysis stay out of scope.
- 57.5 must not collapse back into a Claude session-meta reader.

## Current Implementation Anchors

### `telemetry.cjs` is the closest existing raw-adapter/view layer

`get-shit-done/bin/lib/telemetry.cjs` already contains reusable patterns the new substrate should not duplicate:

- home-directory raw source discovery for Claude artifacts
- corpus loading with project filtering
- enrichment joins (`session-meta` + facets)
- explicit interpretive notes and caveat surfacing
- existing CLI views: `summary`, `session`, `phase`, `baseline`, `enrich`

What it does **not** provide is the registry/query architecture. It is a phase-specific measurement product, not yet a general substrate.

### `kb.cjs` proves the rebuildable cache pattern

`get-shit-done/bin/lib/kb.cjs` already establishes the accepted shape for derived query infrastructure in this repo:

- file-first truth
- rebuildable SQLite cache
- explicit schema initialization
- lazy `node:sqlite` usage
- index/state separation from raw artifacts

This should be copied as a pattern, not overloaded as the measurement store itself. `kb.db` is signal/spike-oriented; extending it into a generic measurement store would couple unrelated freshness semantics and make the phase harder to reason about.

### `automation.cjs` is already a live GSDR measurement surface

`get-shit-done/bin/lib/automation.cjs` and `.planning/config.json` already provide real GSDR-side measurement inputs:

- per-feature `fires`, `skips`, `last_triggered`, `last_skip_reason`
- resolved automation level/config context
- lock/regime/reflection counters

This is the clearest 57.5 path for GSDR extractors serving pipeline-integrity.

### Codex artifacts already expose a safe proof surface

The live local Codex corpus confirms that a low-risk proof extractor is available now:

- `~/.codex/state_5.sqlite` table `threads` contains `rollout_path`, `cwd`, `sandbox_policy`, `model`, `reasoning_effort`, timestamps, git metadata, and agent fields.
- Current rows for this project show `model=gpt-5.4`, `reasoning_effort=xhigh`, concrete `rollout_path`s, and sandbox policies.
- Codex session JSONL starts with `session_meta` carrying `cwd`, `model`, `effort`, runtime identity, CLI version, approval/sandbox policy, and other environment fields.

This is enough to land a Codex metadata/provenance extractor without solving transcript-content analysis or rich Codex parity.

## Recommended Architecture

### 1. Raw Layer

Keep the raw layer native and explicit. Do not pre-normalize artifacts into a single canonical JSON dump.

Recommended raw adapters:

- `RUNTIME`
  - Claude `session-meta`
  - Claude facets
  - parent JSONL for token/human-turn ground truth
  - Codex `state_5.sqlite` thread rows
  - Codex session JSONL session metadata
- `DERIVED`
  - `/insights` products including session-meta/facets write-path and mtime provenance
- `GSDR`
  - `.planning/config.json`
  - `SUMMARY.md`, `VERIFICATION.md`, `PLAN.md`, `CONTEXT.md`
  - signal files / `kb.db`
  - git history and workflow call sites where needed

Planning implication: build adapters/loaders that expose source-family and runtime metadata explicitly. Avoid a loader API that silently treats derived artifacts as runtime truth.

### 2. Extractor Registry

The extractor registry is the actual extensibility seam and should be a first-class module, not an incidental list inside one command file.

Each registry entry should declare at minimum:

- `name`
- `source_family`
- `raw_sources`
- `runtimes`
- `reliability_tier`
- `features_produced`
- `serves_loop`
- `distinguishes`
- `status_semantics` if the extractor uses `not_emitted` or other feature-specific absence states

Recommended architectural split:

- thin router module at `get-shit-done/bin/lib/measure.cjs` or `measurement.cjs`
- internal helpers under a small subdirectory such as `get-shit-done/bin/lib/measurement/`

Recommended internal responsibilities:

- `sources/` or `raw/`: runtime/artifact loaders
- `registry.cjs`: registry definition and lookup
- `extractors/*.cjs`: pure extractor implementations
- `store.cjs`: rebuildable SQLite cache for measurement outputs
- `query.cjs`: JSON interpretation objects and query assembly

This keeps the CLI module pattern intact while preventing one flat file from becoming unmaintainable.

### 3. Measurement Store

**Recommended choice:** a sibling rebuildable measurement cache/store, not `kb.db`, and not query-only-on-demand.

Why not extend `kb.db`:

- `kb.db` is about signals/spikes and their lifecycle indexing.
- 57.5 measurement outputs need runtime/source-family/query semantics that do not belong in the KB schema.
- MEAS-GSDR-05 already identifies a freshness bug around `kb.db`; coupling 57.5 to that store before fixing freshness would magnify the problem.

Why not pure on-demand only:

- retroactive corpus runs across session + artifact corpora need repeatable, testable outputs
- JSON query ergonomics improve if extractor outputs are materialized and diffable
- integration and regression tests need stable artifacts to assert against

Recommended store behavior:

- rebuildable from raw sources
- stores feature rows / extractor outputs, not coarse final reports as canonical truth
- supports query-time composition into interpretations
- keeps aggregates as query products, not canonical stored state

This matches MEAS-ARCH-05's "resolution on demand" requirement while still using SQLite as an indexed cache.

### 4. Query / Interpretation Layer

The query contract should be designed around semantic output, not numeric dumps. 57.5 only needs v1 of the interpretation object:

- `question`
- `scope`
- `runtime_dimension`
- `features`
- `interpretations`
- `distinguishing_features`
- `anomaly_register`
- `provenance`
- `reliability`
- `coverage`
- `freshness`

Recommended status modeling:

- per-feature availability: `exposed` | `derived` | `not_available` | `not_applicable`
- extractor-specific absence where needed: `not_emitted`
- freshness: `fresh` | `stale` | `unknown`
- reliability tier: `direct_observation` | `artifact_derived` | `inferred` | `cross_runtime`

Important planning note: the requirements and roadmap language currently blur "symmetry markers" and "availability statuses." Treat these as **separate concerns** in implementation:

- feature status tells what is available for a runtime/source
- symmetry marker tells whether the feature is shared/asymmetric across runtimes

The planner should not collapse those into a single enum.

## Best Codex Proof Choice

**Recommendation:** make the 57.5 Codex proof extractor a metadata/provenance extractor, not a behavioral/content extractor.

Why this is the right 57.5 proof:

- high extraction certainty from actual local artifacts
- directly serves MEAS-ARCH-04 and MEAS-ARCH-10
- immediately useful for agent-performance stratification later
- low risk of scope bleed into transcript analysis or Phase 60 live sensor work
- enough to prove the registry/query substrate is genuinely cross-runtime

Do **not** choose as the 57.5 proof:

- full Codex token accounting parity
- Codex transcript-content features
- Codex compaction/deeper behavioral features
- anything that depends on fragile inference instead of directly available fields

## How Existing `telemetry` Commands Should Relate To 57.5

The planner should assume the existing `telemetry` surface survives, but its internals stop being a separate architecture.

Recommended relation:

- move reusable raw-loading/enrichment logic into shared measurement loaders or shared helpers
- keep `telemetry` as a compatibility view for existing commands/tests
- have new measurement/query code own the generalized runtime/source semantics
- migrate overlapping telemetry computations to shared extractor logic where feasible

This is the least disruptive path that still satisfies the "no dual architecture" guardrail.

Planning consequence: do not schedule a big-bang deletion/replacement of `telemetry.cjs` in 57.5. Schedule a compatibility wrapper/refactor path.

## Provenance And Freshness Defaults

The query layer should default to explicit provenance rather than implicit trust.

Recommended query defaults:

- every feature returns source family, raw source identifiers, runtime, and reliability tier
- every aggregate returns coverage counts and a bias warning when coverage is non-uniform
- `DERIVED` sources always include last-refresh/write-path provenance
- GSDR/KB-backed answers include freshness state and rebuild requirement when stale or unknown
- cross-runtime queries never silently average away missing runtime fields

Minimum provenance object for 57.5:

- `source_family`
- `runtime`
- `raw_source_refs`
- `reliability_tier`
- `coverage`
- `freshness`
- `write_path` for derived artifacts where relevant
- `notes` / caveats

This keeps the system aligned with the deliberation's "runtime asymmetry is data, not noise" commitment.

## Test Strategy Implications

57.5 should be planned with real retroactive execution as part of validation, not just structural unit coverage. This is directly supported by the active triaged signals warning about second-order effects, tool/API assumptions, and missing model provenance.

Minimum test shape:

- unit tests for registry and status semantics
- unit tests for each core extractor family
- unit tests for query objects and semantic unavailability markers
- integration test that rebuilds the measurement store over a controlled corpus fixture
- at least one real-corpus validation path for:
  - Claude session/token coverage
  - GSDR automation stats
  - Codex runtime metadata extraction
  - stale vs fresh `kb.db` behavior

The planner should explicitly include assumption-verification work where commands, file paths, and schema fields are involved. The active plan-checker signals show that leaving this implicit is a known failure mode.

## Recommended Plan Topology

The planner should likely converge on **4 plans in 2 waves** or **5 plans in 3 waves**, not one giant architecture plan.

Recommended decomposition:

### Wave 1

- measurement substrate + store/query scaffolding
- raw adapters + core registry definition

### Wave 2

- first extractor wave for `RUNTIME`/`DERIVED`
- GSDR extractors + kb freshness fix

### Wave 3 if needed

- telemetry compatibility refactor

The important structural point is that the freshness fix and retroactive corpus run should not be left as last-minute cleanup. They are part of the proof that the architecture actually works.

## Risks

### Risk 1: `telemetry` split-brain

If 57.5 adds a new measurement command family but leaves `telemetry.cjs` as an unrelated measurement island, the project recreates the same architectural narrowing and drift this phase exists to fix.

### Risk 2: Store overreach

If the implementation extends `kb.db` into a general measurement store, freshness and ownership boundaries will become harder, not clearer.

### Risk 3: Codex scope bleed

If the Codex proof chooses a deeper behavioral/content slice, the phase will drift into 57.6/57.7/60 territory.

### Risk 4: Status-model confusion

If the planner collapses runtime symmetry and feature availability into one enum, query semantics will get muddy quickly and later phases will have to unwind it.

### Risk 5: Corpus demo becomes optional

The roadmap explicitly requires retroactive demonstration. If the plan treats this as postlude polish rather than a must-have, the phase can "ship architecture" without proving the architecture.

## Genuine Gaps

No spike-worthy research blocker remains for planning this phase. The remaining uncertainties are implementation choices the planner can carry explicitly:

- exact command naming (`measure` vs `measurement`) and final file path names
- exact boundary between compatibility wrappers and shared extractor logic inside `telemetry.cjs`
- exact measurement store path/schema names

The only live empirical caution is Codex field drift across future CLI versions, but the current local corpus is already strong enough for the recommended metadata/provenance proof extractor.

## Planner Guidance

The planner should optimize for:

- cheap extractor addition (`write pure function + register it`)
- explicit runtime/source/provenance semantics
- rebuildability over cleverness
- compatibility, not sudden CLI breakage
- real-corpus proof, not only fixture proof

The planner should avoid:

- dashboards, REPLs, or text-report ambitions
- broad Codex parity
- content/quality reasoning extractors
- hiding freshness/coverage caveats in comments rather than query output

## Recommended Bottom Line

Plan 57.5 as a rebuildable measurement substrate with a sibling SQLite cache, a first-class extractor registry, a JSON interpretation/query layer, a telemetry compatibility path, a low-risk Codex metadata extractor, and an explicit kb freshness repair. That is the smallest plan shape that still satisfies the actual 57.5 authority chain instead of silently shrinking it.

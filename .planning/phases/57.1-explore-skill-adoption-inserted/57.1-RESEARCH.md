## Summary

This phase adopts five upstream files from `upstream/main` to ship `/gsdr:explore` as a minimal GSDR-branded port of `/gsd:explore`. The installer's existing `replacePathsInContent()` function handles all namespace transforms automatically (`gsd:` → `gsdr:`, `get-shit-done/` → `get-shit-done-reflect/`, `gsd-` agent prefix → `gsdr-`), so the adoption requires only placing files in the correct source directories.

Two of the five files need zero modification beyond what the installer handles: `commands/gsd/explore.md` and `get-shit-done/workflows/explore.md`. One file (`get-shit-done/references/questioning.md`) already exists in the source tree — it must be compared against upstream to determine if an update is needed or if the existing copy is current. One file (`get-shit-done/references/domain-probes.md`) does not yet exist and must be created by copying from upstream. The test file (`tests/explore-command.test.cjs`) requires framework conversion from upstream's `node:test` to the project's vitest ESM format, plus string substitution of `gsd:explore` → `gsdr:explore`.

A prior signal [sig-2026-04-03-discuss-mode-adoption-gap] established that upstream adoptions can silently miss files when the upstream feature is multi-file. This phase avoids that risk by having a locked, enumerated file manifest (D-04) and a test that verifies each file's existence and key content properties post-install.

**Primary recommendation:** Copy four files from upstream, verify/update questioning.md, convert the test to vitest, run `node bin/install.js --local` to verify installer handles the branding, then run `npm test` to confirm the test suite passes.

**What:** The installer copies `commands/gsd/` and `get-shit-done/` wholesale. No per-file registration is needed in `feature-manifest.json` or `install.js` for new command/workflow/reference files.

- `copyWithPathReplacement(skillSrc, skillDest, ...)` copies ALL files under `get-shit-done/` to `get-shit-done-reflect/`

No installer changes are required. Adding a file to these source directories is sufficient for it to be installed.

**Recommendation on CHANGELOG:** Add an entry in this phase under `[Unreleased] > Added`. The command is a visible user-facing feature; batching with other v1.19.x changes risks the entry being forgotten.

### Pattern 4: Wiring Validation Test Impact

The `tests/integration/wiring-validation.test.js` test already validates all @-references in `commands/gsd/*.md` and `get-shit-done/workflows/*.md`. Adding `explore.md` to both directories means this test will automatically check that:
- `commands/gsd/explore.md` references `get-shit-done/workflows/explore.md` (resolves)
- `get-shit-done/workflows/explore.md` references `get-shit-done/references/questioning.md` (resolves)
- `get-shit-done/workflows/explore.md` references `get-shit-done/references/domain-probes.md` (resolves)

**All three references must resolve at time the test runs.** This means domain-probes.md must be created before running the test suite.

### Anti-Patterns to Avoid

- **Adopting files out of order:** Create domain-probes.md and questioning.md before running wiring validation — the workflow references both.
- **Forgetting to check questioning.md freshness:** The file exists in source but may be stale relative to upstream. Diff before assuming it's current.
- **Using `git checkout upstream/main -- <path>`:** This would change the working tree to the upstream state for tracked files; use `git show` to extract content instead.
- **Placing test in wrong directory:** The upstream file is `tests/explore-command.test.cjs`. The adapted file should be `tests/explore-command.test.js` (`.js` not `.cjs`) to match vitest's `include: ['tests/**/*.test.js']` glob.

## Common Pitfalls

### Pitfall 1: questioning.md Assumed Current

**What goes wrong:** `questioning.md` already exists in `get-shit-done/references/`. Executor skips the upstream fetch, assuming the file is up to date. Upstream may have updated the file since it was first adopted.
**Why it happens:** File presence implies currency — but upstream evolves independently.
**How to avoid:** Always diff the existing file against `upstream/main:get-shit-done/references/questioning.md` before deciding no update is needed. If upstream version is longer or structurally different, adopt the upstream version.
**Warning signs:** Diff output shows non-trivial differences in questioning technique sections.

### Pitfall 2: Test File Extension Mismatch

**What goes wrong:** Test file placed as `explore-command.test.cjs` (upstream filename) instead of `explore-command.test.js`. Vitest's `include: ['tests/**/*.test.js']` glob skips `.cjs` files. Tests appear to pass (no failures) but are never run.
**Why it happens:** Upstream filename has `.cjs` suffix. The fork uses `.js` for all test files.
**How to avoid:** Name the adapted test `explore-command.test.js`. Verify test count increases after adding the file.
**Warning signs:** `npm test` passes with same count (472) as before the test was added.

### Pitfall 3: Silent Adoption Gap (learned from sig-2026-04-03)

**What goes wrong:** The file manifest is treated loosely — executor writes 4 of 5 files and verifies them, misses one (e.g., domain-probes.md), and the workflow's @-references resolve to a missing file at runtime.
**Why it happens:** Prior discuss-phase adoption silently missed 6 of 8 required files despite 15/15 verification passing. Files not listed in verification must-haves are invisible to the checker.
**How to avoid:** Verification must-haves should enumerate all 5 files by path, not just the command and workflow. Domain-probes.md and questioning.md are easy to miss because they're shared references, not command-specific.
**Warning signs:** Wiring validation test fails with "Broken @-references in workflows."

### Pitfall 4: Test Location Mismatch

**What goes wrong:** Test placed in `tests/unit/` instead of the root `tests/` directory (where upstream placed it, and where similar command tests would go).
**Why it happens:** Upstream test is at `tests/explore-command.test.cjs` — it's a flat file, not in a subdirectory. Adopter may instinctively put it in `tests/unit/`.
**How to avoid:** Place the adapted test at `tests/explore-command.test.js` matching upstream's location convention for command tests.
**Warning signs:** Test file is unreachable or miscategorized.

### Adapted Test File (vitest ESM)

```javascript
// Source: upstream tests/explore-command.test.cjs, adapted to vitest ESM
import { describe, it, expect } from 'vitest'
import { existsSync, readFileSync } from 'node:fs'
import path from 'node:path'
import { fileURLToPath } from 'node:url'

const __dirname = path.dirname(fileURLToPath(import.meta.url))
const REPO_ROOT = path.resolve(__dirname, '..')

describe('explore command', () => {
  it('command file exists', () => {
    const p = path.join(REPO_ROOT, 'commands', 'gsd', 'explore.md')
    expect(existsSync(p)).toBe(true)
  })

  it('command file has required frontmatter', () => {
    const p = path.join(REPO_ROOT, 'commands', 'gsd', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('name: gsd:explore')
    expect(content).toContain('description:')
    expect(content).toContain('allowed-tools:')
  })

  it('workflow file exists', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    expect(existsSync(p)).toBe(true)
  })

  it('workflow references questioning.md and domain-probes.md', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('questioning.md')
    expect(content).toContain('domain-probes.md')
  })

  it('workflow documents all 6 output types', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('Note')
    expect(content).toContain('Todo')
    expect(content).toContain('Seed')
    expect(content).toContain('Research question')
    expect(content).toContain('Requirement')
    expect(content.includes('New phase') || content.includes('phase')).toBe(true)
  })

  it('workflow enforces one question at a time principle', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('one question at a time')
  })

  it('workflow requires user confirmation before writing artifacts', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(
      content.includes('explicit user selection') || content.includes('Never write artifacts without')
    ).toBe(true)
  })

  it('workflow respects commit_docs config', () => {
    const p = path.join(REPO_ROOT, 'get-shit-done', 'workflows', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('commit_docs')
  })

  it('command references the workflow via execution_context', () => {
    const p = path.join(REPO_ROOT, 'commands', 'gsd', 'explore.md')
    const content = readFileSync(p, 'utf-8')
    expect(content).toContain('workflows/explore.md')
  })
})
```

Note: The test checks `name: gsd:explore` (not `gsdr:explore`) because it runs against the **source** files in `commands/gsd/`, not the installed files in `.claude/commands/gsdr/`. This is the established pattern — tests verify source files, installer handles the rename.

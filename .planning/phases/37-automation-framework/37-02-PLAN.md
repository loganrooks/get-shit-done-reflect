  <action>
Add a module-level constant `FEATURE_CAPABILITY_MAP` near `KNOWN_TOP_LEVEL_KEYS` in gsd-tools.js. This map defines which runtime capabilities each feature needs at which automation levels:

```javascript
const FEATURE_CAPABILITY_MAP = {
  signal_collection: {
    hook_dependent_above: null,  // workflow postlude, not hook-based
    task_tool_dependent: false,
  },
  reflection: {
    hook_dependent_above: null,  // counter-based in workflow
    task_tool_dependent: true,   // spawns reflector as subagent
  },
  health_check: {
    hook_dependent_above: 2,     // session-start nudge needs hooks above level 2
    task_tool_dependent: false,
  },
  ci_status: {
    hook_dependent_above: 1,     // session-start display needs hooks above level 1
    task_tool_dependent: false,
  },
};
```

Extend `cmdAutomationResolveLevel` to implement the full 4-step resolution chain:

1. **Global level** (already implemented in 37-01): Read `automation.level` from config, default 1.

2. **Per-feature override** (AUTO-02): Check `automation.overrides[normalizedFeature]`. If present, use it instead of global. Add reason: `"override: {feature}={N}"`.

3. **Context-aware deferral** (AUTO-04): If `options.contextPct` is provided AND exceeds `automation.context_threshold_pct` (default 60) AND current resolved level >= 3, downgrade to 1 (nudge). Add reason: `"context_deferred: {pct}% > {threshold}% threshold"`.

4. **Runtime capability cap**: If `options.runtime` is provided OR detect runtime heuristically (check if `.claude/settings.json` exists with hooks section), look up the feature in `FEATURE_CAPABILITY_MAP`. If the feature is `hook_dependent_above: N` and runtime lacks hooks, cap effective level at N. If `task_tool_dependent: true` and runtime lacks task_tool, cap at 2. Add reason: `"runtime_capped: {feature} needs hooks above level {N}"`.

   For runtime detection heuristic (when `--runtime` not provided):
   - Check if `path.join(cwd, '.claude', 'settings.json')` exists and contains a `hooks` key -> assume hooks available
   - If no hooks detected, assume constrained runtime

5. **Fine-grained knobs** (AUTO-03): Read `automation[normalizedFeature]` from config (the per-feature knobs object). Include it in the output as a `knobs` field. These knobs are informational -- the resolve-level command reports them but does not use them for level resolution. Consuming workflows read knobs from the output to configure their behavior.

Updated output shape:
```json
{
  "feature": "signal_collection",
  "configured": 1,
  "override": 3,
  "effective": 3,
  "reasons": ["override: signal_collection=3"],
  "knobs": { "auto_collect": true, "context_threshold_pct": 65 },
  "level_names": { "0": "manual", "1": "nudge", "2": "prompt", "3": "auto" }
}
```

Important design notes:
- Resolution order matters: override BEFORE deferral BEFORE runtime cap. Each step can only reduce or maintain the level.
- Context deferral only applies to level 3 (auto). Levels 0-2 are not context-sensitive.
- Runtime capping is the final step -- it reflects hardware/software reality.
- If the feature is not in FEATURE_CAPABILITY_MAP, assume no runtime constraints (all levels achievable).
- Feature name normalization (hyphens to underscores) was added in 37-01; verify it still works with override lookups.

## Summary

This phase addresses five concrete code-level gaps where the GSD harness mishandles Codex CLI: (1) `automation.cjs` does not recognize `codex-cli` as a runtime value, causing every Codex feature to be capped; (2) `sensors.cjs` only discovers `.md` sensor files, missing the `.toml` format Codex uses; (3) `checkAgentsInstalled()` in `core.cjs` similarly only checks `.md` and `.agent.md` extensions, missing `.toml`; (4) `init.cjs` brownfield detection does not exclude `.codex/` directories; and (5) `capability-matrix.md` contains six factually incorrect claims about Codex CLI capabilities, most critically that hooks are unavailable.

Research uncovered a significant finding: Codex CLI hooks are now real and functional, not just structural placeholders. The official OpenAI developer documentation confirms SessionStart, Stop, PreToolUse, PostToolUse, and UserPromptSubmit hooks were added across versions 0.115.0-0.117.0 (March 2026). The current installed version (0.118.0) reports `codex_hooks under development true` in `codex features list`, and the live `config.toml` has `codex_hooks = true`. This means the capability matrix must be updated from "N" to a conditional status, and the runtime capability resolver should treat Codex hooks as conditionally available rather than absent.

The TOML agent/sensor file format used by Codex is simple: top-level key-value pairs with `developer_instructions` in a triple-quoted string. There is no TOML frontmatter in the YAML `---` delimited sense -- the entire file IS TOML. For sensor metadata extraction, only three fields matter (`description`, `sandbox_mode`, and the `developer_instructions` content which contains the sensor contract). The existing YAML frontmatter parser can be supplemented with a minimal inline TOML key-value parser covering the sensor contract fields (`sensor_name`, `timeout_seconds`, `config_schema`). These fields do not appear directly in the TOML file structure; they live inside the `developer_instructions` string. This means the sensor metadata for TOML files must be extracted from within the instructions text, not from top-level TOML keys.

**Primary recommendation:** Implement changes in strict feature-detection style -- add Codex capabilities to the existing resolution system rather than adding runtime-name conditional branches.

## Architecture Patterns

### Pattern 1: Feature-Capability Resolution (Existing Pattern to Extend)

**What:** `automation.cjs` resolves runtime capabilities through a two-stage system: explicit `--runtime` flag, then heuristic fallback.

**Current code (lines 114-130):**
```javascript
if (options.runtime) {
  hasHooks = options.runtime === 'claude-code' || options.runtime === 'full';
  hasTaskTool = options.runtime === 'claude-code' || options.runtime === 'full';
} else {
  // Heuristic: check for .claude/settings.json with hooks
  try {
    const settingsPath = path.join(cwd, '.claude', 'settings.json');
    if (fs.existsSync(settingsPath)) {
      const settings = JSON.parse(fs.readFileSync(settingsPath, 'utf-8'));
      hasHooks = settings.hooks !== undefined;
      hasTaskTool = true;
    }
  } catch { /* assume constrained */ }
}
```

**Required change:** Add `codex-cli` to the explicit runtime branch, and add Codex heuristic detection to the fallback path. Per the feature-detection principle, Codex capabilities should be resolved based on what is detected, not hardcoded by runtime name.

**Codex capability mapping (research-verified):**
| Capability | Value | Evidence |
|------------|-------|----------|
| hasTaskTool | `true` | `codex features list` -> `multi_agent stable true` |
| hasHooks | conditional | `codex features list` -> `codex_hooks under development true`; hooks.json mechanism documented at developers.openai.com/codex/hooks |
| hasMcp | `true` | Live MCP servers configured and working |

**When to use:** Always -- this is the existing resolution pattern, not a new one.

### Pattern 2: Multi-Directory Multi-Format Discovery

**What:** Scan multiple directories for agent/sensor files in multiple formats, deduplicating by logical name.

**Current code (sensors.cjs lines 13-17):**
```javascript
let agentsDir = path.join(cwd, '.claude', 'agents');
if (!fs.existsSync(agentsDir)) {
  agentsDir = path.join(cwd, 'agents');
}
```

**Required change:** Instead of short-circuiting on first directory found, merge results from all directories that exist, with format-aware discovery. Priority order recommendation: `.claude/agents/` > `.codex/agents/` > `agents/` (installed runtimes preferred over dev source).

**Deduplication rule:** When both `.md` and `.toml` versions of the same sensor exist (same logical name), prefer the format matching the detected runtime. In practice, this deduplication will rarely trigger because `.claude/agents/` contains `.md` files and `.codex/agents/` contains `.toml` files -- they live in different directories. However, the project-local `.codex/agents/` in the GSDR dev repo does contain `.toml` sensor files alongside the source `agents/` directory `.md` files, so the dedupe logic is needed for this project specifically.

### Pattern 3: TOML Sensor Metadata Extraction

**What:** Extract sensor contract metadata from TOML agent files.

**Key insight from research:** Codex TOML agent files do NOT have the sensor contract fields (`sensor_name`, `timeout_seconds`, `config_schema`) as top-level TOML keys. These fields exist inside the `developer_instructions` triple-quoted string, embedded in what would be the YAML frontmatter section in the `.md` version. Looking at the installed `.toml` sensor files, the top-level keys are only `description`, `sandbox_mode`, and `developer_instructions`.

**Extraction approach for TOML sensors:**
1. Read the file
2. Extract the `developer_instructions` value (everything between `'''` delimiters)
3. Within `developer_instructions`, look for the same `<inputs>` or contract metadata that exists in the `.md` version
4. For sensor name: derive from filename pattern `gsdr-{name}-sensor.toml` -> `{name}` (same as `.md` approach)
5. For timeout: search `developer_instructions` for `timeout_seconds` mentions, or default to 45

**Alternative approach (simpler, recommended):** Since the sensor name is already derived from the filename (line 50 of sensors.cjs), and timeout_seconds/config_schema are in the YAML frontmatter of `.md` files specifically, for `.toml` files use defaults (timeout=45, config_schema=null) unless the TOML file contains explicit top-level keys for these. This avoids parsing into the developer_instructions string and is consistent with how Codex treats these files.

### Pattern 4: Heuristic Runtime Detection via Config File Presence

**What:** Detect Codex runtime from project-level config files when no explicit `--runtime` flag is provided.

**Detection chain (recommended):**
1. Check `.claude/settings.json` -- if exists with `hooks` key, Claude Code with hooks
2. Check `.claude/settings.json` -- if exists without `hooks`, Claude Code without hooks  
3. Check `.codex/config.toml` -- if exists, Codex CLI
4. Check project-local `.codex/config.toml` -- same as above
5. Neither found -- assume constrained

**For Codex hook detection in heuristic mode:**
Read `.codex/config.toml` (or project `.codex/config.toml`), check for `codex_hooks = true` under `[features]`. This is a simple string search, not a full TOML parse.

**Important:** No reliable env var exists for auto-detecting "running inside Codex." The `CODEX_MANAGED_BY_NPM` env var is set on the Codex CLI process itself but is filtered by the sandbox's `shell_environment_policy` and is NOT accessible from within tool calls. The official Codex docs do not document any session-identifying env var. Runtime detection must rely on config file presence, not env vars.

### Anti-Patterns to Avoid

- **Runtime name branching:** Never add `if (runtime === 'codex-cli')` in capability resolution. Use the existing `hasHooks`/`hasTaskTool` boolean system. When `--runtime codex-cli` is passed, set the capability booleans based on what Codex supports, then let the existing cap logic handle the rest.
- **Parsing full TOML for simple fields:** The sensor files have a flat structure. A regex or line-by-line key extraction is sufficient and avoids adding a TOML parsing dependency.
- **Short-circuit directory discovery:** The current `if (!exists) fallback` pattern misses `.codex/agents/`. Use a merge-and-deduplicate approach instead.

## Common Pitfalls

### Pitfall 1: Treating Codex Hooks as Absent

**What goes wrong:** The drift audit and existing code both treat hooks as "N" for Codex. But hooks are now real and functional (SessionStart, Stop, PreToolUse, PostToolUse, UserPromptSubmit added in v0.115.0-0.117.0).
**Why it happens:** The docs were written when hooks genuinely did not exist. They were not updated when Codex added them.
**How to avoid:** Use `codex features list` output as evidence. Set `hasHooks` based on detected feature state, not a hardcoded false. The capability matrix should say "Y (under development)" not "N".
**Warning signs:** If `codex features list` shows `codex_hooks under development true` but the harness still caps hook-dependent features, the fix is incomplete.

### Pitfall 2: TOML Frontmatter Confusion

**What goes wrong:** Assuming TOML sensor files have YAML-style `---` delimited frontmatter. They do not.
**Why it happens:** The `.md` sensor files use YAML frontmatter. Natural assumption is TOML files use TOML frontmatter. But Codex TOML agent files are entirely TOML -- the top-level keys ARE the metadata.
**How to avoid:** For TOML files, read top-level key-value pairs directly. The file starts with `description = "..."`, not `---`.
**Warning signs:** If the frontmatter regex `(/^---\n([\s\S]*?)\n---/)` returns null on a TOML file, the parser is wrong.

### Pitfall 3: Discovery Directory Short-Circuiting

**What goes wrong:** Current code checks `.claude/agents/`, and if it exists, never checks `.codex/agents/`. On machines with both runtimes installed, sensors from one runtime are invisible to the other.
**Why it happens:** Original design assumed one runtime per project.
**How to avoid:** Collect from ALL existing directories, then deduplicate. The function should build a list of directories to scan, filter to those that exist, then merge results.
**Warning signs:** `sensors list` returns different counts depending on which runtime directory exists.

### Pitfall 4: Project-Local .codex/ vs Global ~/.codex/

**What goes wrong:** Brownfield detection scanning `.codex/` in the project directory (which exists in the GSDR dev repo) and finding `.toml` files, triggering false-positive "existing code" detection.
**Why it happens:** `.codex/` contains agent `.toml` files and skill `.md` files which match code extension patterns.
**How to avoid:** Add `.codex` to `skipDirs` alongside `.claude` in `init.cjs` line 215.
**Warning signs:** `init new-project` detects existing code in a directory that only has GSD/Codex config files.

### Pitfall 5: Updating Docs Without Validation Commands

**What goes wrong:** Documentation corrections are made based on the drift audit, but the underlying runtime keeps evolving. Within weeks, the corrected docs drift again.
**Why it happens:** No automated validation exists -- the parity document has no way to verify itself.
**How to avoid:** The "living document" transformation for cross-runtime-parity-research.md must include executable validation commands that can be re-run. Include: `codex --version`, `codex features list`, artifact shape checks.
**Warning signs:** The parity document has no "last-audited" date or version number.

### Example 2: Multi-Format Sensor Discovery

```javascript
// Source: sensors.cjs cmdSensorsList (replacement for lines 11-57)
function discoverSensorDirs(cwd) {
  const candidates = [
    path.join(cwd, '.claude', 'agents'),
    path.join(cwd, '.codex', 'agents'),
    path.join(cwd, 'agents'),
  ];
  return candidates.filter(d => fs.existsSync(d));
}

function discoverSensors(dirs) {
  const sensorPattern = /^gsdr?-(.+)-sensor\.(md|toml)$/;
  const seen = new Map(); // name -> { file, dir, ext }

  for (const dir of dirs) {
    const files = fs.readdirSync(dir);
    for (const file of files) {
      const match = file.match(sensorPattern);
      if (!match) continue;
      const name = match[1];
      const ext = match[2];
      // First seen wins (priority: .claude > .codex > agents)
      if (!seen.has(name)) {
        seen.set(name, { file, dir, ext });
      }
    }
  }
  return seen;
}
```

### Example 3: TOML Sensor Metadata Extraction

```javascript
// Source: Based on sensors.cjs lines 32-49, extended for TOML
function parseSensorMetadata(filePath, ext) {
  const content = fs.readFileSync(filePath, 'utf8');
  const fmObj = {};

  if (ext === 'md') {
    // Existing YAML frontmatter parser
    const fmMatch = content.match(/^---\n([\s\S]*?)\n---/);
    if (fmMatch) {
      for (const line of fmMatch[1].split('\n')) {
        const kvMatch = line.match(/^(\w[\w_]*):\s*(.+)$/);
        if (kvMatch) {
          let val = kvMatch[2].trim();
          if (val === 'null') val = null;
          else if (val === 'true') val = true;
          else if (val === 'false') val = false;
          else if (/^\d+$/.test(val)) val = parseInt(val, 10);
          fmObj[kvMatch[1]] = val;
        }
      }
    }
  } else if (ext === 'toml') {
    // TOML: sensor metadata derived from filename + defaults
    // Top-level TOML keys are description, sandbox_mode, developer_instructions
    // sensor_name, timeout_seconds, config_schema are NOT top-level TOML keys
    // Use defaults; override only if explicit TOML key found
    for (const line of content.split('\n')) {
      // Match simple TOML key = value (not inside triple-quoted strings)
      const kvMatch = line.match(/^(sensor_name|timeout_seconds|config_schema)\s*=\s*(.+)$/);
      if (kvMatch) {
        let val = kvMatch[2].trim().replace(/^"|"$/g, '');
        if (val === 'null') val = null;
        else if (val === 'true') val = true;
        else if (val === 'false') val = false;
        else if (/^\d+$/.test(val)) val = parseInt(val, 10);
        fmObj[kvMatch[1]] = val;
      }
    }
  }

  return fmObj;
}
```

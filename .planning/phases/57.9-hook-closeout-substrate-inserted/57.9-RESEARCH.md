## Summary

Phase 57.9 can now be planned against current evidence instead of stale hook assumptions. As of 2026-04-21, the live repo still does not install any Claude closeout hook and still exits early for Codex before any hook wiring. But the underlying runtime story has shifted: official Anthropic docs now expose Claude `Stop` and `SessionEnd` hooks, and official OpenAI Codex docs expose Codex `Stop`, `SessionStart`, `UserPromptSubmit`, `PreToolUse`, and `PostToolUse` hooks via `hooks.json`. The old phrase "`SessionStop`" is therefore best treated as a conceptual closeout-hook label, not the literal runtime event name to install.

The practical planning consequence is narrower than the phase wording suggests. Phase 57.9 should target the shared closeout event both runtimes actually document today: `Stop`. On Claude, `SessionEnd` is a possible secondary cleanup surface, but it is not required to satisfy the shared substrate. On Codex, the hook surface is real but still constrained: official docs say `PreToolUse` and `PostToolUse` currently only emit `Bash`, so 57.9 must not promise full tool-interception parity or incident capture through general hook callbacks. The honest path is to install the closeout hook when effective Codex support is present, otherwise emit an explicit waiver, and keep incident/postlude evidence metadata-only.

The missing substrate is therefore four concrete things: normalize artifact language from `SessionStop` to runtime-specific closeout events, extend `bin/install.js` to write the Claude closeout hook and the Codex `hooks.json` closeout hook when supported, add a canonical `.planning`-owned postlude record format plus `loadSessionMetaPostlude()` source module, and align the roadmap / requirements / capability references plus tests so they all describe the same owner and the same runtime vocabulary.

**Primary recommendation:** Plan 57.9 as a strict four-slice closure phase: (1) runtime event-name and ownership alignment, (2) Claude closeout hook install + tests, (3) Codex `Stop` hook install-or-waiver + tests, and (4) metadata-only `session_meta_postlude` source plus Phase 58 re-entry verification mapping.

### Pattern 1: Conceptual Closeout Hook, Runtime-Specific Event Names
**What:** Treat "closeout hook" as the invariant and map it to the runtime event name actually documented today. On Claude, the shared closeout event is `Stop` (with `SessionEnd` reserved for optional end-of-session cleanup). On Codex, the shared closeout event is `Stop`.
**When to use:** Whenever a phase artifact currently says `SessionStop` but the runtime config needs a literal event key.
**Example:**
```json
// Source: https://developers.openai.com/codex/hooks and https://code.claude.com/docs/en/hooks
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node .../gsdr-postlude.js",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

### Pattern 2: Metadata-Only Postlude Records, Then Adapter Loader
**What:** Have the runtime hook append JSONL records under a dedicated `.planning` measurement directory, then let `loadSessionMetaPostlude()` translate those records into the `{ts, gate, result, phase, source_file, ...}` tuples that `gate_fire_events` already expects.
**When to use:** For `postlude-fired`, `error-rate`, `direction-change`, and `destructive-event` markers that must be canonical, testable, and explicit about `not_available`.
**Example:**
```json
// Source: local design derived from 58-19 summary + ci-notices loader pattern
{
  "ts": "2026-04-21T22:30:00Z",
  "phase": "57.9",
  "runtime": "codex-cli",

  "direction_change": { "status": "not_available", "reason": "needs downstream live wiring" },
  "destructive_event": { "status": "not_available", "reason": "needs downstream live wiring" }
}
```

### Pattern 3: Waiver as Verifiable Project State
**What:** When the runtime cannot honestly install Codex closeout hooks, write an explicit project-level waiver marker that verification can inspect without reading runtime internals.
**When to use:** When `codex_hooks` is unavailable, disabled, or otherwise not safe to rely on for the target install.
**Example:**
```json
// Source: local design constrained by 58-16 re-entry conditions
{
  "codex_hooks_waived": true,
  "codex_hooks_waiver_reason": "target runtime lacks effective codex_hooks support",
  "codex_hooks_checked_at": "2026-04-21T22:30:00Z"
}
```

### Anti-Patterns to Avoid
- **Literal `SessionStop` config keys:** current official docs use `Stop` / `SessionEnd` on Claude and `Stop` on Codex, so planning against a nonexistent literal key will ship broken config.
- **Overclaiming Codex hook coverage:** current official Codex docs say `PreToolUse` and `PostToolUse` only emit `Bash`; do not claim file-write, MCP, or general tool interception.
- **Silent Codex no-op installs:** if hooks are not installed, there must be an explicit waiver marker and verification witness.
- **Direct runtime reads in consumers:** Phase 58 consumers should read the canonical postlude source, not parse runtime transcript/config files directly.
- **Doc-only closure:** roadmap or capability-matrix updates without installer/test evidence would recreate the exact false-closure pattern 57.9 exists to prevent.

## Common Pitfalls

### Pitfall 1: Runtime Event Name Drift
**What goes wrong:** Plans and docs say `SessionStop`, but current official runtime docs use `Stop` (both runtimes) and `SessionEnd` (Claude-only).
**Why it happens:** Earlier repo artifacts used `SessionStop` as a conceptual closeout label and that wording propagated into ROADMAP/REQUIREMENTS/Phase 58 artifacts.
**How to avoid:** Treat `SessionStop` as a semantic alias in planning artifacts, but install literal `Stop` on Codex and literal `Stop` or `SessionEnd` only where the runtime actually documents them.
**Warning signs:** Plans that say "write `SessionStop` into settings.json/hooks.json" without citing current runtime docs.

### Pitfall 2: Over-Wiring Codex Beyond What Docs Guarantee
**What goes wrong:** The plan assumes Codex hooks can intercept all writes/tools, then builds incident detection or enforcement on surfaces that do not fire.
**Why it happens:** Codex docs now document hooks, so it is easy to jump from "hooks exist" to "hooks are parity-complete."
**How to avoid:** Keep 57.9 scoped to `Stop`-based closeout and metadata emission; explicitly document that current Codex `PreToolUse`/`PostToolUse` are Bash-only and work in progress.
**Warning signs:** Any 57.9 task that depends on Codex `Write`, `ApplyPatch`, MCP, or WebSearch hook interception.

### Pitfall 3: Effective Codex Capability Misread from the Wrong Config Layer
**What goes wrong:** GSD reads only one `config.toml` location and either over-waives or over-claims actual runtime hook support.
**Why it happens:** In this repo today, the project `.codex/config.toml` lacks `codex_hooks = true`, while the user-global `~/.codex/config.toml` enables it.
**How to avoid:** Plan an explicit verification path for effective Codex capability rather than assuming project-local config alone is authoritative.
**Warning signs:** Installer or verifier logic that only greps `.codex/config.toml` and never checks the runtime-effective path.

### Pitfall 4: Leaving the Test Suite in the Old "Codex Stays Hook-Free" World
**What goes wrong:** Source changes ship but tests still assert that Codex never gets hooks, so the new substrate either fails CI or stays half-migrated.
**Why it happens:** `tests/integration/multi-runtime.test.js` currently has an explicit assertion that Codex remains hook-free.
**How to avoid:** Update tests together with installer behavior and split supported-vs-waived scenarios explicitly.
**Warning signs:** One path tests "Codex hook-free" while another path starts writing `.codex/hooks.json`.

### Pitfall 5: Postlude Source Contract Drifts from the Existing Extractor Seam
**What goes wrong:** The new source writes rich marker records, but `loadSessionMetaPostlude()` or `gate_fire_events` expect a different tuple shape.
**Why it happens:** HOOK-03 needs canonical markers, while `gate_fire_events` currently expects gate-style rows.
**How to avoid:** Keep persisted records metadata-rich but make the loader normalize them to the gate tuple shape already described in `58-19-SUMMARY.md`.
**Warning signs:** Planner tasks create a marker file but never specify how the source module maps it to `{ts, gate, result, phase, source_file}`.

### Pitfall 6: Backward Compatibility Is Deferred Until Verification
**What goes wrong:** The new substrate assumes empty directories, clean installs, or already-enabled flags, then verification discovers old installs or mixed config states require extra work.
**Why it happens:** Installer/harness work often touches pre-existing runtime surfaces and project config with legacy states.
**How to avoid:** Add explicit pre-validation and mixed-state tests during planning, not after execution.
**Warning signs:** Plans that mention only happy-path fresh install checks.

## Open Questions

### Resolved
- **Q1: Under exactly what runtime evidence does Phase 57.9 install Codex hooks versus emit a waiver?**  
  Install when effective Codex support is present, with `codex_hooks = true` as the minimum positive capability witness. Live repo evidence on 2026-04-21 confirms `codex-cli 0.121.0` reports `codex_hooks under development true`. However, this repo also shows a config-layer mismatch (`project_has_flag=no`, `global_has_flag=yes`), so planning must not rely on project-local config alone when deciding whether runtime support exists.
- **Q2: What persisted artifact should `session_meta_postlude` read?**  
  A dedicated `.planning`-owned metadata-only JSONL stream is the safest fit, e.g. `.planning/measurement/session-meta-postlude/*.jsonl`. Hook scripts should append canonical marker rows there, and `loadSessionMetaPostlude()` should adapt them into the `{ts, gate, result, phase, source_file, ...}` tuples already expected by `gate_fire_events`.
- **Q3: Where should explicit Codex waiver evidence live?**  
  `.planning/config.json` should remain the canonical waiver witness because the existing Phase 58 re-entry contract already names it. If the implementation also wants a runtime-local note, treat that as supporting evidence, not the primary witness.

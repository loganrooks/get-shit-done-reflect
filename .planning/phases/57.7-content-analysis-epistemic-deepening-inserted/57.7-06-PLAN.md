Write the privacy-model documentation per CONTEXT.md. Required content:

```markdown
# Measurement Privacy Model (MEAS-ARCH-09 / DC-3 / G-1)

**Governing principle:** no raw content re-exposure. The measurement store holds **derived features only** — counts, names, positional markers, structural indicators. Raw JSONL stays on disk as source; derived features land in `measurement.db` with the source content itself discarded after extraction.

## content_contract enum

Extractors that touch session content declare `content_contract` in their registry entry. Values:

- `derived_features_only` — extractor reads content in-memory to compute a derived feature, then discards the content. Only the derived value is persisted. This is what 57.7 content extractors ship with.
- `metadata_only` — extractor reads session metadata only (session_id, model, timestamps). Never touches content.
- `no_content_access` — extractor reads file paths / directory contents, not session content at all.

Extractors without the field have `content_contract: null` on their frozen registry entry (Plan 04 registry fix preserves this). They operate on pre-derived features or non-content sources.

## What each 57.7 content extractor computes and discards

### tool_invocation_sequence (Plan 04)
- Reads: `assistant.message.content[*].type === 'tool_use'` blocks
- Persists: ordered sequence of tool names (e.g., `['Bash', 'Read', 'Edit']`), per-tool counts, Shannon entropy
- Discards after extraction: tool input arguments, tool_result content, assistant text content, thinking summaries
- Never stored: the verbatim `input` object of any tool_use block

### topic_shift_markers (Plan 04)
- Reads: `/clear` commands (already counted by `countClearInvocations`), `compact_boundary` markers (`extractCompactionEvents`), tool-category transitions
- Persists: counts of each shift type and positional indices (record-array indices)
- Discards after extraction: all content text; only the structural boundary positions remain
- Never stored: any text that triggered the boundary

### intervention_points (Plan 05)
- Reads: user-message text (filtered by `isHumanTurnRecord` to exclude tool_result echoes) to apply `INTERVENTION_MARKER_REGEX`
- Persists: intervention count, positions (record indices), interventions_per_100_turns, mean_intervention_position
- Discards after extraction: the user-message text itself; `sampled_matches` field used only during calibration, NOT persisted in the stored feature value
- Never stored: the user's verbatim intervention text

## Where the privacy model surfaces

Three places per CONTEXT.md:

1. **Registry metadata:** each extractor declares `content_contract: 'derived_features_only'` — Plan 04 registry fix preserves this on the frozen entry, visible in `measurement rebuild` output and in `query.cjs` feature records.
2. **This document (`.planning/measurement/PRIVACY.md`)** — the operator-readable reference.
3. **Report header privacy line:** `measurement report <loop>` outputs a `privacy:` line naming which content-derived features were computed in that report. This is wired in Plan 07 using a count filter on `registry.extractors.filter(e => e.content_contract === 'derived_features_only')`.

## Operator opt-out

57.7 does not ship a runtime opt-out. If an operator wants to exclude content extraction:
- Delete or rename the relevant extractors from `extractors/runtime.cjs` (local edit)
- Or: omit the `--include-content` flag (not implemented in 57.7 — future opt-out mechanism is a Phase 60 concern)

## Future scope

Phase 60 may add hook-derived live measurement collection. If that lands, this privacy model extends but does not relax — content still becomes derived features in real time, not raw stream.

## Cross-references
- CONTEXT.md DC-3 (governing)
- Deliberation §5 Phase 57.7 scope
- Plan 04 extractors (tool_invocation_sequence, topic_shift_markers)
- Plan 04 registry.cjs fix preserving `content_contract` on frozen entry
- Plan 05 extractor (intervention_points) with calibration-gated ship mode
```

**Part B — `.planning/measurement/interventions/README.md`:**

```markdown
# Intervention-Outcome Records

Interpretation objects may be acted on by interventions (GATE shipments, architecture changes, etc.). When that happens, a committed record here carries the predicted outcome at intervention time and the actual outcome when a later revision populates it.

These records are **file-first source of truth** (DC-10 resolution-on-demand). The measurement query layer reads them at query time to compose intervention-outcome status on interpretations — there is no SQL INSERT path, and caching a summary on the interpretation row is forbidden.

## Schema

Each record is a committed markdown file with YAML frontmatter.

### Required frontmatter fields

| Field | Type | Description |
|---|---|---|
| `intervention_id` | string | Stable identifier, `int-YYYY-MM-DD-<slug>` |
| `interpretation_id` | string | References the interpretation this intervention acts on. Two forms accepted: (a) the live-registry interpretation id from `query.cjs` `buildInterpretations` (e.g., `phase_57_5_live_registry_query`); (b) a diagnostic slug (e.g., `2026-04-17-phase-57-vision-drop`). Matching is exact string equality. |
| `intervention_description` | string (multi-line YAML OK) | Human-readable description of what the intervention does |
| `intervention_artifact` | object | `{phase, requirement, commit}` — traceability to what shipped |
| `predicted_outcome` | object | `{summary, measurable_in_terms_of, evaluation_horizon}` — what the intervention was expected to change |
| `actual_outcome` | object | `{summary, evidence_paths}` — populated on later revision; null/omitted if pending |
| `outcome_status` | enum | One of `pending | confirmed | disconfirmed | mixed`. No `presumed_successful` fallback (DC-8). |
| `evaluation_date` | ISO 8601 or null | When actual_outcome was populated; null while pending |

### Optional fields

| Field | Type | Description |
|---|---|---|
| `superseded_by` | string | If this intervention has been invalidated or replaced by a later one |
| `qualifies` | string | If this intervention qualifies an earlier intervention's outcome |

## Naming convention

`YYYY-MM-DD-<short-slug>.md` — slug is hyphenated, ≤40 chars, descriptive of the intervention.

Alternative (for easier interpretation-grouped browsing): `<interpretation_id>-<slug>.md`. This plan leaves the choice to Plan 10 (first intervention record writer); both forms parse the same way.

## Lifecycle

1. Write record with `outcome_status: pending`, `actual_outcome: null`, `evaluation_date: null`
2. Commit to this directory
3. When a later phase/revision can evaluate the intervention against real post-phase data, update `actual_outcome` + `outcome_status` + `evaluation_date`, commit the revision
4. Never delete records — invalidation is via `outcome_status: disconfirmed` or `superseded_by` link

## Governing principles

- **G-2 (pending records are real):** An intervention with `outcome_status: pending` is a real epistemic object, not a placeholder. It proves the schema + workflow work. Don't close pending records by manufacturing outcomes.
- **DC-8 (no "probably worked"):** Only the four enum values. No soft statuses.
- **DC-10 (resolution-on-demand):** The query layer reads these files at query time and composes intervention-outcome status on interpretation objects. No pre-stored summary.
- **G-7 (no success theater):** A `confirmed` status requires the predicted_outcome to be narrow enough that "confirmed" is defensible (bounded, measurable, with cited evidence). If the prediction is too vague to be structurally evaluated, ship `pending` or do not ship at all.

## Consumer

`get-shit-done/bin/lib/measurement/query.cjs` exposes `loadInterventionOutcomes(cwd, interpretationId)` (shipped in Plan 07) — reads this directory, filters by `interpretation_id` via exact string equality, returns array of outcome records joined to the interpretation.

**Interpretation id shape (verified against query.cjs):** `buildInterpretations` at `query.cjs:628` currently returns interpretations with `id: 'phase_57_5_live_registry_query'`. Side-car records link to that id directly OR to a diagnostic slug when the interpretation is diagnostic-derived. Both are valid; match is exact string equality.

## Examples

See `.planning/phases/57.7-content-analysis-epistemic-deepening-inserted/57.7-10-SUMMARY.md` (when written) for the first committed record (GATE-09 intervention with pending outcome).
```

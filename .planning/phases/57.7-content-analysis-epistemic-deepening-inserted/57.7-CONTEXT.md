1. **Content axis.** The measurement system gains session-content structural extractors (tool-invocation sequences, intervention points, topic shifts) under an explicit privacy/storage model with no raw transcript re-exposure, plus the phantom-thinking-token reconciler (MEAS-RUNTIME-05) and reasoning-quality measurement (MEAS-RUNTIME-11). 57.5/57.6 content coverage is metadata-and-markers only; 57.7 is the first time transcripts themselves become a named source.

## Working Model & Assumptions

### Session-content extraction scope and privacy/storage model

**Current state:** [evidenced:cited] 57.5/57.6 content-like features stop at metadata and regex markers over Claude's `thinking_summary` (MEAS-RUNTIME-01/06), plus structural non-content extractors over raw JSONL (`/clear` counts, compaction boundaries). No extractor reads transcript message bodies. The measurement deliberation §5 Phase 57.7 scope names "structural patterns in transcripts — tool invocation sequences, intervention points, topic shifts — with documented privacy/storage model; no raw content re-exposure." ROADMAP.md Phase 57.7 SC-1 restates this verbatim. <!-- Load-bearing for every content-extraction decision below; traceable to .planning/deliberations/measurement-infrastructure-epistemic-foundations.md §5 Phase 57.7 and ROADMAP.md Phase 57.7 entry -->

- [governing:reasoned] "No raw content re-exposure" means the measurement store holds **derived features only** — counts, sequences of tool-invocation names, positional markers, topic-shift indicators — never verbatim user text, tool arguments beyond tool name, or assistant response bodies. Raw JSONL stays on disk as source; derived features land in `measurement.db` (or in the raw_events rows we already preserve) with the content itself discarded after extraction.
- [decided:reasoned] Three content-structural extractors ship in 57.7, one per named pattern class in the requirement text: (a) `tool_invocation_sequence` — ordered list of tool names per session, features: `tool_count_per_tool`, `tool_sequence_entropy`, `distinct_tool_count`; (b) `intervention_points` — count and positions of user messages interrupting multi-turn assistant sequences (post-reasoning corrections, scope-redirects, stop requests), features: `intervention_count`, `interventions_per_100_turns`, `mean_intervention_position`; (c) `topic_shift_markers` — structural shift indicators (sequential tool-category changes, assistant commits closing topic blocks, large token jumps between messages), features: `topic_shift_count`, `mean_shift_interval_turns`. All three operate on JSONL that 57.6 already loads through `sources/claude.cjs`; Codex adapter is asymmetric-only in 57.7 per the same policy 57.6 applied to `/clear`.
- [decided:reasoned] Privacy-model surfaces explicitly in three places: (i) each extractor declares `content_contract: "derived_features_only"` in its registry entry; (ii) a new `docs/measurement-privacy.md` (or equivalent section in an existing measurement doc) documents what is computed, what is discarded, what the operator can opt out of; (iii) the human `measurement report` header block surfaces a `privacy:` line per report stating which content-derived features were computed and whether their source content was retained (default: discarded after extraction).
- [assumed:reasoned] "Topic shift" does not require semantic analysis in 57.7. The deliberation and requirement text frame content-analysis as **structural**; semantic topic-modeling belongs downstream of 57.7 (if ever). Topic-shift detection ships as purely structural indicators (tool-category changes, turn-size discontinuities, explicit `/clear` reset, compaction boundary). If a plan reviewer argues semantic analysis is required, the phase defers semantic work rather than pulling NLP dependencies into the zero-dependency project.
- [assumed:reasoned] "Intervention point" counts user messages that *change direction* mid-sequence, not every user message. Operational definition: a user message is an intervention when it (i) arrives after ≥2 consecutive assistant turns AND (ii) opens with one of {`stop`, `no`, `actually`, `wait`, `let me`, `instead`, `don't`, `can you first`, `before that`, `hold on`} (case-insensitive, regex-matched). The regex set is calibration-prone; it ships with an auto-report of match rate per session so the plan can reject the heuristic if it fires on fewer than, say, 5% of long sessions. If the calibration fails, the feature ships as `not_available` rather than with a silently-wrong definition.
- [projected:reasoned] Phase 58 GATE-09 effectiveness evaluation will cite content-feature aggregates (intervention counts, topic shifts per phase) to detect scope-narrowing cascades structurally rather than from post-hoc audit. 57.7's content extractors must be tagged `serves_loop: ['pipeline_integrity']` so GATE-09 queries reach them without refactor.

### Post-Popperian epistemic deepening — suggestion, intervention, revision, provenance

**Current state:** [evidenced:cited] The 57.5/57.6 interpretation object already carries competing interpretations, distinguishing features (as declared lists per interpretation), anomaly registers, and runtime-dimension symmetry markers. What is missing: (i) *suggestion* — when distinguishing features aren't computed, the system says so but does not propose which registry features would actually discriminate; (ii) *intervention-outcome* — interpretations that have been acted on (GATE-09, 57.5 architecture) do not carry predicted-vs-actual records; (iii) *revision classification* — when a 57.6 diagnostic is revised, the revision is not classified Lakatos-progressive or -degenerating; (iv) *provenance summary* — the interpretation object carries provenance fields but does not aggregate them into the single epistemic-status sentence the deliberation §4.1 specifies ("surviving challenge from N perspectives, grounded in M intervention-outcome pairs, carrying K tracked anomalies, revision history classified as [progressive/degenerating/mixed]"). <!-- Load-bearing for all four post-Popperian sub-claims that follow; traceable to .planning/phases/57.5-measurement-architecture-retroactive-foundation/57.5-VERIFICATION.md Truth 4 (competing interpretations/distinguishing features/anomaly register; corrected by context-checker: was Truth 3 which covers retroactive analysis) and the interpretation-object shape in get-shit-done/bin/lib/measurement/query.cjs and .planning/measurement/diagnostics/README.md -->

- [decided:reasoned] Distinguishing-feature suggestion ships as **registry-traversal**, not LLM-assisted. Given a pair of competing interpretations with declared `distinguishes` lists, the system scans `buildRegistry()` for extractor outputs whose `distinguishes` metadata overlaps with that interpretation-pair and reports the ones that have NOT been computed for the current session/phase scope. This is deterministic, testable, and grounds the suggestion in already-declared registry metadata rather than in an LLM's guess.
- [assumed:reasoned] LLM-assisted suggestion is deferred to the backlog. Once the registry-traversal path is live and usage patterns emerge, an optional LLM-based "suggest features beyond the registry" layer can be added, but shipping that first would formalize before trial (per `feedback_trial_before_formalize`).
- [decided:reasoned] Intervention-outcome tracking ships as **side-car records in `.planning/measurement/interventions/`**, not as new SQL rows. Each record is a committed markdown file with frontmatter declaring: `interpretation_id` (path to the interpretation it acts on), `intervention_description`, `intervention_artifact` (commit, phase, plan, requirement), `predicted_outcome` (what the intervention expected to change, in measurable terms), `actual_outcome` (populated by a later revision), `outcome_status ∈ {pending, confirmed, disconfirmed, mixed}`. The measurement query layer reads these at query time (there is no INSERT path) and joins them to interpretation objects via `interpretation_id`. This mirrors the 57.6 diagnostic-as-markdown pattern and keeps the SQL store file-derivable.
- [decided:reasoned] Revision classification ships as **author-assigned at revision time, verified by a structural check**. When a diagnostic is revised, the revision record carries a `revision_classification ∈ {progressive, degenerating, mixed, unclassified}` field plus a justification paragraph. The structural check: a revision claiming `progressive` must introduce at least one new `distinguishes` entry that previous versions did not carry, AND at least one new anomaly-register entry resolved (moved from open-anomaly to explained-anomaly) by the revision. A revision claiming `degenerating` must explicitly note which anomaly it accommodated and what new prediction (if any) it generates. If neither check passes, the record is `unclassified` and the plan documents the gap rather than forcing a verdict.
- [assumed:reasoned] Fully automatic classification (the system decides progressive vs degenerating without author input) is out of scope. Lakatos's distinction is inherently interpretive; the 57.7 deliverable is a *ledger that makes the distinction explicit and checkable*, not a classifier that makes the distinction for us.
- [decided:reasoned] Full-provenance summary ships as a **derived field on the interpretation object**, computed on query rather than stored. `queryMeasurement()` (or a small helper) composes the summary string from: count of competing interpretations that received at least one distinguishing-feature computation; count of intervention-outcome records with `outcome_status ∈ {confirmed, mixed}`; anomaly-register size; revision-classification distribution across the interpretation's revision history. The string is surfaced in the JSON contract and in the report layer header block. Storing it would pin a stale summary against updated side-car records; computing it on-query preserves the resolution-on-demand principle (MEAS-ARCH-05).
- [governing:reasoned] "An interpretation is never verified" (deliberation §2.3, §4.1, P1) continues to govern output. The full-provenance summary never emits strings like "confirmed" or "validated"; the strongest phrasing any interpretation carries is "surviving challenge from N perspectives, grounded in M interventions, carrying K tracked anomalies."

### Phantom-thinking-token reconciler (MEAS-RUNTIME-05)

**Current state:** [evidenced:cited] REQUIREMENTS.md MEAS-RUNTIME-05 specifies the formula exactly: `raw_thinking_tokens = output_tokens − tokens(visible_output) − tokens(thinking_summary)` for Claude; direct `token_count.reasoning_output_tokens` for Codex; reported on a common `reasoning_tokens` axis. Negative deltas are flagged. The requirement text explicitly notes "blocked on real tokenizer (spike C3); schema is specifiable now." 57.5/57.6 did not specify the schema. <!-- Load-bearing for 57.7's MEAS-RUNTIME-05 delivery shape; traceable to REQUIREMENTS.md line 137 -->

- [decided:reasoned] The reconciler ships as a single extractor named `reasoning_tokens_reconciler` under `extractors/runtime.cjs` with `runtimes: ['claude-code', 'codex-cli']` and explicit per-runtime branches: Claude branch computes the delta formula, Codex branch reads the direct field. Features produced: `reasoning_tokens` (number), `reasoning_tokens_source ∈ {derived_delta, direct_count}`, `negative_delta_flag` (bool), `tokenizer_id` (string). One extractor rather than two per-runtime extractors preserves the unified `reasoning_tokens` axis the deliberation §4.3 mandates.
- [decided:reasoned] Schema lands even if the formula does not. If C3 confirms a usable tokenizer, the formula computes and the feature carries numbers. If C3 does not, the feature returns `not_available` with `skip_reason: 'tokenizer_unavailable'` on Claude sessions and returns direct counts on Codex sessions only (asymmetric but legitimate — Codex exposes the value directly, so there is no reason to withhold it just because Claude cannot match).
- [stipulated:reasoned] "Usable tokenizer" means a tokenizer whose output for the Claude model family in use disagrees with character-count-divided-by-4 heuristic on at least the current session corpus by a median relative error under 15%. The threshold is a stipulation; C3's DECISION.md revises it if the spike finds the bar is wrong. <!-- typed by context-checker: threshold is a picked criterion, not an assumption; corrected from [assumed:reasoned] -->
- [governing:reasoned] The reconciler never silently falls back to a heuristic. If the tokenizer is unusable, the feature is `not_available`, not `estimated`. Silent estimation is the exact verificationism trap §2.2 warns against — a "verified" number backed by a bad tokenizer is worse than an explicit gap.

## Epistemic Guardrails

- **G-1:** [governing:reasoned] Content analysis is structural, not semantic, in 57.7. Tool-invocation sequences, intervention counts, topic-shift markers are structural and defensible. Topic modeling, sentiment analysis, or LLM-summarization of transcripts is semantic and out of scope. A plan proposing semantic content analysis must either revise the scope formally or defer the feature past 57.7 — it cannot slip in under "content extraction."
- **G-2:** [governing:reasoned] Intervention-outcome records with `outcome_status: pending` are real records. They prove the schema and workflow work. A plan that closes pending records by manufacturing outcomes to satisfy a success criterion fails demonstration integrity — ship pending, document the evaluation horizon.
- **G-3:** [governing:reasoned] Revision classification is author-assigned with a structural check; it is never fully automatic. A plan proposing "the system decides progressive vs degenerating" confuses Lakatos's interpretive distinction with a classifier — the 57.7 deliverable is a *ledger*, not a verdict.
- **G-4:** [governing:reasoned] Distinguishing-feature suggestion is registry-traversal, not LLM-guessing, in 57.7. Shipping an LLM suggester first formalizes before trial and couples the measurement substrate to an LLM call on every interpretation query. If registry-traversal proves insufficient after trial, an LLM extension can follow.
- **G-5:** [governing:reasoned] Reasoning-quality feature output is always labelled proxy and always tier `inferred`. No exception regardless of which C5 candidate wins. DC-4 is not negotiable.
- **G-6:** [governing:reasoned] Schema-only delivery is a legitimate outcome for MEAS-RUNTIME-05 if C3 finds no usable tokenizer. Silent estimation via character-count heuristic is worse than explicit `not_available`. DC-8 mirrors this for intervention-outcome: honest gaps beat forced numbers.
- **G-7:** [governing:reasoned] The 57.7 demonstration artifacts are not success theater. If the revised vision-drop diagnostic cannot be honestly classified as progressive, it ships as `unclassified` with a documented gap. If the GATE-09 intervention-outcome record has no post-GATE-09 phases to evaluate against, it ships as `pending`. Fidelity to what the substrate can support governs over checkbox success.
- **G-8:** [governing:reasoned] FTS5 drop is a deletion, not a rewrite. 57.7 does not pre-implement contentless FTS5 "in case we need it later" — that is speculative work. The re-entry path is documented; the current phase deletes.

## Open Questions

### Q1: Does a usable tokenizer exist for the Claude model family currently in use, and under what license / dependency terms?
**Research program:** Spike C3, running as the first (or co-first with C5) plan of Phase 57.7. Candidate libraries to evaluate: `@anthropic-ai/tokenizer` (if published), `tiktoken` (OpenAI's BPE, known to mis-tokenize Claude non-trivially), community reverse-engineered tokenizers. Evaluation method: run each candidate against a sample of 20 sessions where `output_tokens` is known; compute `raw_thinking_tokens = output_tokens − tokens(visible_output) − tokens(thinking_summary)`; measure negative-delta rate and median absolute error. Decision threshold: median relative error under 15% AND negative-delta rate under 20% on the sample. Secondary axis: dependency weight (zero-dependency preferred; WASM/Node-native rejected only if install burden is heavy).
**Downstream decisions affected:** MEAS-RUNTIME-05 ships as full (computed) vs schema-only (`not_available`); whether the reconciler is a 57.7 deliverable or a schema-only deferral to 57.7+.
**Reversibility:** MEDIUM — schema-only ships preserve the extractor seam and can be upgraded without refactor when a tokenizer lands. A wrong "usable" verdict is more costly: it pollutes the reasoning-token axis across the corpus.

### Q2: Which MEAS-RUNTIME-11 candidate mechanism (reference-density / concept-diversity / LLM-as-judge / facets substitute) survives the C5 spike and ships as the reasoning-quality proxy in 57.7?
**Research program:** Spike C5, running as the first (or co-first with C3) plan of Phase 57.7. Design a dual-axis evaluation: (i) **convergent validity** — does the candidate correlate with independent quality markers (human rubric score on a small sample, outcome quality of phases)? (ii) **discriminant validity** — does the candidate distinguish sessions where we independently know reasoning quality varied (e.g., Opus vs Haiku same-phase, different profile levels)? Candidates: (a) reference-density — count of cross-references to source files, deliberations, prior phases per 1k tokens; (b) concept-diversity — TTR or similar lexical-diversity over non-stop tokens in thinking summaries; (c) LLM-as-judge — model-graded rubric score over completed session artifacts; (d) facets-based substitute — reuse of 57.6 `facets_semantic_summary` goals/outcome/friction fields as proxy cluster. Evaluate on the existing session corpus; pick the one with best convergent+discriminant validity that also fits the zero-dependency posture. Reject any mechanism whose implementation would require adding an ML/NLP dependency beyond what the project already carries.
**Downstream decisions affected:** MEAS-RUNTIME-11 extractor implementation, Agent Performance loop's reasoning-quality feature, whether LLM-as-judge enters the project's measurement substrate at all (currently it does not).
**Reversibility:** LOW on the mechanism choice — once shipped, aggregate reasoning-quality proxy rows populate the store and future phases will cite them; switching mechanisms mid-way creates two incomparable axes. HIGH on the schema: all four candidates fit the existing `reasoning_quality_proxy` feature slot, so the outer interpretation-layer consumers do not change.

### Q3: Does the intervention-point heuristic (user message preceded by ≥2 assistant turns AND opening with a change-of-direction marker) fire at a meaningful rate across the session corpus?
**Research program:** Prototype the regex set (seed list documented in working model) against 50 sessions of varied length. Compute intervention rate, distribution by session length and model, and false-positive rate by manually sampling 20 flagged turns. If the rate is below 5% on long sessions or the false-positive rate is above 30%, the heuristic ships as `not_available` (feature registered but skip-gated) and a post-57.7 redesign is required. If the rate is healthy and false-positives are below 30%, the heuristic ships live. Calibration is empirical, not theoretical — 57.7 needs a measurement, not a linguistics debate.
**Downstream decisions affected:** Whether the intervention-points extractor ships as a live feature or as schema-only; whether Phase 58 GATE-09 can cite intervention counts as a scope-narrowing indicator.
**Reversibility:** HIGH — intervention-points can ship as `not_available` and be revisited. The structural shape of the extractor does not change based on the heuristic's success; only the feature's data population does.

### Q4: What is the minimum evidence threshold for classifying the vision-drop diagnostic revision as "progressive" under the 57.7 structural check, and does 57.7's content extraction produce enough evidence to clear it?

If the count is positive, 57.7 writes the progressive revision and commits the revised diagnostic artifact.
**Downstream decisions affected:** Whether the primary 57.7 demonstration artifact classifies its revision as progressive or unclassified; whether 57.7 closure includes a positive demonstration of the revision-classifier or an honest gap documentation.
**Reversibility:** HIGH — honest gap documentation is a legitimate outcome per G-7. Forcing a progressive verdict to close the phase is the worse failure mode.

### Q5: When no post-GATE-09 phases have closed, does a `pending` intervention-outcome record satisfy ROADMAP Phase 57.7 SC-3 ("surface this as the strongest epistemic status available")?

**Reversibility:** MEDIUM — picking the wrong intervention at discuss-phase cascades into execution artifacts; the demonstration material is load-bearing for 57.7 closure.

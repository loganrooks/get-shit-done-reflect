## Summary

Phase 57.7 extends — never rewrites — the Phase 57.5/57.6 measurement substrate along two axes. The **content axis** adds three structural extractors (`tool_invocation_sequence`, `intervention_points`, `topic_shift_markers`) whose privacy commitment is surfaceable because every extractor already declares its contract via `REQUIRED_EXTRACTOR_FIELDS`; we add one optional field (`content_contract: "derived_features_only"`) and the existing header-block primitive in `report/primitives.cjs:92` already accepts new caveat/key lines. The **epistemic axis** deepens `measurement/query.cjs:628-661` — which today emits a single hard-coded interpretation with competing readings — into a real registry-traversal suggester, intervention-outcome joiner, and provenance-summary composer. None of these changes touches `REQUIRED_EXTRACTOR_FIELDS` or `LOOP_DEFINITIONS`.

Two spikes (**C3** tokenizer availability, **C5** reasoning-quality mechanism) run as in-phase first plans. The tokenizer landscape is genuinely constrained: `@anthropic-ai/tokenizer` is explicitly obsolete for Claude 3+ models; the official `messages.countTokens` method requires an API call; `js-tiktoken` is pure-JS-with-zero-dependencies but only an *approximation* for Claude. The stipulated threshold (median rel error < 15%, negative-delta rate < 20%) is the empirical decision-gate — if no candidate clears it, MEAS-RUNTIME-05 ships schema-only per DC-5. C5 evaluates four candidates whose implementability diverges sharply: facets-substitute is a zero-line extractor (reuses `facets_semantic_summary` fields); reference-density needs a new extractor but no dependency; LLM-as-judge violates trial-before-formalize and is pre-emptively deferred in CONTEXT.md.

The FTS5 drop is genuinely simple — `kb.cjs:135` is a single `CREATE VIRTUAL TABLE IF NOT EXISTS signal_fts ... USING fts5(...)` statement with no triggers, no population path, and no current consumer. Migration is a `DROP VIRTUAL TABLE IF EXISTS signal_fts` added to the next `initSchema()` call or a one-shot migration, plus removing the `CREATE` line from `initSchema`. The riskiest content extractor is **intervention_points** (Q3): the heuristic is untested, will behave differently on phase-planning sessions vs execution sessions, and the whole extractor may ship `not_available` if calibration fails. The two-artifact demonstration set has Q4 (whether 57.7 content extraction produces enough evidence for `progressive` classification of the vision-drop revision) and Q5 (whether `pending` intervention-outcome satisfies ROADMAP SC-3) as non-blocking interpretive questions — both resolvable by reading the vision-drop diagnostic and the SC-3 wording verbatim.

**Primary recommendation:** Sequence the phase as C3 || C5 (parallel spikes) → FTS5 drop (smallest, independent) → three content extractors (parallelizable by extractor) → interpretation-layer deepening (must come after content extractors so registry-traversal has something to find) → demonstration artifacts (revised vision-drop, GATE-09 intervention-outcome). Do not introduce `@anthropic-ai/tokenizer` even if C3's evaluation uses it; document the tokenizer landscape and ship schema-only if nothing clears the threshold.

// Helper 1: tool_invocation_sequence — iterate records, pluck tool_use .name only
function scanToolInvocationSequence(records) {
  const sequence = [];  // ordered list of tool names (names only, no args)
  const tool_count_per_tool = {};
  for (const record of records || []) {
    if (!record || record.type !== 'assistant') continue;
    const content = record.message && record.message.content;
    if (!Array.isArray(content)) continue;
    for (const block of content) {
      if (!block || block.type !== 'tool_use' || !block.name) continue;
      sequence.push(block.name);
      tool_count_per_tool[block.name] = (tool_count_per_tool[block.name] || 0) + 1;
    }
  }
  const distinct = Object.keys(tool_count_per_tool);
  // Shannon entropy over the sequence
  const total = sequence.length;
  let entropy = 0;
  for (const name of distinct) {
    const p = tool_count_per_tool[name] / total;
    entropy -= p * Math.log2(p);
  }
  return {
    sequence,
    tool_count_per_tool,
    distinct_tool_count: distinct.length,
    tool_sequence_entropy: total > 0 ? entropy : 0,
  };
}

// Helper 2: intervention_points — user turns that change direction after ≥2 assistant turns
const INTERVENTION_MARKER_REGEX = /^(stop|no|actually|wait|let me|instead|don't|can you first|before that|hold on)\b/i;

function scanInterventionPoints(records) {
  const total_turns = records.length;
  let consecutive_assistant = 0;
  const interventions = [];  // positions (record indices) where an intervention fired

  for (let i = 0; i < records.length; i++) {
    const record = records[i];
    if (!record) continue;
    if (record.type === 'assistant') {
      consecutive_assistant++;
      continue;
    }
    if (record.type !== 'user') {
      consecutive_assistant = 0;
      continue;
    }
    // user turn
    if (!isHumanTurnRecord(record)) {
      consecutive_assistant = 0;
      continue;
    }
    if (consecutive_assistant >= 2) {
      const normalized = normalizeUserContent(record.message ? record.message.content : null);
      const text = normalized.text.trim();
      if (text && INTERVENTION_MARKER_REGEX.test(text)) {
        interventions.push(i);
      }
    }
    consecutive_assistant = 0;
  }

  const intervention_count = interventions.length;
  const mean_position = intervention_count > 0
    ? interventions.reduce((s, p) => s + (p / total_turns), 0) / intervention_count
    : null;
  return {
    intervention_count,
    interventions_per_100_turns: total_turns > 0 ? (intervention_count * 100) / total_turns : 0,
    mean_intervention_position: mean_position,
    total_turns,
    // auto-calibration data for Q3 ship-vs-skip decision
    intervention_positions: interventions,
  };
}

// NEW: Distinguishing-feature suggester — registry-traversal over interpretation-pairs
function suggestDistinguishingFeatures(interpretation, registry, computedFeatures) {
  const interpDistinguishes = new Set(interpretation.distinguishing_features || []);
  const suggestions = [];
  for (const extractor of registry.extractors) {
    // Suggest extractors whose .distinguishes metadata overlaps the interpretation's declared targets
    const overlap = extractor.distinguishes.filter(d => interpDistinguishes.has(d));
    if (overlap.length === 0) continue;
    // Check whether the extractor actually emitted a feature in computedFeatures for this scope
    const computed = computedFeatures.some(f => f.extractor === extractor.name);
    if (!computed) {
      suggestions.push({
        extractor: extractor.name,
        distinguishes: overlap,
        reason: 'registered_but_not_computed_for_current_scope',
      });
    }
  }
  return suggestions;
}

// NEW: Intervention-outcome side-car reader — file-first, computed on query
function loadInterventionOutcomes(cwd, interpretationId) {
  const dir = path.join(cwd, '.planning', 'measurement', 'interventions');
  if (!fs.existsSync(dir)) return [];
  const files = fs.readdirSync(dir).filter(f => f.endsWith('.md'));
  const outcomes = [];
  for (const file of files) {
    const record = parseMarkdownFrontmatter(path.join(dir, file));  // reuse existing frontmatter parser
    if (!record || record.interpretation_id !== interpretationId) continue;
    outcomes.push({
      intervention_artifact: record.intervention_artifact,
      predicted_outcome: record.predicted_outcome,
      actual_outcome: record.actual_outcome || null,
      outcome_status: record.outcome_status || 'pending',
      record_path: file,
    });
  }
  return outcomes;
}

function composeProvenanceSummary(interpretation, interventions, anomalies) {
  const challengeCount = (interpretation.competing_readings || []).length;
  const confirmed = interventions.filter(i => i.outcome_status === 'confirmed').length;
  const mixed = interventions.filter(i => i.outcome_status === 'mixed').length;
  const grounded = confirmed + mixed;
  const anomalyCount = (anomalies || []).length;
  const revisionHistory = interpretation.revision_history || [];
  const classDistribution = revisionHistory.reduce(
    (acc, r) => ((acc[r.revision_classification || 'unclassified'] = (acc[r.revision_classification || 'unclassified'] || 0) + 1), acc),
    {}
  );
  const classSummary = Object.keys(classDistribution).length === 0
    ? 'no_revisions'
    : Object.entries(classDistribution).map(([k, v]) => `${k}=${v}`).join(', ');
  // Governed wording: never uses "verified" or "confirmed to be true"
  return `surviving_challenge_from_${challengeCount}_perspectives | grounded_in_${grounded}_interventions | carrying_${anomalyCount}_tracked_anomalies | revisions=[${classSummary}]`;
}
```

### Anti-Patterns to Avoid

- **Adding `reasoning_tokens` as two separate extractors (one per runtime):** Collapses the unified axis deliberation §4.3 mandates. Use single extractor with per-runtime branches (CONTEXT.md decided).
- **Silently falling back to character-count-divided-by-4 when tokenizer fails:** This is the exact verificationism trap deliberation §2.2 warns against. Return `not_available` with canonical `skip_reason: 'tokenizer_unavailable'`.
- **Storing full-provenance summary on the interpretation object at rebuild time:** Violates DC-10 resolution-on-demand; would pin stale summary against updated side-car records.
- **Classifying a revision as `progressive` without the structural check firing:** Fails G-3 and the structural validator CONTEXT.md decided ships with the classifier.
- **Shipping LLM-assisted distinguishing-feature suggestion as first cut:** G-4 and `feedback_trial_before_formalize` both reject this. Registry-traversal first; LLM extension on the backlog.
- **Extracting tool arguments or assistant content text in content extractors:** Violates `content_contract: derived_features_only` and DC-3 "no raw content re-exposure." Only tool *names* for tool sequences; only regex-markers for intervention points; only structural boundaries for topic shifts.
- **Rewriting FTS5 as contentless mode in 57.7 "in case we need it later":** G-8 prohibits speculative re-implementation. Drop-only; re-entry path documented (Phase 59).

## Common Pitfalls

### Pitfall 1: The intervention-point seed regex fires on false positives from tool-result messages

**What goes wrong:** A user message that contains `<command-stdout>` or a tool-result echo might start with a word like "stop" or "no" if the tool output did. Without filtering via `isHumanTurnRecord()` (already in `sources/claude.cjs:181-192`), the heuristic will count tool-result echoes as interventions and overestimate the rate.

**Why it happens:** Content in assistant-sent `tool_result` blocks can mirror user-terminology; the heuristic doesn't know the difference from text alone.

**How to avoid:** The `scanInterventionPoints` pseudocode above uses `isHumanTurnRecord(record)` before applying the regex — this filters `isMeta`, `isSidechain`, tool_result lists, and command-prefix patterns in one existing rule.

**Warning signs:** Q3 calibration shows intervention rate >30% on long sessions, or clusters of interventions in the middle of tool-heavy execution spans.

### Pitfall 2: Tool-use block iteration skipping thinking/text blocks

**What goes wrong:** An assistant message has mixed content: `[{type:'thinking',...}, {type:'text',...}, {type:'tool_use',...}]`. If the scanner accidentally iterates the whole content array as tool invocations, it corrupts the sequence.

**Why it happens:** `message.content` in Claude JSONL is polymorphic — it can be string (early records) or an array of heterogeneous blocks (tool-using records). `sources/claude.cjs:162` already shows the existing code's awareness of this.

**How to avoid:** Always type-check `block.type === 'tool_use'` explicitly; never iterate without the guard.

**Warning signs:** `tool_sequence_entropy` returning NaN (log of zero); `tool_count_per_tool` with keys like `"thinking"` or `"text"`.

### Pitfall 3: The `distinguishes` metadata is sparsely populated across existing extractors

**What goes wrong:** Registry-traversal suggester returns empty for most competing-interpretation pairs because 57.5/57.6 didn't populate `distinguishes` lists richly enough. The suggester appears to work but never surfaces useful suggestions.

**Why it happens:** The MEDIUM vulnerability logged in CONTEXT.md's dependency table: "Registry-traversal suggester is sufficient → distinguishes metadata populated on enough extractors." Sparse population means traversal has nothing to match against.

**How to avoid:** At plan-phase, include a registry-audit task: walk `registry.extractors`, count distinct `distinguishes` values, cross-reference against the readings in the vision-drop and facets-coverage diagnostics. If the audit finds empty traversals dominate, add a distinct `distinguishes`-backfill plan that extends 57.5/57.6 extractor definitions.

**Warning signs:** Suggester returns empty lists for all four vision-drop competing readings; registry `distinguishes` field coverage analysis shows < 50% of extractors declare project-domain-relevant distinctions.

### Pitfall 4: Codex reasoning-output-tokens field is not guaranteed present

**What goes wrong:** The reconciler's Codex branch reads `token_count.reasoning_output_tokens`. If the field is absent (older Codex versions, some rollout events), the code silently returns `null` or `undefined`, polluting the `reasoning_tokens` axis.

**Why it happens:** Codex telemetry schema evolves; the spike 008 cross-runtime OTel bridge is marked `partial`. Absence is data per G-2 but must be handled as `not_available`, not `0`.

**How to avoid:** Make the Codex branch check explicitly for the field; when absent, return `availability_status: 'not_available'` with `skip_reason: 'codex_reasoning_tokens_field_absent'`.

**Warning signs:** Codex rows showing `reasoning_tokens: 0` when sessions are known to have had extended reasoning; negative-delta-flag firing on Codex (it should only fire on Claude's derived-delta branch).

### Pitfall 5: FTS5 drop missing existing triggers

**What goes wrong:** If at some point (between our last read and the migration) someone added a `CREATE TRIGGER ... AFTER INSERT ON signals ... INSERT INTO signal_fts ...` pattern, the simple `DROP TABLE IF EXISTS signal_fts` will fail because the trigger references it.

**Why it happens:** The current `kb.cjs:133-136` has NO triggers referencing `signal_fts`. But agents modify files; check at migration time.

**How to avoid:** Migration task should first run `SELECT name FROM sqlite_master WHERE type='trigger' AND sql LIKE '%signal_fts%'` and drop any triggers found, then drop the virtual table.

**Warning signs:** Migration task fails with "cannot drop table referenced by trigger" error; existing kb.db schemas show triggers after initSchema is called.

### Pitfall 6: Claude Code JSONL content field variation across eras

**What goes wrong:** Pre-v2.1.x records may have `content: "string"` (plain text); v2.1.69+ records are always `content: [{...}]` arrays. A scanner that assumes array will skip earlier data entirely.

**Why it happens:** MEAS-RUNTIME-10 era-boundary registry explicitly carves the corpus into non-comparable eras. The existing `normalizeUserContent` (`sources/claude.cjs:156-179`) already handles string vs array; new helpers must call it or mirror its logic.

**How to avoid:** All new scan helpers for content extraction must use or mirror `normalizeUserContent` for user-side records and the array-with-type-check pattern for assistant-side.

**Warning signs:** Early-era sessions showing `tool_count_per_tool: {}` when they're known to have used tools; intervention counts dropping to zero for whole chunks of the corpus.

### Pitfall 7: Q4 vision-drop revision classification — forcing "progressive" to close the phase

**What goes wrong:** The revised vision-drop diagnostic classifies its revision as `progressive` even when 57.7's new content extraction doesn't actually resolve a previously-open anomaly. This violates G-7 (artifacts are not success theater).

**Why it happens:** Phase-closure pressure. Author wants SC-4 to pass.

**How to avoid:** The structural check is the honest guard. Run the check; if neither "new distinguishes entry AND resolved anomaly" nor "anomaly accommodated AND new prediction" fires, ship as `unclassified` with documented gap.

**Warning signs:** Structural check in commit-message form: "Revision classified progressive; check bypassed because X." Any bypass = G-7 violation.

### Pitfall 8: Computing full-provenance summary at rebuild time instead of on query

**What goes wrong:** A plan tries to optimize by caching the provenance-summary string on the interpretation row. Days later, a new intervention-outcome file lands. The cached summary is stale but the query layer still returns it.

**Why it happens:** Developer instinct to cache expensive computations.

**How to avoid:** DC-10 resolution-on-demand is explicit. Compose on every `queryMeasurement()` call by reading the live side-car files. This mirrors 57.6's diagnostic-as-markdown pattern.

**Warning signs:** Benchmark work for "interpretation query performance"; proposals to "cache interpretation summaries for fast retrieval."

function normalizeUserContent(content) {
  if (typeof content === 'string') {
    return { kind: 'text', text: content.trim() };
  }
  if (Array.isArray(content)) {
    if (content.some(item => item && item.type === 'tool_result')) {
      return { kind: 'tool_result', text: '' };
    }
    const textParts = content
      .filter(item => item && item.type === 'text' && typeof item.text === 'string')
      .map(item => item.text.trim())
      .filter(Boolean);
    if (textParts.length > 0) {
      return { kind: 'text', text: textParts.join('\n').trim() };
    }
    return { kind: 'structured', text: JSON.stringify(content) };
  }
  return { kind: 'unknown', text: '' };
}
```

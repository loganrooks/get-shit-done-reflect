<context>
The user's core insight is that the self-improvement pipeline produces the wrong kind of output. "Lessons" are declarative knowledge (episteme) but what's needed is practical wisdom (phronesis) — knowing what to do in a particular situation. The pipeline accumulates knowledge without changing practice.

Two audience problem:
1. GSD developers: reflection tells what to build next (patterns → roadmap items). Manual translation.
2. GSD end users: reflection should help work better NOW — through local patches, retrieved lessons at decision points, workflow amendments, config adjustments.

The user explicitly asked to draw on philosophy: pragmatism (Dewey — knowledge only real when it changes practice), phronesis vs episteme (Aristotle), dialectics (Hegel — self-knowledge through contradiction).

Key prior deliberations:
- signal-lifecycle-closed-loop-gap (concluded): fixed HOW lifecycle transitions fire (programmatic), but assumed lessons are the right output
- v1.16 deliberation: designed the full pipeline (DETECT→TRIAGE→REMEDIATE→VERIFY→LESSON), section B1 envisioned "Workflow Intelligence" but deferred it

The question is whether the pipeline endpoint should be "lessons" at all, or something that directly changes system behavior.

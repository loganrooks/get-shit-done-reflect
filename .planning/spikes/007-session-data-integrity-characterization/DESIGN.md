# Spike Design: Session Data Integrity Characterization

**Phase:** 57
**Created:** 2026-04-09
**Designed by:** Orchestrator (pre-execution review)

## Open Question

How widespread are data quality issues across the 268 session-meta corpus, and what is the "clean" subset we can trust for baseline computation?

## Why This Matters

Every Phase 57 baseline metric depends on session-meta data. Spikes A, C, and E all incidentally discovered data quality issues — malformed JSON files (3 found), extreme duration artifacts (19,996 min), sessions with 0 turns, batch-generated metadata with potential staleness, /continue inheritance contaminating JSONL. If these issues are widespread, baseline metrics computed over the full corpus are misleading. If they're rare, we can document and exclude them. We don't know which yet.

## Type

Exploratory — characterize the problem space, not answer a binary question.

## Hypothesis

The majority (>80%) of session-meta files are clean and usable for baseline computation, but a meaningful minority (<20%) have data quality issues that would contaminate aggregate metrics if included undifferentiated.

## Alternative Hypotheses

- **H-alt-1:** Quality issues are rare (<5%) — the 3 malformed files and duration artifact are outliers, not symptoms of systemic problems. If true, simple exclusion rules suffice.
- **H-alt-2:** Quality issues are pervasive (>30%) — session-meta generation is unreliable enough that the corpus cannot support statistical baselines. If true, Phase 57 scope may need to narrow to output_tokens + tool_counts only (the fields Spike A found reliable).
- **H-alt-3:** Quality issues are version-dependent — sessions generated by different Claude Code versions have different schema completeness. If true, baselines should be segmented by Claude Code version.

## Scope Boundaries

**In scope:**
- Structural validity of all 268 session-meta JSON files (parseable, expected fields present)
- Field completeness audit (which fields are null/missing/zero across what proportion of sessions)
- Duration anomaly characterization (>1000 min sessions — are they multi-day, recording artifacts, or legitimate?)
- Batch generation timestamp analysis (when were meta files created relative to sessions? are any stale?)
- Classification of each session into a trust tier (clean / caveated / exclude)

**Out of scope:**
- JSONL data quality (known unreliable per Spike A — separate investigation if needed)
- Facets data quality beyond coverage count (Spike C already characterized the 109-session overlap)
- Repairing malformed files (characterize the damage, don't fix it)
- Claude Code version identification (may not be determinable from session-meta alone)

## Failure Modes (What Could Make This Investigation Misleading)

- **FM-1:** If session-meta files are periodically regenerated by Claude Code (batch timestamps suggest this), our "snapshot" of 268 files may not reflect the state at session time. Staleness could be systemic, not incidental.
- **FM-2:** If field presence depends on Claude Code version, absence of a field doesn't mean "missing data" — it means "not yet implemented." Treating this as data quality failure would overcount issues.
- **FM-3:** If the 268 files represent a biased sample (only recent sessions, only sessions that reached a certain length), the integrity characterization doesn't generalize to future sessions.

## What Inconclusive Looks Like

- If trust tiers are unclear (many sessions are "maybe clean" with ambiguous quality), the spike is inconclusive and a follow-up with stricter criteria is needed.
- If the dominant quality issue is field absence (not corruption), the answer depends on WHICH fields matter for the baseline — and that's a design question, not a data quality question.

## Success Criteria (Multi-Level)

| Level | Finding | Implication |
|-------|---------|-------------|
| Strong | >80% clean, <5% exclude, clear categorization rules | Proceed with baseline on clean subset, document exclusions |
| Moderate | 60-80% clean, categorization requires judgment calls | Proceed but with conservative error bars, flag ambiguous sessions |
| Weak | <60% clean or quality issues are version-dependent | Narrow baseline to output_tokens + tool_counts (Spike A validated fields), defer richer metrics |
| Inconclusive | Trust tiers are ambiguous, dominant issue is field absence | Follow-up needed: determine which fields are required for baseline, accept available-field-specific baselines |

## Experiment Plan

Research mode — analyze existing data only. No BUILD or RUN phases.

1. **Parse all 268 session-meta files.** For each file, record:
   - Parseable? (valid JSON)
   - Field presence: check all expected fields from the schema (session_id, project_path, start_time, duration_minutes, user_message_count, assistant_message_count, tool_counts, languages, git_commits, git_pushes, input_tokens, output_tokens, first_prompt, user_interruptions, user_response_times, tool_errors, tool_error_categories, uses_task_agent, uses_mcp, uses_web_search, uses_web_fetch, lines_added, lines_removed, files_modified, message_hours, user_message_timestamps)
   - File creation timestamp (mtime) vs session start_time — detect batch generation and staleness

2. **Characterize anomalies:**
   - Duration >1000 min: how many? Can message_hours[] gaps identify multi-day vs continuous?
   - Zero-turn sessions (user_message_count=0 or assistant_message_count=0): how many?
   - Token anomalies: input_tokens=0, output_tokens=0 — do these correlate with zero-turn?
   - tool_counts empty/null: how many?
   - user_response_times[] empty: how many? (Spike E found 54% missing — validate)

3. **Classify sessions into trust tiers:**
   - **Clean:** Valid JSON, all core fields present, duration <1000 min or explainable, non-zero turns
   - **Caveated:** Valid JSON but with known limitations (missing optional fields, extreme duration explainable by message_hours pattern, stale meta timestamp)
   - **Exclude:** Malformed JSON, zero turns with zero tokens, or irreconcilable anomalies

4. **Report:** Summary table, per-field completeness, trust tier distribution, specific excluded session IDs with reasons.

## Evidence Sources

- `~/.claude/usage-data/session-meta/*.json` — all 268 files
- File system timestamps (mtime, ctime) for batch generation analysis
- Prior spike findings: Spike A (3 divergence causes), Spike C (3 malformed files identified), Spike E (54% response_times gap, 19,996-min artifact)

## Epistemic Guardrails

- Do NOT treat field absence as corruption without checking whether the field was expected for that session's Claude Code version
- Do NOT assume the 268-file corpus is the complete historical record — Claude Code may prune old session-meta
- Report trust tier distribution with error bars or ranges, not false precision
- The classification rules themselves are provisional — they encode assumptions about what "clean" means that the researcher may challenge

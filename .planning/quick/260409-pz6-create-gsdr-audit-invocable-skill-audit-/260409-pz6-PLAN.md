  <action>
Create `agents/gsdr-auditor.md` following the spike-runner pattern (receives task spec from orchestrator, executes it) combined with the context-checker pattern (explicit epistemic ground rules in the spec itself).

**Frontmatter:**
- `name: gsdr-auditor`
- `description: Executes audit task specs produced by /gsdr:audit — reads ground rules and body template from the task spec, investigates the target, structures output per template, handles escape hatch Rule 4`
- `tools: Read, Write, Bash, Glob, Grep`
- `model: sonnet` (per MEMORY.md: sensors/agents use Sonnet on quality profile; auditor does investigation work similar to sensor agents)
- `color: yellow`

**Role section:**
- Spawned by `/gsdr:audit` command with a task spec path
- Core responsibility: execute the audit described in the task spec, not figure out what to audit (the orchestrator already did that)
- Two modes of operation based on audit family:
  - Structural audits: checklist-driven, run tests, check wiring, produce findings table
  - Epistemic/exploratory audits: investigation-driven, read content not metadata, present findings incrementally
  - Compliance audits: survey-driven, check practice against standard, produce compliance matrix

**Input section** (following spike-runner pattern):
- Receives: task spec path in the session directory
- Task spec contains: ground rules (already copied in), body template, scope, target artifacts, audit type
- The agent reads the task spec and follows it -- it does NOT re-derive ground rules or select templates

**Execution flow (5 steps):**

1. **Load task spec** -- Read the task spec file. Parse: audit type, ground rules section, body template, scope, target artifacts, any predecessor audits. If predecessor audits listed, read them to understand prior findings.

2. **Investigation phase** -- Execute the audit per the task spec scope:
   - For structural: check artifacts exist (Level 1), are substantive (Level 2), are wired (Level 3). Run tests, not just read them (Rule S1). Check connections, not just existence (Rule S2).
   - For epistemic: formulate the investigation question. Read artifacts (Rule E1: content not metadata). For each finding, seek disconfirmation BEFORE writing it (Rule 2). Admit uncertainty explicitly (Rule E2). Verify citations resolve (Rule E3).
   - For compliance: identify the standard being checked. Survey actual practice (Rule C1: practice, not documentation). Build compliance matrix.
   - For exploratory: follow the question wherever it leads. Core rules apply. Add family extensions as the audit's question demands.
   - ALL types: Rule 1 (cite file:line, quote passages), Rule 2 (disconfirmation), Rule 3 (distinguish measure from what it captures).

3. **Structure output** -- Write the audit output file to the session directory using the body template from the task spec. The template is a starting structure, not a straitjacket -- if the audit demands a different structure, use it but note the deviation.

4. **Escape hatch (Rule 4)** -- Before finalizing, answer: "What did you encounter that these ground rules didn't prepare you for?" Write an explicit `## Escape Hatch (Rule 4)` section at the end of the output. If the answer is "nothing," consider whether the ground rules shaped attention so thoroughly that excess was invisible.

5. **Return result** -- Return structured output to orchestrator with: output file path, findings count, severity distribution (if applicable), and whether Rule 4 produced anything notable.

**Critical rules section:**
- DO NOT trust claims from prior audits without re-verifying them (lesson from the gold-standard audit chain)
- DO NOT count without reading (Rule E1) -- "28 signals" is not a finding; reading 5 signals and describing content is
- DO NOT commit -- the orchestrator handles commits
- DO NOT modify the task spec -- it is the audit's provenance record
- If you cannot complete the audit (scope too large, access issues), return a checkpoint describing what was accomplished and what remains

**Output format section** -- two structured returns:
- Completion: `## AUDIT COMPLETE` with output path, findings count, escape hatch summary
- Checkpoint: `## AUDIT CHECKPOINT` with progress, issue, what remains

**Required reading:** `@~/.claude/get-shit-done/references/agent-protocol.md`

Create `commands/gsd/audit.md` following the deliberate.md pattern (conversational skill with context inference) combined with the audit-milestone.md pattern ("This command IS the orchestrator").

**Frontmatter:**
- `name: gsd:audit` (installer converts to `gsdr:audit` via directory rename)
- `description: Run a structured audit with type-aware ground rules, body templates, and artifact preservation`
- `argument-hint: '"topic/scope" [--type phase_verification|milestone|codebase_forensics|cross_model_review|requirements_review|comparative_quality|claim_integrity|adoption_compliance|exploratory] [--auto]'`
- `allowed-tools: [Read, Write, Edit, Bash, Glob, Grep, Task, AskUserQuestion]`

**Objective section:**
- This command IS the orchestrator for standalone audits
- Creates session directory, selects ground rules by type family, generates task spec with ground rules COPIED in, dispatches gsdr-auditor, preserves artifacts, commits
- Operationalizes audit-conventions.md (directory, frontmatter, types, templates) and audit-ground-rules.md (core rules, family extensions, escape hatch)

**Execution context:**
- `@~/.claude/get-shit-done/references/audit-conventions.md`
- `@~/.claude/get-shit-done/references/audit-ground-rules.md`

**Context section:**
- Arguments: $ARGUMENTS
- `@.planning/STATE.md`
- `@.planning/config.json`

**Process section (8 steps):**

**Step 1: Parse Arguments**
- Extract topic/scope from quoted string argument
- Extract `--type` if provided (one of 9 types + exploratory)
- Extract `--auto` flag (skip interactive confirmation, run autonomously)
- If no arguments: check conversation context for what triggered the audit (same pattern as deliberate.md Step 1)

**Step 2: Discovery Phase**
Read conversation context and any provided arguments to determine:

a. **Audit type inference** (if --type not provided):
Map context clues to types using this heuristic table:

| Context clue | Inferred type |
|---|---|
| "phase N", "verification", "does X work" | phase_verification |
| "milestone", "cross-phase", "integration" | milestone |
| "how does X work", "codebase structure", "forensics" | codebase_forensics |
| "review from different model", "external perspective" | cross_model_review |
| "requirements", "coverage", "feasibility" | requirements_review |
| "quality", "comparison", "regression" | comparative_quality |
| "claims", "citations", "integrity" | claim_integrity |
| "adoption", "compliance", "convention" | adoption_compliance |
| None of the above, or user explicitly wants open-ended | exploratory |

If ambiguous, ask 1 clarifying question (max). In --auto mode, use best inference.

b. **Scope determination:** What files/phases/artifacts are the target? Derive from arguments and context.

c. **Present overview** (unless --auto):
```
GSD AUDIT
Type: {inferred type} ({family})
Scope: {what will be audited}
Ground rules: core + {family extensions}
Body template: {family template name}
Proceed?
```

**Step 3: Create Session Directory**
Per audit-conventions.md Section 1:
```bash
AUDIT_DATE=$(date +%Y-%m-%d)
AUDIT_SLUG="{kebab-case-from-scope}"  # max 50 chars
AUDIT_DIR=".planning/audits/${AUDIT_DATE}-${AUDIT_SLUG}"
mkdir -p "$AUDIT_DIR"
```

**Step 4: Select Ground Rules**
Per audit-ground-rules.md Section 4, select based on type family:

| Type family | Ground rule set |
|---|---|
| structural (phase_verification, milestone, codebase_forensics) | Core rules 1-3 + Rule 4 + S1, S2 |
| epistemic (cross_model_review, requirements_review, comparative_quality, claim_integrity) | Core rules 1-3 + Rule 4 + E1, E2, E3 |
| compliance (adoption_compliance) | Core rules 1-3 + Rule 4 + C1 |
| exploratory | Core rules 1-3 + Rule 4 only (add extensions as audit demands) |

**CRITICAL: Copy the actual rule text into the task spec.** Read `~/.claude/get-shit-done/references/audit-ground-rules.md` and extract the rule bodies. Do NOT write "see audit-ground-rules.md" -- the agent needs the rules in its context window, not a pointer to them.

**Step 5: Select Body Template**
Per audit-conventions.md Section 4, select the family template:
- Structural: Goal/Scope, Artifacts Checked, Wiring Verification, Findings table, Gaps Summary
- Epistemic: Question, Methodology, Evidence Examined, Findings (with disconfirmation), Limitations
- Compliance: Standard, Projects Surveyed, Compliance Matrix, Deviations, Recommendations
- Exploratory: Question/Motivation, What Examined, What Found, What It Might Mean, Follow-Up Needed

Read `~/.claude/get-shit-done/references/audit-conventions.md` Section 4 and extract the template for the selected family. Copy the template structure into the task spec.

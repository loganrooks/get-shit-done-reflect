# Post-v1.16 Roadmap Deliberation

**Created:** 2026-03-02
**Last updated:** 2026-03-02
**Status:** Active — milestone ordering and first milestone selection in progress

## Context

After shipping v1.16 (Signal Lifecycle & Reflection), a broad brainstorming session identified 5 milestone-sized themes for the next phase of GSD Reflect development. This document captures the thinking, dependency analysis, and open questions from that conversation.

### What Prompted This

1. CI tests were silently failing throughout v1.16 (5 consecutive failures, all bypassed via admin push) — the self-improvement system failed to catch its own process failure
2. Pending TODOs in STATE.md were partially stale (feature manifest already built, dual-install Phase 2 genuinely pending)
3. Reflection report (2026-03-02) showed 79 signals with 62 untriaged, revealing significant automation gaps
4. System has no meta-observability — sensors don't measure each other, no token/context tracking, no feature utilization data
5. Deliberation and backlog systems exist as infrastructure but are unused in practice
6. Cross-platform support is install-tested but not runtime-tested
7. Conversations like this one (deliberations) have no capture mechanism

## The 5 Milestone Themes

### M-A: Close the Automation Loop (v1.17)

**Core idea:** Make the self-improvement system actually self-triggering instead of manually invoked.

**Key features:**
- Auto-trigger `/gsd:collect-signals` after phase execution (PostToolUse hook on SUMMARY.md writes, or execute-phase postlude)
- CI sensor — detect failed GitHub Actions, bypassed branch protection, test regressions (uses `gh` CLI)
- Auto-trigger reflection after N phases (configurable) or at milestone boundaries
- Health check auto-trigger via hooks (SessionStart for `on-resume`, execute-phase for `every-phase`)
- Fix the immediate CI wiring test failure (wiring-validation.test.js checking `.claude/agents/` which doesn't exist in CI)
- CI status hook at SessionStart — warn if last CI run was red

**Estimated effort:** Medium — mostly hook config, workflow modifications, one new sensor
**Dependencies:** None — this is foundational
**Files touched:** hooks/, workflows/execute-phase.md, workflows/collect-signals.md, agents/ (new CI sensor), wiring tests

### M-B: Meta-Observability (Sensors Measuring Sensors)

**Core idea:** The system can observe and measure its own processes — token efficiency, context consumption, feature utilization.

**Key features:**
- Log sensor implementation (Spike 002 done, paths known, design decided — `~/.claude/projects/{path}/{session}.jsonl`)
- Metrics capture in SUMMARY.md: token usage, context consumption, model used per plan
- Persist context % from statusline to disk for trending (extend gsd-statusline.js to write to a metrics file)
- Feature utilization tracking — which commands get used, how often, invocation logging
- Metrics sensor (envisioned in v1.16 deliberation) — aggregates token/context/duration data
- Meta-measurement: sensors report their own resource consumption so we can evaluate sensor efficiency

**Estimated effort:** Large — log sensor is the biggest item, session JSONL parsing at 442MB scale
**Dependencies:** Partially benefits from M-A (auto-triggering makes metrics collection continuous), but log sensor can be built independently
**Files touched:** agents/gsd-log-sensor.md (rewrite from stub), hooks/gsd-statusline.js, SUMMARY.md templates, new metrics infrastructure

### M-C: Deliberation & Conversation Intelligence

**Core idea:** Formalize how the system captures, processes, and retrieves thinking-in-progress across sessions.

**Key features:**
- Formalize deliberation system: schema, index, freshness tracking, `/gsd:deliberate` command
- Conversation trace capture and summarization from session logs
- Auto-extract decisions, ideas, rejected alternatives, open questions from session JSONL
- Link deliberations to the requirements/phases they produced
- Make backlog actually useful: `/gsd:backlog` slash command, integration with reflect/execute-phase/plan-phase
- Search across past deliberations ("what did we decide about X?")
- Deliberation → milestone pipeline (deliberations surface during `/gsd:new-milestone`)

**Estimated effort:** Large — conversation intelligence is AI-heavy, backlog improvements are moderate
**Dependencies:** Log sensor from M-B (for conversation trace capture). Deliberation formalization is independent.
**Files touched:** .planning/deliberations/ (schema), new commands, workflows/new-milestone.md, backlog CLI integration

### M-D: Cross-Platform Parity as First-Class Concern

**Core idea:** Runtime behavioral testing, not just installation testing. Seamless same-repo multi-runtime operation.

**Key features:**
- Runtime behavioral testing — validate that degraded workflows actually work on Codex/Gemini/OpenCode
- E2E test implementation (currently `it.todo()` stubs)
- Validate capability matrix claims against real runtime behavior
- Bake capability checks into more workflows (currently only execute-phase and hooks)
- Seamless same-repo operation: switch between Claude Code, Codex, Gemini CLI without friction
- Test that signal collection, reflection, spike workflow work under degraded conditions
- Consider: should the installer generate runtime-specific test suites?

**Estimated effort:** Medium-Large — research-heavy for runtime behavioral testing, may need access to other runtimes
**Dependencies:** Largely independent. Benefits from M-A's CI sensor for cross-platform CI.
**Files touched:** tests/, agents/ (capability checks), workflows/ (capability gates), bin/install.js

### M-E: Parallelization & Efficiency

**Core idea:** More work happening in parallel, less context burned per operation.

**Key features:**
- Research Claude Code's worktree capabilities (Feb-Mar 2026 state)
- Research parallel session support across runtimes
- Roadmapper identifies parallelizable phases and marks them
- Execute-phase leverages worktrees for independent work
- Progressive context disclosure adopted broadly (two-pass pattern from reflector)
- Plan checker enhancements (semantic validation: tool API existence, config keys, directory existence, cross-plan deps)
- Context budget tracking per workflow (which workflows burn context fastest?)

**Estimated effort:** Large — research-heavy, architectural changes to execute-phase and roadmapper
**Dependencies:** Benefits from M-B's context metrics. Parallelization research is independent.
**Files touched:** workflows/execute-phase.md, agents/gsd-planner.md, agents/gsd-roadmapper.md, agents/gsd-plan-checker.md

## Dependency Graph

```
M-A: Automation Loop ──────┐
    (foundational)         │
                           ├──→ M-B: Meta-Observability
                           │        (log sensor, metrics)
                           │              │
                           │              ├──→ M-C: Deliberation Intelligence
                           │              │        (needs log sensor for conversation capture)
                           │              │
                           │              └──→ M-E: Parallelization
                           │                       (benefits from context metrics)
                           │
M-D: Cross-Platform ───────┘
    (independent, benefits from CI sensor)
```

### Hard Dependencies
- M-C conversation capture **requires** M-B log sensor
- M-C deliberation formalization is **independent** (can start without log sensor)

### Soft Dependencies (benefits from, doesn't require)
- M-B benefits from M-A auto-triggering (continuous metrics collection)
- M-D benefits from M-A CI sensor (cross-platform CI awareness)
- M-E benefits from M-B context metrics (measuring efficiency improvements)

### Parallelizability Between Milestones

**Can run in parallel (different file domains):**
- M-D (Cross-Platform) + M-A (Automation Loop) — M-D is tests/installer, M-A is hooks/workflows
- M-D (Cross-Platform) + M-C (Deliberation formalization parts) — different file domains entirely

**Should run sequentially:**
- M-A → M-B (auto-triggering informs metrics design)
- M-B → M-C conversation parts (log sensor needed first)

**Worktree feasibility for parallel milestones:**
- GSD currently assumes one active milestone at a time (single ROADMAP.md, STATE.md, config.json)
- Worktree-based parallel milestones would need: separate .planning/ state per worktree, merge strategy for MILESTONES.md, shared KB
- This is itself a feature of M-E (parallelization) — chicken-and-egg problem
- Simpler approach: run M-D in a branch while M-A runs on main, merge when both complete

## Proposed Ordering

**Option 1: Sequential (safe)**
1. M-A → 2. M-B → 3. M-C → 4. M-D → 5. M-E

**Option 2: Partial parallel (ambitious)**
1. M-A + M-D in parallel (branch-based) → 2. M-B → 3. M-C → 4. M-E

**Option 3: Front-load independence**
1. M-A (foundational) → 2. M-D (independent, branch off) + M-B (main) → 3. M-C → 4. M-E

# Post-v1.16 Roadmap Deliberation

**Created:** 2026-03-02
**Last updated:** 2026-03-03
**Status:** Active — v1.17 requirements defined (49), philosophical foundations established, pre-roadmap

## Context

After shipping v1.16 (Signal Lifecycle & Reflection), a broad brainstorming session identified 5 milestone-sized themes for the next phase of GSD Reflect development. This document captures the thinking, dependency analysis, and open questions from that conversation.

### What Prompted This

1. CI tests were silently failing throughout v1.16 (5 consecutive failures, all bypassed via admin push) — the self-improvement system failed to catch its own process failure
2. Pending TODOs in STATE.md were partially stale (feature manifest already built, dual-install Phase 2 genuinely pending)
3. Reflection report (2026-03-02) showed 79 signals with 62 untriaged, revealing significant automation gaps
4. System has no meta-observability — sensors don't measure each other, no token/context tracking, no feature utilization data
5. Deliberation and backlog systems exist as infrastructure but are unused in practice
6. Cross-platform support is install-tested but not runtime-tested
7. Conversations like this one (deliberations) have no capture mechanism

## The 5 Milestone Themes

### M-A: Close the Automation Loop (v1.17)

**Core idea:** Make the self-improvement system actually self-triggering instead of manually invoked.

**Key features:**
- Auto-trigger `/gsd:collect-signals` after phase execution (PostToolUse hook on SUMMARY.md writes, or execute-phase postlude)
- CI sensor — detect failed GitHub Actions, bypassed branch protection, test regressions (uses `gh` CLI)
- Auto-trigger reflection after N phases (configurable) or at milestone boundaries
- Health check auto-trigger via hooks (SessionStart for `on-resume`, execute-phase for `every-phase`)
- Fix the immediate CI wiring test failure (wiring-validation.test.js checking `.claude/agents/` which doesn't exist in CI)
- CI status hook at SessionStart — warn if last CI run was red
- Code review agent — automated review step before merging phase branches to main (or milestone branch). Could run as a post-execution gate that reviews the diff, checks for common issues, and produces a review summary before the user merges.

### Code Review Agent (Design Note, 2026-03-03)

**Origin:** Enabling per-phase branching creates a natural CI + review gate before merging. A code review agent would automate the review step, catching issues that CI alone won't (style, architecture, unnecessary changes, security).

**Possible scope:**
- Post-execution: review the full phase diff against the plan's stated objectives
- Check for over-engineering, unintended side effects, leftover debug code
- Flag deviations between what the plan said and what was built (cross-reference SUMMARY.md)
- Produce a structured review (approve / request-changes / comments) that feeds into the merge decision
- Could be a sensor (review-sensor) or a standalone workflow step in execute-phase

**Open questions:**
- Should it block merge, or advisory only? (Probably advisory initially, like plan-checker)
- Does it run per-plan or per-phase (full branch diff)?
- How does it interact with human code review? (Supplement, not replace)

**Estimated effort:** Medium — mostly hook config, workflow modifications, one new sensor
**Dependencies:** None — this is foundational
**Files touched:** hooks/, workflows/execute-phase.md, workflows/collect-signals.md, agents/ (new CI sensor), wiring tests

### M-B: Meta-Observability (Sensors Measuring Sensors)

**Core idea:** The system can observe and measure its own processes — token efficiency, context consumption, feature utilization.

**Key features:**
- Log sensor implementation (Spike 002 done, paths known, design decided — `~/.claude/projects/{path}/{session}.jsonl`)
- Metrics capture in SUMMARY.md: token usage, context consumption, model used per plan
- Persist context % from statusline to disk for trending (extend gsd-statusline.js to write to a metrics file)
- Feature utilization tracking — which commands get used, how often, invocation logging
- Metrics sensor (envisioned in v1.16 deliberation) — aggregates token/context/duration data
- Meta-measurement: sensors report their own resource consumption so we can evaluate sensor efficiency

**Estimated effort:** Large — log sensor is the biggest item, session JSONL parsing at 442MB scale
**Dependencies:** Partially benefits from M-A (auto-triggering makes metrics collection continuous), but log sensor can be built independently
**Files touched:** agents/gsd-log-sensor.md (rewrite from stub), hooks/gsd-statusline.js, SUMMARY.md templates, new metrics infrastructure

### M-C: Deliberation & Conversation Intelligence

**Core idea:** Formalize how the system captures, processes, and retrieves thinking-in-progress across sessions.

**Key features:**
- Formalize deliberation system: schema, index, freshness tracking, `/gsd:deliberate` command
- Conversation trace capture and summarization from session logs
- Auto-extract decisions, ideas, rejected alternatives, open questions from session JSONL
- Link deliberations to the requirements/phases they produced
- Make backlog actually useful: `/gsd:backlog` slash command, integration with reflect/execute-phase/plan-phase
- Search across past deliberations ("what did we decide about X?")
- Deliberation → milestone pipeline (deliberations surface during `/gsd:new-milestone`)

**Estimated effort:** Large — conversation intelligence is AI-heavy, backlog improvements are moderate
**Dependencies:** Log sensor from M-B (for conversation trace capture). Deliberation formalization is independent.
**Files touched:** .planning/deliberations/ (schema), new commands, workflows/new-milestone.md, backlog CLI integration

#### M-C Scope Expansion: Deliberative Council (2026-03-08)

The reflection output ontology deliberation spawned a significant expansion of M-C's scope. See `.planning/deliberations/deliberative-council-and-epistemic-framework.md` for full details. Summary:

**Council of Agents:** Multi-agent deliberation with specialized perspectives (software engineer, AI researcher, domain philosophers), each backed by MCP servers for retrieving relevant papers, texts, and guides. Guided by philosophies of dialogue and intersubjective communication, potentially with a mediator agent.

**Epistemic Framework for AI Research:** Philosophical navigation of empirical AI claims — scope conditions (Cartwright), programme trajectory (Lakatos), paradigm maturity (Kuhn), social epistemology (Longino), temporal validity, and complexity-instability.

**New philosophers needed for corpus:** Kuhn (paradigm maturity, high priority), Longino (social epistemology, high priority), Hacking (styles of scientific reasoning, medium priority).

**M-C may need splitting:**
- M-C1: Deliberation infrastructure (capture, index, search — original scope)
- M-C2: Deliberative council (multi-agent, MCP-backed, philosophically structured — new scope)

Or M-C2 becomes its own milestone. The design space is open — see the deliberation for preliminary positions (Advisory Council, Structured Debate, Dialectical Encounter).

**Key features:**

**Estimated effort:** Medium-Large — research-heavy for runtime behavioral testing, may need access to other runtimes

### Parallelizability Between Milestones

**Can run in parallel (different file domains):**
- M-D (Cross-Platform) + M-A (Automation Loop) — M-D is tests/installer, M-A is hooks/workflows
- M-D (Cross-Platform) + M-C (Deliberation formalization parts) — different file domains entirely

**Should run sequentially:**
- M-A → M-B (auto-triggering informs metrics design)
- M-B → M-C conversation parts (log sensor needed first)

**Worktree feasibility for parallel milestones:**
- GSD currently assumes one active milestone at a time (single ROADMAP.md, STATE.md, config.json)
- Worktree-based parallel milestones would need: separate .planning/ state per worktree, merge strategy for MILESTONES.md, shared KB
- This is itself a feature of M-E (parallelization) — chicken-and-egg problem
- Simpler approach: run M-D in a branch while M-A runs on main, merge when both complete

### Two-Tier Branching Model (Design Note, 2026-03-03)

**Origin:** During v1.17 Phase 36 execution, commits landed directly on main (branching was `none`). Moving to `phase` branching helps CI gating and code review, but `phase` and `milestone` are mutually exclusive. Parallel phases need per-phase branches for isolation, but also a milestone integration branch before main.

**Proposed model:**
```
main
 └── gsd/v1.17-automation-loop        (milestone branch)
      ├── gsd/phase-36-foundation-fix   (parallel)
      ├── gsd/phase-37-automation-fw    (parallel)
      └── gsd/phase-38-sensor-arch      (parallel)
```

**Design questions:**
- Phase branches merge to milestone branch (squash or merge commit?)
- Milestone branch merges to main (PR with full CI?)
- How does the roadmapper signal which phases are parallelizable?
- Does execute-phase auto-detect the milestone branch as base, or is it configured?
- Conflict resolution when parallel phase branches touch overlapping files

## Design Decisions from v1.17 Deliberation (2026-03-02)

### Unified Automation Level System

**Decision:** Implement automation as a unified level system (0-3) with per-feature overrides and fine-grained knobs, rather than scattered boolean toggles.

| Level | Name | Behavior |
|-------|------|----------|
| 0 | Manual | Everything requires explicit command invocation |
| 1 | Nudge | Statusline indicators, context warnings, suggestions |
| 2 | Prompt | System asks "should I do X now?" at appropriate moments |
| 3 | Auto | System auto-triggers with configurable thresholds |

Each level includes everything below it. Per-feature overrides allow bumping individual features up or down. Fine-grained knobs (thresholds, frequencies) provide control regardless of level.

**Key design decisions:**
- `mode` (yolo/interactive) stays SEPARATE from `automation.level` — they're orthogonal. Mode controls within-workflow approval. Level controls cross-workflow triggering.
- Context-aware auto-triggering: if context usage exceeds threshold, auto downgrades to nudge for that session.
- Runtime-aware effective levels: global level set once, but statusline shows "Level 3 (effective: 2 on Codex)" based on capability matrix.
- Lightweight automation stats (`auto_collects_fired`, `last_triggered`, etc.) serve as both effectiveness data for M-B and watchdog for automation health.

### Health Score: Infrastructure + Workflow

**Decision:** Health is two-dimensional, not just infrastructure.

- **Infrastructure health** (binary): CI green? Config valid? KB accessible? State fresh?
- **Workflow health** (weighted accumulation): Signal debt, pattern density, recurrence rate, deviation trend

Composite health score displayed in statusline (green/yellow/red). Infrastructure problems → immediate red. Workflow problems → gradual drift.

**Weighted signal accumulation for workflow health:**

| Severity | Weight | Solo threshold | Cluster behavior |
|----------|--------|---------------|-----------------|
| Critical | 1.0 | 1 triggers | Always triggers |
| Notable | 0.3 | ~4 needed | 2-3 in same pattern triggers |
| Minor | 0.1 | ~10 needed | Only if clustered with notable+ |

Deduplicate by pattern first — 10 minor signals with the same root cause = 1 pattern with 10 occurrences, not 10 independent contributions.

Signal accumulation triggers REFLECTION, not health checks. Health checks stay infrastructure-focused. The health score combines both.

**Reactive health check triggers:**
- CI goes red (infrastructure)
- Config corruption detected (infrastructure)
- Health score drops below threshold on fresh session start (composite)
- NOT triggered mid-session to avoid consuming context needed for recovery

### Extensible Sensor Architecture

**Decision:** Design CI sensor as the first sensor under a new auto-discovery model, not as another hardcoded addition.

Convention: `agents/gsd-*-sensor.md` files are auto-discovered by collect-signals workflow. Each sensor declares its config key and enable/disable status. Adding a new sensor requires only the agent spec + a config entry — no workflow edits, no wiring test changes.

Standardized sensor contract:
- Input: phase number, project context, config flags
- Output: JSON signal candidates wrapped in `## SENSOR OUTPUT` delimiters
- Error handling: return empty signals array on failure, never crash
- Timeout: configurable per sensor
- Provenance: include `runtime` and `model` fields

Artifact and git sensors retroactively conformed to the contract. `gsd-tools.js sensors list` command shows discovered sensors + status.

**Why this matters for future milestones:**
- M-B log sensor drops in with just an agent spec
- M-B metrics sensor same
- A hypothetical "contradiction sensor" (dialectical analysis) could plug in without touching any workflow code
- Third-party sensors become possible

### Advisory-Only Plan Checker (with escalation path)

**Decision:** Semantic validation ships as advisory only. But the schema supports future adaptive escalation.

Each advisory finding carries a typed ID that can be correlated with execution signals later. If a pattern emerges (advisory warning X → plan fails → fix-fix-fix chain), future milestones could auto-escalate that finding type. v1.17 doesn't implement this escalation, but the data schema supports it.

### Mode vs Automation Level (Orthogonal)

**Decision:** Keep `mode` (yolo/interactive) separate from `automation.level`.

- `mode` controls within-workflow behavior: auto-approve checkpoints, skip confirmations
- `automation.level` controls cross-workflow triggering: auto-collect signals, auto-reflect, health check triggers

They compose independently: Level 3 + interactive = auto-trigger everything, but ask before approving within workflows. Level 1 + yolo = nudge only for cross-workflow, but auto-approve when manually triggered.

## Future Direction: Dialectical System Evolution (Original, 2026-03-02)

**Superseded by Philosophical Foundations section above.** Preserved for continuity.

The current system already has implicit dialectical structure:
- Signal (thesis: raw observation) → Reflection (antithesis: critique + counter-evidence) → Lesson (synthesis: principle that transcends individual signals)
- Lessons feed back into planning (next cycle's thesis), creating a spiral (Aufhebung)

A more explicitly dialectical system might:
- Seek contradictions between signals, not just patterns
- Generate antitheses to its own recommendations
- Track lesson evolution through successive refinements
- Capture the reasoning behind decisions, not just the decisions themselves (deliberation capture → M-C)

Connection to analytic philosophy: Brandom's inferentialism, making explicit what's implicit in practice. The system captures what happened (signals) but not the inferential reasoning behind decisions. The deliberation system (M-C) would begin to address this.

v1.17's extensible sensor architecture enables future dialectical analysis — a "contradiction sensor" could plug into the auto-discovery system without workflow changes.
